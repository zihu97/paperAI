**中文摘要**:
无服务器大语言模型（LLM）通过 GPU 资源共享和“按量付费”的定价模式，已成为部署 AI 服务的一种经济高效的解决方案。然而，冷启动延迟，特别是模型加载阶段的延迟，已成为一个关键的性能瓶颈。
因为它会随着模型大小线性扩展，严重限制了大规模 LLM 服务的实际部署。
本文提出了 Tangram，一个通过高效 GPU 内存重用来加速无服务器 LLM 加载的新型系统。
通过利用未使用的 GPU 内存来保留模型参数，Tangram 显著减少了模型传输时间和冷启动延迟。
其设计包括三个关键组件：
1.  **统一的 GPU 内存池**：用于跨模型进行张量级别的参数共享。
2.  **按需 KV 缓存分配**：用于动态内存管理。
3.  **GPU 亲和性感知调度**：用于最大化资源利用率。
这些技术共同解决了无服务器 LLM 平台中内存使用效率低下和冷启动的关键挑战。我们实现了一个功能齐全的原型，实验表明，与现有先进方法相比，Tangram
的加载速度最高可提升 6.2 倍，并将冷启动期间的首次令牌生成时间（TTFT）减少了 23-55%。

**第1章：引言 (Introduction) 详解**

*   **设计思想**:
    引言的核心思想是定位并凸显无服务器 LLM 平台中的关键性能瓶颈。随着模型规模的急剧增大，传统的“按需加载”模式导致了难以忍受的冷启动延迟，其中模型参数从存储加载到 GPU
    内存的过程（`Load` 阶段）是主要矛盾。现有方法（如 CPU 缓存、并行加载）虽然有一定效果，但无法从根本上解决 PCIe
    带宽瓶颈问题。因此，作者提出，解决问题的关键在于**减少需要传输的数据量**，而不仅仅是加快传输速度。这引出了 Tangram 的核心理念：**在 GPU
    内部重用已存在的模型参数**，避免重复的数据传输。

*   **关键技术**:
    本章提出的关键思路是**利用 GPU 闲置内存**。在典型的无服务器场景中，GPU 并非总是满负荷运行，尤其是在请求间隔期或 KV 缓存未被完全使用时，存在大量未利用的内存空间。
    Tangram 的设计正是要抓住这个机会，将这部分闲置内存用作一个“缓存区”，保留那些可能被再次请求的模型参数（张量）。

*   **实现细节/要点**:
    1.  **问题定义**: 将无服务器 LLM 的高昂部署成本作为背景，引出其“按需付费”模式的优势。
    2.  **瓶颈分析**: 通过对 GPT-20B 模型的 TTFT（Time-To-First-Token）进行分解（图 1），明确指出 `Load`
        阶段占据了冷启动时间的大部分。并通过与几种最先进优化（SLLM-C, SLLM-CM）的对比（图 2），进一步证明即使在优化了 `Init` 和 `Profile`
        阶段后，`Load` 阶段的延迟问题依然突出，并且随模型尺寸线性增长。
    3.  **解决方案提出**: 基于瓶颈分析，提出了 Tangram 系统。它通过 GPU 内存重用和亲和性调度来加速模型加载。
    4.  **贡献总结**: 明确了 Tangram 的三个核心贡献：
        *   设计并实现了张量级别的 GPU 内存重用机制。
        *   提出了一个统一的、感知碎片化的内存管理方案，支持按需 KV 缓存分配。
        *   设计了一个 GPU 亲和性调度器，以最大化内存重用的效益。

**第2章：背景与动机 (Background and Motivation) 详解**

*   **设计思想**:
    本章旨在为 Tangram 的设计提供坚实的理论和事实基础。其思想是深入剖析现有无服务器 LLM
    系统的工作流程和资源管理方式，从中发现不合理或可优化的环节。通过对真实世界工作负载的分析，揭示出两个被当前系统忽略但极具优化潜力的特性：**访问局部性**和**内存的超额预留**。
    这两个观察直接催生了 Tangram 的核心设计目标。

*   **关键技术/分析**:
    1.  **无服务器 LLM 架构**: 介绍了典型的“控制器-工作节点”（Controller-Worker）架构（图 3）。控制器负责接收请求、调度任务；工作节点（Worker）负责在 GPU
        上执行推理。模型实例在空闲一段时间后会被驱逐，以释放资源，这正是导致冷启动的直接原因。
    2.  **LLM 推理剖析**: 详细拆解了 LLM 推理的四个初始化阶段：
        *   `Init`：环境初始化。
        *   `Load`：模型参数加载到 GPU。
        *   `Profile`：性能分析，确定 KV 缓存大小等。
        *   `Prefill`：处理输入提示（prompt），生成初始 KV 对。
        这个过程的延迟共同构成了 TTFT。
    3.  **性能瓶颈的再次确认**: 实验（图 2）再次确认，在集成了 CRIU（上下文快照恢复）和离线分析等优化后，`Init` 和 `Profile`
        的开销已大大降低，使得`Load`阶段的线性增长瓶颈更加凸显。
    4.  **机会发现（关键观察）**:
        *   **观察1：模型访问呈现局部性 (Locality)**。通过分析真实世界的无服务器轨迹（图 4a），发现许多对同一模型的请求是连续或间隔很短的。
            这意味着，如果模型参数能在 GPU 中保留一小段时间，后续请求就能直接重用，避免冷启动。
        *   **观察2：KV 缓存分配过于保守 (Over-conservative)**。当前系统通常为每个请求预留其可能达到的最大 KV 缓存空间，但实际使用量远小于此（图
            4b）。这导致大量 GPU 内存被预留但未使用，造成了严重的资源浪费。这部分浪费的内存完全可以用来缓存更多的模型参数。

**第3章：Tangram 设计 (Design of Tangram) 详解**

*   **设计思想**:
    本章的设计思想是构建一个**以 GPU 内存为中心**的、**细粒度**的、**动态**的资源管理系统。
    *   **以 GPU 内存为中心**: 核心是最大化 GPU 内存的价值，将其从一个仅用于计算的临时存储，转变为一个跨模型、跨请求的**持久化缓存**。
    *   **细粒度**:
        放弃了以整个模型为单位的粗粒度管理方式，选择**张量（Tensor）**作为内存管理和重用的基本单元。这使得系统可以更灵活地组合和保留部分模型，提高了内存利用率。
    *   **动态**: 系统能根据实时的工作负载和内存状况，动态地做出**驱逐**、**分配**和**调度**决策，以实现全局最优。

*   **关键技术与实现细节**:
    1.  **整体架构 (图 5)**:
        *   在每个 Worker 节点上部署一个 **Reuse Store**，它包含一个**统一内存池（Unified Memory Pool）**，用于管理所有可重用的 GPU 内存。
        *   该内存池被 **Tensor Allocator**（张量分配器）和 **Block Allocator**（用于 KV 缓存的块分配器）共同管理。
        *   Controller 节点的**调度器**被增强为 **GPU 亲和性感知**，它会查询 Worker 上的 Reuse Store 来获取重用信息，以做出最佳调度决策。

    2.  **张量级别的模型重用与加载 (Tensor-level Model Reuse and Loading)**:
        *   当一个模型被调度到某个 GPU 时，Reuse Store 会检查其所需的张量是否已存在于统一内存池中。
        *   它通过一个哈希表（Tensor Mapping）来追踪每个张量的“指纹”及其在 GPU 上的地址。
        *   只加载缺失的张量，从而减少了通过 PCIe 总线传输的数据量。

    3.  **可重用内存管理 (Reusable Memory Management)**:
        *   **问题建模**: 当内存不足时，需要决定驱逐哪些已存在的张量来为新张量腾出空间。
            这个问题非常复杂，因为它不仅涉及驱逐，还可能需要移动张量以合并小的空闲内存块（即“碎片整理”）。作者创新地将其形式化为一个**多选多维背包问题（MCMDKP）**（公式
            1），目标是最小化总成本（驱逐成本 + 合并成本）。
        *   **两阶段启发式算法**: 由于 MCMDKP 是 NP-hard 问题，作者设计了一个高效的两阶段启发式算法来求解：
            *   **阶段1：最小成本驱逐 (Minimal-Cost Eviction)**:
                首先，通过一个成本模型（公式 2）来评估驱逐每个张量的“代价”。这个代价综合考虑了模型被再次访问的概率、张量大小、模型加载带宽和用户指定的延迟敏感度。
                然后，采用贪心策略，优先驱逐成本最低的张量，直到释放出足够的空间。
            *   **阶段2：分区增益打包 (Partitioned-Gain Packing)**:
                即使总空闲空间足够，内存碎片也可能导致无法分配大的连续张量。为避免全局合并（成本高昂），该算法（算法 1）尝试通过局部、增益驱动的方式来解决碎片问题。
                它递归地将内存空间划分为子空间，并尝试在这些子空间中打包张量，只在必要时才执行小范围的合并，从而以最小的移动成本完成分配。

    4.  **按需 KV 缓存分配 (On-demand KV cache Allocation)**:
        *   Tangram 集成了 **ElasticKV** 引擎，该引擎借鉴了 vLLM 的 PagedAttention 思想。
        *   KV 缓存不再在运行前一次性预分配全部，而是在推理过程中**按需、逐块地**从统一内存池中申请。
        *   这极大地减少了 KV 缓存的平均内存占用，将更多 GPU 内存释放出来，使其可以被 Reuse Store 用来缓存更多的模型张量，从而提高了内存重用的潜力。

    5.  **GPU 亲和性感知调度器 (GPU Affinity-Aware Scheduler)**:
        *   **调度策略（算法 2）**: 当一个新请求到达时，调度器不再是随机选择一个空闲 GPU，而是会**遍历所有可用的 GPU**。
        *   对于每个 GPU，它会查询其 Reuse Store，计算出如果将请求调度到该 GPU 上，**有多少模型张量可以被重用**。
        *   基于可重用的张量大小，它使用一个简单的加载时间估算模型（公式 3）来预测在每个 GPU 上的预期 `Load` 延迟。
        *   最终，它会选择那个**预期加载延迟最低**的 GPU，即与该模型“亲和性”最高的 GPU。

**第4章：实现 (Implementation) 详解**

*   **设计思想**:
    本章的重点在于展示 Tangram 的可行性和工程实践。思想是**基于现有成熟框架进行扩展**，而不是一切从零开始。通过在 SLLM 和 vLLM
    这两个业界广泛使用的开源系统上进行二次开发，可以快速构建功能完备的原型，并确保其核心推理性能与业界标准保持一致，从而使性能评估更有说服力。

*   **关键技术与实现细节**:
    1.  **技术栈**:
        *   **基础框架**: SLLM [14] (用于无服务器框架) 和 vLLM [23] (用于推理引擎，特别是其 PagedAttention 机制)。
        *   **代码量**: 约 11,829 行 CPU 代码 (Python) 和 725 行 CUDA 代码，表明这是一个具有相当工程复杂度的系统。
    2.  **核心实现点**:
        *   **统一内存池管理**: 使用 `cuda_malloc` 在 GPU 上分配一个大的连续内存块作为统一内存池。所有张量和 KV 缓存块都在这个池中进行亚分配（sub-allocation）。
        *   **张量传输**: 使用 `cuda_mem_copy` 在 CPU 和 GPU 之间，以及在 GPU 内部移动张量。为了提高效率，将多个小的相邻张量合并为一个大的拷贝操作，直到达到一个经验阈值（8MB）。
        *   **与 ElasticKV 的集成**: Tangram 的内存管理器（Reuse Store）通过 CUDA IPC (Inter-Process
            Communication) API 与 ElasticKV 引擎共享 GPU 内存句柄。这使得 ElasticKV 可以在不知道内存池全局布局的情况下，安全地请求和使用由 Tangram
            分配的内存块。
        *   **定制化的 CUDA 核**: 扩展了 vLLM 的 PagedAttention，实现了两个新的 CUDA 核 (`reshape_and_cache_segment` 和
            `segmented_attention`)。这些核函数使注意力计算能够直接使用 **物理地址** 访问 KV 缓存，而不是 vLLM
            中默认的逻辑块索引。这是必要的，因为在 Tangram 中，KV 缓存块的物理位置是由全局的 Reuse Store 动态决定的。
        *   **调度器的扩展**: 扩展了 SLLM 的调度器，使其在做决策前能通过 RPC (Remote Procedure Call) 查询各个 Worker 节点上的 Reuse
            Store，获取实时的张量重用信息。

**第5章：评估 (Evaluation) 详解**

*   **设计思想**:
    本章的评估设计旨在全面、系统地验证 Tangram 的有效性。其核心思想是**多维度、对比性**地进行性能测试。
    *   **多维度**: 不仅测试最终的端到端性能（TTFT），还深入剖析性能增益的来源（加载时间、合并开销等），并考察系统在不同硬件配置（单/多 GPU）、工作负载特性（局部性、批大小）和模型规模下的表现。
    *   **对比性**: 将 Tangram 与当前最先进的基线系统（SLLM, SLLM-C, SLLM-CM）进行直接比较，以量化其带来的具体提升。同时，通过消融实验（Ablation
        Study），逐步增加 Tangram 的组件，以验证每个设计选择的贡献。

*   **关键实验与结论**:
    1.  **实验环境**: 使用了单节点（NVIDIA L40 GPU）和多节点（8x NVIDIA 4090 GPUs on Ray）两种环境，覆盖了从单个大内存 GPU 到分布式多 GPU
        的典型场景。选择了多种来自 OPT, LLAMA, Qwen, Yi 等系列的流行 LLM。
    2.  **总体性能 (Overall Performance, 图 8)**:
        *   **发现**: 与最佳基线 SLLM-CM 相比，Tangram 将整体 TTFT 降低了 14%-60%。对于大模型（如 GPT-20B），`Load`
            时间从基线的 3.3 秒显著降低到 Tangram 的 0.67 秒，加载速度提升了 1.8x - 6.2x。
        *   **结论**: 证明了 Tangram 在解决 `Load` 瓶颈方面的巨大优势。
    3.  **推理吞吐量 (Decode Throughput, 表 1)**:
        *   **发现**: Tangram 的按需 KV 缓存机制引入的吞吐量损失低于 3.2%，几乎可以忽略不计。
        *   **结论**: Tangram 在显著降低延迟的同时，并未牺牲稳态推理性能。
    4.  **性能分解 (Performance Breakdown, 图 9)**:
        *   **消融研究**:
            *   仅加入 **Reuse Store** (`+Reuse`): 在小批量时性能提升显著（2.3x-7.6x），但由于需要为 KV 缓存预留大量空间，大模型（GPT-20B）无法完全装入，导致性能受限。
            *   再加入 **按需KV缓存** (`+ODKV`, 即完整 Tangram): 释放了被 KV 缓存占用的内存，使得更多张量可以被重用。相比 `+Reuse`，性能进一步提升 1.16x-2.6x。
        *   **结论**: 内存重用（Reuse）和按需 KV 缓存（ODKV）是相辅相成的，二者结合才能最大化性能收益。
    5.  **张量分配策略分析 (Tensor Allocation Policies, 图 10)**:
        *   **对比**:
            *   `Rand+GM` (随机驱逐+全局合并): 效果最差，合并开销巨大。
            *   `MCE+GM` (最小成本驱逐+全局合并): 驱逐策略有效，但合并开销依然很高。
            *   `MCE+PGP` (Tangram 策略):
                通过分区增益打包，将合并（Merge）开销降低了 93%，几乎消除了碎片整理的代价。
        *   **结论**: Tangram 提出的两阶段内存管理算法（MCE+PGP）是高效且至关重要的。
    6.  **敏感性分析 (Sensitivity Analysis, 图 12)**:
        *   **发现**:
            *   当工作负载的**局部性降低**或**批大小增加**时，Tangram 的性能优势会减小，因为内存重用的机会变少了。在最坏情况下（无局部性），其性能回退到与 SLLM 相同。
            *   当**模型池的总大小**远超单个 GPU 内存时，Tangram 依然能维持稳定的低加载延迟，因为它总能保留最高价值的张量。
        *   **结论**: Tangram 的性能具有鲁棒性，在各种场景下都能带来收益或至少不比基线差。
    7.  **多 GPU 性能 (Multi-GPU Performance, 图 13)**:
        *   **发现**: 在多 GPU 环境下，Tangram 的 GPU 亲和性调度器能有效选择最佳 GPU，相比 SLLM-CM 的随机选择策略，P99 尾延迟降低了 8%-54%。
        *   **结论**: GPU 亲和性调度在多 GPU 环境下效果显著，进一步放大了内存重用的优势。

**第6章：相关工作 (Other Related Work) 详解**

*   **设计思想**:
    本章的目的是将 Tangram 置于现有研究的坐标系中，通过与相关工作的比较来明确其独特性和创新性。其思想在于梳理两个主要研究方向——**无服务器 LLM 加载优化**和 **GPU
    内存共享**——的代表性工作，并逐一分析它们与 Tangram 的异同，从而凸显 Tangram 的核心贡献。

*   **关键技术/工作的比较**:
    1.  **与无服务器 LLM 加载优化工作的比较**:
        *   **SLLM [14]**: 在 CPU 内存中缓存模型，并使用并行传输来加速加载。
            *   **对比**: SLLM 的瓶颈仍在 PCIe 带宽。Tangram 则通过在 GPU 内部重用，直接**减少了需要传输的数据量**，从根本上缓解了带宽压力。
        *   **Prism [48]**: 协调多个 GPU 之间的传输以获得更好的带宽。
            *   **对比**: Prism 同样受限于 PCIe 总带宽。
        *   **Tidal [13] / Sui et al. [36]**: 通过预加载（pre-loading）来隐藏延迟。
            *   **对比**: 预加载受限于模型大小和对未来请求的准确预测。如果模型太大或预测不准，则无法有效工作。Tangram 的重用机制是被动的、机会主义的，不依赖于预测，适用性更广。
        *   **正交性**: 作者强调，Tangram 的 GPU 重用机制与这些技术（如预加载、并行传输）是**正交的**，可以结合使用以获得更进一步的性能提升。

    2.  **与 GPU 内存共享工作的比较**:
        *   **vLLM [23]**: 在**不同序列（sequence）**的 KV 缓存之间共享 GPU 内存，但它假定**每个 GPU 系统中只有一个模型**。
            *   **对比**: vLLM 未解决多模型场景下的内存管理和冷启动问题。Tangram 则将共享的概念从**单模型内的 KV 缓存**扩展到了**多模型间的模型参数（张量）**，并统一管理了张量和
                KV 缓存。
        *   **Choi et al. [11]**: 首次提出在多 GPU 服务器上对机器学习应用进行时空共享。
        *   **Prism [48]**: 允许多个 LLM 在单个 GPU 上并发运行。
            *   **对比**: 这两项工作主要关注并发服务（serving）时的资源利用率，而不是优化冷启动（loading）。Prism 以模型为粒度管理内存，每次切换仍需加载完整的模型参数。
                Tangram 则通过**张量级的重用**和**碎片感知的管理**，实现了更高效的内存利用和更快的模型加载。
