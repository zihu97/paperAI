# 内存受限边缘 TPU 上的多租户推理协作处理 (Collaborative Processing for Multi-Tenant Inference on Memory-Constrained Edge TPUs)

---

## 论文整体骨架

**论文文件名**: 2602.17808v1.pdf
**英文标题**: Collaborative Processing for Multi-Tenant Inference on Memory-Constrained Edge TPUs
**中文标题**: 内存受限边缘 TPU 上的多租户推理协作处理

### 中文摘要
物联网（IoT）应用日益依赖设备端 AI 加速器以确保高性能，特别是在连接受限和安全关键的场景中。
然而，这些加速器有限的片上内存迫使推理运行时在主机和加速器内存之间交换模型段，导致延迟大幅增加。
虽然通过在 CPU 和加速器资源之间划分模型处理的协作处理可以减轻加速器内存压力，但简单的划分可能会
因为将过多计算转移到 CPU 或未能充分抑制交换而使端到端延迟恶化，这一问题在多租户和动态环境中更为突出。
为解决这些问题，我们提出了 SwapLess，这是一个针对内存受限边缘 TPU 的自适应、多租户 TPU-CPU 协作推理系统。
SwapLess 利用解析排队模型捕捉与划分相关的 CPU/TPU 服务时间，以及跨不同工作负载组合和请求率的
模型内和模型间交换开销。利用该模型，SwapLess 在线持续调整划分点和 CPU 内核分配，以最小的决策开销
最小化端到端响应时间。在配备边缘 TPU 的平台上的实现表明，与默认的 Edge TPU 编译器相比，SwapLess
在单租户工作负载下将平均延迟降低了高达 63.8%，在多租户工作负载下降低了高达 77.4%。

### 各章节主旨内容
1.  **引言 (I. Introduction)**: 介绍设备端 AI 推理的必要性及边缘加速器（如 Google Edge TPU）面临的
    内存约束挑战，提出通过 CPU-TPU 协作处理来解决内存交换开销的思路。
2.  **背景与动机 (II. Background and Motivation)**: 量化边缘 TPU 的内存交换开销，分析单模型及多模型
    并发运行时的性能瓶颈，探讨 CPU 卸载的可能性与挑战。
3.  **SwapLess 设计 (III. SwapLess Design)**: 详细描述 SwapLess 的系统架构，包括离线分析阶段和在线编排引擎，
    并推导了用于预测延迟的解析排队模型及贪婪爬山资源分配算法。
4.  **实现 (IV. Implementation)**: 介绍 SwapLess 基于 TensorFlow Lite 的具体实现细节，包括模型划分工具、
    量化处理及在线监控机制。
5.  **实验评估 (V. Experimental Evaluation)**: 在真实硬件上验证排队模型的准确性，并与基线方案对比，
    展示 SwapLess 在静态和动态工作负载下的优越性能。
6.  **相关工作 (VI. Related Work)**: 回顾边缘 TPU 推理优化、跨处理器推理和多租户推理的现有研究。
7.  **结论 (VII. Conclusion)**: 总结 SwapLess 的贡献和实验结果。

### 中文结论
本文提出了 SwapLess，这是一个专门为内存受限的边缘 TPU 设备设计的多租户 TPU-CPU 协作推理系统。
SwapLess 采用了解析排队模型，能够准确刻画多租户环境下协作推理的性能特征，为资源分配提供指导。
评估结果表明，通过动态优化模型划分和 CPU 内核分配，SwapLess 能够显著降低推理延迟，相比于传统的
Edge TPU 编译器，在多租户场景下平均延迟降低了高达 77.4%。

## 章节详解

### 第一章：引言
在物联网场景中，由于带宽限制、延迟要求或隐私安全，AI 推理正从云端转向边缘设备。虽然专用的加速器（如 
Edge TPU）能够显著提升性能，但其硬件设计往往在 SRAM 容量上做出了妥协。例如，Edge TPU 仅提供 8MB SRAM 用于
缓存权重，这对于 ResNet50 等主流模型来说远远不够。当模型超出内存限制时，驱动程序会进行“段交换”，这会
引入巨大的 IO 延迟。作者指出，虽然利用闲置 CPU 核心进行协作是一个潜在路径，但如何划分模型是一个多维
优化问题，特别是在多个模型共享资源的复杂环境下。

### 第二章：背景与动机
本章通过实验量化了交换开销的严重性。
- **模型内交换 (Intra-model Swapping)**: 对于 InceptionV4，交换开销竟占总推理时间的 62.4%。
- **模型间交换 (Inter-model Swapping)**: 在多租户环境下，由于 SRAM 空间的竞争，频繁的上下文切换导致
  权重被反复剔除和重新加载。实验显示，这可能增加高达 49% 的额外延迟。
- **CPU 卸载机会**: 作者观察到，神经网络的后期层（如全连接层或后期卷积层）计算密度相对较低，在通用 
  CPU 上的执行速度有时能与加速器持平甚至更好。这为通过将后期层移至 CPU 来释放 TPU 内存空间提供了依据。

### 第三章：SwapLess 设计
这是论文的核心，提出了一套严谨的数学框架。
- **解析排队模型**: 
    - **TPU 建模**: 将其视为 $M/G/1/FCFS$ 排队系统，使用 Pollaczek-Khinchine 公式计算期望等待时间。
      该模型创新性地将“权重缺失概率” $\alpha$ 纳入服务时间预测，从而捕捉内存压力的影响。
    - **CPU 建模**: 将每个模型的 CPU 推理视为独立的 $M/D/k_i$ 过程，其中 $k_i$ 是分配给该模型的物理内核数。
    - **端到端延迟公式**: 综合考虑了输入传输、TPU 排队与执行、权重加载（基于缺失率）、中间张量传输以及 
      CPU 排队与执行的时间。
- **资源分配算法**: 面对非线性的整数规划问题，SwapLess 放弃了高开销的求解器，转而采用一种“贪婪爬山”启发式算法。
  算法从全 CPU 执行开始，逐步将模型段向 TPU 迁移，并在每一步重新平衡 CPU 内核分配，以找到局部最优解。
  这种方法在资源受限的边缘设备上具有极低的运行开销。

### 第四章：实现细节
SwapLess 的工作流程分为两个阶段：
- **离线阶段**: 使用 `GraphSurgeon` 遍历 TensorFlow 计算图，在所有可能的算子边界进行切分，生成一系列
  “前缀（TPU 运行）”和“后缀（CPU 运行）”对。所有段均经过 8 位量化处理，并由 TPU 编译器进行预编译。
  此外，还需预先测量每个段在目标硬件上的基础执行时间。
- **在线阶段**: 包含一个请求协调器。它通过滑动窗口监测请求速率的变化。一旦检测到负载波动，就会触发 
  Greedy 算法重新配置。推理过程中，TPU 核心处理前缀，生成的中间激活值通过共享内存传递给专用的 CPU 线程池。

### 第五章：实验评估
- **模型验证**: 实验在 Google Coral + Raspberry Pi 5 平台上进行。结果显示，排队模型对单租户的预测
  误差 (MAPE) 仅为 1.9%，对多租户的误差为 6.8%，证明了数学建模的鲁棒性。
- **对比基线**: 
    - 相比 **Edge TPU 编译器**: SwapLess 通过减少交换显著胜出。
    - 相比 **基于阈值的静态划分**: SwapLess 能根据请求率动态调整，避免了静态方案在负载变动时的性能崩溃。
- **动态响应**: 实验演示了当一个模型的请求率上升时，SwapLess 如何主动将其他较轻负载的模型迁移到 CPU，
  从而腾出 TPU 的 SRAM 给高频模型使用。

### 第六、七章：总结与思考
论文最后总结了 SwapLess 填补了现有研究在单设备多租户、内存受限加速器优化方面的空白。通过将复杂的
内存管理和资源调度问题转化为一个在线的数学优化问题，SwapLess 为未来更复杂的边缘异构计算提供了范式。
作者也承认，目前的工作主要集中在卷积神经网络，对于具有动态注意力的 Transformer 模型支持仍是未来的
一个研究方向。
