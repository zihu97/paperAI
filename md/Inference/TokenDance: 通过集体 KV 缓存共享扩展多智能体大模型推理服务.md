# TokenDance: Scaling Multi-Agent LLM Serving via Collective KV Cache Sharing
2604.03143v1.pdf
## 中文标题：TokenDance: 通过集体 KV 缓存共享扩展多智能体大模型推理服务

### 中文摘要
多智能体大语言模型（LLM）应用通常以同步轮次的形式组织执行，其中中央调度器汇总所有智能体的输出并重新分发组合后的上下文。这种“全收集”（All-Gather）通信模式导致了巨大的 KV 缓存（KV Cache）冗余，因为每个智能体的提示词都包含相同的共享输出块，但现有的重用方法无法有效利用这一点。本文提出了 TokenDance 系统，通过利用全收集模式进行集体 KV 缓存共享来扩展并发智能体的数量。TokenDance 的 KV 收集器（KV Collector）在一次集体步骤中完成整个轮次的 KV 缓存重用，使得共享块的重用成本与智能体数量无关。其差分感知存储（Diff-Aware Storage）将兄弟缓存编码为针对单个主副本（Master）的块稀疏差分（Diffs），在代表性工作负载下实现了 11-17 倍的压缩。实验评估表明，在满足服务水平目标（SLO）的前提下，TokenDance 支持的并发智能体数量比带有前缀缓存的 vLLM 多出 2.7 倍，将每个智能体的 KV 缓存存储空间减少了多达 17.5 倍，并实现了比逐请求位置无关缓存高出 1.9 倍的预填充加速。

### 各章节主旨内容
- **Chapter 1: Introduction** - 阐述多智能体 LLM 推理面临的内存瓶颈，分析“全收集”模式导致的 KV 缓存冗余根源。
- **Chapter 2: Background** - 形式化定义全收集模式，并讨论现有 KV 缓存重用技术（如前缀缓存和位置无关缓存 PIC）的局限性。
- **Chapter 3: Overview** - 概述 TokenDance 的核心设计理念：将优化单元从单请求提升到多智能体轮次，包括轮次感知接口、集体重用和差分存储。
- **Chapter 4: TokenDance Design** - 详细介绍系统的三大支柱：轮次感知提示接口、集体 KV 缓存重用算法、差分感知存储方案及融合差分还原。
- **Chapter 5: Implementation** - 描述 TokenDance 在 vLLM 和 LMCache 基础上的实现细节，包括 3000 行 Python 和 500 行 CUDA 代码。
- **Chapter 6: Evaluation** - 在 A100 GPU 上通过 GenerativeAgents 和 AgentSociety 工作负载验证系统的扩展性、计算效率和内存节省。
- **Chapter 7: Related Work** - 回顾 LLM 推理优化、KV 缓存压缩和多智能体系统的相关研究。
- **Chapter 8: Conclusion** - 总结 TokenDance 的贡献，强调通信模式感知在 LLM 推理栈中的重要性。

### 中文结论
TokenDance 系统通过观察到多智能体交互中存在的“全收集”模式，提出了集体 KV 缓存共享方案。该设计遵循两个核心观察：一是现有的重用方法在处理智能体轮次时效率低下；二是即使在轮次级别的重用之后，生成的 KV 缓存仍然高度相似且可压缩。TokenDance 通过集体 KV 缓存重用、主-镜像（Master-Mirror）存储结构和融合差分还原，显著降低了多智能体推理的端到端延迟和内存占用。在代表性工作负载下，它相比 vLLM 减少了 94% 的存储需求。本文的研究表明，通信结构应成为 LLM 推理系统的一等公民，为构建未来高效的大规模智能体平台奠定了基础。

# 章节详细解读

## 1. 引言与背景 (Introduction & Background)

### 1. 设计思想 (Design Philosophy)
- **痛点分析**：多智能体应用（如社交模拟、协作编程）的 KV 缓存压力巨大。由于智能体之间的交互遵循“全收集”模式，所有智能体在下一轮都会收到相同的共享输出块。然而，因为每个智能体的私有历史（Private History）长度不同，这些共享块在提示词中的绝对位置不同，导致传统的前缀缓存（Prefix Caching）失效。位置无关缓存（PIC）虽能重用，但需为每个智能体独立执行昂贵的 RoPE 旋转和位置校正，且每个智能体仍需存储完整的冗余缓存副本。
- **核心动机**：作者认为应当从“轮次”（Round）而非“请求”的角度来优化多智能体推理。如果多个智能体共享大部分上下文，那么重用计算应当是集体的，存储应当是差分的。
- **设计目标**：在有限的 GPU 内存预算下，最大化支持的并发智能体数量，同时降低推理延迟。

### 2. 关键技术 (Key Technologies)
- **All-Gather 模式形式化**：定义第 $t+1$ 轮智能体 $i$ 的提示词为 $P_i^{t+1} = H_i^t || \Pi_i(O^t)$，其中 $H_i^t$ 是私有历史，$O^t$ 是上一轮所有智能体的输出集合，$\Pi_i$ 是调度器定义的布局。这种结构决定了跨请求的高相似度。
- **现有系统的局限**：
    - **前缀缓存**：仅能匹配从位置 0 开始的相同前缀，一旦私有历史发散，后续共享块无法重用。
    - **位置无关缓存 (PIC)**：虽然支持任意偏移量重用，但对 $N$ 个智能体需重复执行 $N$ 次 RoPE 旋转和重要位置选择。

## 2. TokenDance 核心设计 (TokenDance Design)

### 1. 轮次感知提示接口 (Round-Aware Prompt Interface)
- **架构拆解**：引入预留的分隔符令牌 `<TTSEP>` 来标记逻辑块的边界。
- **逻辑流程**：应用层在组装提示词时，在私有历史和各共享输出块之间插入 `<TTSEP>`。
- **关键创新点**：将推理系统从处理“平铺的令牌流”转变为处理“逻辑块序列”。这使得运行时能够使用基于段（Segment-based）的哈希而非固定大小的块哈希，从而准确识别即便处于不同位置的相同内容块。

### 2. 集体 KV 缓存重用 (Collective KV Cache Reuse)
- **架构拆解**：KV 收集器（KV Collector）负责协调具有相同活跃提示长度和缓存跨度的请求。
- **算法流程**：
    - **分组**：将兼容的请求归类。
    - **层级集体重用**：在每一层，将所有请求的 Q 和 K 拼接成一个大张量，执行单次批处理 RoPE 调用。
    - **检查层分析**：在配置的检查层，一次性对比整个组的旋转 Key 与缓存 Key，识别出显著发散的“重要位置”。
    - **差异更新**：后续层直接复用这些重要位置集合，仅对每个请求特定的私有位置进行刷新。
- **设计目标**：将重用分析的开销从随智能体数量线性增长转变为子线性增长。

### 3. 差分感知存储 (Diff-Aware Storage)
- **设计思想**：利用 Master-Mirror 布局消除内存冗余。
- **实现细节**：
    - **Master 副本**：存储一个完整的、稠密的 KV 缓存作为基准（通常选偏差最小的请求）。
    - **Mirror 副本**：仅存储该请求与 Master 之间的块稀疏差分（Block-sparse diffs）。由于共享块占主导，Mirror 通常仅占用完整缓存大小的 10%-20%。
    - **块稀疏表示**：差异位置往往集中在私有历史段或块边界，块粒度表示降低了元数据开销。

### 4. 融合差分还原 (Fused Diff Restore)
- **关键技术**：在 KV 缓存从外部存储搬运到 GPU 显存的层级传输路径中，直接应用差分更新。
- **实现细节**：使用双缓冲区（Ping-pong buffers）交替进行 Master 块加载和原位纠正（In-place correction）。通过 CUDA 扩展实现的 Paired K/V Diff 内核，在执行 RoPE 恢复和写入 PagedAttention 内存之前完成数据融合。
- **对比与权衡**：避免了“先还原完整稠密缓存再读取”的额外内存访问，确保了压缩收益能转化为在线推理的速度提升。

## 3. 实验评估与结论 (Evaluation & Conclusion)

### 1. 关键实验发现
- **扩展能力**：在 A100 上，TokenDance 能支持 8 个满足 SLO 的并发智能体，而 vLLM 仅能支持 2 个。
- **存储压缩**：在 Qwen2.5-14B 模型上实现了 17.5 倍的压缩率。对于 10 个智能体，仅需约 1.5 个完整缓存的显存开销，而非 10 个。
- **计算提速**：由于减少了冗余的重用分析（如 RoPE 旋转），在预填充阶段实现了高达 2.57 倍的加速（10 智能体规模下）。

### 2. 实现细节与代价
- **技术栈**：基于 Python 和 CUDA 实现，集成到 LMCache 和 vLLM。
- **准确性权衡**：由于采用了类似 CacheBlend 的选择性重计算（PIC）机制，会引入极其微小的数值摄动，但在贪婪解码下对模型输出的一致性影响极小（多轮模拟后仅有 3%-12% 的发散，且属于随机性范畴）。

### 3. 核心权衡
- **以管理开销换内存空间**：TokenDance 引入了块管理和差分计算的复杂性，但在多智能体的高冗余场景下，这种权衡带来的内存节省远超其计算开销，是扩展大规模智能体系统的必要路径。
