### 论文基本信息

*   **论文文件名:** 2309.06180v1.pdf
*   **英文标题:** Efficient Memory Management for Large Language Model Serving with PagedAttention
*   **中文标题:** vLLM：使用 PagedAttention 为大型语言模型服务提供高效的内存管理
*   **中文摘要:**
    大型语言模型（LLM）服务的高吞吐量需要将足够多的请求进行批处理。然而，现有系统在这种场景下举步维艰，因为每个请求的键值缓存（KV
    cache）都非常大，并且其大小是动态变化的。如果管理不善，这部分内存会因为碎片化和冗余重复而造成严重浪费，从而限制了批处理的大小。为解决这个问题，我们提出了
    PagedAttention，这是一种受到操作系统中经典虚拟内存和分页技术启发的注意力算法。在此基础上，我们构建了 vLLM，一个
    LLM 服务系统。vLLM
    实现了几乎零浪费的 KV 缓存内存管理，并且支持在请求内部和跨请求之间灵活共享 KV
    缓存，以进一步减少内存使用。我们的评估表明，与现有最先进的系统（如 FasterTransformer 和 Orca）相比，在相同延迟水平下，vLLM
    能将流行 LLM 的吞吐量提高 2-4 倍。这种改进在处理长序列、大模型和复杂解码算法时更为显著。

*   **各章节主旨内容:**
    *   **第 1 章：引言 (Introduction):**
        介绍了在 LLM 服务中，内存管理，特别是 KV 缓存的管理，是限制系统吞吐量的关键瓶颈。现有系统由于内存碎片化和管理效率低下，导致资源浪费严重。为此，论文提出
        PagedAttention 机制来解决该问题。

    *   **第 3 章：LLM 服务中的内存挑战 (Memory Challenges in LLM Serving):**

        KV 缓存管理器，通过“块表”（Block Table）来维护逻辑块到物理块的映射。最后，详细阐述了如何利用这套机制，通过写时复制（Copy-on-Write）等技术高效支持并行采样、集束搜索等多种解码策略，并实现了基于交换（Swapping）和重计算（Recomputation）的请求抢占机制。

    *   **第 7 章：消融研究 (Ablation Studies):**

    *   **第 8 章：讨论 (Discussion):**
        探讨了将虚拟内存和分页思想应用于其他 GPU
        工作负载的可能性，并指出了 vLLM

    *   **第 9 章：相关工作 (Related Work):**

    *   **第 10 章：结论 (Conclusion):**

### 章节详解

*   **第 1 章：引言 (Introduction)**
    大型语言模型（LLM）的应用日益广泛，但其服务成本高昂。提升服务吞吐量是降低成本的关键。现有方法通过批处理（Batching）多个请求来提高
    GPU 利用率，但批处理的大小受到 GPU
    内存的严重制约。特别是用于存储序列历史状态的键值缓存（KV
    Cache），它在推理过程中动态增长，消耗大量内存。
    现有系统（如 FasterTransformer,
    Orca）为每个请求预先分配一块连续的物理内存来存储
    KV Cache，但这导致了严重的内存浪费：
    1.  **内部碎片化 (Internal Fragmentation):**
        由于无法预知输出序列的长度，系统通常按最大可能长度（如
        2048）来分配内存。然而，实际生成的序列长度往往远小于最大值，导致大量已分配但未使用的内存被浪费。
    2.  **外部碎片化 (External Fragmentation):**
        不同请求预留的内存块大小不一，当这些块被频繁申请和释放后，会产生许多不连续的小块空闲内存，难以满足新请求的分配需求。
    3.  **内存共享困难:**
        在并行采样或集束搜索（Beam
        Search）等场景下，多个生成序列共享相同的前缀（Prompt）。若采用连续内存存储，共享这部分
        KV
        Cache 会变得非常复杂或需要大量数据拷贝。
    为了解决这些问题，论文提出了 **PagedAttention**
    的核心思想：借鉴操作系统中虚拟内存和分页的机制，将 KV
    Cache 存放在非连续的物理内存块中。这使得内存管理更加灵活，几乎完全消除了内存碎片，并能轻松实现跨序列的内存共享。基于此思想构建的
    **vLLM** 系统，实现了 2-4 倍的吞吐量提升。

*   **第 2 章：背景 (Background)**
    本章为后续内容铺垫了技术基础。
    *   **自回归模型 (Autoregressive Model):** LLM
        的生成过程是自回归的，即逐个 token
        地生成。每个新 token
        的生成都依赖于它之前所有 token
        的隐藏状态。
    *   **KV Cache:** 在 Transformer
        模型中，这些历史状态表现为自注意力机制中的键（Key）和值（Value）向量。为了避免每次生成新
        token 时都重复计算整个序列的 K 和 V
        向量，服务系统会将它们缓存起来，这就是 KV
        Cache。因此，每生成一个新 token，KV Cache
        就会增长一点。
    *   **LLM 推理的两个阶段:**
        1.  **预填充/提示阶段 (Prefill/Prompt Phase):**
            系统并行处理输入的整个提示（Prompt），为其所有
            token 生成 KV Cache。这个阶段计算密集度高。
        2.  **解码/生成阶段 (Decoding/Generation Phase):**
            系统逐个生成输出 token。这个阶段是内存密集型（Memory-bound），GPU
            的计算单元常处于空闲状态。
    *   **批处理技术 (Batching):** 为了改善解码阶段的 GPU
        利用率，现有系统采用**持续批处理**（Continuous
        Batching）或**迭代级调度**（Iteration-level
        Scheduling）。其核心思想是在每个生成步骤（Iteration）后，动态地将新请求加入批次，并将已完成的请求移出批次，从而让
        GPU 在任何时候都能处理尽可能多的请求。然而，这种方法的批处理大小最终还是受限于管理
        KV Cache 的内存效率。

*   **第 3 章：LLM 服务中的内存挑战 (Memory Challenges in LLM Serving)**
    本章深入剖析了 KV Cache 管理低效的根源。
    *   **巨大的内存占用:** KV Cache
        本身非常大。例如，对于一个 13B
        的OPT模型，单个 token 的 KV Cache 就需要 800KB，一个长度为
        2048 的序列将占用高达 1.6 GB 的内存。这使得单个 GPU
        能容纳的并发请求数非常有限。
    *   **连续内存分配的弊端:**
        现有深度学习框架要求张量（Tensor）在物理内存中是连续的。LLM
        服务系统沿用了这一设计，为每个请求的 KV Cache
        分配一个巨大的连续内存块。这种做法导致了三种内存浪费（如图
        3 所示）：
        1.  **预留内存 (Reserved Memory):**
            为未来可能生成的 token
            预留的内存空间，在请求的生命周期早期完全处于闲置状态。
        2.  **内部碎片 (Internal Fragmentation):**
            请求的实际输出长度小于预分配的最大长度，造成了空间浪费。
        3.  **外部碎片 (External Fragmentation):**
            不同大小的内存块释放后，形成不连续的空闲空间，难以再利用。
    *   **复杂解码策略带来的挑战:**
        1.  **并行采样 (Parallel Sampling):**
            一个输入产生多个输出。这些输出共享了输入的
            KV Cache。若内存连续，则必须为每个输出都复制一份共享数据，造成浪费。
        2.  **集束搜索 (Beam Search):**
            在生成过程中维护多个候选序列（Beam）。这些序列之间可能存在复杂的
            KV Cache 共享关系（部分前缀相同），并且这种共享关系是动态变化的。连续内存模型难以高效处理这种动态共享。

*   **第 4 章：方法 (Method)**
    本章是论文的核心，详细阐述了 PagedAttention 和 vLLM
    系统的设计。
    *   **4.1 PagedAttention 算法:**
        *   **核心思想:** 解除 KV Cache
            在逻辑上和物理上的连续性约束。它将每个序列的 KV
            Cache 分割成固定大小的**块 (Block)**。这些块在物理内存中可以不相邻，就像操作系统的物理页（Physical
            Page）一样。
        *   **工作原理:**
            注意力计算被修改为可以操作这些非连续的块。在计算注意力分数时，GPU
            核函数会根据一个名为**块表（Block
            Table）**的元数据，找到每个逻辑块对应的物理块，然后从正确的物理地址读取
            K 和 V 向量。
    *   **4.2 KV Cache 管理器:**
        *   **类比操作系统:** vLLM
            的内存管理器模仿了操作系统的虚拟内存管理。它将请求看作“进程”，将
            KV Cache 块看作“页”，将块表看作“页表”。
        *   **逻辑块与物理块:**
            每个请求的 KV Cache 由一系列**逻辑块**组成。管理器维护着一个**块表**，负责将这些逻辑块映射到
            GPU 显存中的**物理块**。
        *   **动态分配:** 当序列增长需要更多空间时，管理器只需分配一个新的物理块，并更新块表即可，无需进行大规模的数据复制。这从根本上消除了内部和外部碎片。
    *   **4.3 & 4.4 高效支持复杂解码策略:**
        *   **内存共享与写时复制 (Copy-on-Write):**
            当多个序列共享同一个前缀时（如并行采样或共享系统提示），它们的块表可以指向相同的物理块。vLLM
            为每个物理块维护一个引用计数。当某个序列需要修改一个共享块时（例如，在共享前缀后生成了自己独有的
            token），vLLM 会触发**写时复制**：为该序列分配一个新的物理块，复制共享块的内容，然后更新其块表指向新块。这样既实现了最大化的内存共享，又保证了序列间的隔离。
        *   **集束搜索 (Beam Search):**
            在集束搜索中，当一个候选序列被丢弃时，其占用的物理块的引用计数会减少。当引用计数归零时，该物理块被回收，可用于其他序列。这种机制极大地减少了传统实现中频繁的内存拷贝开销。
    *   **4.5 调度与抢占 (Scheduling and Preemption):**
        *   **调度策略:** 采用简单的先进先出（FCFS）策略保证公平性。
        *   **抢占:** 当 GPU
            物理块耗尽时，vLLM
            需要抢占某些正在运行的请求。它采用“序列组”（Sequence
            Group）的**“全有或全无”**策略，即一个请求的所有相关序列要么全部保留，要么全部被抢占。
        *   **恢复机制:** 被抢占的请求有两种恢复方式：
            1.  **交换 (Swapping):**
                将其 KV Cache
                块从 GPU 显存交换到 CPU
                内存，待有空闲资源时再换回。
            2.  **重计算 (Recomputation):**
                直接丢弃其 KV Cache，待恢复时将已有 prompt 和
                生成内容拼接成一个新的 prompt
                重新计算，一次性恢复 KV Cache。
    *   **4.6 分布式执行:**
        vLLM 支持基于 Megatron-LM
        的张量并行。在分布式环境中，所有 GPU Worker
        共享一个中心的 KV Cache
        管理器（位于调度器中）。每个 Worker
        虽然只存储模型参数和 KV Cache
        的一部分，但它们都持有完整的块表信息，因此能够正确地找到并访问自己需要的数据。

*   **第 5 章：实现 (Implementation)**
    本章关注 vLLM 的工程实践。
    *   **系统架构:** vLLM
        是一个端到端的服务系统，包含一个基于 FastAPI 的
        OpenAI 兼容前端接口和一个高性能的后端推理引擎。
    *   **定制 CUDA 核函数:** 为了最小化 PagedAttention
        引入的开销并提升性能，vLLM
        团队开发了多个关键的 CUDA 核函数：
        1.  **Fused Reshape and Block Write:**
            将新生成的 KV
            向量进行形状变换并写入指定物理块的操作合并成一个核函数。
        2.  **Fusing Block Read and Attention:**
            修改注意力核函数，使其能直接根据块表读取非连续的
            KV Cache 块并即时计算注意力，避免了额外的访存开销。
        3.  **Fused Block Copy:**
            将多个离散块的复制操作（由写时复制触发）合并到一次核函数调用中，减少启动开销。
    *   **统一的解码算法接口:** 系统内部抽象出 `fork`, `append`, `free`
        三种基本操作来管理序列的生命周期。`fork`
        从一个现有序列创建新序列（用于并行采样或集束搜索），`append`
        向序列追加新 token，`free`
        删除序列。所有复杂的解码算法都可以通过组合这三种操作来实现，大大简化了系统的设计。

*   **第 6 章：评估 (Evaluation)**
    本章通过详实的实验数据验证了 vLLM 的性能优势。
    *   **实验设置:**
        *   **模型:** 使用了不同规模的 OPT 模型（13B, 66B, 175B）和
            LLaMA-13B 模型。
        *   **数据集:** 使用了 ShareGPT（代表长对话场景）和 Alpaca
            （代表短指令场景）构建合成负载。
        *   **基线系统:**
            *   **FasterTransformer (FT):**
                NVIDIA
                官方优化过的推理引擎，但调度功能简单。
            *   **Orca:**
                当时最先进的开源服务系统，实现了持续批处理，但内存管理仍是基于连续分配。作者复现了
                Orca 的三种变体：**Orca
                (Max)**（按最大长度预分配）、**Orca
                (Pow2)**（按2的次幂向上取整分配）和理想化的
                **Orca (Oracle)**（假设能预知准确输出长度）。
    *   **主要发现:**
        *   **吞吐量显著提升:** 在所有测试场景下，vLLM
            的吞吐量都远超基线系统，通常能达到 Orca (Oracle) 的
            1.7-2.7 倍，是 Orca (Max) 的 2.7-8
            倍。这证明了 PagedAttention
            在提高内存利用率方面的巨大成功，使得系统可以容纳更大的批处理规模。
        *   **长序列和复杂解码优势更明显:** 在 ShareGPT
            这种长序列负载，以及并行采样和集束搜索等需要大量内存共享的场景下，vLLM
            的性能优势更加突出。例如，在 beam width 为 6
            的集束搜索任务中，vLLM 的吞吐量提升可达 2.3 倍。
        *   **内存节省:**
            通过物理块共享，vLLM
            在并行采样中节省了高达 30.5% 的内存，在集束搜索中节省了高达
            66.3% 的内存。

*   **第 7 章：消融研究 (Ablation Studies)**
    本章深入探究了 vLLM
    内部机制的影响。
    *   **PagedAttention 核函数开销:**
        与高度优化的 FasterTransformer
        的注意力核函数相比，PagedAttention
        由于需要额外的查表和分支逻辑，带来了约
        20-26%
        的延迟开销。但作者认为，这点计算开销与通过内存优化带来的巨大吞吐量提升相比，是完全值得的。
    *   **块大小 (Block Size) 的影响:**
        块大小是一个重要的超参数。块太小，会导致元数据（块表）开销增大，且不利于
        GPU 并行读取；块太大，则会重新引入内部碎片问题。实验表明，对于大多数工作负载，块大小设置为
        16 是一个在 GPU
        利用率和内存浪费之间的良好平衡点。
    *   **重计算 vs. 交换:**
        对抢占的恢复机制进行了微基准测试和端到端测试。结果显示：
        *   当块较小时，交换操作由于涉及大量小数据的
            PCIe 传输，效率低下，此时重计算更优。
        *   当块较大时，交换的效率更高。
        *   在块大小为 16-64
            的中间区域，两者性能相当。这为系统根据实际情况动态选择恢复策略提供了依据。

*   **第 8 章：讨论 (Discussion)**
    本章对 vLLM
    的技术思想进行了延伸和反思。
    *   **通用性:**
        虽然分页思想在 LLM
        服务中效果显著，但它并非适用于所有 GPU
        工作负载。例如，在张量形状固定的 DNN
        训练中，内存可以被静态优化，引入分页反而会带来不必要的开销。该技术更适用于那些内存使用模式动态且性能受限于内存容量的场景。
    *   **LLM 服务的特殊性:** vLLM
        并非简单地照搬操作系统的分页机制，而是根据 LLM
        服务的特性进行了深度定制和优化。
        *   **“全有或全无”的交换策略:**
            利用了“一个请求的所有 token
            在计算时都必须在场”的特性。
        *   **重计算恢复:**
            这在通用操作系统中是不可行的，但在 LLM
            服务中却是一种有效的恢复手段。
        *   **核函数融合:**
            通过将内存访问操作与计算操作融合，大大减轻了内存虚拟化带来的性能开-销。

*   **第 9 章：相关工作 (Related Work)**
    本章将 vLLM
    置于更广阔的研究背景中进行比较。
    *   **通用模型服务系统:**
        像 TensorFlow Serving、Clipper
        等系统提供了通用的服务框架，但没有针对 LLM
        的自回归特性和 KV Cache 进行特殊优化。
    *   **专用 Transformer 服务系统:**
        许多系统为 Transformer
        的推理做了优化，如使用专门的 CUDA 核函数、模型并行等。
    *   **与 Orca 的关系:** Orca
        通过迭代级调度优化了**计算**的流水线，而 vLLM
        通过 PagedAttention 优化了**内存**的利用。论文指出，这两者是**互补**的。一个高效的调度器（如
        Orca）会让内存管理变得更具挑战性，从而凸显出
        vLLM 的 PagedAttention
        机制的重要性。将两者结合，有望实现更进一步的性能提升。
    *   **其他内存优化工作:**
        虽然有其他工作研究内存优化（如压缩、交换等），但
        vLLM 是第一个系统性地将“分页”思想引入
        LLM 在线服务并解决其内存碎片和共享难题的工作。

*   **第 10 章：结论 (Conclusion)**
    论文重申了其核心贡献：识别并解决了 LLM 服务中 KV
    Cache 管理的核心痛点。通过提出 PagedAttention 算法和构建 vLLM
    系统，成功地将操作系统的经典内存管理思想适配到 LLM
    服务场景中，实现了显著的性能提升，为构建更高效、更经济的
    LLM 服务奠定了坚实的基础。
