## 中文标题：LLMServingSim 2.0：异构且解耦的 LLM 推理基础设施统一模拟器

### 中文摘要
大语言模型（LLM）推理基础设施正朝着异构化和解耦化的方向演进。现代部署越来越多地集成多样化的加速器和近内存计
算技术，同时在系统软件层面将计算、内存和模型组件分离，以提升扩展性和效率。然而，现有的模拟器难以在统一框架
内共同建模异构硬件和解耦推理技术。本文提出了 LLMServingSim 2.0，这是一个统一的系统级模拟器，旨在使异构和解
耦推理环境中的软硬件交互变得显式且可分析。它将推理决策和硬件行为嵌入单一运行时循环，支持对批处理、路由、放
置、卸载、内存管理和功耗的交互感知建模。验证结果显示，该模拟器能以 0.97% 的平均误差复现性能、内存和功耗指
标，为下一代 LLM 推理基础设施的软硬件协同设计提供了实践桥梁。

### 各章节主旨内容
- **Chapter 1: Introduction** - 阐述了 LLM 推理基础设施从同构 GPU 集群向集成 NPU、TPU 及 PIM/PNM 的异构解耦架构
  转化的趋势，指出当前模拟器在建模软硬件复杂交互方面的局限。
- **Chapter 2: Background** - 介绍了 LLM 推理的两阶段工作流（Prefill & Decode）、现有的各种优化技术（如 PD 分离、
  MoE 等）以及异构加速器与内存层级的现状。
- **Chapter 3: Motivation** - 分析了建模 LLM 推理动态特性的复杂性，指出静态模型无法捕获请求流带来的时变资源竞争，
  强调了对集成硬件保真度、内存动态和推理策略的统一模拟器的迫切需求。
- **Chapter 4: LLMServingSim 2.0** - 详细描述了模拟器的架构，包括输入规范、基于算子的剖析（Profiling）、推理工作
  流、执行规划器、模型推理组（MSG）以及系统模拟器核心。
- **Chapter 5: Serving Techniques and Use-case Modeling** - 展示了如何利用该模拟器建模异构设备卸载、MoE 专家路由、
  PD 分离部署以及多级前缀缓存（Prefix Caching）等前沿技术。
- **Chapter 6: Methodology** - 说明了实验基准设置，涵盖 RTX A6000、H100 和 TPU-v6e 等多种平台，以及算子级剖析的具体
  配置。
- **Chapter 7: Evaluation** - 通过与真实系统对比验证了模拟器在性能、功耗和内存使用方面的准确性，并展示了其在探索
  多实例前缀缓存等场景下的能力。
- **Chapter 8: Related Work** - 回顾并对比了硬件级、基础设施级及系统级 LLM 推理模拟器的相关研究。
- **Chapter 9: Conclusion** - 总结了 LLMServingSim 2.0 作为学术界和工业界探索下一代 LLM 推理基础设施设计空间的工具
  价值。

### 中文结论
LLMServingSim 2.0 通过将算子驱动的执行、推理动态和异构内存系统整合进统一的运行时循环，成功解决了理解异构与解
耦 LLM 推理系统中复杂软硬件交互的挑战。该模拟器不仅在复现真实系统行为方面表现出极高的准确度（平均误差 0.97%），
而且具备良好的可扩展性，能够支持 TPU、PIM 等新兴硬件的快速建模。它为研究人员提供了一个实用的平台，能够在无需支
付高昂物理测试成本的情况下，系统地探索和协同设计未来的 LLM 推理架构。

# 章节详细解读

## Chapter 4: LLMServingSim 2.0 系统架构

### 1. 设计思想 (Design Philosophy)
- **痛点分析**：现有模拟器要么侧重于底层硬件微架构（如 LLMCompass），忽略了动态请求导致的运行时决策；要么侧重于
  系统级调度（如 Vidur），对异构内存层级和新兴加速器（如 PIM）的支持有限，且无法捕获软硬件之间的反馈循环。
- **核心动机**：作者主张“交互感知”（Interaction-awareness）。LLM 推理的性能不再由孤立的硬件或软件决定，而是
  由调度、数据迁移和互连行为在运行时的交互决定。因此，必须将推理决策（如批处理、卸载）与硬件性能模型耦合在同一
  循环中。
- **设计目标**：构建一个能够同时处理异构设备池、多级存储架构、以及复杂并行/路由策略的系统级模拟器，同时保持较高
  的模拟效率。

### 2. 关键技术 (Key Technologies)
- **模型推理组 (Model Serving Group, MSG)**：这是模拟器中的逻辑执行单元。每个 MSG 负责管理一个可自定义的设备池
  （Device Pool），支持 GPU、NPU、CXL 附加内存及 PIM 模块的混合配置。
- **交互式剖析器 (Operator-level Profiler)**：基于 PyTorch/HuggingFace 开发，仅需在单台设备上运行一次剖析任务
  （例如 Llama-3 70B 在 H100 上约 2.1 小时），即可获取不同算子的延迟和功耗曲线，供模拟器重复使用。
- **运行时驱动循环**：模拟流程并非静态的 DAG 执行。`Request Router` 将请求分发给 MSG，MSG 内部的 `Batch Scheduler` 
  根据实时内存状态决定批处理，`Operation Mapper` 映射算子到异构设备，最后生成执行图交给系统模拟器评估。

### 3. 实现细节 (Implementation Details)
- **算子映射逻辑**：当设备池包含 GPU 和 PIM 时，`Operation Mapper` 可以根据策略将 Attention 算子映射到 PIM 执
  行，而将其他线性层保留在 GPU 上。模拟器会自动在执行图中插入 GPU 与 PIM 之间的数据迁移操作。
- **内存模型扩展**：扩展了传统的内存模型，显式建模 HBM、DRAM、CXL 内存池之间的带宽竞争和 KV Cache 迁移开销。
- **功耗建模**：采用三状态模型（Idle, Active, Standby）对加速器建模，并分解为 CPU、DRAM、网卡等 7 个主要组件，
  利用剖析数据实时计算总能耗。

### 4. 权衡与对比 (Trade-offs & Comparison)
- **对比分析**：与 Vidur 等轻量级模拟器相比，LLMServingSim 2.0 增加了对 PIM、CXL 等异构硬件的底层建模；与 TokenSim 
  相比，它支持多实例和复杂的 PD 分离、MoE 并行策略。
- **优劣势考量**：由于采用了更详细的内存和互连建模，其模拟时间通常在 10 分钟左右，虽慢于仅使用分析模型的模拟器，
  但在捕获“竞争放大”等复杂动态现象时具有不可替代的保真度。

## Chapter 5: 服务技术与用例建模

### 1. 设计思想 (Design Philosophy)
- **痛点分析**：解耦式推理（如 PD 分离、MoE 专家离线）引入了跨节点的频繁数据交换，其瓶颈往往在网络而非计算。
- **核心动机**：通过在模拟器中显式支持多种解耦策略，允许研究人员在投入昂贵的基础设施建设前，先行评估不同硬件组合
  下的通信开销。

### 2. 关键技术 (Key Technologies)
- **PD 分离建模**：支持将 Prefill MSG 和 Decode MSG 配置在不同类型的硬件上（例如用 H100 做 Prefill，用 PIM 做 
  Decode），并自动处理 KV Cache 的层级传输。
- **多级前缀缓存 (Multi-tier Prefix Caching)**：实现了基于 Radix Tree 的缓存机制，支持设备级、主机内存级及全
  局 CXL 存储级的三级缓存共享，精确模拟 Cache Hit 对端到端延迟的削减作用。

### 3. 实现细节 (Implementation Details)
- **MoE 专家路由**：模拟器内部实例化了一个 `Expert Router`，模拟门控行为。根据负载平衡策略（如轮询、随机或用户
  定义），动态决定每个 Token 在各个专家层之间的路由路径。
- **PIM 加速执行**：不仅仅是简单的延迟加成，模拟器会将 PIM 算子拆解为具体的数据加载、计算和回写指令，以便在异构
  背景下评估其对内存总线的真实压力。

## Chapter 7: 实验验证与结果分析

### 1. 验证目标
- **准确性**：在不同 GPU（RTX A6000, H100）和 TPU 平台上，与使用 vLLM 框架的真实部署对比吞吐量、延迟和功耗。
- **有效性**：通过 PIM 和 CXL 的用例研究，证明其辅助架构探索的能力。

### 2. 核心实验发现
- **吞吐量误差**：在 A6000 上平均误差为 5.66%，在 H100 上仅为 2.98%。这主要得益于其对运行时批处理和排队行为的准
  确模拟。
- **功耗预测**：在不同张量并行（TP）度下，功率曲线与真实测量值高度契合，总能量消耗误差约为 1.34%。
- **内存与缓存**：在多实例前缀缓存场景下，模拟器准确复现了随请求积累而增加的缓存命中率和内存占用步进式增长，误差
  保持在 0.41% 以内。

### 3. 权衡与对比 (Trade-offs & Comparison)
- **性能权衡**：在处理复杂的 PD 分离和 MoE 场景时，LLMServingSim 2.0 耗时约 478.2 秒，而 APEX 和 Vidur 因为缺乏
  相应功能或简化了抽象，往往无法运行或产生巨大误差。
- **架构增益评估**：实验显示，利用 PIM 加速 Decode 阶段可以将请求完成后的单位 Token 能耗降低 32.3%，这直接验证了
  异构硬件在解决内存受限任务中的潜力。
