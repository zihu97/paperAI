中文标题: MegaScale-Infer：以解耦专家并行化大规模服务MoE模型
中文摘要: 混合专家模型 (MoE) 在提升大型语言模型 (LLM) 性能并降低计算复杂度方面展现出巨大潜力。
然而，其稀疏激活架构导致推理过程中前馈网络 (FFN) 从计算密集型变为内存密集型，
从而导致 GPU 利用率显著降低和运营成本增加。本文提出了 MegaScale-Infer，
一个高效且经济划算的服务大规模 MoE 模型的系统。MegaScale-Infer 在每个模型层
中将注意力 (Attention) 和 FFN 模块解耦，实现了模块的独立扩展、定制并行策略
以及异构部署。为了充分利用 MoE 稀疏性下的解耦优势，MegaScale-Infer 引入
了乒乓管线并行机制，将请求批次划分为微批次，并在注意力模块和 FFN 模块之间
穿梭以进行推理。结合每个模块独特模型并行化，MegaScale-Infer 有效地隐藏
了通信开销并最大化了 GPU 利用率。为适应解耦的注意力和 FFN 模块并最小化
数据传输开销，MegaScale-Infer 提供了一个高性能 M2N 通信库，消除了不必要
的 GPU-to-CPU 数据拷贝、组初始化开销和 GPU 同步。实验结果表明，
MegaScale-Infer 比现有解决方案实现了高达 1.90 倍的每 GPU 吞吐量提升。

指定章节详解:

*   1 引言 (Introduction):
    *   设计思想: 本章首先指出，尽管 MoE 模型在扩展大型语言模型 (LLM) 方面潜力巨大，
        但在实际推理服务中，其稀疏激活架构导致前馈网络 (FFN) 变为内存密集型，
        而非计算密集型，这严重降低了 GPU 利用率，并增加了运营成本。为了解决
        这一核心问题，MegaScale-Infer 被提出，旨在提供一个高效且经济的 MoE 模型服务系统。
    *   关键技术: MegaScale-Infer 的设计核心在于“解耦”思想。它将 LLM 的每个层中的
        注意力模块 (Attention) 和 FFN 模块进行解耦，允许它们独立扩展，并应用
        定制化的并行策略和异构部署。这种解耦使得系统能够更灵活地分配资源，
        以适应不同模块的计算和内存需求。
    *   实现细节: 为了克服解耦带来的挑战，MegaScale-Infer 引入了“乒乓管线并行”机制，
        将请求批次分解为微批次，并在注意力模块和 FFN 模块之间进行高效穿梭。
        同时，为了最小化数据传输开销，系统开发了一个高性能的 M2N (Many-to-Many)
        通信库，该库专注于消除不必要的 GPU-to-CPU 数据拷贝、组初始化开销和
        GPU 同步，以确保通信效率和稳定性。

*   2 背景与动机 (Background and Motivation):
    *   设计思想: 本章深入探讨了 LLM 推理的固有特性以及 MoE 模型在大规模服务中面临的挑战，
        为 MegaScale-Infer 的设计提供了理论基础和实际动机。它指出，传统 LLM 推理的自回归
        性质以及注意力机制和 FFN 在不同阶段（预填充和解码）的计算/内存特性差异，
        是导致 GPU 利用率低下的主要原因。
    *   关键技术:
        *   LLM 推理特性: 强调了 Transformer 架构中注意力模块在解码阶段的内存密集性
            和 FFN 模块在稀疏 MoE 模型中因小批量大小而导致的计算利用率低下。
        *   MoE 服务问题: 详细分析了 MoE 模型的稀疏性虽然能降低计算复杂度，
            但也会导致 FFN 模块的有效批量小，进而降低 GPU 利用率。图 1 形象地展示了
            MoE 模型 (b) 相对于稠密模型 (a) 在 FFN GPU 利用率上的劣势。
        *   机遇与挑战: 提出了通过“独立扩展”和“异构部署”来优化 MoE 服务的设想。
            独立扩展允许注意力模块和 FFN 模块根据各自特性独立分配资源；
            异构部署则允许为不同类型的模块选择最适合的硬件（例如，内存优化的 GPU
            用于注意力，计算优化的 GPU 用于 FFN）。实现这些优势面临的挑战包括
            如何处理解耦引入的空闲周期以及如何高效管理 M2N 通信。
    *   实现细节: 本章通过具体案例（如 Mixtral 8x22B）和屋顶模型 (Roofline Model)
        的分析，量化了 FFN 模块在 GPU 利用率方面的瓶颈，并为后续章节中 MegaScale-Infer
        的解决方案提供了强有力的动机。例如，对于 A100 GPU，即使批处理大小达到 156
        个令牌，FFN 的 MFU (Model Flops Utilization) 仍然很低。

*   3 MegaScale-Infer 概述 (MegaScale-Infer Overview):
    *   设计思想: MegaScale-Infer 的核心设计思想是通过“解耦专家并行化”来克服 MoE 模型在
        大规模服务中的效率低下问题。其基本理念是认识到注意力模块和 FFN 模块在 LLM 推理中
        具有不同的计算和内存特性，因此应将其视为独立实体进行管理和优化。
    *   关键技术:
        *   解耦专家并行化: 这是一种混合并行策略，将 MoE 模型中的专家模块分配到不同的 GPU
            上，形成一个“专家并行组”。同时，注意力模块的参数在所有注意力节点上进行复制 (数据并行)，
            而 FFN 模块则通过专家并行进行扩展。这种方式使得每个模块可以独立扩展，并选择最
            适合其特性的并行策略。
        *   乒乓管线并行化: 为了解决解耦架构可能导致的模块间空闲时间问题 (例如，注意力模块等待
            FFN 计算结果，反之亦然)，MegaScale-Infer 引入了乒乓管线并行。它将请求批次分解
            为多个微批次，并以流水线方式在注意力模块和 FFN 模块之间传输。这样，当一个模块
            处理一个微批次时，另一个模块可以处理另一个微批次，从而实现计算和通信的重叠，
            隐藏通信开销并提高 GPU 利用率。
        *   高性能 M2N 通信: 解耦架构导致注意力模块和专家模块之间需要 M2N (Many-to-Many)
            通信，而非传统的 All-to-All 通信。MegaScale-Infer 开发了一个专门的 M2N 通信库，
            旨在解决现有通信库（如 NCCL）在处理这种特定模式时存在的性能瓶颈。
            该库专注于减少不必要的 GPU-to-CPU 数据拷贝、消除组初始化开销和 GPU 同步，
            从而提高通信效率和稳定性。
    *   实现细节: 系统的整体架构如图 3 所示，清晰地展示了注意力节点和专家节点如何通过
        乒乓管线并行和 M2N 通信协同工作。通过这种方式，MegaScale-Infer 实现了注意力
        模块和 FFN 模块的独立扩展和异构部署，显著提高了系统效率并降低了服务成本。

*   4 解耦专家并行化 (Disaggregated Expert Parallelism):
    *   设计思想: 本章深入探讨了 MegaScale-Infer 中解耦专家并行化的设计细节，
        特别是乒乓管线并行策略和部署计划生成方法。核心思想是识别并克服解耦架构中
        固有的空闲时间和通信开销，通过智能调度和通信优化实现 GPU 的持续高利用率。
    *   关键技术:
        *   乒乓管线并行: 如图 4 所示，一个请求批次被分解为 `m` 个微批次，
            在注意力节点和专家节点之间形成乒乓管线。每个微批次在 MoE 层中
            会经过两次前向计算和中间结果交换。通过这种方式，计算和通信
            可以重叠，从而提高 GPU 利用率。为了实现高效的乒乓管线并行，
            论文提出了三个关键约束条件：
            *   `Ta ≈ Te`: 注意力节点和专家节点的微批次计算时间应近似相等，
                以最小化计算依赖导致的 GPU 空闲时间。
            *   `Tc < Tf`: 微批次的通信时间应小于其前向计算时间，
                以确保通信能够被计算完全隐藏。
            *   `m × Tf ≥ 2 × (Tf + Tc)`: 微批次的总计算时间应足够长，
                以覆盖一个 MoE 层中两次通信和计算的开销，
                确保流水线能够持续运行并隐藏通信延迟。
        *   部署计划搜索: MegaScale-Infer 采用一种搜索算法（算法 1）来找到
            给定硬件设置和模型配置下的最优部署计划。搜索空间包括注意力节点和
            专家节点的张量并行大小 (`tpa`, `tpe`)、注意力节点数量 (`na`)、
            微批次数量 (`m`) 和全局批处理大小 (`B`)。目标是最小化单位成本吞吐量，
            同时满足服务级别目标 (SLO)。该算法通过模拟和二分查找来确定最优参数。
    *   实现细节:
        *   迭代延迟估计: MoE 层的单次解码迭代延迟 (`Titer`) 和全局批处理的总迭代延迟
            (`Ttotal`) 通过公式 (4) 和 (5) 进行估计，考虑了注意力节点和专家节点之间
            计算的不平衡性以及通信开销。
        *   性能模拟: 为了精确模拟 `Ta`, `Te`, `Tc`，论文分析了 MoE 层中使用的 GEMMs
            （通用矩阵乘法）的输入和参数形状（表 2）。`Ta` 和 `Te` 的计算被建模为
            线性函数，其中的系数通过剖析和插值获得。`Tc` 通过对网络带宽利用率和
            消息大小之间的关系进行剖析来估计。
        *   约束条件: 除了上述乒乓管线并行约束外，搜索过程还需满足总体迭代时间
            小于 SLO (`Titer < SLO`) 和 GPU 内存容量限制 (`4mbashL/g + 2Pa < tpaCa`)。

*   5 高性能 M2N 通信 (High-Performance M2N Communication):
    *   设计思想: 本章的核心设计思想是为解耦专家并行化架构中的 M2N 通信开发
        一个定制化的高性能通信库，以克服现有通用通信库（如 NCCL）在 MoE 推理中
        令牌分发和聚合时的局限性和低效性。
    *   关键技术:
        *   M2N 发送方: M2N 发送方的设计（图 6）遵循流式编程模型，
            利用 CUDA 事件确保前一个核函数执行完毕且待传输张量已填充，
            然后通过 CUDA 驱动程序操作阻塞流，保证下一个核函数在传输完成后才开始。
            数据传输通过定制的 CPU 通信库 (Core Sender) 实现高效的 RDMA 写入。
            发送方通过轮询完成队列 (CQ) 确认数据已写入远程缓冲区。最后，
            通过更新共享内存标志解阻塞流，允许其他核函数继续使用已注册内存。
            这种设计避免了复杂的 GPU 同步、GPU-to-CPU 拷贝和组初始化开销。
        *   M2N 接收方: M2N 接收方（图 7）的设计与发送方类似，
            也等待 CUDA 事件并阻塞流。接收方通过轮询相关 CQ 验证数据是否
            成功从发送方传输。为了确保 GPU 级别的数据一致性，
            定制的 CPU 通信库 (Core Receiver) 也被利用。
        *   流量导向优化: 为了提高通信效率和稳定性，引入了多项流量导向优化：
            *   高优先级 ACK: 通过将 ACK 包分配到高优先级队列，并经验性微调相关权重，
                解决了双向通信中 ACK 包排队或低优先级传输导致的延迟问题。
            *   拥塞控制微调: 针对不平衡通信场景（每个接收方的发送数据量差异大），
                微调拥塞控制算法以最小化限速效应并加速收敛。
        *   与 DeepEP 的比较: 论文将 MegaScale-Infer 的通信策略与 DeepEP 进行比较。
            MegaScale-Infer 采用 CPU 进行节点间通信，避免了 GPU 资源消耗，
            而 DeepEP 则采用 GPU-to-GPU 直接通信。在数据量较大时
            (数百 KB 级别)，单线程 CPU 足够饱和带宽，CPU 方式更优。
    *   实现细节: 通过图 6 和图 7 展示了 M2N 发送方和接收方的组件和交互，
        突出了消除传统通信库中不必要开销的关键机制。实验结果（图 5）表明，
        现有 NCCL 解决方案存在显著的开销和不稳定性问题，而定制的 M2N 通信库
        显著降低了延迟并提高了稳定性。

*   6 实现 (Implementation):
    *   设计思想: 本章重点介绍了 MegaScale-Infer 在实现层面如何通过
        “融合内核”和“高性能 M2N 通信库”来进一步提升效率和降低延迟，
        同时解决真实世界 MoE 服务中的负载均衡问题。
    *   关键技术:
        *   融合内核:
            *   计算与通信重叠: 第一类融合内核旨在重叠张量并行 (TP) 的通信与相邻计算。
                例如，通过 Flux 框架将 all-gather 操作与随后的 GEMM (通用矩阵乘法)
                操作融合到一个内核中，即使在节点内高速互连如 NVLink 中，
                也能有效减少通信开销。
            *   内存密集型操作融合: 第二类融合内核专注于融合序列化的内存密集型操作。
                MoE 模型包含多个小型内存密集型操作序列（例如，注意力节点选择
                top-k 专家、计算中间结果、数据移动到相应专家）。通过将这些步骤
                与 gating 计算融合，减少了核函数启动和内存访问开销。
        *   高性能 M2N 通信库:
            *   PyTorch 扩展: MegaScale-Infer 的通信库是作为一个 PyTorch 扩展实现的，
                代码量约为 4900 到 5000 行 C/C++ 和 Python 代码。
            *   技术支持: 该库支持 GPUDirect 和 GDRCopy 等技术，
                这些技术有助于高效地在 GPU 和 NIC 之间传输数据。
            *   网络监控工具: 系统还精心设计了网络监控工具，
                用于深入分析网络和流量相关的优化。
        *   负载均衡: 在实际流量中，不同专家的负载可能显著不同。
            为了实现热专家和冷专家之间的负载均衡，MegaScale-Infer
            通过考虑专家流行度在专家部署中分配 M 个专家到 N 个节点，
            目标是最小化 `maxj=1..N Cj`，其中 `Cj` 表示计算成本。
            该算法采用贪婪近似策略，根据先前的流量数据生成专家计划。
    *   实现细节:
        *   融合内核通过在单个操作中包含多个逻辑步骤来减少了 GPU 的空闲时间和
            核函数启动开销，从而提高了计算效率。
        *   M2N 通信库的实现着重于低级优化，以消除传统通信库的瓶颈。
        *   负载均衡机制则通过对专家进行分类和预测其流行度，
            从而动态调整资源分配，确保所有专家都能得到充分利用。

*   7 评估 (Evaluation):
    *   设计思想: 本章旨在通过全面的实验评估，验证 MegaScale-Infer 在大规模 MoE
        模型服务中的端到端性能、通信效率以及各项优化策略的有效性。
        通过与现有最先进系统（如 vLLM 和 TensorRT-LLM）的对比，
        量化 MegaScale-Infer 在吞吐量、延迟、成本效益和功耗方面的优势。
    *   关键技术:
        *   实验设置:
            *   测试平台: 实验在两个不同的集群上进行。
                *   第一个集群包含 8 个节点，每个节点配备 8 个 NVIDIA 80GB Ampere GPU，
                    128 个 CPU，2TB 主机内存和 8 个 200 Gbps Infiniband 网卡。
                    节点内 GPU 通过 400GB/s NVLINK 互连。
                *   第二个集群是异构设置，包含 NVIDIA H20 和 L40S 两种 GPU。
                    H20 节点具有 900GB/s NVLINK 和 4 个 400 Gbps 网卡，
                    而 L40S 节点则使用 PCIe 进行节点内通信，并配备 2 个 400 Gbps 网卡。
            *   模型与工作负载: 使用 Mixtral 8x22B、DBRX 和 Scaled-MoE 模型进行评估，
                它们的参数量分别为 141B、132B 和 317B。权重、激活和 KV cache 的数据
                类型均为 bfloat16。工作负载使用生产数据集，中位输入/输出长度分别为 571 和 159 个令牌。
            *   基线: 对比系统包括 vLLM 和 TensorRT-LLM，它们都支持常见的 LLM 服务技术，
                如 FlashAttention、PagedAttention 和连续批处理。
            *   评估指标: 主要评估指标为单位成本解码吞吐量 (per-GPU decoding throughput)，
                即每秒生成的令牌数除以 GPU 数量。对于异构部署，使用单位成本吞吐量。
                目标是在满足指定令牌间时间 (TBT) 延迟约束（设置为 150 毫秒）下，
                最大化吞吐量。
        *   端到端实验:
            *   同构部署: 在 NVIDIA Ampere GPU 上，MegaScale-Infer 相比 vLLM 和 TensorRT-LLM
                实现了显著更高的每 GPU 解码吞吐量（图 8a），对于 Scaled-MoE 更是达到了 1.90 倍的提升。
                这归因于解耦架构能够聚合来自多个注意力模块的请求，增加 FFN 模块的有效批处理大小。
                在令牌间时间方面，MegaScale-Infer 与基线系统相当（图 8b）。
                考虑到预填充阶段的端到端吞吐量，MegaScale-Infer 仍能实现 1.18 倍的提升（图 8c）。
            *   异构部署: 在 H20 和 L40S 异构集群上，MegaScale-Infer 通过将 H20 用于注意力模块
                （高内存带宽和容量）和 L40S 用于专家模块（高计算能力）来最大化优势。
                这带来了高达 3.24 倍的单位成本解码吞吐量提升（图 9a），并且在单位功耗方面也有显著提升
                （图 10）。
        *   M2N 通信性能:
            *   数据大小: M2N 通信在不同数据大小下的性能（图 11）显示，MegaScale-Infer
                在所有数据大小下均优于 NCCL，尤其在小数据量下，延迟降低了 80.8%，尾延迟降低了 96.2%。
            *   发送/接收方数量: 在不同发送方和接收方数量下（图 12），MegaScale-Infer 始终优于 NCCL，
                并展现出更好的稳定性，尤其是在数量增加时。
        *   消融研究:
            *   解耦专家并行化和 M2N 优化: 如图 13 所示，解耦专家并行化本身带来了高达 4.66 倍的吞吐量提升。
                结合 M2N 通信库，进一步将通信开销降低，并通过乒乓管线并行实现通信与计算的完全重叠，
                额外带来了 1.53 倍的吞吐量提升。
            *   乒乓管线并行化: 通过改变微批次数量 (`m`) 进行评估（图 14）。当 `m=1` 时，乒乓管线并行关闭，
                导致吞吐量较低。将 `m` 增加到 2 显著提高了吞吐量。对于较大的模型，增加 `m` 会带来更显著的收益。
            *   部署计划的影响: 如图 15 所示，最佳部署计划对于最小化空闲时间和最大化 GPU 利用率至关重要。
                例如，对于 DBRX 模型，当数据并行 (DP) 程度较低时，注意力模块是瓶颈；当 DP 程度较高时，
                专家模块成为瓶颈。
    *   实现细节: 本章通过提供大量的图表和数据，详细支持了 MegaScale-Infer 的性能优势。
        所有实验均在真实硬件上运行，确保了结果的可靠性和实用性。

*   8 部署经验 (Deployment Experience):
    *   设计思想: 本章旨在分享 MegaScale-Infer 在实际生产环境中的部署经验，
        以验证其在大规模、异构集群下的实际效果，并提供对专家和注意力负载
        分布的深入见解，从而指导未来的系统优化。
    *   关键技术:
        *   规模与成本效益: MegaScale-Infer 已在拥有近 10,000 个 GPU 的公司生产推理服务中部署。
            在异构部署下，根据工作负载特性，它将服务相同流量的成本降低了 1.5 到 2.0 倍。
            这验证了 MegaScale-Infer 在实际应用中的经济效益。
        *   专家负载均衡: 在实际流量中，不同专家的负载分布可能存在显著不平衡。
            如图 16a 所示，每个专家处理的令牌数量在不同模型层中差异很大。
            图 16b 和 16c 显示，在解码阶段专家负载相对稳定，但在预填充阶段
            则波动更大。这些观察结果表明，在解码阶段可以采用静态或周期性
            的专家负载均衡策略，而在预填充阶段则需要更频繁地调整专家计划。
        *   注意力负载均衡: 注意力侧也存在负载不平衡。由于序列长度的变化，
            即使处理相同大小的批次，注意力节点的计算时间也可能显著不同。
            为了解决这一问题，MegaScale-Infer 对不同序列长度和批处理大小下的
            关键操作运行时进行剖析，然后根据预定义的执行时间目标组合每个
            注意力节点上的批次，从而平衡节点间的工作负载。
    *   实现细节: 本章通过展示实际生产环境中的负载分布图（图 16），
        直观地揭示了专家和注意力负载的动态特性。这些经验对于理解和优化
        大规模 MoE 服务系统具有重要指导意义，例如需要根据不同的阶段
        （解码或预填充）调整负载均衡策略。

*   9 相关工作 (Related Work):
    *   设计思想: 本章旨在将 MegaScale-Infer 的工作置于 LLM 服务、
        资源解耦、MoE 优化和集合通信领域的现有研究背景中，
        通过对比分析突出 MegaScale-Infer 的独特贡献和优势。
    *   关键技术:
        *   LLM 服务: 回顾了 Orca、vLLM、Sarathi-Serve、LoongServe、Llumnix、
            ServerlessLLM 和 dLoRA 等工作，这些系统通过迭代级调度、KV 缓存管理、
            分块预填充、弹性序列并行化和请求迁移等技术优化 LLM 推理性能。
            然而，这些工作主要集中在稠密模型上，或未能充分解决 MoE 模型的稀疏性挑战。
        *   资源解耦: 讨论了将硬件资源解耦到独立资源池的系统，
            如 Splitwise、FAST-DECODE、Lamina、MoE-Lightning 等。
            这些系统通过解耦预填充和解码阶段、将计算卸载到更便宜的设备等方式来提高效率。
            MegaScale-Infer 也采用解耦方法，但更侧重于 MoE 效率，
            并解决了 MoE 稀疏性带来的挑战。
        *   MoE 优化: 概述了 MoE 优化方面的研究，包括负载均衡和高效通信。
            这包括了 DeepEP、Janus 等工作。MegaScale-Infer 通过解耦和专门的
            通信库解决了 MoE 稀疏性带来的独特低效率问题。
        *   集合通信: 回顾了分布式机器学习中高性能集合通信库，如 NVIDIA NCCL、SCCL、
            TACCL、TE-CCL、CoCoNET 和 Centauri。这些库旨在通过优化通信算法、
            重叠通信与计算来提高性能。MegaScale-Infer 识别并消除了现有解决方案中
            M2N 通信存在的开销和不稳定性。
    *   实现细节: 本章通过列举大量参考文献，展示了 MegaScale-Infer 在相关领域
        的广泛调研和对现有技术的理解。通过明确指出 MegaScale-Infer 如何弥补现有
        解决方案的不足，进一步强调了其创新性和重要性。
