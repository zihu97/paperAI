# 2602.21788v1.pdf - DHP: Efficient Scaling of MLLM Training with Dynamic Hybrid Parallelism

## 中文标题：DHP：利用动态混合并行实现多模态大语言模型训练的高效扩展
和《FLYING SERVING：大语言模型服务中的在线并行策略动态切换系统.md》这篇文章有相同的思想，使用动态并行策略
### 中文摘要
扩展长上下文能力对于多模态大语言模型（MLLM）至关重要。然而，现实中的多模态数据集具有极强的异构性。现有的
训练框架主要依赖于静态并行策略，在数据异构的情况下，这些策略往往面临严重的负载不均衡、冗余通信以及硬件利用
率低下的问题。本文提出了动态混合并行（DHP），这是一种高效的并行策略，能够根据训练过程中的需求自适应地重新
配置通信组和并行度。我们推广了非 2 的幂次并行度，并开发了一种多项式时间算法，能够在每个训练批次仅产生毫秒
级开销的情况下生成近乎最优的并行策略。即使在极端的数据变异情况下，DHP 也能保持高硬件效率。实验结果表明，
DHP 显著优于 Megatron-LM 和 DeepSpeed，在保持大规模 NPU 集群近线性扩展效率的同时，训练吞吐量提升高达 1.36 倍。

### 各章节主旨内容
- **Chapter 1: Introduction** - 介绍了 MLLM 训练中由于数据异构性（如视频时长差异）导致的负载不平衡挑战，并提出了
  动态混合并行（DHP）框架，通过动态调整并行度和通信组来优化硬件利用率。
- **Chapter 2: Related Work** - 回顾了分布式训练中的静态并行（DP, TP, PP, SP, CP）和动态并行研究，并讨论了
  MLLM 训练加速的相关算法和系统优化。
- **Chapter 3: Preliminaries** - 定义了 MLLM 的基本架构和注意力机制，并详细阐述了现有的分布式训练范式，包括
  数据并行、模型并行（张量并行、流水线并行）以及序列维度切分（CP 和 SP）。
- **Chapter 4: Method** - DHP 的核心设计，包括问题建模（最小化最大执行时间）、成本估计模型（内存和时间）以及
  解决该 NP-hard 问题的两阶段多项式时间算法（基于 BFD 的原子序列分组和基于 2D-DP 的最优资源分配）。
- **Chapter 5: Overall Workflow and Implementation** - 详细描述了 DHP 的工作流，包括微批次规划、内存感知
  序列打包、2D-DP 求解以及执行器的重分配过程，并强调了其非侵入式实现和异步调度机制。
- **Chapter 6: Experiments** - 展示了 DHP 在不同数据集（MSRVTT, InternVid, OpenVid）和模型（InternVL3,
  Qwen3VL）上的端到端性能提升、扩展性分析以及求解器开销验证。
- **Chapter 7: Conclusion** - 总结了 DHP 的贡献，强调了其通过灵活的非 2 幂次并行度和毫秒级调度算法显著消除了
  MLLM 训练中的负载不平衡和冗余通信。

### 中文结论
本文提出了 DHP 框架，旨在解决异构多模态数据集在训练过程中的负载不平衡问题。通过引入非 2 的幂次并行度和
毫秒级开销的调度算法，DHP 有效地减少了同步等待和冗余通信。在 64 个 NPU 的实验中，DHP 实现了显著的吞吐量
提升，并证明了其在处理大规模、长序列数据时的卓越扩展性和稳健性。

# 章节详细解读

## Chapter 4: Method (核心方法论)

### 1. 设计思想 (Design Philosophy)
- **痛点分析**：MLLM 训练数据集（特别是视频）呈现极端的长尾分布。传统的静态并行（如 Megatron-LM 的固定 CP 度）
  将所有序列强行放入大小一致的并行组中。对于短序列，高并行度会导致过度通信开销；对于长序列，低并行度则会导致
  显存溢出。这种“一刀切”的方法导致大量 NPU 处于闲置状态，等待组内最慢的任务完成。
- **核心动机**：作者认为最优的并行策略应该是“量体裁衣”。通过根据序列长度动态调整并行度 $d_p$，让计算密集的
  长序列使用更多资源，通信密集的短序列使用较少资源，从而在集群范围内平衡负载。
- **设计目标**：在满足单设备显存限制 $E$ 的前提下，通过动态构建通信组和分配并行度，最小化单个批次（Makespan）
  的最大执行时间，同时确保调度算法本身的开销可忽略不计。

### 2. 关键技术 (Key Technologies)
- **非 2 幂次并行度限制的突破**：传统的序列并行（SP/CP）通常限制并行度为 2 的幂次（如 1, 2, 4, 8），这大大
  限制了资源分配的粒度。DHP 采用 Ring-style CP，允许并行度为任意正整数，极大地扩展了搜索空间，能够更精准地
  匹配序列长度。
- **成本估计模型**：
    - **内存估计**：$M(C_p) = \sum |s_k| \cdot M_{token} + M_{ms}$。考虑了激活显存（与序列长度成正比）
      和模型状态显存（由 ZeRO-3 优化后的固定开销）。
    - **时间估计**：$T(C_p) = \mathcal{T}_{cp} + \mathcal{T}_{cm} - \min(\mathcal{T}_{cpa}, \mathcal{T}_{cma})$。
      该公式考虑了计算成本、通信成本以及 Ring Attention 中计算与通信的重叠部分。
- **两阶段求解算法 (Two-stage Approximation)**：
    - **阶段 1：基于 BFD 的原子序列分组**：为了减少决策变量数量，先用“最佳适应递减”（Best-Fit Decreasing）
      策略将短序列打包进长序列的“空隙”中，形成若干原子组 $G_k$。
    - **阶段 2：2D 动态规划资源分配**：定义 $DP[i][j]$ 为前 $i$ 个原子组利用 $j$ 个 rank 时的最小最大
      执行时间。状态转移方程：$DP[i][j] = \min_{d \in [d_{min,i}, j-d']} \max(DP[i-1][j-d], T(G_i, d))$。

### 3. 实现细节 (Implementation Details)
- **数据结构**：维护一个原子组列表，每个组包含最小并行度 $d_{min}$ 和对应的预估执行时间查找表。
- **算法复杂度**：2D-DP 算法的时间复杂度为 $O(K'N^2)$，其中 $K'$ 是原子组数量，$N$ 是总 Rank 数。在实际
  测试中，对于 64 个 NPU 的集群，求解时间仅需约 86ms。

### 4. 权衡与对比 (Trade-offs & Comparison)
- **对比分析**：
    - 与 FlexSP 相比：FlexSP 将序列切分视为复杂的 Mixed Integer Linear Programming (MILP) 问题，求解
      极其耗时。DHP 通过 2D-DP 将其简化为多项式时间问题。
    - 与静态 CP 相比：静态 CP 的通信开销是固定的，而 DHP 能在处理短序列时缩减组大小，减少不必要的环路通信。
- **优劣势考量**：
    - **优势**：显著提升异构数据下的硬件利用率，调度算法与计算异步执行，无额外延迟。
    - **开销**：需要预先进行 Profiling（性能画像）以获得准确的成本系数 $\alpha, \beta$。

## Chapter 5: Overall Workflow and Implementation (系统实现)

### 1. 设计思想 (Design Philosophy)
- **非侵入性 (Non-intrusiveness)**：这是 DHP 设计的核心原则。作者不希望修改 Megatron-LM 或 PyTorch
  的底层核心代码，而是将其实现为插件式的模块，确保与现有生态兼容。
- **解耦调度与训练**：调度逻辑如果在训练循环内同步执行，会成为新的瓶颈。因此，DHP 采用生产者-消费者模式，
  在 NPU 计算当前批次的同时，CPU 异步计算下一个批次的调度计划。

### 2. 关键技术 (Key Technologies)
- **动态通信组管理与池化**：在分布式训练中，频繁创建新的进程组（Process Group）会导致巨大的开销甚至崩溃。
  DHP 实现了一个“通信组池”，缓存已经创建过的并行度组合。当需要一个新的 CP 组时，先检查池中是否有可重用的。
- **异步调度流水线**：
    - **Profiler**：在训练开始前，对不同长度和并行度的序列进行采样，拟合出线性/平方成本曲线。
    - **Scheduler**：负责执行 BFD 分组和 2D-DP 求解。
    - **Executor**：负责在每个批次开始前，根据 Scheduler 的结果重新分发数据和更新 MPU 状态。

### 3. 实现细节 (Implementation Details)
- **技术栈**：基于 PyTorch Distributed 和 Megatron-LM。通信后端使用华为的 HCCL（针对昇腾 NPU 优化）。
- **执行流程**：
    1. CPU 读取下一批次的所有序列长度。
    2. 运行 BFD 将序列预打包。
    3. 运行 2D-DP 分配并行度。
    4. 在下一个步进开始前，Executor 瞬间切换通信拓扑（通过重用缓存的组）。

### 4. 权衡与对比 (Trade-offs & Comparison)
- **对比分析**：ByteScale 等方案依赖启发式调度，难以保证全局最优；FlexSP 虽然尝试最优但由于 MILP 太慢无法
  在线使用。DHP 找到了最优性与实时性之间的平衡点。
- **优势**：支持任意整数并行度（非 2 幂次），使得资源分配极其精细。例如，可以用 3 个 NPU 处理一个中等长度
  序列，而不是非要在 2 或 4 之间做次优选择。

## Chapter 6: Experiments (实验验证)

### 1. 设计思想 (Design Philosophy)
- **全面性**：在多种模型规模（2B, 4B, 8B）和多种真实异构数据集（视频和图像混合）上验证。
- **基准对比**：直接对比业界最强的静态并行框架 Megatron-LM (PTD-P) 和 DeepSpeed (Ulysses)。

### 2. 关键技术 (Key Technologies)
- **端到端加速比**：在 InternVL3-8B 模型上，相比静态并行实现了 1.35x 到 1.36x 的加速。
- **扩展性验证**：从 8 NPU 扩展到 64 NPU 时，DHP 的吞吐量曲线保持了近乎线性的增长，甚至在某些情况下由于
  更优的通信组划分，相对效率有所提升。

### 3. 实现细节 (Implementation Details)
- **硬件配置**：华为 Ascend 910B NPU，单机 8 卡，100 Gbps InfiniBand 互联。
- **数据集特性**：MSRVTT（较均匀）、InternVid（中等异构）、OpenVid（极端异构，包含大量长短不一的视频）。

### 4. 权衡与对比 (Trade-offs & Comparison)
- **实验结论**：在数据集越异构（如 OpenVid）的情况下，DHP 的优势越明显。
- **误差分析**：成本估计器的平均误差保持在 8% 以内，足以支持 2D-DP 做出正确的并行决策。
- **求解器开销**：在 64 Rank 规模下，Solver 耗时仅 86ms，远低于单步训练时间（数秒级），证明了其在线应用的
  可行性。
