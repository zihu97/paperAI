## 第一部分：论文整体骨架

### 1. 论文基本信息
*   **英文标题**：JANUS: Disaggregating Attention and Experts for Scalable MoE Inference
*   **中文标题**：JANUS：面向可扩展MoE推理的注意力与专家解耦架构
*   **作者机构**：香港中文大学（深圳）、中国电信人工智能研究院（TeleAI）、哈尔滨工业大学（深圳）等。
*   **发布时间**：2025年12月16日 (arXiv v2)

### 2. 中文摘要
大型混合专家（MoE）模型的推理面临着资源需求高和负载动态变化的双重挑战。现有的解决方案通常将整个模型作为一个
单一的整体单元进行部署，对注意力（Attention）模块和专家（Expert）模块应用统一的资源配置，尽管它们的需求截然不同，
这导致了可扩展性受限和资源效率低下的问题。本文提出了 **JANUS**，这是一个可扩展的 MoE 推理系统，它将注意力层和专家层
解耦到独立的 GPU 子集群上，使得每个模块能够被独立管理和扩展。

JANUS 包含三个针对高效解耦 MoE 推理的关键设计：
1.  **自适应两阶段通信方案**：利用节点内和节点间的带宽层级结构，实现低延迟的数据交换。
2.  **激活负载均衡调度器**：针对 MoE 模块的内存受限特性，引入轻量级调度器并以 GPU Kernel 形式实现，以极小的开销平衡
    跨 GPU 的专家激活数量，从而降低推理延迟。
3.  **细粒度资源管理**：动态调整专家放置，并独立扩展注意力和 MoE 资源，以提高整体效率。

评估表明，JANUS 在满足每 Token 延迟要求的同时，实现了比最先进系统（如 SGLang）高出 **3.9倍** 的单 GPU 吞吐量。

### 3. 各章节主旨内容

*   **1. Introduction (引言)**
    *   阐述了 MoE 模型在 LLM 中的主导地位及其推理面临的挑战（动态负载、巨大显存占用、层间异构需求）。
    *   指出现有“单体（Monolithic）”部署方式的局限性：资源配置粗粒度，导致资源过度配置和利用率低。
    *   介绍了 JANUS 系统及其三个核心贡献：解耦架构、两阶段通信、负载均衡调度。

*   **2. Background and Motivation (背景与动机)**
    *   **MoE 推理特征**：分析了 MoE 层通常是内存受限（Memory-bound）的，其延迟与激活的专家数量呈线性关系，而对
        Batch Size 不敏感；相比之下，注意力层在 Decoding 阶段主要是带宽受限，但行为模式不同。
    *   **现有方案局限**：单体架构限制了扩展性（需重启/重新加载模型）且资源效率低；现有的解耦方案缺乏细粒度的专家管理。
    *   **关键洞察**：应将 Attention 和 MoE 解耦；平衡 GPU 间的激活专家数量是性能关键。

*   **3. System Design (系统设计)**
    *   **3.1 概览**：系统分为 Attention 节点和 MoE 节点，分别独立管理。
    *   **3.2 自适应两阶段通信**：解决 m-to-n 通信开销问题。通过节点内聚合（NVLink）减少跨节点（InfiniBand）的小包传输次数。
    *   **3.3 激活负载均衡调度**：提出 AEBS 算法，在 GPU 上以微秒级开销运行，动态分配请求到负载最小的专家副本。
    *   **3.4 资源与专家管理**：基于“偏度（Skewness）”和“共激活（Co-activation）”模式，动态调整专家副本数量和放置位置，
        并独立缩放子集群规模。

*   **4. Implementation (实现)**
    *   基于 SGLang 框架开发，增加约 3.5k 行 Python 和 300 行 CUDA 代码。
    *   通信层基于 UCX 构建，利用 GPUDirect RDMA 实现零拷贝传输。

*   **5. Evaluation (评估)**
    *   **实验设置**：使用 NVIDIA H100 集群，测试 DeepSeek-V2 等模型。
    *   **主要结果**：相比 SGLang 和普通解耦方案，JANUS 显著提升了吞吐量（最高 3.9x）并降低了延迟。
    *   **消融实验**：验证了两阶段通信和负载均衡调度各自的性能贡献。
    *   **真实负载模拟**：在动态负载下比 SGLang 节省 25% 的 GPU 资源。

*   **6. Discussion and Conclusion (讨论与结论)**
    *   讨论了对异构硬件的支持、与其他并行策略（如 TP）的结合潜力。
    *   总结全文，强调 JANUS 通过解耦和精细化管理解决了 MoE 推理的资源效率与延迟权衡问题。

### 4. 中文结论
本文展示了 JANUS，一个将注意力和 MoE 层解耦到不同 GPU 子集群的可扩展 MoE 推理系统。JANUS 旨在最大化资源效率和
单 GPU 吞吐量，同时满足 Token 级的 SLO（服务等级目标）。为此，它优化了子集群间的通信，最小化了每 GPU 的激活专家数量
以降低延迟，并支持细粒度、模块特定的扩展以适应负载动态。评估结果表明，JANUS 在满足 SLO 的前提下，相比最先进的系统
将单 GPU 吞吐量提升了高达 3.9 倍。

## 第二部分：指定章节详解

### 1. 核心洞察：MoE 与 Attention 的异构性（Section 2.2 & 2.4）

JANUS 的设计始于对 MoE 模型中两种核心组件——注意力层（Attention）和专家层（MoE）——性能特征的深入剖析。

*   **MoE 层的特性：内存受限（Memory-bound）**
    *   **理论分析**：根据 Roofline 模型，MoE 层的算术强度（Arithmetic Intensity）非常低。在 H100 GPU 上，要达到计算受限
        状态，Batch Size 需要达到 18k tokens，这在在线推理中是不现实的（通常 Batch Size < 100）。
    *   **实验观察**：MoE 层的执行延迟与 Batch Size 关系不大，而是**线性地取决于被激活的专家数量**。这意味着，只要能
        减少单个 GPU 上需要处理的激活专家数量，就能直接降低延迟。

*   **Attention 层的特性**
    *   在 Decoding 阶段，Attention 层也是内存带宽受限的，但其延迟在负载较低时相对稳定，直到 Batch Size 很大时才会激增。
    *   **结论**：两者对资源的需求和扩展行为截然不同。将它们捆绑在同一个 GPU 上（单体架构）会导致资源错配——为了满足
        MoE 的显存需求而增加 GPU，会导致 Attention 计算能力的浪费。

### 2. 关键技术一：自适应两阶段通信（Section 3.2）

解耦架构带来的最大挑战是通信开销。Attention 节点需要将中间激活值发送给 MoE 节点，这是一个多对多（m-to-n）的通信模式。

*   **问题**：如果每个 Attention 实例直接发送数据给每个 MoE 实例（全互联），会产生大量的小数据包。由于网络传输的头部
    开销，这种方式效率极低。
*   **JANUS 的解决方案**：
    *   **第一阶段（节点内聚合）**：同一台物理机（Node）内的多个 Attention 实例，利用高带宽的 NVLink 进行 `All-Gather`，
        将去往同一目标 MoE 节点的数据聚合在一起。
    *   **第二阶段（节点间传输）**：聚合后的数据包通过 RDMA 网络批量发送给目标 MoE 节点。这大大减少了网络传输的次数（Message Count）。
    *   **反向路径**：从 MoE 返回结果给 Attention 时，同样先在 MoE 节点内通过 `All-Reduce` 聚合结果，再发回。
*   **MoE 侧门控（Gating on MoE Side）**：
    *   JANUS 选择将 Gating 网络放在 MoE 侧（即全量发送 Token 激活值，在 MoE 侧决定路由）。
    *   虽然这增加了数据量，但避免了在 Attention 侧处理复杂的路由元数据和内存重排（Memory Re-layout），且简化了通信模式，
        最终降低了端到端延迟。

### 3. 关键技术二：激活负载均衡调度（Section 3.3）

为了降低 MoE 层的延迟，必须确保没有任何一个 GPU 因为处理过多的专家而成为瓶颈。

*   **挑战**：MoE 的执行通常只有几百微秒，调度算法必须极快（微秒级），且不能依赖 CPU 与 GPU 之间的频繁同步。
*   **AEBS 算法（Activated-Expert-Balanced Scheduling）**：
    *   这是一个贪心算法。对于每个 Token 激活的专家，如果该专家有多个副本（Replica），算法会将其分配给当前负载
        （已分配激活专家数）最小的那个 GPU 实例。
    *   对于只有单个副本的专家，则只能分配给特定实例。
*   **GPU Kernel 实现**：
    *   JANUS 将此调度逻辑直接实现为一个 GPU Kernel。
    *   **无同步设计**：所有 MoE 实例运行相同的确定性算法，输入相同的元数据（副本布局），因此它们能独立计算出一致的
        调度方案，无需跨 GPU 协商。这消除了昂贵的分布式协调开销。

### 4. 关键技术三：资源与专家管理（Section 3.4）

JANUS 不仅在微秒级进行调度，还在分钟/小时级进行资源重配置。

*   **专家复制与放置（Replication & Placement）**：
    *   **偏度感知（Skewed Popularity）**：利用历史统计数据，识别“热”专家（Hot Experts），并为它们创建更多副本，
        防止单点过载。
    *   **共激活感知（Co-activation）**：有些专家经常被同一个 Token 同时激活。JANUS 尽量将这些“共生”专家放置在
        **不同**的 GPU 上。
    *   **原因**：如果经常同时激活的专家在同一个 GPU 上，该 GPU 的瞬时负载会激增。分散它们可以平摊负载，降低延迟。
    *   **算法**：通过求解一个组合优化问题（启发式算法），最小化每台 GPU 上的最大共激活负载。

*   **独立资源缩放（Resource Scaling）**：
    *   Attention 和 MoE 子集群运行独立的控制器。
    *   系统监控请求队列、延迟和利用率。当检测到 SLO 违规风险时，基于性能模型预测需要增加多少 Attention 节点或 MoE 节点。
    *   这种解耦使得系统可以只增加 MoE 节点（应对计算/显存压力）而不增加 Attention 节点，或者反之，从而节省大量 GPU 资源。

### 5. 实验结果分析（Section 5）

*   **对比基线**：SGLang（单体架构 SOTA）和 DisAgg（普通的解耦实现，无 JANUS 的优化）。
*   **性能数据**：
    *   **吞吐量**：在不同 Batch Size 下，JANUS 的每 GPU 吞吐量比 SGLang 高出 1.5倍 到 3.9倍。
    *   **延迟**：在高负载下，DisAgg 因为通信瓶颈无法满足 200ms 的 TPOT SLO，而 JANUS 始终能满足。
*   **资源效率**：在模拟的真实流量（两天的 traces）中，JANUS 能够跟随流量波峰波谷进行细粒度伸缩，比 SGLang 节省了 25% 的 GPU 时间。
*   **微基准测试**：
    *   两阶段通信（2-Phase Comm）在大 Batch Size（512）下将延迟降低了 18%。
    *   负载均衡调度（LB Scheduling）在中小 Batch Size 下进一步降低了 7% 的延迟，并显著减少了 GPU 间的负载不平衡（从差值 8 降至 4）。
