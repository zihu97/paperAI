# 错峰批处理调度：面向高效大语言模型推理的协同优化首字延迟与吞吐量
# (Staggered Batch Scheduling: Co-optimizing Time-to-First-Token and Throughput for High-Efficiency LLM Inference)

## 第一部分：论文整体骨架

### 1. 论文基本信息
*   **英文标题**：Staggered Batch Scheduling: Co-optimizing Time-to-First-Token and Throughput for High-Efficiency LLM Inference
*   **中文标题**：错峰批处理调度：面向高效大语言模型推理的协同优化首字延迟与吞吐量
*   **作者机构**：Jian Tian 等 (百度公司)
*   **发表时间**：2025年12月18日 (arXiv:2512.16134v1)

### 2. 中文摘要
随着大语言模型（LLM）向复杂的分布式架构演进——特别是 P/D 分离（Prefill/Decode 分离）的大规模 DP+EP（数据并行+专家并行）范式——引入了独特的调度挑战。与调度器可以将实例视为黑盒的传统部署不同，DP+EP 架构表现出极高的内部同步成本。我们发现，在这些系统中，直接请求分发（Immediate Dispatch）会导致严重的**引擎内排队（In-engine Queuing）**和**并行化气泡（Parallelization Bubbles）**，从而降低首字延迟（TTFT）。

为了解决这个问题，我们提出了**错峰批处理调度（Staggered Batch Scheduling, SBS）**，这是一种通过故意缓冲请求以形成最佳执行批次的机制。这种时间上的解耦在不牺牲吞吐量的情况下消除了内部排队气泡。此外，利用缓冲创建的调度窗口，我们引入了一种**负载感知全局分配（Load-Aware Global Allocation）**策略，平衡了 Prefill 和 Decode 阶段各 DP 单元的计算负载。在服务于 DeepSeek-V3 的生产环境 H800 集群上的部署表明，与最先进的直接调度基线相比，该系统将 **TTFT 降低了 30-40%**，并将**吞吐量提高了 15-20%**。

### 3. 各章节主旨内容

*   **1. 引言 (Introduction)**
    *   阐述 LLM 架构从单体向 P/D 分离及 DP+EP 混合并行架构的演进。
    *   指出传统“持续服务”假设在 DP+EP 架构下的失效，因为 Prefill 阶段实际上是“非抢占式离散批处理过程”。
    *   直接分发导致“队头阻塞（HOL Blocking）”，即请求在引擎内部排队等待，而此时其他实例资源可能闲置。
    *   提出 SBS 核心理念：通过“等待来加速（Waiting to accelerate）”，利用调度侧排队替代设备侧排队。

*   **2. 相关工作 (Related Work)**
    *   回顾 P/D 分离架构（如 Splitwise, DistServe）及其优化。
    *   讨论混合并行下的调度策略（如 D²LPM, Llumnix）及其在处理动态负载时的局限性。
    *   提及前缀缓存感知调度（Prefix Cache-Aware Scheduling）。

*   **3. 分析 (Analysis)**
    *   **调度粒度变化**：从单体实例转变为 DP-Attention 单元，调度需深入到子实例级别。
    *   **排队动力学**：分析为何 Prefill 阶段在直接分发下表现为 Gated M/D/1 系统（导致高延迟），而 SBS 将其虚拟化为 M/D/S 系统（显著降低等待时间）。
    *   **同步开销**：解释 DP+EP 中的 All-to-All 通信如何导致“落后者效应（Straggler Effect）”，以及批处理窗口如何提供全局优化机会来消除这些气泡。

*   **4. 设计与实现 (Design and Implementation)**
    *   **SBS 架构**：包含控制面、状态面和资源面的闭环反馈系统。
    *   **自适应调度间隔**：提出算法动态计算最佳调度间隔 $I_{opt}$，并设计多层状态同步协议确保系统鲁棒性。
    *   **批处理 Prefill 分发**：建立细粒度 DP 容量模型，提出“优先批处理分配算法（PBAA）”，采用“注水法”填补负载空隙。
    *   **双目标 Decode 调度**：针对 Decode 阶段的显存和计算负载耦合问题，提出 IQR 感知的字典序调度算法，平衡 KV Cache 和 Batch Size。

*   **5. 实验 (Experiments)**
    *   在 H800 集群上基于 DeepSeek-V3 模型进行评估。
    *   **TTFT 优化**：在 80% 负载以下，TTFT 降低约 40%。
    *   **负载均衡**：Prefill Chunk 利用率从 52% 提升至 88%，QPS 提升 12.9%-22.8%；Decode 吞吐量提升 15%。

*   **6. 结论 (Conclusion)**
    *   总结 SBS 将调度范式从连续分发转变为离散的基于窗口的分配。
    *   强调离散调度同步和全局视图负载整形是下一代万亿参数模型推理系统的可扩展路径。

### 4. 中文结论
本文解决了 P/D 分离、大规模 DP+EP 推理架构中固有的调度低效问题。我们证明了直接分发策略与高同步并行执行的耦合是导致延迟退化的根本原因。我们提出的解决方案——错峰批处理调度（SBS），从根本上将调度范式从连续分发转变为离散的、基于窗口的分配。通过主动缓冲请求，SBS 消除了内部排队延迟和并行化气泡，将 TTFT 降低了高达 40%。此外，利用批处理窗口，我们在两个阶段都部署了全局分配机制。通过解决 KV-cache 方差和 Batch size 不平衡的耦合挑战，它通过装箱算法（Bin-packing）将 Prefill 吞吐量提高了 12.9%–22.8%，并将 Decode 吞吐量提高了约 15%。随着 LLM 继续向万亿参数专家混合模型扩展，本文提出的离散调度同步和全局视图负载整形原则为下一代推理系统提供了一条可扩展的路径。

## 第二部分：指定章节详解

### 第一章：引言 (Introduction)
本章阐述了 LLM 推理架构演进带来的新挑战。
*   **架构背景**：模型参数从千亿级迈向万亿级，推动了架构从单体向 **P/D 分离（Prefill/Decode Disaggregation）** 和 **DP+EP（数据并行+专家并行）** 混合模式的转变。以 DeepSeek-V3 为例，这种架构利用大规模并行组（如 Decode 阶段 EP=320）来处理高并发。
*   **核心问题**：现有的调度器基于“持续服务”假设，即请求到达即分发。这在单体实例中有效，但在 P/D 分离的 DP+EP 集群中失效。
*   **关键发现**：Prefill 阶段实际上是一个 **“非抢占式、离散的批处理过程” (non-preemptive, discrete batch process)**。
    *   如果调度器忽略引擎的“忙碌状态”而直接分发请求，请求会积压在引擎内部的队列中（Device-side Queuing）。
    *   这导致了 **队头阻塞 (Head-of-Line, HOL blocking)**：请求在某个忙碌的引擎队列中死等，而其他引擎可能有空闲资源。
*   **解决方案**：提出 **错峰批处理调度 (SBS)**。通过反直觉的策略——**“等待以加速 (waiting to accelerate)”**——在调度侧缓冲请求形成最佳批次，消除了设备侧排队，从而在不牺牲吞吐量的情况下大幅降低 TTFT。

### 第二章：相关工作 (Related Work)
本章综述了当前 LLM 推理优化的三个主要方向及其局限性。
*   **P/D 分离架构**：
    *   Splitwise 开创了 Prefill/Decode/Hybrid 三层池设计，通过网络传输 KV cache 提升吞吐量。
    *   后续工作引入了 Chunked Prefill（解决长短请求混合问题）、Instance Flipping（动态角色切换）等。
*   **混合并行下的调度**：
    *   随着模型规模增长，DP+EP 成为主流（如 DeepSeekMoE）。这引入了 All-to-All 通信挑战和单点故障影响。
    *   现有工作如 **D²LPM**（基于分布式前缀匹配）和 **Llumnix**（跨实例重调度和 KV 迁移）虽然改善了负载均衡，但往往缺乏对运行时工作负载动态的响应能力，或者迁移开销较大。
*   **前缀缓存感知调度**：
    *   针对多轮对话和 RAG 场景，Preble 和 SGL-Router 等工作优化了前缀缓存共享。
    *   这些工作标志着从简单负载均衡向智能路由的转变，但未解决 DP+EP 特有的离散批处理同步问题。

### 第三章：深度分析 (Analysis)
本章为 SBS 提供了理论支撑，对比了新旧调度模式的数学本质。

#### 3.1 调度粒度：从单体到 DP-Attention 单元
*   传统调度将整个实例视为原子单元（1:1 映射）。
*   在 DP+EP（如 DeepSeek-V3，TP=4, DP=8）中，最细粒度的调度单元变成了 **单个 DP-Attention 处理单元**。调度器必须深入到子实例级别（Sub-instance level）进行资源管理。

#### 3.2 排队动力学：HOL 阻塞的根源
*   **直接分发 (Immediate Dispatch)**：
    *   模型：$N$ 个隔离的 **Gated M/D/1** 队列。
    *   后果：请求被过早绑定到特定实例。由于 Prefill 的非抢占性，请求必须等待当前批次完成（耗时 $T$）。平均等待时间 $T/2$。
    *   本质：不可观察、不可管理的设备侧排队。
*   **错峰批处理 (SBS)**：
    *   模型：虚拟化的统一 **M/D/S** 系统 ($S=N$)。
    *   机制：在调度侧缓冲 $T/N$ 时间。
    *   后果：请求只在资源就绪时进入引擎。平均等待时间降至 $T/(2N)$。
    *   对于 $N>10$ 的集群，这是数量级的延迟优化。

#### 3.3 同步开销与批处理窗口
*   DP+EP 架构的同步屏障导致 **落后者效应 (Straggler Effect)**。整个系统的速度受限于最慢的 DP 单元。
*   直接分发导致贪婪的局部最优，产生大量 **并行化气泡 (Parallelization Bubbles)**。
*   SBS 的 **批处理窗口 (Batching Window)** 将调度问题转化为 **小批量全局优化问题 (Mini-batch global optimization)**。这使得使用装箱算法（Bin-packing）等高级策略成为可能，从而填补气泡，实现计算与通信的完美重叠。

### 第四章：设计与实现 (Design and Implementation)
本章详述了 SBS 的三大核心机制。

#### 4.1 自适应调度间隔 (Adaptive Scheduling Interval)
*   **动态控制回路**：
    *   目标：使调度间隔 $I_{opt}$ 与集群聚合处理能力同步。
    *   公式：$I_{opt} = (\bar{T}_{fwd} + L_{net}) / N_{active}$。
    *   实现：使用滑动窗口平滑前向传播时间 $\bar{T}_{fwd}$，快速适应流量波动。
*   **状态同步协议**：
    *   为了防止死锁和漂移，设计了三层保障：
        1.  **Quiescence Polling**（初始化/空闲时）：队列为空立即触发。
        2.  **Async Completion Signaling**（常态）：实例完成时主动推送信号（EndForward）。
        3.  **Liveness Watchdog**（兜底）：超时强制重置，处理网络分区或节点故障。

#### 4.2 批处理 Prefill 分发 (Batched Prefill Dispatching)
*   **容量建模**：精确计算每个 DP 单元的剩余 Token 容量，考虑在途和排队请求。
*   **算法 2：优先批处理分配 (PBAA)**：
    *   **Phase 1：防饥饿**。严格 FCFS 处理旧请求。
    *   **Phase 2：注水法 (Water-Filling)**。基于 **贪婪装箱 (Greedy Bin Packing)**。将最长的 Sequence 分配给剩余容量最大的 DP 单元。
    *   **效果**：主动平衡负载，最小化层内同步等待时间。
    *   **缓存优化**：在 Cache 场景下，目标函数调整为最大化 Cache Hit，即最小化“有效计算量”。

#### 4.3 双目标 Decode 调度 (Dual-Objective Decode Scheduling)
*   **耦合挑战**：Decode 阶段面临 **KV Cache 显存压力**（由长序列引起）和 **计算负载/Batch Size**（由请求数引起）的双重不平衡。
*   **算法 3：IQR 感知字典序调度**：
    *   **Masking (掩码)**：利用四分位距 (IQR) 识别并屏蔽 KV Cache 负载过高的“危险”节点（Stragglers）。
    *   **Lexicographical Selection (字典序选择)**：
        *   在安全节点集合中，优先优化 **Batch Size ($B_i$)** 以最大化并行度。
        *   次级优化 **KV Cache 长度 ($K_i$)** 以均衡内存。
    *   **Fill-the-valley**：优先填补负载最低的节点。

### 第五章：实验 (Experiments)
本章在真实生产环境中验证了 SBS 的效果。
*   **实验设置**：NVIDIA H800 集群，DeepSeek-V3 模型 (3P1D 配置：3 Prefill, 1 Decode)。TP=4, DP=8, EP=32。
*   **5.1 TTFT 优化**：
    *   在 80% 负载以下，SBS 将 TTFT 降低了 **40%**。
    *   在长上下文（3K-64K tokens）场景下，SBS 依然保持显著优势，有效抑制了长尾延迟。
*   **5.2 负载均衡效果**：
    *   **Prefill**：通过“注水”填补气泡，Chunk 利用率从 52% 飙升至 **88%**。系统最大 QPS 提升了 **12.9%-22.8%**。
    *   **Decode**：IQR 感知调度将 KV Cache 的负载方差（$\pm 1\sigma$）范围压缩了 40%，消除了极端离群点。由于减少了同步等待，Decode 吞吐量提升了 **15%**。

### 第六章：结论 (Conclusion)
*   总结了 SBS 的贡献：识别了 DP+EP 架构中直接分发与同步并行执行的冲突。
*   **核心范式转变**：从“连续分发”转向“离散、基于窗口的分配”。
*   **价值**：
    *   消除内部排队 -> 降低 TTFT。
    *   全局视图负载整形 -> 提升 Prefill 和 Decode 吞吐量。
*   **展望**：随着模型迈向万亿参数 MoE，这种“离散调度同步”和“全局视图”的设计将成为下一代推理系统的基石。
