### **引言**

随着大型语言模型（LLM）在自然语言处理、代码生成、对话系统等领域的广泛应用，如何高效地为大量用户提供 LLM 服务成为了一个关键问题。
当前的 LLM 服务系统面临着几大挑战：首先，模型规模巨大，导致极高的内存和计算资源消耗；其次，自回归式的生成过程使得并行计算变得困难，
难以在满足低延迟（latency）要求的同时实现高吞吐量（throughput）；最后，对于专家混合（MoE）等更复杂的模型结构，资源管理和负载均衡问题变得更加突出。
为了应对这些挑战，研究者们提出了多种优化技术，但它们往往只关注于问题的某个方面。本文提出了 OmniInfer，一个系统级的统一加速框架，
它从调度、计算和请求处理等多个层面进行协同优化，旨在全面提升 LLM 服务的效率。

### **OmniPlacement - 负载感知的 MoE 调度**

专家混合（MoE）模型通过将模型参数分散到多个“专家”网络中，能够在不显著增加计算量的情况下，极大地扩展模型规模。然而，MoE 模型的服务效率在很大程度上取决于“门控网络”（gating network）如何将输入 token 分配给不同的专家。
如果分配不均，某些专家可能会成为性能瓶颈，而另一些专家则处于空闲状态。

OmniPlacement 正是为解决这一问题而设计的。它的核心思想是实现负载感知和动态调度。具体来说，OmniPlacement 会持续监控每个专家的计算负载和等待队列的长度。
当一个新的计算请求到达时，调度器会根据实时的负载信息，选择一个或多个当前负载较轻的专家来处理该请求。此外，OmniPlacement 还可能结合专家的“专长”信息，
即某些专家可能对特定类型的输入更“擅长”。通过这种智能调度，OmniPlacement 能够最大化专家的并行度，减少请求的等待时间，从而提高整个 MoE 模型的推理吞吐量。

### **OmniAttn - 稀疏注意力加速**

在 Transformer 架构中，注意力机制是模型性能的关键，但其计算复杂度与输入序列长度的平方成正比，是 LLM 推理的主要性能瓶颈之一。
OmniAttn 的设计基于一个重要的观察：注意力矩阵通常具有稀疏性，这意味着一个 token 的输出只受到少数几个其他 token 的显著影响。

OmniAttn 利用了这种稀疏性来加速计算。它可能采用多种技术来实现这一点，例如：
1.  **动态剪枝**：在计算注意力分数后，动态地将低于某一阈值的分数置为零，从而在后续的加权求和计算中跳过这些“不重要”的 token。
2.  **固定模式稀疏**：采用一些预定义的稀疏模式，如局部注意力（local attention）或步长注意力（strided attention），来强制限制每个 token 的注意力范围。
3.  **可学习的稀疏模式**：通过训练一个小的网络来预测每个 token 应该关注哪些位置，从而生成一个稀疏的注意力掩码。

通过这些方法，OmniAttn 能够将密集的矩阵乘法运算转化为稀疏的计算，这不仅减少了所需的浮点运算次数（FLOPs），也降低了内存带宽的压力，
因为不再需要加载和存储完整的注意力矩阵。这对于处理长序列输入（long-context）的场景尤其有效。

### **OmniProxy - 解耦感知的请求调度**

为了进一步提升资源利用率和可扩展性，现代 LLM 服务系统开始采用解耦架构，例如将 KV Cache（键值缓存）从 GPU 内存中分离出来，
存放到专用的内存服务器上。这种架构虽然灵活，但也引入了新的挑战：如何高效地调度和路由请求，以最小化数据在不同物理节点之间传输的开銷。

OmniProxy 就是一个专为这种解耦架构设计的智能请求调度器。它扮演着前端代理的角色，接收所有用户请求，并根据后端资源的状态做出最优的调度决策。
OmniProxy 的“解耦感知”体现在以下几个方面：
1.  **数据局部性感知**：OmniProxy 会跟踪每个请求所需的 KV Cache 数据存储在哪个物理位置。在调度时，它会优先将请求发送到距离其所需数据最近的计算节点上，以减少网络传输延迟。
2.  **计算与I/O协同**：OmniProxy 能够预测一个请求的计算量和数据传输量，并据此进行协同调度。例如，当网络带宽成为瓶颈时，
它可能会推迟那些需要大量数据传输的请求，优先处理计算密集但数据量小的请求。
3.  **批量化与优先级**：与传统的请求调度器一样，OmniProxy 也支持将多个请求组合成一个批次（batch）进行处理，以提高 GPU 利用率。
同时，它也支持基于优先级的抢占式调度，确保高优先级的请求能够被及时响应。

通过这种精细化的调度，OmniProxy 能够有效地掩盖数据传输带来的延迟，使得解耦架构的优势得以充分发挥。

### **评估**

为了验证 OmniInfer 的有效性，论文作者进行了一系列详尽的实验。他们将 OmniInfer 与当前最先进的 LLM 服务系统（如 vLLM、TensorRT-LLM 等）在多个方面进行了比较，
包括吞吐量、延迟、以及在不同负载下的性能表现。实验采用了多种不同规模的 LLM 模型（如 Llama、Mixtral 等）和多种典型的真实世界工作负载。

实验结果清晰地表明，OmniInfer 在几乎所有的测试场景中都展现出优越的性能。具体来说，与基线系统相比，OmniInfer 能够将平均吞吐量提升高达 2-3 倍，
同时将 P99 延迟（99% 的请求延迟）降低 50% 以上。这些显著的性能提升，证明了 OmniInfer 中各项优化技术（OmniPlacement、OmniAttn 和 OmniProxy）的有效性，
以及将它们整合到一个统一框架中的协同效应。

### **结论**

大型语言模型的高效服务是推动其广泛应用的关键。本文提出的 OmniInfer 框架，通过在系统层面进行多维度、细粒度的优化，
为解决 LLM 服务中的核心挑战提供了一个创新的解决方案。OmniInfer 整合了负载感知的 MoE 调度（OmniPlacement）、稀疏注意力加速（OmniAttn）
和解耦感知的请求调度（OmniProxy），这三者协同工作，显著提升了 LLM 服务系统的吞吐量和延迟表现。实验证明了 OmniInfer 相比于现有系统的优越性。
未来，OmniInfer 的设计思想可以被进一步扩展，以支持更多样化的模型架构和更复杂的部署环境，为构建下一代高性能 AI 服务平台奠定基础。
