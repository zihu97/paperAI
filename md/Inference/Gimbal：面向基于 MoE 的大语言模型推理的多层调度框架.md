# 2602.21626v1.pdf - Multi-Layer Scheduling for MoE-Based LLM Reasoning

## 中文标题：Gimbal：面向基于 MoE 的大语言模型推理的多层调度框架

### 中文摘要
大语言模型（LLM）在各类任务中表现卓越，但其大规模高效推理由于巨大的计算和延迟需求仍具挑战。现有的推理框架
（如 vLLM、SGLang）多依赖简单的调度策略（如引擎层的 FCFS 和调度层/协调层的轮询 RR），往往无法充分利用系统资源，
并可能面临队头阻塞（HoL）和负载不均衡问题。混合专家模型（MoE）的兴起因专家并行（EP）和路由复杂性带来了新的调度挑战。
本研究提出了 Gimbal，这是一个专为基于 MoE 的 LLM 推理量身定制的多层调度框架。它针对三个层面进行调度：请求级、
引擎级和专家级。在请求级，探索了最短作业优先（SJF）和优先级感知的老化算法以提升吞吐量并降低延迟。在引擎级，
设计了感知负载的分发策略，考虑前缀 Token 负载、KV 缓存利用率和用户粘性。在专家级，重点在于缓解专家热点，
并战略性地放置跨层专家依赖以平衡负载并提高路由效率。实验结果表明，Gimbal 在多种负载分布下均优于 vLLM，
TTFT 降低高达 17.8%，TPOT 降低高达 13.3%。

### 各章节主旨内容
- **Chapter I: Introduction** - 介绍大模型推理面临的效率挑战，指出 MoE 架构虽然降低了单 Token 计算量，但也引入了
  复杂的并行和路由调度难题。引出 Gimbal 框架及其三个层面的优化目标。
- **Chapter II: Background and Motivation** - 概述 LLM 和 MoE 的基础架构、现有的推理服务架构（引擎层、请求层、
  专家层）及其局限性，特别是现有系统在处理专家并行和负载均衡方面的不足。
- **Chapter III: System Design** - 详述 Gimbal 的核心架构，包括数据并行（DP）引擎负载均衡模块、SJF 调度器和
  专家动态替换模块的设计逻辑。
- **Chapter IV: Implementation** - 描述 Gimbal 在 vLLM（0.9.1版本）基础上的实现细节，包括扩展 ZeroMQ 通信和
  集成专家并行负载均衡（EPLB）等。
- **Chapter V: Performance Evaluation** - 在 A100 GPU 环境下，使用 BurstGPT 和 ShareGPT 数据集对 Gimbal 的
  TTFT、TPOT 和前缀缓存命中率进行全面评估。
- **Chapter VI: Related Work** - 综述现有 LLM 推理调度、多级反馈队列以及专家级调度的相关研究。
- **Chapter VII: Conclusions and Future Work** - 总结 Gimbal 的贡献，并指出未来在更大规模集群和在线专家
  亲和性估计方面的研究方向。

### 中文结论
Gimbal 证明了通过多层协同调度可以显著提升 MoE 模型的推理性能。通过在请求级引入前缀长度感知的 SJF、在引擎级
引入 KV 缓存感知的负载均衡以及在专家级引入跨层亲和性感知的位置策略，Gimbal 成功解决了 MoE 推理中的资源利用
不均和延迟瓶颈。在真实负载测试中，Gimbal 不仅大幅降低了首字延迟（TTFT）和每 Token 延迟（TPOT），还提高了
前缀缓存的命中率，且未损失系统吞吐量。

# 章节详细解读

## I. 背景与动机 (Background and Motivation)

### 1. 设计思想 (Design Philosophy)
- **痛点分析**：
    - **引擎层（Engine Level）**：传统 RR（轮询）策略忽视了不同引擎的实际负载（如前缀 Token 数量、KV 缓存占用），
      导致某些引擎过载而另一些空闲。
    - **请求层（Request Level）**：FCFS（先来先服务）在面对长短请求混合的负载时，容易产生队头阻塞（HoL），
      使短请求被长请求拖累。
    - **专家层（Expert Level）**：现有系统未考虑跨层专家之间的亲和性（Affinity）和依赖模式，导致跨设备通信
      开销巨大，且容易产生 GPU 本地热点。
- **核心动机**：MoE 模型的推理具有独特的多级特性（Data Parallel 和 Expert Parallel 结合）。作者认为，
  仅在单一层面优化是不够的，必须通过跨层的负载状态感知和亲和性保持来综合提升效率。

### 2. 关键技术 (Key Technologies)
- **多层架构拆解**：
    - **(1) LLM 引擎层**：负责具体推理。
    - **(2) 调度器层**：负责请求排序。
    - **(3) 专家层**：负责 MoE 专家的路由和位置分配。

## II. 系统设计 (System Design)

### 1. 引擎级负载均衡 (DP Engine Load Balancing)
- **设计思想**：
    - **KV 缓存感知**：优先考虑 GPU 内存中的 KV 缓存余量，防止因内存溢出导致的重计算或请求拒绝。
    - **用户粘性（User Stickiness）**：通过将会话固定在同一引擎，最大化前缀缓存（Prefix Caching）的重用率。
- **算法逻辑（Algorithm 1）**：
    - 首先检查各引擎的 KV 使用率差异。如果差异超过阈值 $\theta_{diff}$，则将请求路由至 KV 使用率最低的引擎。
    - 若 KV 差异在容差范围内，则比较运行负载 `RunningLoad`（以 Token 计数为准，而非请求数）。
    - 引入“用户亲和性”：在无严重负载不均的情况下，将来自同一用户的连续请求发往同一引擎，以复用上下文。

### 2. SJF 调度器与老化机制 (SJF Scheduler with Aging)
- **关键技术**：
    - **前缀长度估计**：由于输出长度不可知，作者利用 prefill 阶段（计算密集型）的 prefill Token 数作为请求
      成本的稳定代理，实现近似的 SJF（最短作业优先）。
    - **老化算法（Algorithm 2）**：为防止长请求被无限期推迟（Starvation），设定阈值 $\theta_{age}$。
      一旦等待时间超过该值，请求将被提拔至高优先级队列。
- **权衡与对比**：相比于基于预测输出长度的 SJF（如 FastServe），Gimbal 基于 prefill 长度的方法更具模型
  无关性和预测稳定性。

### 3. 专家动态替换模块 (Expert Dynamic Replacement Module)
- **设计思想**：
    - **跨层专家亲和性**：MoE 模型中，某些专家在层间往往协同激活。将这些具有强依赖关系的专家放置在同一 GPU
      上，可以显著减少跨 GPU 的全对全（All-to-All）通信。
- **数学建模**：
    - 将专家放置问题建模为多约束图划分任务。
    - **目标函数**：最小化跨 GPU 的通信成本 $Cut$，同时保持每一层的计算负载平衡。
    - **MILP 形式化**：定义激活矩阵 $A$ 和层间通信权重 $W$。
- **启发式策略（Algorithm 3）**：
    - **锚点安置（Affinity Placement）**：识别具有强跨层依赖的专家对，将其固定在同一“锚点”GPU 上。
    - **贪心负载均衡**：对剩余专家，按激活强度降序排列，依次放置在当前负载最轻的 GPU 上。

## III. 实验评估 (Performance Evaluation)

### 1. 实验配置 (Implementation & Setup)
- **硬件**：双路 A100-80GB，通过 NVLink 互联。
- **软件**：基于 vLLM v0.9.1 扩展。
- **数据集**：BurstGPT（真实负载分布）和 ShareGPT（评估前缀缓存）。
- **对比基准**：原生 vLLM，以及 Gimbal 的消融版本（DPLB：仅引擎均衡；SJFS：仅请求均衡；EDR：仅专家均衡）。

### 2. 核心实验结果
- **TTFT（首字延迟）**：
    - 在 1.4 RPS（高负载）下，Gimbal 在多种分布下平均降低了 17.76% 的 TTFT。
    - 消融实验显示，DPLB（引擎负载均衡）贡献了最大比例的性能提升。
- **TPOT（每 Token 延迟）**：
    - Gimbal 平均降低了 13.34% 的 TPOT。
    - 这表明多层调度不仅在排队阶段有效，在推理执行阶段也通过减轻热点和优化通信带来了收益。
- **前缀缓存（Prefix Cache）**：
    - Gimbal 的用户亲和性策略使前缀缓存命中率提升了约 4.4%。
    - 实验观察到，vLLM 的轮询策略由于异步分发，即使相同顺序的请求也可能导致不确定的引擎分配，
      而 Gimbal 通过用户绑定确保了确定的行为。

## IV. 讨论与局限性 (Discussion & Limitations)

### 1. 权衡考量 (Trade-offs)
- **通信与计算平衡**：在专家放置时，严格的负载均衡可能会破坏跨层亲和性。Gimbal 通过参数 $\alpha, \beta$
  控制这两者的平衡，但在实验中发现，为了减少昂贵的跨设备通信，牺牲一点点极端的负载均衡是值得的。
- **离线与在线**：当前的专家亲和性矩阵是离线收集的。虽然这对稳定运行的模型有效，但对于在线演进的负载，
  其有效性可能会随时间推移而下降。

### 2. 未来方向
- **在线亲和性估计**：开发更轻量级的在线监测机制，动态更新专家关系。
- **超大规模集群**：目前的实验仅在 2x A100 环境下进行。在拥有数百个节点和分层网络拓扑的生产环境中，
  多层调度的复杂性和收益潜力都会指数级增加。
