# 论文整体解析

### **标题**

面向多厂商 GPU 的大语言模型服务解耦式 Prefill 和 Decoding 推理系统

### **摘要**

随着大语言模型（LLM）在各行业的广泛应用和模型规模的日益增大，为服务提供商构建高效的 LLM 推理系统成为一个亟待解决的问题。推理过程分为 Prefill 和 Decode 两个特性不同的阶段，它们在推理过程中会相互干扰。为此，研究人员提出了 P-D（Prefill-Decode）解耦的推理框架。当前的研究主要在同构 GPU 上进行，缺乏基于业务场景的部署方案。与同构 GPU 相比，使用异构 GPU 构建推理系统能更好地提高资源利用率并降低成本。即使使用来自不同供应商的 GPU 构建推理系统，也能在降低成本的基础上提高资源利用率，并减少对单一供应商的依赖。因此，论文设计了一个基于异构 GPU 的 P-D 解耦推理系统，并设计了系统中的异构兼容传输模块以解决异构 GPU 的数据兼容性问题。然后，提出了并行策略和实例数量分配的联合优化算法以获得最优部署方案。最终的实验结果表明，该 P-D 解耦推理系统能很好地解决来自不同供应商的异构 GPU 的混合推理问题，并且该联合优化算法能获得最优的部署方案。

### **各章节主旨内容**

*   **第一章：引言 (Introduction)**
    本章介绍了大语言模型推理的背景，指出推理速度是关键的用户体验指标。LLM 的推理过程包含计算密集型的 Prefill 阶段（生成第一个 token）和显存密集型的 Decode 阶段（生成后续 tokens）。传统的推理系统将这两个阶段放在一起处理，导致资源利用率低下。因此，论文提出了将 Prefill 和 Decode 阶段分离（P-D 解耦）并部署在来自不同供应商的异构 GPU 上的想法，以提升系统吞吐量、优化资源利用并降低成本。

*   **第二章：相关工作 (Related Work)**
    本章回顾了 LLM 推理优化的相关研究。首先介绍了并行策略，如流水线并行（GPipe）和张量并行（Megatron-LM），这些策略最初用于模型训练，后被应用于推理。接着，重点介绍了 P-D 解耦推理系统，如 ByteDance 的 KVDirect 和 Mooncake 架构，它们通过分离 Prefill 和 Decode 集群来优化 KV Cache 的传输和管理。最后，本章指出现有研究大多基于同一供应商的同构或异构 GPU，而跨不同供应商 GPU 的 P-D 解耦系统是一个亟待解决的问题。

*   **第三章：系统设计 (System Design)**
    本章详细阐述了所提出的异构 P-D 解耦推理系统的架构和工作流程。
    *   **系统架构**：由全局调度器、服务器、引擎、异构兼容模块和 KV 传输模块组成。计算能力强的 GPU 用于 Prefill 实例，而显存访问能力强的 GPU 用于 Decode 实例。
    *   **工作流程**：用户请求由全局调度器分发给合适的 P 和 D 实例。P 实例生成第一个 token 的 KV 数据和隐藏状态，然后 D 实例读取这些数据以生成剩余的 tokens。
    *   **异构兼容传输模块**：这是核心模块，用于解决不同厂商 GPU 之间的数据共享问题。它包含三个关键组件：RDMA 传输模块、VRAM 管理对齐组件（解决不同 GPU 的数据布局和块大小差异）和异构并行策略对齐组件（处理 P 和 D 实例可能采用不同并行策略的问题）。
    *   **联合优化算法**：提出了一个数学模型，旨在通过联合优化并行策略和 P/D 实例比例，以最大化系统吞吐量，同时满足 TTFT（首个 token 生成时间）和 TPOT（后续 token 生成时间）的服务等级目标（SLO）。

*   **第四章：性能评估 (Performance Evaluation)**
    本章通过实验评估了所提出系统的性能。实验平台使用了两种来自不同供应商的 GPU（GPU A 和 GPU B），并使用 Llama2-7B 模型进行测试。
    *   **不同上下文长度的影响**：实验表明，随着输入和输出长度的增加，TTFT 和 TPOT 都会增加，导致吞吐量下降。
    *   **不同 P-D 比例的影响**：实验结果显示，增加 P 实例数量可以显著降低 TTFT，增加 D 实例数量可以降低 TPOT，但两者存在相互制约，需要找到最佳比例。
    *   **异构 P-D 解耦的优势**：与将 P/D 集成在同一 GPU 上的方案相比，P-D 解耦方案在长上下文和高 QPS（每秒查询数）场景下表现出显著的性能优势，吞吐量提升了 17% 到 30%。这证明了解耦架构可以打破单个 GPU 的性能瓶颈。

### **结论 (Conclusion)**

论文提出了一个 P-D 解耦的异构推理框架，旨在提高闲置资源利用率并降低成本。该框架中的核心设计是一个异构兼容性模块，用以解决由不同 VRAM 管理方式和不同并行策略引起的兼容性问题。基于此，论文建立并解决了一个关于并行策略和实例比例的联合优化问题，并通过详细的模拟器进行了验证。实验结果（在 TTFT、TPOT 和吞吐量方面）表明，所提出的异构 P-D 解耦系统具有更优的性能。同时，论文根据实验数据给出了异构 P-D 解耦的优势场景。论文也指出，由于资源限制，没有在更大的模型上进行实验验证，未来的工作将会在更大模型上持续改进实验，并优化 KV 传输延迟。

# 第三章：系统设计 (System Design) 详解

这一章是整篇论文的基石，它详细描述了作者如何构建一个能够跨越不同厂商 GPU、高效运行大语言模型推理的系统。其核心思想是通过“P-D 解耦”和“异构兼容”两大策略来最大化硬件资源的利用率。

我们可以将第三章的设计分为四个主要部分：

## 1. 系统整体架构 (System Architecture)

论文提出的系统架构如论文图 1 所示，主要由以下几个模块构成：

*   **全局调度器 (Global Scheduler)**：这是整个系统的入口和大脑。它接收来自用户的 HTTP 请求，并根据各个 GPU 集群的实时负载情况，智能地为每个请求分配一个 **P 实例 (Prefill Instance)** 和一个 **D 实例 (Decode Instance)**。
*   **服务器 (Server)**：它作为调度器和推理引擎之间的中间人，接收调度指令，并将推理任务放入对应引擎的请求队列中。
*   **引擎 (Engine)**：这是执行具体推理任务的工作单元，它在 GPU 上运行 Prefill 或 Decode 计算。
*   **KV 传输模块 (KV Transfer Module)**：负责在 P 实例和 D 实例之间高速传输 KV Cache 数据。为了追求极致的性能，该模块采用 **RDMA (远程直接内存访问)** 技术，可以绕过 CPU，实现网络设备和内存之间的高速数据传输。
*   **异构兼容模块 (Heterogeneous Compatible Module)**：**这是整个设计的灵魂**。它的作用是“屏蔽”不同厂商 GPU 之间的差异，确保 P 实例（例如，在一个 A 厂商的 GPU 上）产生的数据能被 D 实例（例如，在一个 B 厂商的 GPU 上）正确理解和使用。

在这个架构中，系统会根据 GPU 的特性来分配任务：将计算密集型的 Prefill 任务交给计算能力强的 GPU（作为 P 实例），将显存访问密集型的 Decode 任务交给显存带宽大的 GPU（作为 D 实例）。

## 2. 系统工作流程 (Workflow Design)

论文图 2 清晰地展示了一个请求从进入到完成的完整生命周期：

1.  **请求分发**：全局调度器收到用户请求，选择一个 P 实例和一个 D 实例。
2.  **Prefill 阶段**：调度器将请求（包含 D 实例的位置信息）发送给 P 实例。P 实例的引擎执行 Prefill 计算，生成第一个 token 的 **KV Cache** 和 **隐藏状态 (hidden state)**。
3.  **数据暂存**：P 实例将计算出的 KV Cache 和隐藏状态复制到自己的 **CPU 内存缓冲区**中。
4.  **Decode 阶段**：调度器通知 D 实例处理该请求。D 实例根据 P 实例的地址，通过 RDMA 读取之前存放在 P 实例 CPU 缓冲区中的 KV Cache 数据，并加载到自己的 GPU 显存中。
5.  **生成后续 Token**：D 实例利用这些 KV Cache 数据，高效地逐个生成剩余的 tokens，并将最终结果返回给用户。

## 3. 异构兼容传输模块设计 (Heterogeneous Compatible Transmission Module Design)

这是本章最关键、最具创新性的部分，它解决了跨厂商 GPU 协作的核心难题。该模块主要由三个子组件构成：

### VRAM 管理对齐组件 (VRAM Management Alignment Component)
*   **问题**：不同厂商的 GPU（甚至同一厂商不同代次的 GPU）其底层硬件架构不同，导致它们存储张量（Tensor）的数据布局（Layout）可能不一样，例如行优先（Row-major）或列优先（Column-major）。如果直接传输，D 实例可能无法正确解析数据。
*   **解决方案**（如论文图 3 所示）：论文提出了一种通用方法。在数据传输前，P 实例先将多维的张量数据“拉平”成一个标准的**一维数组**。传输完成后，D 实例再根据自己需要的内存布局，将这个一维数组重新恢复成多维张量。通过这种“先标准化，再个性化”的方式，屏蔽了底层硬件的布局差异。

### 异构并行策略对齐组件 (Heterogeneous Parallel Strategy Alignment Component)
*   **问题**：为了最大化效率，P 实例和 D 实例可能会采用不同的并行策略。例如，P 实例可能使用 4 路张量并行（TP=4），而 D 实例为了节省显存可能只使用 2 路张量并行（TP=2）。这时，P 实例产生的数据分片和 D 实例需要的数据分片是不匹配的。
*   **解决方案**（如论文图 4 所示）：该组件充当一个“数据适配器”。在上面的例子中，当 D 实例（TP=2）需要数据时，它会主动从 P 实例（TP=4）的第 1 和第 2 个分片中读取数据并**合并**，作为自己第 1 个分片的数据；同时读取 P 实例的第 3 和第 4 个分片的数据并合并，作为自己第 2 个分片的数据。反之，如果 D 实例的并行路数更多，它则会执行**拆分**操作。

### RDMA 传输模块 (RDMA Transmission Module)
*   这个模块负责执行高效的数据传输。论文提到，为了减少数据在 CPU 和 GPU 之间来回拷贝的开销，系统在初始化时就分配了**锁页内存 (Pinned Memory)** 作为 CPU 缓冲区。这种内存可以直接被 RDMA 访问，从而实现“零拷贝 (Zero-copy)”传输，极大地降低了数据传输延迟。

## 4. 联合优化算法 (Joint Optimization of Parallel Strategy and P-D Ratio)

这部分是关于如何为系统做出最优的“部署规划”。仅仅搭建好系统还不够，还需要决定**用多少个 P 实例和 D 实例（P-D 比例）**，以及**每个实例应该采用什么样的并行策略**，才能在满足服务质量（如响应时间）的前提下，实现吞吐量最大化。

*   **数学建模**：论文为此建立了一个优化模型。
    *   **目标**：最大化整个系统的总吞吐量（每秒生成的 token 数）。
    *   **约束条件**：
        1.  首个 token 的生成时间（TTFT）必须低于设定的阈值。
        2.  后续 token 的生成时间（TPOT）也必须低于设定的阈值。
        3.  任何一个 GPU 的显存占用都不能超过其物理上限。
*   **模拟器设计 (Simulator Design)**：
    *   由于可能的配置组合非常多，无法一一实际部署测试。因此，作者设计了一个分层模拟器（如论文图 5 所示）。这个模拟器从底层的 Transformer 理论模型出发，逐层考虑硬件特性（GPU 算力、显存带宽等）、推理框架特性（如 vLLM 的 PagedAttention）和算子库性能，最终能够相当准确地预测出在任意一种配置下系统的延迟和显存占用。
    *   通过这个模拟器，系统可以在部署前就“推演”出各种方案的优劣，从而找到那个能让吞吐量最大化的最佳配置。
