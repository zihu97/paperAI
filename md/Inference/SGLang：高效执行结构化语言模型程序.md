### 论文基本信息
- **论文文件名:** 2312.07104v2.pdf
- **英文标题:** SGLang: Efficient Execution of Structured Language Model Programs
- **中文标题:** SGLang：高效执行结构化语言模型程序

### 中文摘要
大型语言模型（LLM）正越来越多地用于完成需要多次调用生成、高级提示技术、控制流和结构化输入/输出的复杂任务。
然而，目前缺乏用于编写和执行这些应用的有效系统。论文介绍了 SGLang，一个旨在高效执行复杂语言模型程序（LM Programs）的系统。
SGLang 由一个前端语言和一个后端运行时组成。前端通过为生成和并行控制提供原语，简化了编程过程。
后端运行时则通过新颖的优化来加速执行，例如用于 KV 缓存重用的 RadixAttention 和用于加速结构化输出解码的压缩有限状态机。
实验表明，在包括智能体控制、逻辑推理、少样本学习基准测试、JSON 解码、检索增强生成（RAG）流水线和多轮对话等任务中，
SGLang 在多种大型语言和多模态模型上，相比最先进的推理系统，实现了高达 6.4 倍的吞吐量提升。

### 各章节主旨内容
*   **第一章：引言 (Introduction)**
    *   介绍了随着 LLM 能力的增强，其应用从简单的聊天转向了需要多次、有依赖的 LLM 调用的复杂“语言模型程序”。
    指出了当前系统在编写和高效执行这些程序方面面临的编程繁琐和执行效率低下的两大挑战。
    最后，概述了 SGLang 作为解决方案，它通过一个前端语言和优化的后端运行时来系统性地利用 LM 程序的结构以提升效率。

*   **第二章：编程模型 (Programming Model)**
    *   详细介绍了 SGLang 作为 Python 内嵌的领域特定语言（DSL）。通过一个多维度论文评审的实例，展示了 SGLang 
    如何使用 `gen`（生成）、`select`（选择）、`fork`（并行分支）等核心原语，以及与 Python 
    原生语法的结合，来简化复杂提示流的编写。同时，介绍了其解释器执行模式，并指出了可利用的运行时优化机会。

*   **第三章：使用 RadixAttention 高效重用 KV 缓存 (Efficient KV Cache Reuse with RadixAttention)**
    *   针对 LM 程序中普遍存在的共享提示前缀问题，提出了 RadixAttention 这一核心优化。该技术将 KV 
    缓存视为一个传统的缓存系统，并使用基数树（Radix Tree）来存储和管理所有请求的 KV 
    缓存，实现了跨不同调用和实例的自动、高效的缓存重用。同时，设计了缓存感知的调度策略以提高缓存命中率。

*   **第四章：使用压缩有限状态机的高效约束解码 (Efficient Constrained Decoding with Compressed Finite State Machine)**
    *   为了解决生成特定格式（如 JSON）输出时的低效问题，提出了一种快速约束解码技术。该技术通过分析正则表达式（regex）
    约束，构建一个压缩的有限状态机（FSM）。这个压缩 FSM 能够将多步的、无分支的解码路径合并为单步，从而允许一次性解码多个 token，显著加速了结构化输出的生成过程。

*   **第五章：使用 API 推测执行的高效端点调用 (Efficient Endpoint Calling with API Speculative Execution)**
    *   针对只提供黑盒 API 访问的模型（如 GPT-4），提出了一种名为“API 推测执行”的优化。它通过在单次 API 
    调用中，让模型在遇到停止符后继续多生成一些 token。这些被“推测”生成的额外 token 
    如果能匹配后续的生成请求，就可以直接重用，从而节省一次 API 调用的延迟和成本。

*   **第六章：评估 (Evaluation)**
    *   通过在多种 LLM 工作负载（如 MMLU, ReAct Agent, JSON 解码等）和多种模型（Llama、Mixtral、LLaVA 
    等）上的大量实验，展示了 SGLang 相较于 vLLM、Guidance、LMQL 
    等基线系统的性能优势。结果表明，SGLang 显著提升了吞吐量并降低了延迟。同时通过消融实验验证了 RadixAttention 
    等关键组件的有效性。

*   **第七章：相关工作 (Related Work)**
    *   将 SGLang 与现有的 LLM 编程框架（如 LangChain, DSPy）和推理优化技术（如 vLLM, 
    PagedAttention）进行了比较。强调了 SGLang 的独特性在于其创新的、基于基数树的 KV 缓存管理机制，以及前端语言和后端运行时之间的协同设计。

*   **第八章：未来方向与结论 (Future Directions and Conclusion)**
    *   总结了 SGLang 的主要贡献，即作为一个高效编程和执行结构化 LM 
    程序的框架，通过新颖的优化显著提升了复杂工作流的性能。同时，指出了未来可能的研究方向，
    包括支持更多输出模态、模糊语义匹配的缓存重用、以及更高级的编译器优化等。

### 中文结论
论文介绍了 SGLang，一个用于高效编程和执行结构化语言模型程序的框架。通过 RadixAttention、压缩有限状态机和语言解释器等新颖优化，SGLang 
显著提升了复杂 LM 程序的吞吐量和延迟。它是一个用于开发高级提示技术和智能体工作流的宝贵工具。

### **各章节详解**

#### **第一章：引言 (Introduction)**
本章首先描绘了大型语言模型（LLM）应用的发展趋势。当前，LLM 的使用已经超越了简单的单轮问答或文本生成，
演变为更复杂的“语言模型程序”（LM Programs）。这些程序通常包含多个、相互依赖的 LLM 调用，并与控制流（如循环、条件判断）
交织在一起，以完成如自主智能体（Agent）、思维链（Chain-of-Thought）推理、多工具使用等高级任务。
然而，开发和执行这些 LM 程序面临两个主要挑战：
1.  **编程困难**：开发者需要进行繁琐的字符串操作、手动管理并行执行、解析非结构化的模型输出，这使得代码可读性差且难以维护。
2.  **执行低效**：现有的推理系统（如 vLLM, TensorRT-LLM）虽然对单次生成任务进行了优化，但它们对整个 LM 
    程序的上层结构一无所知。这导致了大量的冗余计算，尤其是在 Key-Value (KV) 缓存方面。例如，在多轮对话或少样本学习（few-shot 
    learning）中，大量共享的前缀（如系统提示、历史对话、样本示例）对应的 KV 缓存被反复重新计算，造成了严重的性能浪费。
针对这些挑战，论文提出了 SGLang。其核心思想是系统性地利用 LM 程序的内在结构来优化执行。SGLang 
包含两个协同工作的部分：一个前端语言和一个后端运行时。前端语言嵌入在 Python 中，旨在简化编程；后端运行时则引入了多项创新优化，
旨在加速执行。

#### **第二章：编程模型 (Programming Model)**
本章详细阐述了 SGLang 的编程模型，它被设计成一个内嵌于 Python 的领域特定语言（DSL），使得开发者可以利用熟悉的 Python 
语法和生态。
*   **核心设计**：SGLang 将提示（prompt）视为一个可异步操作的流（stream）。开发者可以通过 `+=` 运算符向这个流中添加文本、图片、视频，或者调用 SGLang 的原语。
*   **关键原语**：
    *   `gen("var_name", ...)`: 调用 LLM 进行文本生成，并将结果存入变量 `var_name`。它支持多种参数，如 `stop` 
        （停止字符串）和 `regex`（正则表达式约束）。
    *   `select("var_name", choices=[...])`: 让 LLM 从给定的 `choices` 列表中选择概率最高的一项，结果存入 `var_name`。
    *   `fork(N)`: 将当前的提示流复制 N 份，返回 N 个新的流对象。这使得并行处理（如对多个维度进行独立评估）变得异常简单。
    *   `image(path)` 和 `video(path)`: 用于在多模态模型中处理图像和视频输入。
*   **执行模式**：SGLang 程序通过解释器执行。所有的原语调用都是非阻塞的，类似于异步发起 CUDA 
    核函数。这使得 Python 
    代码可以继续执行，而 LLM 的生成任务则在后台由一个专门的线程管理，从而实现了程序内（intra-program）的并行。当代码需要访问某个生成结果（
    如 `s["var_name"]`）时，程序会阻塞直到该结果就绪。
通过一个“多维度评估图片配文”的例子（见论文图2），论文展示了 SGLang 相比于传统的 OpenAI 
API 调用方式，代码行数可以减少一半以上，并且代码逻辑更清晰、更易于管理并行。

#### **第三章：使用 RadixAttention 高效重用 KV 缓存**
这是 SGLang 最核心的创新之一。KV 缓存是 LLM 推理过程中的中间计算结果，重用它可以避免对相同前缀的重复计算。LM 
程序中充满了重用 KV 缓存的机会（如系统提示、多轮对话历史、few-shot 示例等）。
*   **问题**：现有系统要么完全不重用 KV 缓存，要么只支持非常有限的、需要手动配置的重用场景。它们无法自动处理 LM 
    程序中动态、复杂的树状共享结构。
*   **SGLang 的方案：RadixAttention**
    1.  **数据结构**：SGLang 运行时维护一个全局的、以 token 序列为键的基数树（Radix 
        Tree）。树的每个节点都存储着对应 token 序列的 KV 缓存。基数树相比于普通前缀树（Trie）空间效率更高，能有效表示长序列。
    2.  **自动重用**：当一个新请求到达时，系统会在基数树中进行前缀匹配。如果找到匹配的前缀，对应的 KV 
        缓存就会被直接重用，模型只需从不匹配的位置开始计算。
    3.  **缓存管理**：SGLang 将 KV 缓存真正当作一个“缓存”来管理。它实现了 LRU（Least Recently 
        Used）驱逐策略，当 GPU 内存不足时，会优先驱逐最久未被使用的缓存条目（树的叶子节点）。
    4.  **缓存感知调度**：在等待队列中有多个请求时，调度器会优先选择那些与当前缓存内容有更长共享前缀的请求来执行，
        以最大化缓存命中率，提升批处理（batching）效率。这被称为“最长共享前缀优先”（longest-shared-prefix-first）策略。
RadixAttention 的设计是通用且自动的，能够系统性地捕获所有 KV 缓存重用的机会，且与 PagedAttention 等技术兼容。

#### **第四章：使用压缩有限状态机的高效约束解码**
在很多应用中，需要 LLM 的输出严格遵守某种格式，例如 JSON。这通常通过正则表达式（regex）来约束。
*   **问题**：传统的约束解码方法是将 regex 转换为有限状态机（FSM），然后在每一步解码时，根据 FSM 
    的当前状态过滤掉无效的下一个 token。这种逐个 token 
    解码的方式是低效的，尤其当约束中包含一长串确定的字符时（例如 `"summary": "`）。
*   **SGLang 的方案：压缩 FSM**
    1.  **FSM 压缩**：SGLang 首先分析原始的 FSM，并将其中所有连续的、唯一的转换路径（即没有分支的路径）压缩成一条边。
        例如，字符串 `{"summary":` 在原始 FSM 中可能对应 10 多个状态转换，但在压缩 FSM 
        中只是一条边。
    2.  **多 Token 解码**：在解码过程中，如果遇到这样一条压缩边，SGLang 运行时知道接下来的一长串 token 
        是确定且唯一的。因此，它可以一次性地将这些 token 全部“解码”（实际上是直接作为下一次前向传播的输入），
        而无需逐个 token 地调用模型，从而将多次解码操作合并为一次，极大地提升了速度。
这种方法通用地适用于所有正则表达式，并能显著加速结构化数据的生成。

#### **第五章：使用 API 推测执行的高效端点调用**
对于像 GPT-4 这样只能通过 API 访问的模型，无法修改其内部推理过程，因此 RadixAttention 和压缩 FSM 
等优化无法直接应用。
*   **问题**：一个 LM 程序中可能包含多个连续的 `gen` 调用，例如 `s += "name:" + gen("name") + "job:" + gen("job")`。
朴素的实现需要两次独立的 API 调用。
*   **SGLang 的方案：API 推测执行**
    1.  **推测生成**：在执行第一个 `gen("name")` 调用时，SGLang 会指示 API 在生成 `name` 之后，即使遇到了停止符，也继续多生成几个
        token。
    2.  **匹配与重用**：当程序执行到第二个 `gen("job")` 时，SGLang 
        会检查之前推测生成的额外 token 是否恰好能满足这个请求（例如，模型可能在生成完 `name` 
        后，直接生成了 `\njob: Software Engineer`）。如果匹配，那么第二个 `gen` 
        调用就可以直接使用这些本地已有的 token，无需再次发起昂贵的 API 请求。
这种方法在提示设计得比较好的情况下，能够以很小的代价换来显著的延迟和成本节约。

#### **第六章：评估**
本章提供了详尽的实验数据来证明 SGLang 的有效性。
*   **设置**：实验覆盖了多种模型（Llama-7B/70B, Mixtral-8x7B, LLaVA）、硬件（NVIDIA A10G/A100）和工作负载（
    MMLU, HellaSwag, 智能体, 思维树, JSON 解码, RAG 等）。对比的基线系统包括 vLLM, Guidance, LMQL 
    等。
*   **主要结果**：
    *   **端到端性能**：SGLang 在所有测试负载上都表现出巨大的性能优势，吞吐量最高提升了 6.4 倍，延迟最多降低了 3.7 倍。
    *   **性能归因**：在少样本学习（MMLU）和多轮对话等场景，性能提升主要来自 RadixAttention 的 KV 缓存重用。在 JSON 
    解码场景，性能提升主要来自压缩 FSM。在思维树等需要并行的场景，SGLang 的 `fork` 原语和并行执行模型发挥了关键作用。
    *   **消融研究**：通过禁用 SGLang 的不同优化组件（如关闭缓存、使用简单的哈希表缓存代替基数树、使用 FCFS 
    调度代替缓存感知调度），实验证明了 RadixAttention 及其各个子组件（树结构、调度策略、前端提示）都是实现最佳性能不可或缺的。
    *   **多模态和生产环境**：实验还表明 SGLang 的优化同样适用于多模态模型，并且已经在 Chatbot Arena 
    等生产环境中部署，取得了显著的缓存命中率和延迟降低。

#### **第七章：相关工作**
本章将 SGLang 放置在更广阔的研究背景下进行比较。
*   **与 LLM 编程系统的比较**：像 LangChain 和 DSPy 属于更高级的框架，它们关注的是链的构建和自动提示优化，
而 SGLang 是一个低级的、关注执行效率的系统。SGLang 可以作为 DSPy 等高级系统的后端来加速执行。
与 LMQL 和 Guidance 相比，它们都属于低级 DSL，但 SGLang 的主要区别和优势在于其共同设计的、包含 RadixAttention 
等新颖优化的运行时。
*   **与 KV 缓存优化工作的比较**：虽然 vLLM 的 PagedAttention 和一些并发工作也探索了 KV 
    缓存重用，但它们通常局限于简单的共享场景（如仅共享系统提示），无法处理 SGLang 所能处理的复杂的、动态的、多层次的树状共享模式。
RadixAttention 是第一个将 KV 缓存作为一个完整的、带有 LRU 策略的树状缓存系统来系统性管理的方案。

#### **第八章：未来方向与结论**
本章对全文进行了总结，并展望了未来的研究。
*   **结论**：SGLang 作为一个集成了前端编程语言和高效后端运行时的框架，通过 RadixAttention 和压缩 FSM 
等创新优化，成功地解决了结构化语言模型程序执行效率低下的问题，是开发复杂 LLM 应用的强大工具。
*   **未来方向**：
    *   扩展 SGLang 以支持更多输出模态（如生成图片）。
    *   研究在 RadixAttention 中支持模糊语义匹配，而不仅仅是精确的 token 匹配。
    *   修复缓存感知调度中可能存在的饥饿问题。
    *   增强 SGLang 的编译器，使其能进行更高级的静态优化，如自动调度和内存规划。
