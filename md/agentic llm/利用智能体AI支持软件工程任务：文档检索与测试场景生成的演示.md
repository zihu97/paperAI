# 论文解析：利用智能体 AI 支持软件工程任务

## 第一部分：论文整体骨架

### 基本信息
- **文件名**：2602.04726v1.pdf
- **英文标题**：SUPPORTING SOFTWARE ENGINEERING TASKS WITH AGENTIC AI: DEMONSTRATION ON DOCUMENT RETRIEVAL AND TEST SCENARIO GENERATION
- **中文标题**：利用智能体 AI 支持软件工程任务：文档检索与测试场景生成的演示

### 中文摘要
大语言模型（LLMs）的引入引发了软件开发模式的重大变革。本文介绍了两种针对特定软件工程任务的智能体 AI（Agentic AI）
解决方案：
1. **自动化测试场景生成**：从详细的需求描述中自动生成测试场景。该方案采用星型拓扑结构，由一个监督智能体（Supervisor Agent）
   管理多个专业化的工作智能体（Worker Agents）。
2. **软件工程文档检索**：针对软件开发过程中的文档库，支持搜索、问答、变更追踪和长文档总结。每个用例由专门的 LLM 智能体处理。
本文通过真实案例演示了这些系统的功能，并探讨了未来研究方向。

### 各章节主旨内容
- **1. 引言 (Introduction)**：阐述了 LLM 在软件工程自动化中的潜力，指出 SE 过程中存在大量非结构化或半结构化文档（如 NL 需求、
  邮件、评论等），导航和利用这些文档具有挑战性。
- **2. 相关工作 (Related work)**：回顾了利用 LLM 生成测试场景（如 RAGTAG, GeneUS, ScenGen）以及文档处理与检索（IR）的
  现有研究，分析了当前方案在幻觉控制、成本和特定领域适应性方面的局限。
- **3. 我们的方法 (Our approach)**：详细介绍了基于 LangChain 和 LangGraph 构建的两种智能体架构。
    - **3.1 测试场景生成**：详细描述了 6 智能体构成的星型拓扑架构及其工作流程。
    - **3.2 文档处理**：介绍了包含搜索、问答、追踪和阅读四种能力的智能体解决方案。
- **4. 结论与未来工作 (Conclusion and future work)**：总结了系统在实际软件公司中的部署情况，并提出了未来进行基准测试评估
  和利用使用数据持续优化系统的计划。

### 中文结论
本文提出了两种实用的、基于 LLM 的智能体 AI 解决方案，用于解决软件工程中的测试场景生成和文档检索任务。这些方案目前已在
一家中型软件工程公司投入日常运行。通过将任务分解给专门的智能体并引入反馈循环（如事实核查），系统能够有效缓解 LLM 的
幻觉问题。未来的工作将集中在利用标准基准数据集进行全面评估，并根据实际运行数据优化模型和架构。

## 第二部分：章节详解

### 1. 自动化测试场景生成的设计思想与架构

#### 设计背景与痛点
在软件开发生命周期（SDLC）中，根据功能规格说明书（FSD）手动创建测试场景是一项耗时且昂贵的工作。FSD 通常包含几十页
文字和图表，人工编写需要覆盖每一个细节。传统的自动化工具难以处理非结构化的自然语言描述。

#### 核心架构：星型拓扑智能体集群
作者设计了一个由 **6 个智能体** 组成的系统，核心是**监督智能体（Supervisor Agent）**，它像交响乐指挥家一样协调
其他 5 个专业智能体：
- **设计思想**：为了应对 LLM 上下文窗口的限制，每个智能体只在其特定上下文中运行。监督智能体只处理来自工作智能体的
  自然语言响应，而将 FSD、Markdown 格式的场景等大型工件存储在外部，确保核心上下文的简洁。
- **拓扑结构**：星型拓扑使得扩展变得容易。工作智能体互不感知，只与监督者通信，这种解耦降低了复杂性。

#### 关键组件与交互流程
1. **检索智能体（Retriever Agent）**：负责将 FSD 预处理为纯文本，并利用 VLM 处理图像。它根据监督者的指令定位特定章节。
2. **测试场景编写智能体（Test Scenario Writer Agent）**：基于检索到的文本生成符合预定义模板的 Markdown 测试场景。
3. **事实核查智能体（Fact Checker Agent）**：这是**质量控制**的关键。它将生成的场景与原始 FSD 内容进行比对。如果发现
   不一致或幻觉，它会向监督者报告，从而触发反馈循环（Fact-checking loop），要求编写者修正。
4. **翻译智能体（Translator Agent）**：根据客户需求将场景翻译成目标语言。
5. **Excel 编写智能体（Excel Writer Agent）**：非 LLM 驱动的工具，将 Markdown 转换为结构化的 Excel 文档。

### 2. 文档处理系统的多维能力拆解

#### 设计理念：从简单检索到深度理解
软件工程文档库（如需求、设计文档、不同版本的协议等）往往组织杂乱，且随时间演变。作者构建了一个集成化应用（App），
通过 **Delegator（分发智能体）** 识别用户意图，并调用以下四个专业工具：

#### 关键技术方案
- **通用搜索（General Search）**：
    - **实现细节**：生成数据库查询语句，在 Qdrant 向量数据库中检索。智能体不仅返回列表，还会对每个文档进行摘要总结，
      并根据相关性将结果分为“最相关文档”和“补充材料”。
- **问答系统（Q&A）**：
    - **交互逻辑**：采用先检索、后过滤、再汇总的策略。每个检索到的文档被单独用于尝试回答问题，最后由智能体汇总所有
      部分答案并标注引用来源。如果无法回答，系统会明确告知用户，避免盲目猜测。
- **变更追踪（Tracking Changes/Trace）**：
    - **痛点解决**：在 SDLC 中追踪某个需求的演变非常困难。该工具通过检索特定需求的所有历史版本，构建一个按时间排序的
      演变史，描述需求如何随时间发生变化。
- **长文档阅读（Document Reading）**：
    - **关键算法**：采用**分而治之（Divide-and-conquer）策略**。当文档长度超过上下文窗口时，智能体将其分块阅读。
      智能体在维护一个全局“笔记”的同时，逐块更新理解并清除临时上下文，最后汇总生成完整响应。

### 3. 实现细节与权衡（Trade-offs）

#### 技术栈选择
- **框架**：基于 **LangChain** 和 **LangGraph**。LangGraph 提供的状态管理能力对于实现事实核查循环和复杂的智能体编排
  至关重要。
- **模型**：采用了混合模式，结合了本地部署（On-premise）和云端 LLM。这在数据安全（软件公司的敏感规格说明）与处理
  能力（云端强力模型处理复杂逻辑）之间取得了平衡。

#### 设计权衡
1. **确定性 vs. 灵活性**：在最后一步生成 Excel 时，作者选择了**硬编码**的结构而非 LLM 生成，以确保测试工件的格式
   绝对符合行业规范，牺牲了部分灵活性以换取生产环境的稳定性。
2. **上下文管理**：为了防止“大海捞针”效应（Lost in the middle）和上下文溢出，系统强制将工件存储在外部。虽然这增加了
   系统的工程复杂度（需要频繁的 I/O 和状态同步），但显著提高了处理长篇 FSD 的准确度。
3. **人工干预**：论文强调自动化并不能完全取代专家。生成的测试场景仍需人工确认，AI 的角色是提供高质量的初稿和
   自动化的交叉核对。

### 4. 实验演示与实际价值

#### 真实案例展示
论文展示了一个“密码变更”模块的测试场景生成。系统不仅生成了**主路径（Main Flow）**（如：导航至登录页 -> 点击忘记密码），
还生成了**替代路径（Alternative Flow）**（如：输入无效邮箱后的错误提示）以及**缺陷记录示例**。这证明了智能体架构能够
捕捉业务逻辑中的异常分支，而不仅仅是字面翻译。

#### 部署现状
该系统已在实际的软件工程公司中部署，日常用于处理真实的业务文档。这种从实验室原型到生产环境的跨越，证明了 Agentic AI
在处理垂直领域复杂任务时的实用性。
