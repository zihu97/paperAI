
### 论文基本信息
- **论文文件名**: 2601.16839v1.pdf
- **英文标题**: AI builds, We Analyze: An Empirical Study of AI-Generated Build Code Quality
- **中文标题**: AI构建，我们分析：一项关于AI生成构建代码质量的实证研究

### 中文摘要
随着AI编码代理在软件开发中的迅速普及，其生成的代码质量和可维护性引发了广泛关注。
先前研究主要集中在AI生成的源代码上，而AI编码代理对构建系统（软件生命周期中一个关键但未被充分研究的环节）的影响在很大程度上仍是未解之谜。
本研究利用首个大规模、开源的，包含由AI代理创建的拉取请求（Agentic-PRs）的数据集AIDev，
旨在探究(RQ1)AI编码代理是否会生成存在质量问题（如代码异味）的构建代码，
(RQ2)AI代理在多大程度上能消除构建代码中的异味，以及(RQ3)开发者在多大程度上会接受这些由AI代理生成的PR。
研究识别出364个不同严重等级的可维护性和安全性相关的构建异味，
表明AI生成的构建代码既可能引入质量问题（如缺少错误处理、硬编码路径或URL），
也可能通过重构（如模块上移、属性外部化）移除现有异味。
值得注意的是，超过61%的Agentic-PRs在极少的人工干预下被批准并合并。
这种双重影响凸显了未来研究AI感知构建代码质量评估的必要性，以系统性地评估、指导和规范AI生成的构建系统代码。

### 各章节主旨内容
- **引言 (Introduction)**: 介绍研究背景，即AI编码代理在软件开发中日益普及，但其对构建系统代码质量的影响尚不明确。
提出三个核心研究问题（RQs）：AI是否会产生有异味的构建代码？AI能否移除构建代码中的异味？开发者对AI生成的构建代码接受度如何？

- **背景 (Background)**: 解释研究涉及的关键概念。包括构建系统（如Maven, Gradle, Cmake）在现代软件开发中的作用，
代码异味（Code Smells）作为不良设计和实现策略的指标，以及重构（Refactoring）作为改善代码质量的手段。
同时介绍了用于检测构建代码异味的静态分析工具Sniffer。

- **相关工作 (Related Works)**: 综述了在构建维护（Build Maintenance）和基于AI的代码生成（AI-based Code Generation）两个领域已有的研究。
指出先前工作主要关注源代码，而本研究首次聚焦于AI生成的**构建代码**的质量。

- **方法论 (Methodology)**: 详细描述了研究设计和执行步骤。
研究基于AIDev数据集，筛选出涉及构建文件（pom.xml, .gradle, CMakeLists.txt等）修改的387个PR。
使用Sniffer工具分析AI修改前后的代码，以评估代码异味的增减情况。
通过手动标注和分析，深入探究代码异味被消除的方式（回答RQ2）以及开发者对这些PR的接受与合并情况（回答RQ3）。

- **结果 (Results)**: 展示了对三个研究问题的发现。
  - **RQ1**: AI代理确实会引入新的构建代码异味，主要是可维护性相关的问题，如“通配符依赖”和“缺少错误处理”。
  - **RQ2**: AI代理同样能够通过重构来移除代码异味，例如“移除未使用的依赖”和“将硬编码的属性外部化”。
  - **RQ3**: 开发者对AI生成的构建代码PR接受度很高（超过61%被合并），且通常只需极少的人工审查和修改。

- **讨论 (Discussion)**: 探讨研究结果的意义。
AI在构建代码上表现出双重性（既能引入问题也能修复问题），这强调了开发者监督的必要性。
高接受率表明AI代理有潜力提高开发效率，但也可能引入技术债。

- **有效性威胁 (Threats To Validity)**: 承认研究的潜在局限性。
包括数据集本身的偏差、静态分析工具Sniffer可能存在的误报，以及手动分析过程中的主观性风险。

- **伦理考量 (Ethical Considerations)**: 声明本研究使用的数据均来自公开的GitHub仓库，所有结果均以聚合形式报告，不涉及任何私人或敏感数据。

- **结论与未来工作 (Conclusion & Future Work)**: 总结研究的核心发现，即AI代理已经对构建代码产生实际影响，有好有坏。
未来工作应致力于开发能感知代码异味的“smell-aware”AI代理，并集成自动化质量检查，以防止技术债的累积。

### 中文结论
AI代理如今在现代软件工程中被广泛使用，并越来越多地参与到软件构建代码的生成中。
本研究表明，AI代理能够有意义地重构构建脚本并消除某些构建异味，尽管它们偶尔也会引入新的异味。
我们进一步观察到，大多数由AI代理生成的PR在极少的人
工干预下就被合并，这表明了实践中对该技术的早期信任和采纳。
展望未来，我们的研究将探索能感知异味和以重构为导向的AI代理行为，将分析扩展到更多的构建系统，
并集成自动化的质量检查，以防止通过自动化的构建修改而累积代码问题。

### **各章节详解**

#### **第一章：引言 (Introduction)**
本章首先明确了研究的动机和重要性。在软件工程领域，大型语言模型（LLM）和AI编码代理正被迅速采纳。
虽然已有大量研究关注AI生成的*源代码*的质量、正确性和可维护性，但对于同样至关重要的*构建系统*（如Maven, Gradle, Make）
所受的影响，学术界却鲜有探索。构建系统是自动化依赖管理、编译、打包和部署的核心，其代码质量直接影响整个软件生命周期的效率和稳定性。
本研究旨在填补这一空白，通过实证分析来回答以下三个核心问题（Research Questions, RQs）：
1.  **RQ1**: AI编码代理是否会生成存在“代码异味”（code smells）的构建代码？这些异味是代码中潜在质量问题的标志。
2.  **RQ2**: AI编码代理在多大程度上能够通过重构来移除构建代码中已有的异味？
3.  **RQ3**: 开发者对AI代理自动生成的、与构建相关的代码变更（以Pull Request形式提交）的接受程度如何？

通过回答这些问题，本研究首次对AI生成的构建代码质量进行了实证评估，为理解AI在软件开发全流程中的影响提供了新的视角。

#### **第二章：背景 (Background)**
这一章为后续的分析提供了理论和技术基础，主要介绍了四个核心概念：
1.  **构建系统 (Build Systems)**: 解释了构建系统是现代软件开发的关键组成部分，它们通过配置文件（如Maven的`pom.xml`，Gradle的`.gradle`文件）
来自动化管理项目的构建过程。本研究聚焦于Maven, Gradle, CMake, 和Make这四种主流的构建系统。
2.  **代码异味 (Code Smells)**: 将代码异味定义为代码中预示着深层设计问题的“症状”。这些异味虽然不一定是程序错误（bug），
但会增加未来维护的难度或引入错误的风险。
3.  **重构 (Refactoring)**: 定义为在不改变代码外部行为的前提下，调整其内部结构以提高可读性、可维护性和设计质量的过程。
重构是消除代码异味、偿还技术债的主要手段。
4.  **Sniffer**: 介绍了Sniffer，这是一个专门用于检测构建代码中异味的静态分析工具。
它可以识别出多种与可维护性和安全性相关的异味，例如“硬编码的依赖”、“不安全的URL”和“过时的依赖”等，并报告其严重等级。
这个工具是本研究进行代码质量评估的核心。

#### **第三章：相关工作 (Related Works)**:
本章将当前研究置于现有学术背景之下，回顾了两个主要领域的相关文献：
1.  **构建维护 (Build Maintenance)**: 这一领域的研究强调了随着软件系统日益复杂，维护构建脚本的重要性。
已有工作识别了构建系统中的技术债类型（如库限制和依赖管理问题），并提出了一个构建代码的重构分类体系。
然而，这些研究都未涉及AI在其中的作用。
2.  **基于AI的代码生成 (AI-based Code Generation)**: 此前的研究主要评估AI生成*源代码*的可用性、是否存在漏洞以及代码质量问题。
例如，有研究探讨了GitHub Copilot引入安全漏洞的风险，以及ChatGPT生成的代码质量。
本章指出，所有这些相关工作都局限于源代码层面，而忽略了构建代码。因此，本研究是第一个系统性评估AI代理生成的*构建代码*质量的实-
证研究，具有开创性。

#### **第四章：方法论 (Methodology)**
本章详细阐述了研究的设计和实施步骤，确保了研究的科学性和可复现性。
1.  **数据集 (Dataset)**: 研究使用了`AIDev`数据集，这是一个包含约93.3万个由AI代理（如OpenAI Codex, GitHub Copilot）
在真实GitHub项目中生成的PR的大规模数据集。
2.  **数据预处理 (Data Preprocessing)**: 研究团队首先从`AIDev`数据集中筛选出那些修改了构建系统配置文件的PR。
经过初步自动筛选和随后的人工核查（以排除噪声和误报），最终确定了387个与构建相关的PR作为分析对象，
这些PR共涉及945个构建文件的修改。
3.  **构建代码质量评估 (Build Code Quality Assessment)**: 为了回答RQ1和RQ2，研究者使用`Sniffer`工具对每个被修改的构建文件进行分析。
他们比较了AI修改*前*和修改*后*两个版本的代码，以确定AI的变更是**引入了新的异味**、**移除了已有的异味**，还是**没有改变**。
这个对比量化了AI对构建代码质量的影响。
4.  **定性分析 (Qualitative Analysis)**:
    *   为了深入理解RQ2（AI如何移除异味），研究者手动检查了那些质量得到改善的PR，分析其变更内容、提交历史和开发者的讨论，
    以确定异味是通过何种重构手法被消除的。
    *   为了回答RQ3（开发者的接受度），研究者分析了所有387个PR的最终状态（是否被合并）以及合并前的审查历史。
    他们关注开发者是否进行了修改、是否有正面或负面的评论，从而评估开发者对AI生成内容的信任度和接受度。

#### **第五章：结果 (Results)**
本章详细报告了实证分析的发现，直接回应了引言中提出的三个研究问题。
- **RQ1: AI生成的构建代码质量如何？**
  研究发现，在945个被修改的构建文件中，有66个文件在AI修改后引入了新的代码异味，总计364个。
  这些异味主要与**可维护性**相关，其中最常见的三个是：
  1.  **通配符依赖 (Wildcard Usage)**: 使用 `+` 或 `*` 来指定依赖版本，可能导致构建不稳定。
  2.  **缺少错误处理 (Lack of Error Handling)**: 构建脚本中缺乏对潜在失败情况的处理。
  3.  **过时/已弃用的依赖 (Outdated/Deprecated Dependencies)**: 依赖了不再维护或有已知问题的库版本。
  在安全性方面，“硬编码的路径/URL”也时有出现。不同AI代理引入异味的频率差异显著，其中Copilot引入的异味最多。

- **RQ2: AI在多大程度上能消除代码异味？**
  分析发现，AI的变更也带来了积极影响。在31个构建文件中，共有54个代码异味被成功移除。
  通过手动分析，研究者发现约65%的异味消除是源于有目的性的、提升质量的**重构**行为。
  AI代理执行的重构类型包括：
  1.  **移除未使用/冗余代码 (Remove Unused/Redundant code)**: 清理了不再需要的依赖项。
  2.  **提取并上移变量 (Extract and Pull Up Variable)**: 将分散的配置（如版本号）集中管理。
  3.  **属性外部化 (Externalize Properties)**: 将硬编码的敏感信息（如密码）或可变配置（如URL）移到外部属性文件中，
  从而消除了安全异味并提高了可移植性。

- **RQ3: 开发者对AI生成的PR接受度如何？**
  结果显示，开发者对AI生成的构建代码变更接受度非常高。在分析的387个PR中，有**61.4%**（238个）被项目维护者**批准并合并**。
  更重要的是，大多数合并过程非常迅速，通常在自动化检查通过后就立即合并，几乎没有人工修改。
  审查评论中充满了积极反馈，如“LGTM (Looks Good To Me)”或“你太棒了！可以抢我饭碗了”。
  这表明开发者普遍认为AI生成的构建代码变更既高效又值得信赖。

#### **第六章：讨论 (Discussion)**
本章对研究结果进行了深入解读，并探讨了其对开发者的实际意义。
核心观点是，AI在修改构建代码时表现出一种**双重性**：它既能像有经验的开发者一样通过重构来改善代码质量，
也可能无意中引入新的技术债。这种不一致性凸显了**开发者监督（human-in-the-loop）的必要性**。
尽管AI生成的代码可能看起来“足够好”并且被迅速接受，但开发者仍需保持警惕，仔细审查变更，以防止潜在问题的累积。
研究结果同时指向了一个重要的未来方向：需要改进AI模型的训练和引导方式，使其能够产出持续高质量的重构代码，
而不是偶尔引入问题。

#### **第七章：有效性威胁 (Threats To Validity)**
本章诚实地评估了研究中可能影响结论可靠性的几个因素：
- **外部有效性威胁**: 使用的`AIDev`数据集虽然规模大，但可能无法完全代表所有AI编码场景，结论的普适性可能受限。
研究只关注了四种构建系统，结果可能不适用于其他系统。
- **内部有效性威胁**:
  - `Sniffer`工具作为静态分析器，可能存在**假阳性**（将没问题的代码识别为异味）或**假阴性**（未能识别出所有异味）。
  不过，文献表明其准确率较高。
  - 对PR的手动分析过程（用于RQ2和RQ3）可能带有**主观性**。为了缓解这一问题，研究采用了多位研究者独立标注并计算“评估者间信度”
  （inter-rater agreement）的方法，结果显示一致性很高，增强了结论的可靠性。

#### **第八章：伦理考量 (Ethical Considerations)**
本章简要说明了研究遵循的伦理准则。所有分析的数据均来源于**公开的GitHub项目**，属于公共领域信息。
研究在报告结果时采用了**聚合**的方式，没有透露任何可能识别到个人或私有项目的具体信息，充分保护了个人隐私和数据安全。

#### **第九章：结论与未来工作 (Conclusion & Future Work)**
本章对整个研究进行了总结，并展望了未来的研究方向。
- **结论**: 本研究首次实证分析了AI生成构建代码的质量，发现AI代理既能通过有意义的重构移除代码异味，
也可能引入新的可维护性和安全性问题。开发者对这些AI生成的变更表现出高度的信任和采纳率。
- **未来工作**: 基于以上发现，研究者提出了三个未来方向：
  1.  **开发能感知异味和指导重构的AI代理 (Smell-aware and refactoring-guided AI agent)**: 让AI不仅能写代码，更能理解和改善代码质量。
  2.  **将分析扩展到更多构建系统 (Extend analyses to additional build systems)**: 验证在其他技术栈中是否也有类似发现。
  3.  **集成自动化质量检查 (Integrate automated quality checks)**: 在AI生成代码的工作流中加入自动质检环节，
  防止带有异味的代码被轻易合并，从而有效控制技术债的累积。
