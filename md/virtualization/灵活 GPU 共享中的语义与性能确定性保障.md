# 论文解析：灵活 GPU 共享中的语义与性能确定性保障

---

## 第一部分：论文整体骨架

### 论文信息
- **文件名**：2603.15042v2.pdf
- **英文标题**：Guaranteeing Semantic and Performance Determinism in Flexible GPU Sharing
- **中文标题**：灵活 GPU 共享中的语义与性能确定性保障

### 中文摘要
GPU 共享对于最大化现代数据中心的硬件利用率至关重要。然而，现有方法面临严峻权衡：粗粒度的时间复用（temporal 
multiplexing）会导致交互式服务的尾延迟剧增，而细粒度的空间分区（spatial partitioning）通常需要侵入性的内核修改，
从而破坏了行为等价性。

本文提出了 DETSHARE，一个优先考虑确定性和透明性的新型 GPU 共享系统。DETSHARE 确保了语义确定性（未修改的内核
产生相同结果）和性能确定性（可预测的尾延迟），同时保持完全透明性（零代码修改）。DETSHARE 引入了 GPU 协程
（GPU coroutines），这是一种将逻辑执行上下文与物理 GPU 资源解耦的新抽象。这种解耦通过轻量级上下文迁移实现了灵活、
细粒度的资源分配。

评估表明，与时间共享相比，DETSHARE 将训练吞吐量提高了多达 79.2%。在混部场景下，它优于最先进的基准方案，在不牺牲
吞吐量的前提下将 P99 尾延迟降低了 15.1%。此外，通过感知负载的任务放置和 TPOT-FIRST 调度策略，DETSHARE 将平均
推理延迟降低了 69.1%，并将单 Token 生成时间（TPOT）的 SLO 违规率相对于默认策略降低了 21.2%。

### 各章节主旨内容

1. **引言 (Introduction)**：分析 AI 应用对 GPU 资源的巨大需求与极低利用率之间的矛盾，指出现有时间/空间共享技术
在语义确定性、性能预测和透明性方面的三大挑战。
2. **背景与动机 (Background and Motivation)**：回顾 AI 模型推理/训练的特点，阐述现有 GPU 共享机制（如 MPS、MIG）
及其局限性，深入探讨浮点运算非结合性导致的数值偏差。
3. **GPU 共享机制的挑战 (Challenges of GPU Sharing Mechanisms)**：详细定义三大挑战：语义确定性（数值与统计发散）、
性能确定性（微架构竞争与调度无感）以及透明性（工程税与验证爆炸）。
4. **DETSHARE 设计 (DETSHARE Design)**：介绍系统核心架构，包括虚拟上下文（vCtx）、物理上下文（pCtx）以及 
GPU 协程抽象。详细说明动态绑定、协作式抢占、状态管理与优化迁移机制。
5. **实现细节 (Implementation)**：描述 DETSHARE 的代码实现（C++/CUDA），解释通过 `LD_PRELOAD` 实现的透明拦截层和
全局调度守护进程。
6. **评估 (Evaluation)**：在 A800 和 H20 GPU 上验证训练吞吐量、推理延迟、SLO 合规性以及系统开销。
7. **相关工作 (Related Work)**：对比时间共享、空间共享、LLM 调度以及内存管理方面的现有研究。
8. **结论 (Conclusion)**：总结 DETSHARE 在解决 GPU 共享确定性与利用率矛盾方面的贡献。

### 中文结论
DETSHARE 成功解决了现代 GPU 共享中的核心矛盾。通过引入 GPU 协程这一抽象，它在物理资源分配波动时依然能保持计算的
数学等价性，从而保证了语义确定性。同时，通过轻量级的协作式抢占和感知 SLO 的调度算法，它在提高硬件利用率的同时，
为延迟敏感型任务提供了极强的性能保障。最重要的是，DETSHARE 实现了完全的软件透明性，无需修改任何内核或模型代码，
使其具备极高的工业部署价值。

## 第二部分：章节详解

### 1. 深度分析：GPU 共享的“三大陷阱” (C1, C2, C3)

#### 设计思想与痛点分析
作者敏锐地观察到，尽管工业界迫切需要 GPU 共享来平抑 LLM 推理中 Prefill（计算密集）和 Decode（访存密集）阶段的
资源波动，但现有的空间共享方案（如内核原子化 Kernel Atomization）存在一个致命缺陷：它们通过动态重塑内核（Kernel 
Reshaping）来适应碎片化的资源。

- **语义确定性陷阱 (C1)**：这是深度学习训练中最微妙的痛点。现有的细粒度共享往往通过改变 Batch Size 或修改内核
执行顺序（违反浮点加法结合律）来实现共享，这会导致数值发散（Numerical Divergence）。对于强化学习（RLHF）等对
一致性极度敏感的场景，微小的预测偏差可能导致策略漂移。
- **性能确定性陷阱 (C2)**：标准 GPU 硬件调度器是“吞吐量导向”的，对高层 SLO 无感。一个庞大的后台训练内核可能会
产生头端阻塞（Head-of-line blocking），导致前端推理请求出现数倍的延迟抖动。
- **透明性陷阱 (C3)**：任何需要重写内核代码的方案都会面临“验证爆炸”的问题。在模型迭代极快的背景下，手动优化内核
的工程税（Engineering Tax）高得无法接受。

### 2. 核心技术：GPU 协程 (GPU Coroutines)

#### 设计思想
DETSHARE 的核心思想是**逻辑与物理的彻底解耦**。它不试图去修改内核代码来适应资源，而是通过“协程”的方式，在内核
边界处进行透明的物理资源切换。

#### 关键组件拆解
- **vCtx (Virtual Context)**：面向应用的持久句柄。应用视角下的 CUDA 上下文是静止且完整的。
- **pCtx (Physical Context)**：硬件支撑的临时容器，具有严格的 SM（流处理器）配额限制（如 25%、50% 等）。
- **映射机制 (Dynamic Binding)**：DETSHARE 维护一张绑定表。当拦截到 `cuLaunchKernel` 时，调度器根据当前负载
决定将 vCtx 绑定到哪个 pCtx 上。

#### 确定性保障逻辑
作者提出了“抽象即确定性”的原则。当资源可用性波动时，DETSHARE **不对内核进行原子化切分**，而是采用“迁移即产出”
（Migrate-to-yield）机制。计算函数始终保持 $Result = F(D, S)$，其中 $F$ 是不可变的算术逻辑，而 $R(t)$ 
（瞬时资源状态）仅影响执行速度，不影响执行逻辑。

### 3. 实现细节：协作式抢占与 RCK 注入

#### 协作式抢占 (Cooperative Preemption)
传统的硬件级强行中断会导致上下文切换开销巨大。DETSHARE 创新地使用了一种**驻留控制内核 (RCK, Resident Control 
Kernel)**。
- **RCK 机制**：向每个活动的 pCtx 中注入一个极轻量级的单线程内核（占用 SM 资源 < 0.1%）。
- **信号传递**：当调度器需要回收资源时，它在共享内存中设置一个标志位。RCK 检测到该标志后，触发协作式放弃资源。
- **粒度控制**：利用 CUDA Stream 的依赖关系，在内核边界或指令块边界处暂停执行。这保证了状态迁移时数据是一致的。

#### 状态管理优化
为了降低迁移延迟，DETSHARE 实现了三种技术：
1. **精细化使用追踪**：不再快照整个 GPU 内存，而是监控“脏页”和内存访问模式，仅同步必要的活跃内存足迹。
2. **内省式输入分析**：通过分析内核启动参数中的指针，推导计算所需的最小内存闭包。
3. **需求驱动的延迟拷贝**：仅同步关键执行状态，剩余工作集在恢复执行后按需异步迁移。

### 4. 调度策略：感知 LLM 语义的 TPOT-FIRST

#### 设计权衡 (Trade-offs)
在 LLM 推理中，有两个关键指标：TTFT（首字延迟）和 TPOT（单 Token 生成时间）。
- **DEFAULT 策略**：平衡资源公平性。
- **TPOT-FIRST 策略**：作者发现，一旦 Token 生成开始，保持平滑的生成节奏对用户体验至关重要。该策略会优先保障 
Decode 阶段的资源配额，甚至不惜稍微推迟新请求的 Prefill 任务（增加 TTFT）。

#### 对比结果
实验显示，在极端爆发性流量（BurstGPT 负载）下，TPOT-FIRST 策略虽然让 TTFT 的 SLO 违规率从 4.33% 上升到 6.37%，
但将 TPOT 的 P90 延迟从 847.7ms 骤降至 262.9ms。这种权衡有效地隔离了突发流量对正在进行的生成任务的干扰。

### 5. 实验验证与对比分析

#### 吞吐量与延迟
- **训练场景**：在双 BERT-Large 任务中，DETSHARE 的归一化吞吐量远超 MPS 和 MIG。这是因为 DETSHARE 能利用
“执行气泡”（Execution Bubbles），在毫秒级间隙识别并填补空闲插槽。
- **混部场景**：将 ResNet 训练与 Llama-3 推理混部时，DETSHARE 将推理的 P99 延迟降至 4.5 个单位，而 Orion 为 12.0，
MIG 为 10.0。

#### 系统开销
DETSHARE 的开销极低：
- **上下文切换开销**：仅约 4%。
- **抢占开销**：约 12%，主要源于 RCK 的同步成本。考虑到其带来的 4.7 倍尾延迟收益，这被认为是极具性价比的交换。

可以在无需修改硬件或编译器的情况下，达成 GPU 共享的“确定性”圣杯。结论部分强调的“零代码修改”是其能够直接支持
