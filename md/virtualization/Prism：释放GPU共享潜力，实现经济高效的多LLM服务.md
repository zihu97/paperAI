## 各章节详解

### 第1章：引言 (Introduction)

本章首先指出了当前大型语言模型（LLM）推理服务，特别是对于需要托管成千上万个模型的云服务提供商而言，
面临着巨大的成本压力。为了保持竞争力和盈利能力，降低推理成本同时保证服务质量（如延迟目标SLO）成为核心议题。
论文通过对生产数据的分析，总结出多LLM服务场景下的四个关键工作负载特征：
1.  **长尾模型流行度 (Long-tail model popularity)**：少数热门模型占据了绝大部分请求，而大量模型（尾部模型）的请求量非常稀疏。
2.  **频繁的空闲周期 (Frequent idle periods)**：即便是热门模型，也经常出现没有请求的空闲时间窗口。
3.  **快速的工作负载波动 (Rapid workload fluctuations)**：模型的请求速率变化迅速且剧烈，可能在短时间内出现数倍的增长。
4.  **多样的服务等级目标 (Diverse service-level objectives, SLOs)**：不同模型或用户场景具有不同的延迟要求，从几秒到十几秒不等。

这些特征表明，为每个模型静态分配专用GPU会导致严重的资源浪费。
因此，通过GPU共享来提高利用率成为一个极具吸引力的优化方向。
然而，现有GPU共享技术（如静态分区或固定策略）难以适应这种动态变化的工作负载，
从而无法同时实现成本效益和SLO保证。
本章明确提出，核心挑战在于**缺乏跨模型的内存协调机制**，这阻碍了资源的灵活按需分配。
Prism系统正是为了解决这一挑战而设计的。

### 第2章：背景 (Background)

本章为后续的技术讨论铺垫了必要的背景知识。
-   **LLM推理 (LLM Inference)**：一个完整的LLM推理过程包含两个阶段。
    -   **预填充 (Prefill)**：处理用户输入的提示（prompt），并生成第一个输出令牌（token）。这个阶段计算量大，但通常只做一次。
    -   **解码 (Decode)**：自回归地生成后续的token，每次生成一个。这个阶段计算量小，但访存密集，且迭代次数不确定。
-   **KV缓存 (KV Cache)**：在LLM推理过程中，会产生大量的中间状态数据（Key和Value的张量），
    这些数据必须保存在GPU内存中以供后续token生成使用，其大小与序列长度成正比，是GPU内存的主要消耗者。
-   **PagedAttention**：这是一种先进的KV缓存管理技术，它借鉴了操作系统中的虚拟内存和分页思想。
    它将GPU内存划分为固定大小的物理页（page），并为每个请求按需分配页面来存储KV缓存，
    从而解决了内部碎片化问题，显著提高了GPU内存的利用率。
    PagedAttention已成为vLLM、SGLang等主流推理引擎的核心组件。

### 第3章：动机 (Motivation)

本章深入分析了现有GPU共享方案的不足，从而引出Prism的设计动机。
通过对来自Hyperbolic、Novita AI和Chatbot Arena的四个真实生产环境数据轨迹的分析，
论文揭示了工作负载的动态性对系统性能的巨大影响。
-   **工作负载特征的量化分析**：通过图表（图2）直观展示了长尾分布、空闲周期和请求速率的剧烈波动（用变异系数CV衡量）。
    这些数据证实了引言中提出的观点，即固定、静态的资源分配策略是低效的。
-   **现有方法的局限性**：论文通过一个具体的双模型工作负载场景（图3）对比了几种典型的共享方法：
    -   **静态分区 (Static Partition)**：如NVIDIA MIG，资源隔离死板，一个模型无法使用另一个模型的空闲资源，利用率低。
    -   **MuxServe**：允许多个模型共享GPU，但模型一旦加载便常驻内存，即使空闲也占用资源，无法将资源转移给更需要的模型。
    -   **QLM**：通过模型换入换出实现时间共享，但其交换开销巨大（表现为KV缓存大小周期性跌落至零），容易导致SLO违规。

结论是，一个理想的系统必须能够**灵活地结合空间共享和时间共享**，并**根据实时的负载和SLO需求动态调整策略**。
这需要一个高效、灵活的跨模型内存协调机制，而这正是现有系统所缺失的。

### 第44章：系统概述 (Overview)

本章描绘了Prism系统的整体架构和核心设计思想（图5）。
Prism旨在通过释放GPU共享的全部潜力，来提升多LLM服务的成本效益和SLO达成率。
其架构主要包括：
-   **前端 (Frontend)**：接收来自所有LLM应用的推理请求，并将其路由到对应的后端LLM进行处理。
-   **灵活的共享策略**：Prism不采用固定的共享模式，而是动态组合空间共享（多个低负载模型共置于一个GPU）
    和时间共享（将空闲模型换出到CPU DRAM，在需要时快速换回）。
-   **两大核心设计**：为了实现上述灵活性和需求感知能力，Prism引入了两个关键设计：
    1.  **按需内存分配机制 (§5)**：这是实现灵活性的基础。它允许模型根据实际需求动态获取和释放内存，
        为快速的策略调整和模型激活提供了可能。
    2.  **两级需求感知调度 (§6)**：这是实现高效资源协调的核心。
        -   **全局调度器**：在集群的所有GPU上进行模型布局，目标是平衡各GPU的负载压力。
        -   **本地调度器**：在单个GPU内部，通过一个共享的请求队列和基于优先级的准入控制，来协调共置模型的内存竞争。

通过这种设计，Prism能够持续地根据实时工作负载调整其共享策略，从而在保证SLO的前提下最大化资源利用率。

### 第5章：实现灵活的GPU共享 (Enabling Flexible GPU Sharing)

本章深入探讨了Prism如何实现其核心的内存灵活性。
-   **5.1 跨模型内存协调 (Cross-Model Memory Coordination)**：
    -   **问题**：传统推理引擎（如vLLM）在初始化时为每个模型静态预留一大块GPU内存，这阻碍了内存的跨模型动态再分配。
    -   **解决方案: `kvcached`**：Prism设计了一个名为`kvcached`的“垫片层”（shim layer）。
        它位于PagedAttention之下，利用CUDA的虚拟内存管理API（VMM）将虚拟地址空间与物理内存解耦。
        推理引擎在启动时只保留一个大的虚拟地址空间，给人一种内存可用的“错觉”。
        而物理内存页则在真正需要时（如分配KV缓存）才被动态分配并映射到虚拟地址上。
        这种设计使得Prism能够以极小的代码修改（约20行）就兼容现有引擎，同时实现按需内存分配。
        `kvcached`还通过内存池和页面复用等优化来降低运行时开销。
-   **5.2 快速模型激活 (Fast Model Activation)**：
    -   **问题**：模型交换（特别是重新激活）的速度直接影响共享策略的灵活性。传统的激活过程包括初始化引擎和从CPU加载模型权重，
        耗时可达数十秒，远超在线服务的SLO要求。
    -   **解决方案**：
        1.  **可复用引擎池 (Reusable engine pools)**：Prism解耦了引擎和模型的生命周期。
            它在每个GPU上维护一个预先初始化好的“引擎池”。当需要激活一个模型时，直接从池中取出一个热引擎，
            避免了引擎初始化的开销。模型被换出时，其引擎被回收到池中供未来使用。
        2.  **并行模型权重加载 (Parallel model weight loading)**：为了加速从CPU到GPU的数据传输，
            Prism将模型权重分块，并通过多个GPU的NVLink高速互联进行并行加载，然后聚合到目标GPU上。
            这种方法显著利用了节点内的高带宽互联，相比传统的`cudaMemcpyAsync`有数倍的提速。

通过这两项技术，Prism能够将一个8B模型的激活时间缩短到0.7秒，为动态、灵活的调度策略提供了坚实的基础。

### 第6章：两级需求感知调度 (Two-Level Demand-Aware Scheduling)

本章介绍了Prism如何利用第5章建立的灵活性，来制定高效的资源共享策略。其调度系统分为两个层面：
-   **6.2 全局模型放置调度 (Global Model Placement Scheduling)**：
    -   **目标**：决定每个模型应该被放置在哪个GPU上，以实现整个集群的负载均衡，避免出现部分GPU过载而部分空闲的情况。
    -   **核心启发式指标：KV压力比 (KV Pressure Ratio, KVPR)**：
        为了量化GPU的“拥堵”程度，Prism定义了KVPR = (SLO加权请求速率) / (可用KV缓存内存)。
        这个指标同时考虑了模型的需求量（请求速率）和需求的紧迫性（SLO），值越高代表GPU压力越大。
    -   **算法**：全局调度器周期性地运行一个贪心算法（算法1）。它首先按加权请求率对所有模型排序，
        然后依次为每个模型选择能使其加入后KVPR最小的GPU。
        为了避免因微小收益而频繁迁移模型，还设置了一个迁移阈值。
    -   **模型换出与激活**：当模型长时间空闲，调度器会将其换出（evict）。当被换出的模型收到新请求时，
        调度器会立即为它选择一个当前KVPR最低的GPU来重新激活。
-   **6.3 GPU本地请求调度 (GPU-Local Request Scheduling)**：
    -   **目标**：当多个模型被放置在同一个GPU上时，协调它们对有限KV缓存的竞争，保证高优先级的请求能及时得到服务。
    -   **机制**：Prism在每个GPU上设立一个**共享请求队列**。所有进入该GPU的请求都先放入此队列。
        一个基于**Moore-Hodgson算法**的调度器（算法2）负责从队列中取出请求并派发给对应的模型引擎。
        该算法的核心思想是，优先处理**截止时间（arrival_time + TTFT_SLO）**早的请求。
        如果接纳一个新请求会导致无法在截止时间内完成，调度器会从当前已接纳的请求中，“牺牲”掉那个**预估执行时间最长**的请求，
        从而为更多“短”请求腾出空间，以最大化总体SLO达成率。

这种两级调度设计，使得Prism既能从宏观上平衡集群负载，又能在微观上精细化管理资源竞争，从而实现高效且SLO感知的服务。

### 第7章：实现与评估 (Implementation and Evaluation)

本章详细描述了Prism的实验设置和评估结果，以验证其有效性。
-   **7.1 实现 (Implementation)**：Prism的原型基于Python和C++构建，后端使用了广泛采用的SGLang推理引擎。
    `kvcached`库通过CUDA VMM API实现，并以Python绑定的形式供上层调用。
    前端使用Redis队列缓存请求，全局调度器作为一个独立的Python进程，通过ZeroMQ与各引擎通信。
-   **7.2 实验设置 (Experimental Setup)**：
    -   **测试平台**：一个包含4个节点、每个节点8张NVIDIA H100-80G GPU的集群。
    -   **基线系统**：
        1.  **静态分区 (S-Partition)**：模拟MIG或固定内存划分。
        2.  **MuxServe++**：是MuxServe的一个增强版，移植到了SGLang上并集成了Prism的按需内存分配能力，以进行更公平的比较。
        3.  **QLM**：一个代表性的时间共享系统。
    -   **工作负载**：使用了§3中分析的Hyperbolic和Arena-Chat两个真实轨迹，并对其进行缩放以模拟不同负载水平。
-   **7.3-7.5 评估结果**：
    -   **端到端性能 (图7)**：在不同请求速率、SLO要求和GPU数量下，Prism在TTFT（首token时间）和TPOT（每token时间）
        的SLO达成率上始终显著优于所有基线。例如，在Hyperbolic轨迹上，Prism能比MuxServe++多处理2.3倍的请求，
        同时保持99%的SLO达成率。
    -   **性能分析 (图8-11)**：通过一系列微基准测试（ablation studies），论文分别验证了Prism各个组件的有效性：
        -   **灵活的跨模型内存协调**：展示了按需内存分配相比静态分区，能够在模型间动态转移内存，从而提升吞"吐量。
        -   **模型激活速度**：证明了可复用引擎池和并行加载技术可以将模型激活延迟降低一个数量级（例如，8B模型从7.1s降至1.3s）。
        -   **全局和本地调度**：分别展示了启用全局调度器能有效平衡GPU负载，避免热点；
            启用本地调度器能通过优先级管理显著提升高优（严格SLO）模型的SLO达成率。
    -   **大规模部署 (图12)**：在一个包含58个模型和32个GPU的大规模场景下，
        Prism仅用16个GPU就达到了99%的TTFT SLO达成率，而MuxServe++需要32个GPU，
        其他基线则需要更多或根本无法达到。这直观地展示了Prism在**节省成本**方面的巨大优势（超过2倍）。

总的来说，评估结果强有力地证明了Prism的设计能够在真实多LLM服务场景中，实现性能和成本的大幅优化。
