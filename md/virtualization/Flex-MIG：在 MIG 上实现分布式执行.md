`2511.09143v2.pdf` - `Flex-MIG: ENABLING DISTRIBUTED EXECUTION ON MIG`

**中文标题**
Flex-MIG：在 MIG 上实现分布式执行

**中文摘要**
在多租户环境中，GPU 集群常因资源未被充分利用而效率低下，因此 GPU 共享技术至关重要。其中，NVIDIA 的多实例 GPU (MIG) 技术因其提供硬件级隔离、支持并发工作负载无干扰而备受关注。然而，MIG 的硬件刚性与传统的一对一分配模型共同导致了严重的资源碎片化和集群范围的低利用率。本文提出了 Flex-MIG，这是一个纯软件框架，它用一对多分配模型取代了一对一模型，并实现了跨 MIG 实例的主机共享内存集合通信，且无需硬件修改。Flex-MIG 消除了需要清空 GPU 的重新配置过程，减少了资源碎片，并在多种追踪测试中将任务完成时间（makespan）缩短了高达 17%。这表明，通过将 MIG 的操作模型重塑为一个软件协调层，可以显著提高集群效率。

**各章节主旨内容**
*   **第1章 引言 (INTRODUCTION)**：介绍了在多租户 GPU 集群中高效利用资源的核心挑战，指出尽管 MIG 提供了硬件隔离优势，但其固定配置和传统分配方式导致了利用率低下和碎片化问题，并引出 Flex-MIG 作为解决方案。
*   **第2章 背景与动机 (BACKGROUND & MOTIVATION)**：详细阐述了 MIG 的工作原理、优势（硬件隔离）和四大核心限制（固定的实例配置、树状合并约束、无跨 GPU 聚合、需要清空节点的重新配置）。本章分析了这些限制在传统一对一分配模型下如何导致过量分配、内外碎片化和高昂的重新配置成本。最后，提出了转向“一对多”分配模型的基本原理。
*   **第3章 建议的方法 (PROPOSED APPROACH)**：介绍了 Flex-MIG 的三层（用户/应用层、编排层、运行时层）系统架构。核心思想是将所有 GPU 静态划分为最小单位的 MIG 实例，并通过一个智能的编排层将多个实例逻辑上聚合起来分配给单个作业，从而实现灵活的资源分配。
*   **第4章 实现 (IMPLEMENTATION)**：描述了 Flex-MIG 的具体实现细节。编排层基于 Kubernetes 实现，负责调度和分配。关键的创新在于运行时层，通过修改 NCCL 通信库，引入了 MIG 感知的对等发现和合成总线 ID 标记机制，从而在同一主机上的不同 MIG 实例间（即使在同一物理 GPU 上）实现了高效的基于主机共享内存 (SHM) 的集合通信。
*   **第5章 评估 (EVALUATION)**：通过一个校准过的模拟器和真实环境的实验，将 Flex-MIG 与两种基线模型（动态 MIG 和静态 MIG）进行对比。结果显示，Flex-MIG 在各种工作负载下显著降低了作业等待时间，将集群总完成时间缩短了高达17%，并提高了整体资源利用率。
*   **第6章 相关工作 (RELATED WORK)**：讨论了与 GPU 共享和虚拟化相关的其他技术，如软件层面的共享（MPS, TGS）和在 MIG 之上动态组合的其他方案（MIGER, ParvaGPU），并阐明了 Flex-MIG 在将 MIG 从静态分区器转变为可组合的运行时资源层这一独特思路上的不同之处。
*   **第7章 结论 (CONCLUSION)**：总结了 Flex-MIG 的贡献：它通过创新的“一对多”软件定义模型，将物理上刚性的 MIG 实例转变为逻辑上灵活的资源池，解决了 MIG 的核心痛点，显著提升了多租户 GPU 集群的效率，同时保留了 MIG 的硬件隔离优势。

**中文结论**
我们重新审视了 MIG 一对一分配模型的根本局限性，并证明了固定的实例配置、树状合并约束以及需要清空节点的重新配置共同降低了集群的整体利用率。Flex-MIG 引入了一种纯软件的“一对多”模型，该模型将 GPU 固定划分为最小的“叶子”实例，并为每个作业聚合多个实例。通过将主机内的所有 MIG 资源“扁平化”，Flex-MIG 实现了跨实例的逻辑聚合，解决了资源碎片化问题，并消除了物理重新配置的需求。它还扩展了 NCCL 以支持 MIG 实例间的共享内存集合通信，从而在保持隔离的同时实现了高效的内部 GPU 通信。我们通过真实测量验证的评估表明，与动态和静态 MIG 基线相比，Flex-MIG 将任务完成时间缩短了高达17%，且每个作业的开销适度。这些结果凸显出，将 MIG 重新思考为一个逻辑上可组合、由软件管理的资源层，可以在多租户 GPU 集群中释放可观的效率收益。

**第1章 详解：引言 (INTRODUCTION)**
本章首先指出了问题的背景：随着 AI 工作负载需求的激增，在多租户（即多个用户或任务共享）GPU 集群中实现高资源利用率变得至关重要。许多常见的 AI 模型（如 ResNet）实际上无法用满一整张高端 GPU，导致了严重的资源浪费。为了解决这个问题，NVIDIA 推出了多实例 GPU (MIG) 技术，它通过硬件分区（隔离流式多处理器 SM、L2 缓存和显存）来提供比早期技术（如 MPS）更强的性能隔离。然而，作者通过测量发现，MIG 的实际应用效果并不理想。其根本原因在于 MIG 自身的硬件限制和业界通行的资源分配模式。
作者将这些问题归纳为四点硬件/操作约束 (C1-C4)：
*   **C1 - 固定配置**：MIG 只能按预设的几种规格进行分区，无法任意调整大小。
*   **C2 - 树状合并约束**：MIG 实例的合并受到其在硬件上的树状拓扑结构的限制，不是任意相邻实例都能合并。
*   **C3 - 无跨 GPU 聚合**：无法将不同物理 GPU 上的 MIG 实例组合起来给一个任务使用。
*   **C4 - 需要清空节点的重新配置**：更改 MIG 分区需要先暂停并清空目标 GPU 上的所有任务，代价高昂。
当这些约束与传统“一个任务分配一个 MIG 实例”的一对一模型 (M) 结合时，会导致三种效率低下的问题 (I1-I3)：
*   **I1 - 过量分配**：如果任务需求不完全匹配任何一种固定配置，就必须分配一个更大的实例，造成浪费。
*   **I2 - 内/外部碎片化**：由于合并和跨 GPU 聚合的限制，即使集群中有足够的空闲资源总量，也可能因为它们分散在各处而无法组合成一个足够大的实例来满足新任务的需求。
*   **I3 - 重新配置成本**：为了匹配任务需求，调度器频繁触发代价高昂的重新配置流程。
基于以上痛点，本章引出了本文的核心贡献：**Flex-MIG** 框架，旨在通过软件层面的创新，将 MIG 从一个刚性的硬件分区工具转变为一个灵活的、可组合的资源层，从而在不牺牲硬件隔离的前提下，大幅提升集群效率。

**第2章 详解：背景与动机 (BACKGROUND & MOTIVATION)**
本章深入解释了 MIG 的工作机制以及当前使用模式的低效根源。
首先，它定义了研究范围：专注于使用 MIG 的多租户 GPU 集群，目标工作负载是那些受益于空间分区（即将 GPU 硬件切分给不同任务）的中小型模型，排除了本身就需要多 GPU 或能占满单张 GPU 的超大型模型。
接着，文章图文并茂地解释了 MIG 的架构：一张 A100 GPU 可以被划分为多达 7 个独立的硬件实例，每个实例拥有自己专用的 SM、L2 缓存和显存，从而实现性能隔离。
本章的核心是 **“一对一分配模型的低效性”** 分析：
*   **过量分配 (Over-provisioning)**：通过图表（Figure 2）直观展示了当一个任务的资源需求（例如需要 3g/15GB 显存）落在两个预设配置（例如 2g.10gb 和 4g.20gb）之间时，系统只能分配那个更大的 4g.20gb 实例，图中红色的区域就代表了被浪费的计算和显存资源。
*   **资源碎片化 (Resource Fragmentation)**：分为两种情况（Figure 3）：
    *   **内部碎片化**：在同一张 GPU 内部，由于 MIG 的资源呈树状结构，即使有两个相邻的空闲实例，如果它们不属于同一个父节点，也无法合并成一个更大的实例。
    *   **外部碎片化**：在同一台服务器上，不同物理 GPU 上的空闲实例无法被聚合起来服务于一个需要更大资源的任务。
*   **需要清空节点的重新配置 (Drain-required Reconfiguration)**：在 Kubernetes 等容器化环境中，更改 MIG 配置需要借助`mig-manager`等工具，而这个过程要求目标 GPU 完全空闲。这意味着必须先暂停 GPU 上的所有任务、保存状态、执行耗时很长（作者测量为 100-120 秒）的重新配置、再重启任务，整个流程开销巨大。
最后，本章提出了解决这些问题的**基本思路（Rationale for One-to-Many）**：放弃“一对一”的思路，转而采用“一对多”。具体做法是：将集群中所有的 GPU 都预先、静态地分割成最小的单位（例如 1g.5gb），形成一个同质化的、扁平的资源池。当一个任务到来时，不再是为它寻找一个匹配的、单一的大实例，而是根据其需求，为它分配**多个**这样的小实例。通过这种方式，可以实现：
1.  **资源扁平化**：消除了不同配置的异质性，任何资源需求都可以通过组合小单元来满足，解决了过量分配问题。同时，由于是在逻辑层面聚合，打破了物理上的树状合并和跨 GPU 限制，减少了内外部碎片化。
2.  **避免重新配置**：因为所有 GPU 的分区是静态固定的，所以完全不需要进行代价高昂的物理重新配置。

**第3章 详解：建议的方法 (PROPOSED APPROACH)**
本章详细阐述了 Flex-MIG 的系统设计。Flex-MIG 是一个纯软件框架，它引入了“一对多”的分配模型，允许单个作业跨多个 MIG 实例运行，从而提高集群效率。
**系统概览 (System Overview)**：Flex-MIG 建立在一个典型的三层集群抽象之上：
1.  **用户/应用层 (User/Application Layer)**：在这一层，用户可以提交具有更精细粒度资源需求的作业（例如，5g.15gb），而无需关心底层固定的 MIG 配置。Flex-MIG 虽然不能完全精确匹配（因为硬件的最小单位是固定的），但相比原来粗粒度的配置，能更接近用户的实际需求，从而减少资源浪费。
2.  **编排层 (Orchestration Layer)**：这是 Flex-MIG 的“大脑”。它负责管理作业队列、接纳新作业，并为作业选择和分配 MIG 实例。其核心职责是决定为一个作业分配哪些、多少个、以及位于何处的 MIG 小实例。
3.  **运行时层 (Runtime Layer)**：这一层负责实际的作业执行。它创建进程，并将编排层分配的 MIG 实例绑定给这些进程。Flex-MIG 的关键创新在于，它实现了跨多个 MIG 实例的高效集合通信，这是“一对多”模型能够高效工作的先决条件。

**编排层设计 (Orchestration Layer)**：
从“一对一”到“一对多”的转变给调度器带来了新的挑战：需要决定分配哪种类型的实例（例如，A100 上有 1g.5gb 和 1g.10gb 等不同最小单元）以及如何放置这些实例（是在同一个 GPU 上还是分散在不同 GPU 上）。Flex-MIG 提出了一套包含两个启发式规则的实例选择策略：
*   **尺寸感知的实例优先级 (Size-Aware Instance Prioritization)**：通过实验发现，对于小任务（例如只用1个MIG实例），使用性能更好、显存带宽更高的实例类型（如 1g.10gb）能提升性能；但对于需要多个实例的大任务，由于跨实例同步的开销存在，性能瓶颈在于最慢的那个实例，因此混合使用不同类型的实例没有好处。基于此，Flex-MIG 的策略是：优先将 1g.10gb 这样的“优质小实例”分配给单实例作业，而将 1g.5gb 这样的“普通小实例”用于需要多个实例的作业。
*   **拓扑感知的放置策略 (Topology-Aware Placement)**：实验表明，将一个作业所需的多个 MIG 实例均匀地分布在不同的物理 GPU 上，比将它们集中在少数几个 GPU 上性能更好（因为可以更好地利用多条 PCIe链路的带宽）。因此，Flex-MIG 采用轮询 (round-robin) 策略，将实例均匀地分配到节点内的各个 GPU 上。

**运行时层设计 (Runtime Layer)**：
“一对多”模型最大的技术挑战在于实现跨 MIG 实例的高效通信。标准的 NCCL 库在设计时并未考虑这种情况，当它发现来自同一个物理 GPU 的多个 MIG 实例（它们共享相同的 PCIe 总线 ID）试图加入同一个通信组时，会误判为重复设备而报错。
Flex-MIG 通过修改 NCCL 解决了这个问题：
*   在对等发现（peer discovery）阶段，为每个 MIG 实例嵌入一个唯一的 `mig_id` 标识符。
*   修改 NCCL 的设备识别逻辑：只有当两个实例的 PCIe 总线 ID 和 `mig_id` **同时**相同时，才将它们视为同一个设备。
*   为具有相同物理总线 ID 的不同 MIG 实例分配一个**合成的（synthetic）**、唯一的总线 ID。
这些修改使得 NCCL 能够正确识别来自同一物理 GPU 的不同 MIG 实例，并能在它们之间建立基于主机共享内存（SHM）的高效通信路径。这避免了需要回退到更慢的网络传输（如 RDMA/IP），是保证 Flex-MIG 高性能的关键。

**第4章 详解：实现 (IMPLEMENTATION)**
本章介绍了 Flex-MIG 的具体实现细节，它主要由两大组件构成：
1.  **编排层 (Orchestration Layer)**：
    *   该层基于主流的容器编排平台 **Kubernetes** 实现。
    *   **调度器 (Scheduler)**：作为一个独立的 Python 程序，它通过 Kubernetes 的客户端 API 监控全局作业等待队列。当调度器决定执行一个作业时，它会根据第3章介绍的“尺寸感知”和“拓扑感知”策略，选择一组具体的、当前空闲的 MIG 实例。
    *   **作业执行器 (Job Executor)**：接收调度器传来的作业描述（包含分配好的 MIG 实例列表）。它的任务是创建一个 Kubernetes Pod 的定义文件。关键操作是设置 `NVIDIA_VISIBLE_DEVICES` 这个环境变量，将分配给该作业的所有 MIG 实例的 UUID（全局唯一标识符）都写入这个变量。这样，当 Pod 内的容器启动时，它就只能“看到”并使用这些被指定的 MIG 实例。最后，执行器通过 Kubernetes API 启动这个 Pod。

2.  **运行时层 (Runtime Layer)**：
    *   该层基于 NVIDIA 的 **NCCL** 库进行了少量源码修改，以实现 MIG 感知的集合通信。
    *   **进程级初始化**：当作业的 Pod 启动后，它会为每个分配到的 MIG 实例生成一个工作进程。每个进程通过 `LOCAL_RANK` 环境变量识别自己的身份。在容器内部，一个启动脚本会读取 `NVIDIA_VISIBLE_DEVICES` 中的 MIG UUID 列表，并将对应于当前进程的那个 UUID 导出为两个新的环境变量：`CUDA_VISIBLE_DEVICES`（这是 CUDA 运行时用来绑定物理设备的标准变量）和 `NCCL_MIG_ID`（这是 Flex-MIG 新增的、用于 NCCL 识别的变量）。这个过程确保了每个工作进程都正确地与一个 MIG 实例绑定，并且 NCCL 能获得区分不同实例所需的信息。
    *   **MIG 感知的对等发现 (MIG-aware Peer Discovery)**：在 NCCL 初始化，各个工作进程交换对等信息的阶段，Flex-MIG 增加了一个 `mig_id` 字段。每个进程将自己的 `NCCL_MIG_ID` 值填入此处。NCCL 的设备查重逻辑被修改为：只有当两个进程的 PCIe 总线 ID 和 `mig_id` 都相同时，才认为它们绑定到了同一个 GPU 实例。这解决了原生 NCCL 会因总线 ID 相同而错误中止的问题。
    *   **合成总线 ID 标记 (Synthetic Bus-ID Labeling)**：在 NCCL 构建通信拓扑时，它需要为每个 GPU 设备创建一个唯一的节点。为了解决多个 MIG 实例共享同一个物理 PCIe 总线 ID 的问题，Flex-MIG 引入了一个动态标记机制。当一个新的 MIG 实例（例如来自 Worker 3）加入，而它的总线 ID（例如 `00:48:00.0`）已经被另一个实例（例如 Worker 0）注册时，系统会为这个新的实例动态生成一个带后缀的合成 ID（例如 `00:48:00.1`）。这确保了在 NCCL 的拓扑图中，每个 MIG 实例都是一个独一无二的节点，从而可以正确建立通信路径。这个合成 ID 只在 NCCL 内部使用，在与驱动交互前会被移除。

通过这些实现，Flex-MIG 成功地让多个原本无法直接通信的 MIG 实例能够通过高效的共享内存（SHM）进行集合操作，使得“一对多”分布式执行模型在操作上成为可能且高效。

**第5章 详解：评估 (EVALUATION)**
本章通过一系列实验来验证 Flex-MIG 的有效性。由于在真实集群中反复进行需要清空 GPU 的 MIG 配置变更实验非常困难且耗时，作者采用了一种**模拟器与真实实验相结合**的方法。
**实验设置 (Setup)**：
*   **环境**：一台配备两张 NVIDIA A100-40GB PCIe GPU 的服务器。
*   **基线模型**：为了进行对比，设置了两个基线调度策略：
    1.  **Dynamic-MIG (DM)**：一种动态策略，会根据作业请求实时地重新配置 MIG 实例。但为了模拟真实服务场景，对于推理任务，禁止会中断服务的 GPU 清空操作。
    2.  **Static-MIG (SM)**：一种静态策略，将 GPU 预先划分为固定的几种常用实例组合（如 1g.10gb, 2g.10gb, 4g.20gb）。如果请求的实例类型不可用，它可以分配一个更大的空闲实例。
*   **Flex-MIG (FM)**：即本文提出的方法，将所有 GPU 预先划分为最小实例。
*   **工作负载**：使用了一系列真实的深度学习模型（如 ResNet, BERT, T5）的训练和推理任务，并构建了多种合成追踪（synthetic traces）来模拟不同场景（如小任务为主、大任务为主、均衡型）。
*   **评估指标**：主要关注五个集群级指标：总完成时间 (Makespan)、平均作业完成时间 (Average JCT)、平均等待时间、平均外部碎片化延迟、集群利用率。

**模拟器准确性验证 (Validation of Simulator Accuracy)**：
首先，作者通过在真实硬件上运行 108 个合成追踪来收集 Flex-MIG 的性能数据。然后，他们构建了一个模拟器，该模拟器使用真实测量的单个作业的 JCT 数据，并采用与真实系统完全相同的调度逻辑。通过对比模拟结果和真实测量结果，发现初始模拟器低估了约 14% 的 Makespan。这是因为模拟器未能捕捉到多个作业并发运行时产生的轻微性能干扰。为了修正这一点，作者为每个作业的执行时间引入了一个 1.06 的恒定缩放因子作为校准。校准后，模拟器预测的结果与真实世界的测量结果高度吻unt（如图 Figure 6 所示），证明了模拟器的可靠性。

**Flex-MIG 的有效性 (Effectiveness of Flex-MIG)**：
使用校准后的模拟器，作者比较了 FM、DM 和 SM 在仅训练场景下的表现（Figure 7）：
*   **等待时间**：Flex-MIG 的平均等待时间显著低于 DM 和 SM。相比于 DM，平均降低了 11%。这是因为 FM 通过逻辑聚合消除了碎片，使得作业几乎总能立即找到可用资源，无需等待物理重配或一个完整的大实例被释放。
*   **作业完成时间 (JCT)**：Flex-MIG 的单个作业 JCT 比 SM 略高，因为 SM 可能会“奢侈地”分配一个远超需求的大实例，从而使得单个任务运行得更快。但这种“慷慨”是以牺牲队列中其他任务的等待时间为代价的。相比 DM，Flex-MIG 的 JCT 平均只高出 4%，增幅很小。
*   **总完成时间 (Makespan)**：这是衡量集群吞吐量的最重要指标。Flex-MIG 在所有场景下都取得了最短的 Makespan。尤其是在大任务为主的追踪中，它比 DM 快了约 15%。原因在于，DM 在面对大任务时，常常因为资源碎片化而无法聚合足够资源，导致任务长时间等待；而 FM 则可以轻松地将分散在各处的小实例逻辑上组合起来。
在混合了训练和推理任务、并采用更激进的回填调度策略的场景下（Figure 8），Flex-MIG 依然保持了显著优势。它在 80-90% 的场景中，JCT 增幅相对于 DM 不超过 7-10%，但等待时间、Makespan 和集群利用率均有大幅优化。

**作业级分析 (Job-level Analysis)**：
*   **拓扑感知放置的重要性**：实验（Figure 9）证明，将一个作业所需的 6 个 MIG 实例均匀地分布在两张 GPU 上（每张卡 3 个）比将它们不均匀地分布（例如一张卡 4 个，另一张卡 2 个）性能更好。这验证了第3章中提出的拓扑感知放置策略的正确性。
*   **SHM vs. NET 通信**：实验（Figure 10）表明，当多个作业并发运行时，Flex-MIG 采用的基于共享内存（SHM）的通信方式，其性能受到的干扰远小于基于网络（NET-RDMA）的通信，表现出更强的稳定性和效率。

综上所述，评估章节通过坚实的数据证明了 Flex-MIG 相比现有的 MIG 使用模式，在提升集群整体吞吐量和资源利用率方面具有显著优势，且代价（单个任务性能的轻微下降）完全可以接受。

**第6章 详解：相关工作 (RELATED WORK)**
本章将 Flex-MIG 置于现有研究的背景下，通过对比来凸显其独特性和创新点。
**1. 软件层面的 GPU 共享**：
*   **NVIDIA Multi-Process Service (MPS)**：这是一个早期的 GPU 共享技术，它允许多个 CUDA 进程共享同一个 GPU 上下文，从而实现并发。但 MPS 的问题在于所有进程共享 GPU 的计算资源、L2 缓存和显存带宽，缺乏硬件层面的隔离。这意味着一个“坏邻居”进程可能会严重干扰其他进程，导致性能不稳定。
*   **TGS 和 Orion**：这些是后续研究提出的更精细的软件共享方案。它们试图通过更智能的调度策略来减少任务间的干扰。
*   **与 Flex-MIG 的对比**：这些软件共享方案的共同点是它们在**没有硬件隔离**的 GPU 上工作。而 Flex-MIG 是建立在 **MIG 提供的硬件隔离基础之上**的，它不是在单个 GPU 实例内部进行共享，而是在多个**硬件隔离的** MIG 实例之间进行协同工作。Flex-MIG 保留了 MIG 的隔离性优势，解决了这些方案的根本缺陷。

**2. 基于 MIG 的动态组合方案**：
*   **MIGER**：这是一个认识到 MIG 刚性分区问题的研究。它的策略是，在 MIG 分区的基础上，再对每个 MIG 实例内部启用 MPS，实现“多对一”（多个进程共享一个 MIG 实例）的共享，从而“回收” MIG 实例中未被充分利用的资源。MIGER 的核心是联合决策 MIG 分区大小、任务放置以及每个任务的 MPS 共享比例。
*   **ParvaGPU**：与 MIGER 类似，它也在每个 MIG 实例内部显式地启用 MPS，并通过扩展进程数量来最大化利用单个 MIG 实例的内部资源。
*   **与 Flex-MIG 的对比**：这些方法可以看作是“MIG + MPS”的混合模式。它们试图解决 MIG 实例**内部**的资源浪费问题。而 Flex-MIG 的思路完全不同，它着眼于解决 MIG 实例**之间**的组合问题。Flex-MIG 采用“一对多”模型（一个作业使用多个 MIG 实例），而 MIGER/ParvaGPU 实际上是在 MIG 的“一对一”模型基础上，通过 MPS 实现了实例内部的“多对一”共享。Flex-MIG 的核心创新在于它**重新定义了 MIG 的操作模型**：它不是把 MIG 看作一个静态的分区工具，而是把它看作一个可以**在运行时逻辑组合**的资源层。这种思路从根本上避免了因分区不匹配而导致的物理重新配置，也解决了跨实例的碎片化问题。

**本研究的定位 (Our position)**：
作者最后明确总结了 Flex-MIG 的独特定位：不同于上述所有方法，Flex-MIG 将 MIG 从一个**静态的分区器**（static partitioner）转变为一个**可组合的运行时资源层**（composable runtime resource layer）。它通过将一个作业分散到多个固定的最小 MIG 实例上执行，既保持了 MIG 的硬件隔离性，又避免了代价高昂的重新配置。这种将物理上固定的分区通过软件层赋予逻辑上的组合灵活性的思想，是 Flex-MIG 最核心的贡献。

**第7章 详解：结论 (CONCLUSION)**
本章是对整篇论文工作的总结和升华。
作者首先重申了研究的起点：NVIDIA MIG 虽然提供了宝贵的硬件隔离功能，但其固有的三大限制——**固定的实例配置、树状的合并约束、需要清空节点的重新配置**——在与传统“一对一”分配模型结合时，严重影响了整个 GPU 集群的资源利用率和吞E量。

接着，本章概括了 **Flex-MIG** 的核心贡献和解决方案。Flex-MIG 作为一个纯软件框架，提出并实现了一种创新的**“一对多”**操作模型：
1.  **资源扁平化与逻辑聚合**：通过将所有 GPU 预先、静态地分割成最小的“叶子”实例，Flex-MIG 在主机内部创建了一个同质化的、扁平的 MIG 资源池。这使得调度器可以**逻辑上**聚合任意数量、任意位置的实例来满足作业需求，从而有效解决了物理层面的资源碎片化问题，并彻底消除了对昂贵的物理重新配置的需求。
2.  **高效的跨实例通信**：为了支撑“一对多”模型的分布式执行，Flex-MIG 对 NCCL 进行了扩展，使其能够支持在不同 MIG 实例（即使在同一物理 GPU 上）之间建立基于主机共享内存（SHM）的集合通信。这一关键技术创新，使得跨实例的通信高效进行，同时完整保留了 MIG 提供的硬件隔离优势。

最后，作者强调了研究的成果和意义。通过与动态 MIG (DM) 和静态 MIG (SM) 两种基线方案的对比评估（该评估的有效性已通过真实测量数据校准和验证），结果表明 Flex-MIG 能够将集群的**总完成时间（makespan）提升高达 17%**，而对单个作业性能的负面影响（overhead）非常小。

本章的结论点明了这项工作的深远启示：我们不应将 MIG 仅仅视为一个静态的硬件分区工具，而应该重新思考其使用模式。通过将其视为一个**逻辑上可组合、由软件管理的资源层**，可以解锁其巨大潜力，在现代多租户 GPU 集群中实现显著的效率提升。
