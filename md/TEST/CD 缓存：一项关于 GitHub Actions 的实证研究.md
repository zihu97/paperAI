# 2604.13129v1.pdf - How Developers Adopt, Use, and Evolve CI/CD Caching: An Empirical Study on GitHub Actions

## 中文标题：开发者如何采用、使用和演进 CI/CD 缓存：一项关于 GitHub Actions 的实证研究

### 中文摘要
持续集成/持续交付 (CI/CD) 缓存被广泛用于减少重复计算并提高 CI/CD 效率，但维持有效的缓存需要持续的维护工作。
本文提出了第一项关于开发者如何在 GitHub Actions (GHA) 工作流中配置和演进缓存的实证研究。通过分析 952 个 GitHub
代码仓库（266 个缓存采用者和 686 个非采用者），本文比较了代码库特征，刻画了作业（job）和步骤（step）级别的缓存
用法，揭示了缓存配置演进的模式，并识别了驱动缓存相关变更的因素。分析涵盖了 1,556 个工作流文件、10,373 个提交
以及 17,185 个工作流配置变更。主要观察结果包括：(1) 采用缓存的代码库比非采用者更活跃且更受欢迎；(2) 缓存通过
多种机制而非单一标准方法应用于多种 CI/CD 作业类型；(3) 缓存配置通过频繁、重复的维护模式演进，在构建和测试作业
中更新迅速，而在其他作业类型中演进较慢；(4) 缓存相关修改受不同维护需求驱动：参数更新主要由人工驱动以修复问题，
而版本更新发生较晚且通常由机器人驱动以进行依赖维护。研究量化了 CI/CD 缓存涉及的大量维护工作，并强调了改进可靠
性和工具支持的机会。

### 各章节主旨内容
- **Chapter 1: Introduction** - 阐述了 CI/CD 缓存作为性能优化手段的重要性及其带来的隐形成本和维护负担。
- **Chapter 2: Background** - 介绍了 GitHub Actions 中的缓存机制、配置参数（path 和 key）以及维护活动术语。
- **Chapter 3: Data Collection and Preparation** - 描述了数据集的构建过程，包括元数据提取、作业类型映射和缓存
变更检测。
- **Chapter 4: RQ1: Characteristics of cache-adopting repositories** - 分析了采用缓存的代码库在活跃度、语言分布、
作业类型及缓存策略上的特征。
- **Chapter 5: RQ2: Evolution of caching configurations** - 利用马尔可夫模型量化了不同作业类型下缓存配置的动态
演进模式和时间间隔。
- **Chapter 6: RQ3: Drivers of cache-related modifications** - 构建了驱动缓存变更的分类法，区分了人工和机器人
驱动的不同动机。
- **Chapter 7: Implications** - 为从业者和工具开发者提供了建议，强调缓存应被视为演进的 CI 配置组件。
- **Chapter 8-10: Threats, Related Work, and Conclusion** - 讨论了研究的局限性、相关研究背景并总结全篇。

### 中文结论
本文通过对 GitHub Actions 缓存实践的首次大规模实证研究，证明了缓存并非“一次性”配置，而是需要持续维护的复杂系统。
研究发现缓存配置（尤其是参数）的调整非常频繁且主要由人工完成，旨在解决缓存未命中或工作流失败等稳定性问题。现有的
自动化工具（如 Dependabot）主要局限于版本号的提升，在更复杂的参数优化和结构重构方面支持不足。未来需要更智能的
工具（如基于 Agent 的系统）来辅助开发者进行缓存键的设计和验证。

# 章节详细解读

## 第一章：引言 (Introduction)

### 1. 设计思想 (Design Philosophy)
- **痛点分析**：随着项目增长，CI/CD 工作流的复杂度和执行频率不断增加，重复安装依赖和重建中间产物成为性能瓶颈。
虽然缓存能提速，但其配置并非易事，错误的缓存键（key）设计会导致缓存未命中或重复使用过时产物，反而增加维护负担。
- **核心动机**：现有的研究多将缓存视为二元状态（开启或关闭），忽略了其作为持续维护对象的动态特性。作者旨在填补
这一空白，深入探讨缓存如何随项目演进。

## 第二章：背景 (Background)

### 1. 关键技术 (Key Technologies)
- **GHA 缓存机制**：基于 YAML 文件配置，核心参数包括 `path`（要缓存的文件路径）和 `key`（唯一标识符）。通常
使用文件哈希作为 key 的一部分，以确保依赖变化时缓存失效。
- **维护活动分类**：定义了缓存启用 (EC)、版本更新 (C_up)、添加新缓存 (C_add)、删除缓存 (C_rm)、参数更新 (P_up)
等七类活动，为后续定量分析奠定基础。

## 第四章：RQ1: 缓存采用代码库的特征 (Characteristics of cache-adopting repositories)

### 1. 关键技术与观察
- **活跃度对比**：采用缓存的代码库在 Star 数、Fork 数、贡献者数和提交数上均显著高于非采用者。这表明缓存通常是
在高负荷、高参与度的项目中为了管理 CI 负载而引入的。
- **语言差异**：PHP (54.2%) 和 Java (50.7%) 的缓存采用率最高，而 Python (22.3%) 和 Ruby (17.6%) 相对较低。
这可能与不同语言生态中依赖管理工具的成熟度和缓存便利性有关。
- **策略分布**：显式缓存（使用 `actions/cache`）占主导地位 (70.9%)，包管理器内置缓存占 24.5%。

### 2. 实现细节
- **作业类型映射**：作者将开发者自定义的作业名规范化为八类：build, test, integration, lint, release, linux,
analyze, sync。
- **缓存流行度**：缓存最常出现在 build (37.3%) 和 integration (36.6%) 作业中，sync (5.1%) 最少。

## 第五章：RQ2: 缓存配置如何随时间演进 (Evolution of caching configurations)

### 1. 设计思想 (Design Philosophy)
- **演进模型**：使用一阶马尔可夫模型 (First-order Markov model) 来建模缓存维护活动的序列。节点代表活动类别，
边代表连续活动之间的转换概率。
- **差异化分析**：针对不同作业类型分别建模，以揭示构建/测试与发布等作业在维护节奏上的不同。

### 2. 关键技术 (Key Technologies)
- **转换模式**：发现缓存演进并非单向过程，而是包含大量的迭代（self-loop）。例如，`P_up -> P_up`（连续参数更新）
在构建作业中非常频繁，中位时间间隔仅 4.08 天。
- **“移除-重加”现象**：`C_rm -> C_add` 的转换概率在测试作业中高达 30.19%，表明开发者经常由于初始配置不当
而立即替换缓存策略。

## 第六章：RQ3: 开发者为何修改缓存配置 (Drivers of cache-related modifications)

### 1. 关键技术 (Key Technologies)
- **驱动因素分类法**：通过对提交说明和 PR 的定性分析，识别出 12 种驱动因素。排名前三的是：更新依赖 (91次)、
改进缓存性能 (47次) 和机器人告警 (40次)。
- **人工 vs 机器人**：参数更新（P_up）几乎全由人工驱动，旨在修复缓存失效或调整路径；版本更新（C_up）则多由
机器人（如 Dependabot）触发。

### 2. 权衡与对比 (Trade-offs & Comparison)
- **维护延迟**：缓存启用后的首次参数更新发生较快（约 106 天），而首次版本更新则平均延迟 384 天。这反映了开发者
对核心组件版本更新的谨慎态度，通常直到机器人提醒或必须更新时才动作。

## 第七章：启示 (Implications)

### 1. 核心建议
- **对待缓存的态度**：不应将其视为简单的开关，而应视其为“演进的配置组件”，需要长期的监控和微调。
- **工具开发方向**：现有的自动化工具在“缓存键合成”、“缓存有效性验证”和“恢复键设计”方面几乎是空白。未来的智能
助手应能感知工作流上下文，自动优化缓存参数。
- **移除不必要的缓存**：研究发现约 21% 的缓存被移除，表明并非所有缓存配置都是有益的，有时下载缓存的开销超过了
其节省的时间。
