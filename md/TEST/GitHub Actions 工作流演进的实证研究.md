# 2602.14572v2.pdf - An Empirical Study of the Evolution of GitHub Actions Workflows

## 中文标题：GitHub Actions 工作流演进的实证研究

### 中文摘要
持续集成与持续交付 (CI/CD) 实践在协同软件开发中发挥着至关重要作用，通过自动化测试、构建、质量检查以及依赖和安全管理等
重复性任务来提高效率。GitHub Actions (GHA) 作为 GitHub 集成的 CI/CD 工具，已成为开发者自动化工作流的首选。本文通过
混合方法分析，研究了 GHA 工作流随时间的变化情况。研究首先通过对 439 个修改后的工作流文件进行定性分析，识别出七种概念
上的变更类型。随后，针对 2019 年 11 月至 2025 年 8 月期间的 4.9 万多个 GitHub 公共仓库进行了大规模定量分析，涵盖 26.7
万个工作流历史记录和 340 万个文件版本。研究发现，仓库通常包含三个工作流文件，每周约有 7.3% 的文件会发生变动。工作流
的变更趋向于细微，约四分之三的变更仅包含单处修改。绝大多数观察到的变更涉及作业（jobs）中的任务配置和任务规范。此外，
研究并未发现大语言模型 (LLM) 编码工具或其他重大技术变革对工作流创建及维护频率产生显著影响。研究结果强调了开发更精细
的维护工具（如 AI 辅助的安全和质量保障工具）以及广泛采用依赖管理工具的必要性。

### 各章节主旨内容
- **Chapter 1: Introduction** - 阐述 CI/CD 的重要性及 GHA 的崛起，指出目前缺乏对 GHA 配置语法层级演变的深入研究。
- **Chapter 2: Related Work** - 回顾 CI/CD 实证研究、配置即代码 (CaC) 的演进，以及关于 Travis CI 和 GHA 的历史研究。
- **Chapter 3: Methodology** - 介绍 GHA 语法结构、数据集来源（4.9 万仓库）以及用于提取工作流差异的工具 `gawd`。
- **Chapter 4: RQ1 (频率分析)** - 量化工作流文件的添加、修改、删除和重命名频率，并识别“突发式”修改行为。
- **Chapter 5: RQ2 (概念变更分析)** - 通过定性分析识别出任务规范、任务配置、文档、运行配置等七类核心概念变更。
- **Chapter 6: RQ3 (变更类型分析)** - 定量分析工作流实体的增加、删除和修改比例及其随时间的演进趋势。
- **Chapter 7: RQ4 (语法实体分析)** - 识别变更频率最高的语法“热点”路径，如 `jobs`、`steps`、`uses` 和 `run`。
- **Chapter 8: Discussion** - 对比先前研究、讨论 LLM 工具的影响、自动化工具的作用以及安全性提升的趋势。
- **Chapter 9: Threats to Validity** - 讨论在构建、内部、外部和结论效度方面存在的潜在威胁。
- **Chapter 10: Conclusion** - 总结研究贡献，并提出未来结合执行日志和 AI 辅助代理的研究方向。

### 中文结论
本文通过对海量 GitHub 仓库的实证研究证明，GHA 工作流并非“一劳永逸”的配置，而是作为一种动态维护的软件组件持续演进。
工作流的维护行为主要体现为针对任务规范和配置的精细化、迭代式修改。研究揭示了开发者正逐渐转向更安全的版本锁定实践
（如使用 commit hash），且自动化依赖更新工具（如 Dependabot）已在其中扮演重要角色。尽管 LLM 技术普及，但尚未观察到
其对工作流变更频率的显著冲击。未来应致力于开发支持细粒度维护、安全加固和质量监测的智能化工具。

# 章节详细解读

## 1. RQ2: 工作流的概念性变更 (Conceptual Changes)

### 1.1 设计思想 (Design Philosophy)
- **痛点分析**：仅关注文件级别的添加或删除无法揭示开发者对 CI/CD 逻辑修改的真实意图。作者需要了解在 YAML 配置文件内
部，开发者究竟在“改变什么”。
- **核心动机**：通过对手工抽样的 439 个提交进行定性分析（Qualitative Analysis），构建一套分类体系（Taxonomy），以
便将底层的语法变更映射到高层的开发意图上。
- **设计目标**：建立一套涵盖所有可能工作流变更的概念框架，从而为后续的大规模定量分析提供理论支撑。

### 1.2 关键技术 (Key Technologies)
- **框架法 (Framework Method)**：作者采用了该定性分析方法，分为熟悉、识别框架、索引、图表化和解释五个阶段。
- **七大概念分类**：
    - **任务规范 (Task Specification, TS)**：定义“做什么”和“怎么做”，涉及作业/步骤的增减、顺序调整及矩阵策略。
    - **任务配置 (Task Configuration, TC)**：定义具体步骤的设置，如 Action 的版本选择和参数传递。
    - **文档 (Documentation, D)**：通过 `name` 字段或注释提供描述信息，提高可读性。
    - **运行配置 (Runtime Configuration, RC)**：控制“何时”和“何地”运行，如环境变量、操作系统 (runs-on) 和权限管理。
    - **执行触发器 (Execution Triggers, ET)**：定义激活机制，如 `on: push`。
    - **执行规则 (Execution Rules, ER)**：设置运行条件，如 `if` 判断、`needs` 依赖和超时设置。
    - **格式化 (Formatting, F)**：仅涉及缩进、引号等不影响执行的美容式变更。

### 1.3 实现细节 (Implementation Details)
- **饱和度测试**：研究人员首先随机抽取 389 个案例进行独立分类，随后每次追加 10 个案例，直到不再出现新的类别（最终在
439 个案例处达到饱和）。
- **共识机制**：由两名研究员独立分类，若存在分歧则由第三人介入裁定。

### 1.4 权衡与对比 (Trade-offs & Comparison)
- **对比**：相比于以往对 Travis CI 的研究（主要关注“构建逻辑”），本研究发现 GHA 开发者在“任务配置”和“重用 Action”上
投入了更多精力。
- **权衡**：定性分析虽然深度极高，但受限于人力无法处理数百万个提交，因此本文将其作为定量分析的“罗盘”。

## 2. RQ4: 频繁变更的语法实体 (Syntactic Entities)

### 2.1 设计思想 (Design Philosophy)
- **痛点分析**：即便知道了变更的概念，仍需精准定位到 YAML 文件中的哪些具体字段（Key）是“热点”，以便为工具开发者提
供改进方向。
- **核心动机**：量化不同语法路径（Workflow Paths）的变更频率，识别出最容易出错或最需要频繁调整的配置项。

### 2.2 关键技术 (Key Technologies)
- **路径归一化 (Path Normalization)**：
    - **匿名化**：将用户定义的作业 ID（如 `build-and-test`）统一替换为 `<id>`。
    - **位置遮蔽**：将数组索引（如 `steps[3]`）替换为 `<nr>`。
    - **示例**：`jobs.build.steps[3].if` 归一化为 `jobs.<id>.steps.<nr>.if`。
- **gawd 工具**：作者自研的 Python 工具，能够计算两个 YAML 树之间的结构化差异，检测实体的增加、删除、修改、移动和重
命名。

### 2.3 实现细节 (Implementation Details)
- **热点统计**：
    - **Jobs 层级**：91.05% 的变更发生在此，尤其是 `steps` 内部（70.1%）。
    - **Uses 字段**：占据总变更的 22.4%，其中 99.8% 是修改（Modification），多为版本更新。
    - **Run 字段**：占据 12.99%，主要涉及脚本内容的迭代。
- **自动化占比**：研究发现 Dependabot 覆盖了 69.2% 的工作流历史，主要负责 `uses` 路径下的版本自动升级。

### 2.4 权衡与对比 (Trade-offs & Comparison)
- **控制 vs 复用 (Trade-off)**：分析显示存在 `run` 与 `uses` 互相转换的现象。开发者在追求“控制力”（转为 custom scripts）
与“易用性/模块化”（转为重用 Actions）之间不断权衡。
- **安全性趋势**：研究观察到 `permissions` 字段的新增（尤其是在 2021 年 4 月 GHA 引入精细权限控制后）显著增加，表
明社区对“最小权限原则”的重视度在提升。

## 3. LLM 对工作流演进的影响分析

### 3.1 设计思想 (Design Philosophy)
- **痛点分析**：业界普遍认为 Copilot 等 LLM 工具会改变编码效率，但其是否真的改变了配置文件的维护模式（例如是让配置
变得更正确、减少调试，还是导致更多低质量配置产生）尚无定论。
- **核心动机**：通过“前-后对比试验” (Before-After Analysis)，在宏观统计层面验证 LLM 工具的普及是否导致工作流变更
频率或“突发式”修改行为的变化。

### 3.2 关键技术 (Key Technologies)
- **时间窗口设计**：设置了两个 695 天的对比期，中间包含一个 821 天的“排除期”（对应 Copilot/ChatGPT 普及期），以确
保对比样本具有代表性。
- **Mann-Whitney U 检验**：用于检测两组样本在变更频率、突发次数等指标上是否存在统计学上的显著差异。
- **Cliff’s δ 效应量**：即便存在统计差异，仍需通过效应量衡量差异的实际规模。

### 3.3 实现细节 (Implementation Details)
- **工作流阶段划分**：将工作流分为“早期阶段”（创建后的前 3 个月）和“后期阶段”（第 9 到 12 个月），以分别观察 LLM
在创建期和维护期的影响。
- **统计结果**：虽然在某些维度上 H0 假设被拒绝，但所有的 Cliff’s δ 值均处于“极小” (negligible, < 0.147) 范围。

### 3.4 权衡与对比 (Trade-offs & Comparison)
- **结论考量**：作者坦诚 LLM 的影响可能存在“局部性”或“延迟性”，宏观统计数据目前无法捕捉到决定性的变化趋势。
- **未来方向**：需要结合更精细的启发式算法（如 [51] 提出的检测 AI 生成代码的方法）进行深度溯源。
