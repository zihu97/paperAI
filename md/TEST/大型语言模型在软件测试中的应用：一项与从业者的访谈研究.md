# 大型语言模型在软件测试中的应用：一项与从业者的访谈研究

## 中文摘要
背景：大型语言模型（LLM）在软件测试中的应用正在迅速增长，它们支持从测试用例生成到自动化和文档的众多任务。然而，它们的采用
通常依赖于非正式的实验，而非结构化的指导。
目的：本研究旨在调查软件测试专业人员在实践中如何使用 LLM，以提出一个初步的、由从业者提供的指南，支持他们将其集成到测试工作流中。
方法：我们对来自不同角色和领域的 15 名软件测试人员进行了一项定性研究。通过半结构化访谈收集数据，并使用扎根理论的流程进行分析，
重点关注主题分析。
结果：测试人员描述了一个迭代和反思的过程，包括定义测试目标、应用提示工程策略、优化提示、评估输出和持续学习。他们强调了人工监督
和仔细验证的必要性，特别是考虑到 LLM 的已知局限性，如幻觉和不一致的推理。
结论：LLM 在软件测试中的采用正在增长，但仍受到不断发展的实践和对风险的谨慎态度的影响。本研究为在测试环境中结构化 LLM 
的使用提供了一个起点，并邀请未来的研究在不同团队、工具和任务中完善这些实践。

## 各章节主旨内容
*   **第 1 章：引言** - 介绍了在软件工程和测试领域中 LLM 应用日益增长的趋势，指出现有实践多依赖非正式探索，缺乏系统性指导，
并由此引出本研究的核心问题：软件测试者如何在实际工作中有效利用 LLM。
*   **第 2 章：LLM辅助下的测试** - 综述了将 LLM 用于软件测试的相关研究，涵盖了其在单元测试生成、缺陷复制、测试脚本编写等方面的应用，
同时也指出了输出结果不一致、存在幻觉以及验证机制缺失等挑战。
*   **第 3 章：研究方法** - 详细说明了本研究采用的定性访谈设计，包括通过便利性抽样和雪球抽样招募了15名有 LLM 
使用经验的测试从业者，并采用半结构化访谈进行数据收集，最后通过扎根理论指导下的主题分析法进行数据分析。
*   **第 4 章：结果** - 呈现了访谈的核心发现，总结了 LLM 在测试用例设计、测试自动化、研究学习等方面的具体用途，从业者感知的优势
（如效率提升），以及他们面临的挑战（如数据隐私、幻觉问题）。最重要的是，本章提炼出一个五步实践指南。
*   **第 5 章：讨论** - 深入探讨了研究发现的意义，强调 LLM 目前在测试中扮演的是“辅助”而非“替代”人类的角色。
讨论了研究结果对学术研究和行业实践的启示。
*   **第 6 章：有效性威胁** - 讨论了本研究作为定性研究的局限性，如样本量较小，不追求普遍性结论，但强调其理论可转移性，
即研究结果可为其他相似情境提供参考。
*   **第 7 章：结论** - 总结了研究贡献，重申了测试人员与 LLM 互动的迭代过程，并强调了所提出的五步指南作为一个实践基础，
旨在促进对 LLM 更审慎和有效的整合。

# 各章节详解

## 第 1 章：引言 (INTRODUCTION)
本章首先确立了研究背景：大型语言模型（LLM）作为一种强大的 AI 系统，已从执行小型、孤立的编程任务（如代码生成、调试）扩展到更广泛地
集成到软件开发工作流中。开发者不仅用它来加速重复性工作，还用它探索解决方案、理解代码。在软件测试领域，LLM 
同样展现出巨大潜力，可用于单元测试生成、缺陷复现、测试脚本编写等。然而，现有研究多集中于功能测试层面，
而测试规划、设计以及非功能测试（如性能、安全）等领域尚未得到充分探索。

尽管 LLM 带来了减少重复劳动、降低技术门槛等好处，但其“幻觉”输出、模糊推理和缺乏验证机制等问题也引发了业界的担忧。
其不确定性的行为模式，给依赖可预测性和精确质量保证方法的结构化测试带来了挑战。

目前，行业内对 LLM 的使用主要依赖于非正式的实践、社区建议和“灰色文献”，缺乏成熟的框架或标准来指导其集成。
为了填补这一空白，本研究提出以下研究问题（RQ）：
**“软件测试人员如何在工作中使用 LLM？从他们的经验中可以提炼出哪些步骤来指导 LLM 在测试活动中的集成？”**

为回答此问题，研究者访谈了15名软件测试专业人士，分析了他们在日常测试任务中使用 LLM 的实际经验、遇到的挑战以及形成的有效策略。
本研究的主要贡献在于：
1.  一份关于软件测试专业人员如何在工业界使用 LLM 的实证报告。
2.  一个基于真实世界经验的、初步的 LLM 集成指南。

## 第 2 章：LLM辅助下的测试 (TESTING WITH LLMS)
本章回顾了现有文献，探讨了 LLM 在软件测试中的应用、好处与局限性。研究表明，LLM 正在成为解决测试自动化、维护和知识传递等长期挑战
的有前途的方向。与需要大量精力来适应变化的传统自动化测试方法相比，LLM 提供了灵活的、基于提示的交互方式。

**主要应用领域包括：**
*   **单元测试生成**：这是研究最广泛的应用之一，模型根据源代码或方法描述生成可编译的测试用例。
*   **缺陷复现**：LLM 根据自然语言描述的缺陷报告生成可复现问题的输入。
*   **测试预言（Oracle）创建与测试文档**：辅助定义预期的测试结果和生成文档。

**挑战与局限性：**
*   **语义错误**：生成的代码可能语法正确，但与预期的软件行为在语义上不符，或包含“幻觉”出的断言和错误逻辑。
*   **一致性与确定性**：LLM 输出的非确定性使其难以在严格的质量保证流程中使用。
*   **对上下文的依赖**：LLM 的效用高度依赖于任务、编程语言和领域。

当前，从业者主要以非正式和探索性的方式采用 LLM，通常将其用作“协作助手”而非独立的自动化工具，尤其是在编程背景有限的测试人员中，
LLM 帮助他们弥合技术差距。然而，由于缺乏正式的指导方针，这种临时的采用方式引发了对过度依赖、信任和集成的担忧。

## 第 3 章：研究方法 (METHOD)
本研究采用定性访谈方法，旨在深入了解软件测试从业者如何将 LLM 集成到他们的工作流程中。

**研究设计分为三个阶段：**
1.  **抽样与招募**：研究者结合了**便利性抽样**（从研究者的职业网络和测试社区开始）和**雪球抽样**（邀请早期参与者推荐同事）。
为了确保样本多样性，研究者还运用了**理论抽样**，主动寻找能够扩展或对比已有发现的参与者。最终成功招募了15名参与者，
他们来自不同行业，拥有不同的资历水平和角色（手动测试、自动化工程师、QA分析师、测试经理等）。许多人因公司保密协议（NDA）
或内部限制而拒绝参与。

2.  **数据收集**：在2024年10月至2025年1月期间，研究者通过在线方式进行了一对一的同步访谈。每次访谈持续25到45分钟，
并经参与者许可后进行录音和转录。访谈采用**半结构化**形式，围绕 LLM 的使用场景、任务、优缺点、组织实践等开放性问题展开。
访谈指南经过了两位专业人士的试行和定性方法专家的反馈，以确保问题质量。

3.  **数据分析**：分析过程采用了**扎根理论指导下的主题分析**，包含开放式编码、聚焦式编码和理论编码。
    *   **开放式编码**：第一作者逐行审查访谈记录，识别与 LLM 使用相关的离散事件、实践和判断。
    *   **聚焦式编码**：将相似的开放编码归纳为更广泛的类别（例如，“评估输出”）。
    *   **理论编码**：将这些类别关联并组织成一个五步结构，反映了参与者描述的 LLM 集成流程。
为保证分析的严谨性，作者之间会讨论模糊的编码案例，并保留了详细的编码簿。

**伦理考量**：研究方案获得了机构研究伦理委员会的批准。所有参与者在知情同意后参与，并且个人身份信息被完全匿名化处理。

## 第 4 章：结果 (RESULTS)
本章展示了从15名不同背景的测试从业者访谈中获得的核心发现。

**4.1 LLM 在测试中的常见用途和采纳趋势**
LLM 主要用于测试用例设计、自动化和学习，以执行重复性任务和提高效率。具体用途包括：
*   **测试用例与场景创建 (8/15)**：根据需求、用户故事或系统规格生成结构化的测试用例。
*   **测试自动化 (7/15)**：生成、优化和重构自动化测试脚本（如 Selenium, Cypress），并辅助调试。
*   **研究与学习 (7/15)**：快速获取信息、解决技术问题、了解最佳实践，替代了传统的论坛搜索。
*   **测试文档 (5/15)**：起草和标准化测试模板、报告和执行计划。
*   其他频率较低的用途还包括：测试执行、验收标准生成、测试证据分析、需求生成、缺陷描述等。

**4.2 LLM 如何赋能软件测试：报告的优势**
*   **时间效率 (9/15)**：在寻找解决方案、创建测试用例时节省大量时间。
*   **提升生产力 (4/15)**：通过自动化任务和改进测试设计来提高工作效率。
*   **改进测试覆盖率 (3/15)**：通过生成多样化的场景来扩展测试覆盖面。
*   **学习与培训支持 (3/15)**：为新团队成员提供结构化指导，帮助他们更快地独立工作。

**4.3 在软件测试中采纳 LLM 的挑战**
*   **准确性与幻觉 (9/15)**：LLM 经常生成不正确、不完整或逻辑不一致的测试工件，需要人工验证。
*   **数据隐私与安全 (7/15)**：使用包含内部系统细节的提示存在数据泄露风险。
*   **提示词与工具采纳的困难 (3/15)**：学习如何构建精确的提示词以获得理想输出需要时间。
*   **认知上的过度依赖风险 (3/15)**：担心过度依赖 LLM 会削弱测试人员的分析判断能力。
*   **组织与流程障碍 (1/15)**：将 LLM 辅助的测试与现有质量保证政策对齐并获得管理层批准存在挑战。

**4.4 一个初步的、从业者启发的 LLM 使用指南**
基于访谈，研究者提炼出一个反映从业者实践的五步指南：
1.  **定义测试目标并构建提示 (Define the Testing Objective and Structure the Prompt)**：明确测试目标，并构建能够清晰表达该目标的提示。
2.  **应用提示工程技术 (Apply Prompt Engineering Techniques)**：使用如 CARE 方法、少样本（few-shot）示例等结构化技术来优化提示，减少模糊性。
3.  **迭代和优化测试提示 (Iterate and Refine Testing Prompts)**：当 LLM 输出不理想时，通过增加上下文、约束或期望的输出格式来逐步调整提示，
类似于回归测试或探索性测试。
4.  **评估、扩展和集成 LLM 输出 (Evaluate, Expand, and Integrate LLM Outputs)**：在将 LLM 生成的材料（如代码、测试用例）集成到工作流之前，
评估其准确性、可读性和可维护性。
5.  **参与持续学习和知识共享 (Engage in Continuous Learning and Knowledge Sharing)**：持续学习新的提示策略，并与团队分享经验，
以在不牺牲分析严谨性的前提下提高测试效率。

## 第 5 章：讨论 (DISCUSSIONS)
本章对研究结果进行了深入的解读。研究发现，与文献中常强调的 LLM 
的自动化能力（如测试生成）不同，在实际应用中，测试人员更多地将 LLM 
视为一种**辅助工具**，用来增强而非取代他们的专业判断。LLM 
被用于解释错误信息、起草测试证据等适应性强的日常任务中。

这种“人机协作”模式反映了从业者在实践中形成的策略，他们通过调整提示、交叉验证输出来应对 LLM 
的局限性（如幻觉、领域知识不足）。这表明 LLM 
的有效使用远非“即插即用”，而是需要使用者发展出一套与之互动的特定技能。

本研究提出的五步指南，其新颖之处在于它并非一个僵化的流程，而是对从业者与 LLM 
互动行为的结构化表达。它源于实践，反映了从确定任务、迭代优化提示到最终评估集成的完整闭环。这个指南为反思和改进 LLM 
在测试中的应用提供了基础。

最终，本研究将焦点从“LLM能做什么”转移到“测试人员如何使用LLM”，强调了 LLM 采纳过程中的“人的因素”。
成功的 LLM 集成不仅取决于模型的能力，同样也依赖于从业者的专业知识和批判性思维。

## 第 6 章：有效性威胁 (THREATS TO VALIDITY)
本章坦诚地讨论了研究的潜在局限性。
*   **非普适性 (Generalization)**：作为一项定性研究，本工作不追求统计上的普适性。研究结果的价值在于其**理论可转移性**
（theoretical transferability），即其他组织或个人可以在理解其特定上下文的基础上，考虑、调整或实施这些发现。
*   **数据保密性**：由于保密性问题（尤其是一些参与者所在公司不鼓励使用 LLM），研究者无法分享完整的访谈记录。
为了保证透明度，研究在论文中包含了匿名的引文来支撑结论。
*   **饱和度**：数据收集在达到**理论饱和度**时停止，即新的访谈不再产生新的核心概念，而是不断强化现有类别。
这种做法加强了研究模式的一致性，并符合定性研究的指导原则。

## 第 7 章：结论 (CONCLUSIONS)
本研究通过对15名专业测试人员的访谈，探索了他们如何将 LLM 集成到日常工作中。核心结论是，测试人员并非将 LLM 
视为完全自主的工具，而是通过一个迭代的过程来使用它。

这个过程始于**明确定义测试目标**并构建相应的提示。接着，从业者会运用**提示工程**来提高输出质量。当输出不理想时，
他们会**迭代优化提示**，直到模型交付满意的结果。最后，输出结果经过**评估、调整和集成**到测试工件中。整个过程伴随着**持续学习**，
以不断提升使用效率。

考虑到 LLM 在软件工程中存在的幻觉、推理模糊和上下文保留能力不足等已知局限，建立明确、透明的使用实践至关重要。
本研究提出的初步指南正是为了响应这一需求，它为从业者在采纳 LLM 时提供了一个基于实践的起点。该指南旨在鼓励对当前实践的反思、
提高对潜在风险的认识，并支持更有意识地使用 LLM，为在测试工作流中集成 LLM 提供了坚实的实践基础。
