# NecoFuzz：通过模糊测试专用虚拟机对嵌套虚拟化进行有效模糊测试

---

## 第一部分：论文整体骨架

### 1. 论文基本信息
*   **论文文件名**：2512.08858v1.pdf
*   **英文标题**：NecoFuzz: Effective Fuzzing of Nested Virtualization via Fuzz-Harness Virtual Machines
*   **中文标题**：NecoFuzz：通过模糊测试专用虚拟机对嵌套虚拟化进行有效模糊测试

### 2. 中文摘要
嵌套虚拟化目前已被主流云供应商广泛支持，它允许用户在云端利用基于虚拟化的技术。然而，支持嵌套虚拟化显著增加了宿主管理程序（Hypervisor）的复杂性，并在云平台中引入了新的攻击面。虽然已有许多针对管理程序的模糊测试研究，但由于生成有效的虚拟机（VM）实例面临巨大的状态空间挑战，此前并未有研究专门针对嵌套虚拟化进行测试。

本文提出了 **NecoFuzz**，这是第一个系统性地针对管理程序中嵌套虚拟化逻辑的模糊测试框架。NecoFuzz 合成可执行的“模糊测试专用虚拟机”（fuzz-harness VMs），其内部状态处于有效与无效的边界附近，并由硬件辅助虚拟化规范的近似模型进行引导。鉴于嵌套虚拟化中的漏洞通常源于对意外 VM 状态的不正确处理，这种基于规范引导的、面向边界的生成方法显著提高了不同管理程序中安全关键代码的覆盖率。

作者在 Intel VT-x 和 AMD-V 平台上实现了 NecoFuzz，并通过扩展 AFL++ 来支持模糊测试专用虚拟机。实验结果显示，NecoFuzz 在 Intel VT-x 和 AMD-V 上分别实现了 84.7% 和 74.2% 的嵌套虚拟化特定代码覆盖率，并在包括 KVM、Xen 和 VirtualBox 在内的三个管理程序中发现了六个此前未知的漏洞，其中两个已被分配 CVE 编号。

### 3. 各章节主旨内容总结

*   **第 1 章：引言 (Introduction)**
    *   阐述了嵌套虚拟化在云计算中的重要性（如机密计算、遗留系统迁移）及其带来的安全风险。
    *   指出当前针对管理程序的模糊测试工具（如针对 I/O、MMIO、DMA 的工具）无法有效覆盖硬件辅助虚拟化接口，主要原因是输入空间（完整的 VM 状态）过于庞大且复杂。
    *   概述了 NecoFuzz 的核心贡献：通过生成处于有效性边界的完整 VM 实例来有效触发管理程序的错误行为。

*   **第 2 章：背景 (Background)**
    *   介绍了硬件辅助虚拟化（Intel VT-x/AMD-V）的基本概念，包括根模式/非根模式、VMCS（虚拟机控制结构）等。
    *   解释了嵌套虚拟化的工作原理：L0（宿主）、L1（客户机管理程序）、L2（嵌套客户机）的层级关系，以及 L0 如何通过模拟硬件行为来支持 L1 运行 L2（特别是 VMCS 的隐藏与模拟）。

*   **第 3 章：设计 (Design)**
    *   定义了威胁模型：攻击者控制云端的 L1 VM，试图通过恶意的硬件辅助虚拟化指令攻击 L0。
    *   详细介绍了 NecoFuzz 的三个核心组件：
        1.  **VM 执行工具 (VM Execution Harness)**：作为 L1 管理程序运行，执行指令序列以触发 VM 退出。
        2.  **VM 状态验证器 (VM State Validator)**：基于规范模型修正随机生成的 VMCS 状态，使其既能通过基本检查又处于边界状态。
        3.  **vCPU 配置器 (vCPU Configurator)**：改变 vCPU 的功能特性（如启用/禁用 EPT），以覆盖不同的代码路径。

*   **第 4 章：实现 (Implementation)**
    *   描述了基于 AFL++ 的具体实现细节。
    *   介绍了运行在宿主机上的代理程序（Agent），用于连接 Fuzzer 和目标 VM。
    *   详细说明了如何从 Bochs 移植并改造验证逻辑，以及如何构建 UEFI 格式的执行工具。

*   **第 5 章：评估 (Evaluation)**
    *   **RQ1 代码覆盖率**：在 KVM 上对比了 Syzkaller、IRIS 等工具，NecoFuzz 展现出显著的覆盖率提升（Intel 上 1.4倍，AMD 上 11.0倍）。
    *   **RQ2 组件有效性**：通过消融实验验证了状态验证器、执行工具和配置器对覆盖率的贡献。
    *   **RQ3 跨管理程序能力**：在 Xen 上进行了测试，证明了框架的通用性。
    *   **RQ4 漏洞发现**：展示了在 KVM、Xen、VirtualBox 中发现的 6 个新漏洞及其成因。

*   **第 6 章：讨论 (Discussion)**
    *   讨论了当前的局限性，如未涉及异步事件（中断）、多 vCPU 交互及更深层嵌套（L3）的支持。
    *   分析了现有 Sanitizer 在检测逻辑漏洞方面的不足。

*   **第 7 章：相关工作 (Related Work)**
    *   对比了虚拟 CPU 模糊测试（Blackbox/Non-blackbox）和虚拟设备模糊测试的研究现状，强调 NecoFuzz 填补了嵌套虚拟化模糊测试的空白。

*   **第 8 章：结论 (Conclusion)**
    *   总结了 NecoFuzz 的创新点和实验成果，确认其为嵌套虚拟化安全研究提供了有效的基石。

### 4. 中文结论
本文提出了 NecoFuzz，这是首个专门针对管理程序中嵌套虚拟化功能的模糊测试框架。NecoFuzz 引入了一种 VM 生成器，能够高效合成完整的 VM 实例（称为 fuzz-harness VMs），专门用于探索有效与无效 VM 状态之间的安全关键逻辑。该生成器包含三个组件：触发嵌套虚拟化路径的 VM 执行工具、基于硬件规范引导输入生成的 VM 状态验证器，以及系统性探索硬件特性组合的 vCPU 配置器。

我们在 Intel VT-x 和 AMD-V 上实现了 NecoFuzz，并将其应用于 KVM、Xen 和 VirtualBox 的模糊测试。评估表明，NecoFuzz 在代码覆盖率上显著优于现有工具（包括唯一支持嵌套虚拟化的 Syzkaller）。此外，NecoFuzz 发现了 6 个此前未知的漏洞，涵盖三个主流管理程序，证明了其在挖掘嵌套虚拟化这一未探索攻击面方面的实用性和有效性。

## 第二部分：指定章节详解

### 第 1 章：引言 (Introduction)
*   **背景与挑战**：随着云计算的发展，嵌套虚拟化（在虚拟机中运行虚拟机）已成为 Azure 等云平台的关键特性，用于支持机密计算（如 Intel TDX）和系统迁移。然而，由于 CPU 本身不原生支持多层嵌套，宿主管理程序（L0 Hypervisor）必须通过软件模拟复杂的硬件虚拟化接口（如 VMCS 操作）。这带来了巨大的实现负担和新的攻击面。
*   **现有问题**：传统的 Fuzzing 技术主要关注 I/O 接口（如 virtio），无法触及核心的虚拟化指令处理逻辑。现有的 CPU 模拟器 Fuzzing 技术则难以生成符合复杂硬件约束的有效 VM 状态（VM State），导致测试无法深入。
*   **解决方案**：NecoFuzz 提出通过生成“模糊测试专用虚拟机”作为输入，直接在 L0 上运行。它不依赖随机变异，而是利用对硬件规范的理解，生成处于“有效-无效”边界的特殊状态，从而高效触发 L0 的异常处理逻辑。

### 第 2 章：背景 (Background)
*   **硬件辅助虚拟化**：详细解释了 Intel VT-x/AMD-V 的机制。核心是 **VMCS (VM Control Structure)**，它保存 CPU 状态。VM 的启动（VM Entry）和退出（VM Exit）都围绕 VMCS 的状态切换进行。VMCS 包含成百上千个字段，且字段间存在严格的一致性检查（Consistency Checks）。
*   **嵌套虚拟化机制**：
    *   **L0 (Host)**：物理机上的 Hypervisor（如 KVM）。
    *   **L1 (Guest)**：运行在 L0 上的 VM，它自己也是一个 Hypervisor。
    *   **L2 (Nested Guest)**：运行在 L1 上的 VM。
    *   **VMCS 模拟**：L1 认为自己在直接操作硬件，维护 `VMCS12`（虚拟的 VMCS）。L0 需要将 `VMCS12` 的内容翻译并合并到真实的 `VMCS02`（L0 用于运行 L2 的结构）中。这个“影子 VMCS”的同步和模拟是漏洞的高发区。

### 第 3 章：设计 (Design)
本章是核心，详细阐述了如何解决“巨大的状态空间”和“复杂的依赖约束”这两个难题。

#### 3.1 威胁模型
假设攻击者拥有 L1 VM 的完全控制权（例如购买了云主机），攻击者可以在 L1 内部加载自定义的内核模块或裸机程序，执行任意硬件虚拟化指令（如 `vmptrld`, `vmlaunch`），试图让底层的 L0 Hypervisor 崩溃或执行未授权操作。

#### 3.2 系统概览
NecoFuzz 的核心是一个 **VM 生成器**，它生成一个二进制的 **Fuzzing Input**，该输入被分割并分发给三个子组件：

1.  **VM 执行工具 (VM Execution Harness)**：
    *   这是一个 UEFI 二进制程序，作为 L1 Hypervisor 运行。
    *   **初始化阶段 (Initialization Phase)**：利用变异的输入参数执行标准的虚拟化初始化序列（如 `vmxon` -> `vmptrld` -> `vmlaunch`）。这里使用“模板化”策略，防止因破坏基本流程而导致测试过早终止。
    *   **运行时阶段 (Runtime Phase)**：在进入 L2 后，循环执行指令以触发 VM Exit。这些指令（如 `cpuid`, `vmcall`）的参数由 Fuzzing 输入控制，迫使 L0 执行复杂的模拟逻辑。

2.  **VM 状态验证器 (VM State Validator)**：
    *   **设计思想**：随机生成的 VMCS 几乎肯定因为违反硬件约束而被 CPU 直接拒绝，无法触发 L0 的深层逻辑。完全合法的 VMCS 又难以触发错误。
    *   **边界变异策略**：
        1.  从 Fuzzing 输入中读取原始数据。
        2.  **Rounding (修约)**：利用从 Bochs 提取的逻辑，将非法值强制修正为“最近的合法值”。
        3.  **Selective Mutation (选择性变异)**：在修正后的合法状态基础上，再次根据输入翻转少量比特。
    *   **效果**：生成的 VMCS 既能通过大部分基础检查，又可能在某些边缘条件下违反约束，从而测试 L0 是否能正确处理这些微妙的错误状态。

3.  **vCPU 配置器 (vCPU Configurator)**：
    *   L0 的行为高度依赖于 L1 vCPU 的特性（如是否开启 EPT、是否支持大页等）。
    *   配置器在 VM 启动前，根据 Fuzzing 输入动态调整这些特性组合。这确保了测试覆盖了各种硬件特性开启/关闭的排列组合场景。

### 第 4 章：实现 (Implementation)
*   **基础框架**：基于 AFL++。由于 AFL++ 面向用户态程序，作者开发了一个 **Agent** 程序运行在 Host (L0) 上。
*   **Agent 的作用**：
    1.  接收 AFL++ 的变异数据。
    2.  启动 QEMU/KVM 加载 UEFI Harness。
    3.  通过共享内存将覆盖率数据（Intel PT 或 KCOV）反馈给 AFL++。
    4.  监控系统日志（dmesg）和 Sanitizer（KASAN/UBSAN）输出以检测崩溃。
*   **验证器移植**：作者从开源模拟器 **Bochs** 中提取了约 2500 行高保真的 VMCS 检查代码。将这些 C++ 代码解耦并集成到 NecoFuzz 的 C 环境中，用于在运行时动态修正 VMCS。
*   **适配性**：为 KVM 编写了脚本适配器，通过命令行参数传递 vCPU 配置；Agent 设计为与 Hypervisor 内部实现解耦，便于移植到 Xen 等其他平台。

### 第 5 章：评估 (Evaluation)
*   **实验设置**：在 Linux 6.5 (KVM) 上测试，使用 Intel i9-12900K 和 AMD Ryzen Threadripper。
*   **覆盖率分析 (RQ1)**：
    *   相比 Syzkaller（通用的内核系统调用 Fuzzer），NecoFuzz 在 Intel 上的覆盖率从 61.4% 提升到 84.7%。
    *   相比 IRIS（基于记录重放的 Fuzzer），NecoFuzz 不仅稳定而且覆盖率更高。
    *   特别是在 AMD 平台上，由于缺乏专门工具，Syzkaller 仅覆盖 7%，而 NecoFuzz 达到了 74.2%。
*   **组件贡献 (RQ2)**：
    *   移除 **VM State Validator** 导致覆盖率下降最严重（Intel 上下降 16.9%），证明了“构造边界状态”是测试嵌套虚拟化的关键。
    *   移除 **vCPU Configurator** 也会导致显著下降，说明单一配置无法覆盖所有逻辑。
*   **漏洞挖掘 (RQ4)**：
    *   **KVM CVE-2023-30456**：L1 设置了 `IA-32e` 模式但未开启 `PAE`（物理地址扩展），Intel 手册要求必须开启，但硬件默默容忍了。KVM 却字面理解规范，导致页表损坏。这是一个典型的“硬件行为与规范不一致”导致的漏洞。
    *   **VirtualBox CVE-2024-21106**：未检查 MSR 加载值的规范性（Canonical Check），导致宿主机出现一般保护错误（#GP）。
    *   **Xen 漏洞**：涉及 VMCS `Activity State` 字段的盲目复制，导致宿主机死锁。

### 第 6 章：讨论 (Discussion)
*   **Bug 检测的局限**：目前的 Sanitizer（如 KASAN）主要检测内存错误。但逻辑错误（如 L1 成功欺骗 L0 访问了受限资源，或者 L0 错误地向 L1 注入了异常）很难被发现。作者建议未来开发针对虚拟化语义的检测器。
*   **异步事件**：当前只测试了指令触发的同步退出。中断、NMI 等异步事件的时序更难控制，是未来的方向。

### 第 7 章：相关工作 (Related Work)
*   **虚拟 CPU Fuzzing**：现有的 EmuFuzzer 等工具主要针对仿真器（如 QEMU 的 TCG），而不是硬件辅助虚拟化（KVM/VT-x）。
*   **虚拟设备 Fuzzing**：Nyx, VDF 等工具非常成熟，但它们只关注 Virtio/MMIO，完全不触及 `vmexit` 处理逻辑。
*   **结论**：NecoFuzz 开辟了“嵌套虚拟化接口”这一新的 Fuzzing 领域。

### 第 8 章：结论 (Conclusion)
NecoFuzz 成功证明了针对嵌套虚拟化的专门 Fuzzing 是必要且可行的。通过“在 VM 内部运行 Fuzzer”并结合“硬件规范引导的变异”，它能够深入到 Hypervisor 最复杂、最晦涩的代码路径中。发现的多个 CVE 证实了该方法在工业界实际应用中的价值。
