## **各章节详解**

### **第 1 章：引言 (Introduction)**

本章首先明确了软件 DevOps 是一个贯穿软件开发、交付、部署和管理全过程的关键且劳动密集型的流程。尽管近期的 AI 代理和大型语言模型（LLM）在开发环节（如代码生成、修复 GitHub 问题）已展现出强大的自动化能力，但后续的运维阶段（如系统构建、部署、监控）仍需大量人工干预。

作者指出，自动化这些复杂的运维任务面临着与传统代码生成截然不同的挑战。它要求 AI 代理具备分析系统运行时行为、与领域特定工具交互以及执行多步规划的能力。例如，诊断内存泄漏需要代理反复调用 `ps`、`iostat` 等监控工具，并解读输出以发现异常信号。这类依赖复杂工具使用、动态环境交互和连贯决策的任务，使其成为自主 AI 代理的绝佳应用领域。

然而，现有的 LLM 和 AI 代理基准测试大多集中于软件开发任务，如代码生成（HumanEval）、问题解决（SWE-bench）等，而对运维侧的关注有限，且往往局限于模拟环境或狭窄的基础设施设定。因此，当前迫切需要一个能够评估 AI 代理端到端 DevOps 能力的基准测试。

为了填补这一空白，作者推出了 **DEVOPS-GYM**。这是首个旨在全面评估 AI 代理在真实软件项目和多样化 DevOps 工具环境中跨核心工作流能力的基准测试。它具备以下特点：
1.  **覆盖四个基本 DevOps 阶段**：构建与配置、监控、问题解决、测试生成。
2.  **真实世界的任务**：任务来源于真实的 GitHub 问题或模拟真实复杂场景的合成任务。
3.  **面向代理的工作流**：任务需要广泛的工具使用和多步规划。
4.  **工具增强的动态评估环境**：提供标准化的工具调用接口和详细的评估指标。

最后，本章给出了通过 DEVOPS-GYM 评估得出的初步结论：即便是最先进的 AI 系统，其表现也远未达到理想水平，这说明在实现 DevOps 全流程自动化方面，仍有大量基础性研究工作亟待完成。

### **第 2 章：相关工作 (Related Work)**

本章系统梳理了与本研究相关的两类基准测试：**编码基准测试** 和 **DevOps 相关基准测试**，旨在明确 DEVOPS-GYM 的独特性和贡献。

1.  **编码基准测试 (Coding benchmarks)**
    作者首先提到函数级别的编码基准（如 HumanEval），它们是评估 LLM 代码生成能力的基础。随后，研究焦点转向了更复杂的仓库级别（repository-level）评估，如 RepoCod，这类基准测试要求模型处理多文件代码生成。
    另一条线的工作是将通用代码生成扩展到特定的真实世界任务。例如，SWE-bench 及其后续工作专注于使用 Python 解决真实 GitHub 仓库中的问题。此外，还有针对特定领域的基准测试，如基础设施即代码（IaC）、后端开发、数据库交互和机器学习任务等。
    尽管这些基准测试提供了更真实的评估，但它们存在共同的局限性：
    *   **仅覆盖开发侧**：几乎所有工作都集中在 DevOps 的“Dev”部分，而忽略了“Ops”部分。
    *   **非代理中心**：它们主要评估模型的推理和生成能力，而非代理系统进行顺序决策和工具调用的“代理”能力。

2.  **DevOps 相关基-准测试 (DevOps-related benchmarks)**
    作者指出，在代码开发之外，学术界也开始努力为 DevOps 周期的其他步骤（如测试用例生成、构建与配置、监控）建立基准。
    *   **测试生成**：SWT-bench 评估 Python 中的测试用例生成，关注功能性和代码覆盖率。
    *   **构建与配置**：现有基准主要关注项目的初始配置，通过静态检查和编译成功与否来评估。
    *   **监控**：IT-bench 在项目运行期间构造监控和解决问题的任务；AIOpsLab 则专注于微服务环境和云原生应用的故障检测与定位，并评估代理的效率和成本。

    **局限性 (Limitations)**:
    作者总结说，目前还没有一个基准测试能够覆盖真实仓库中的端到端 DevOps 周期。现有工作大多聚焦于孤立的任务（编码、测试或监控）。此外，很少有基准是为“代理系统”明确设计的，这些系统需要执行结合了代码推理、生成和工具调用的多步工作流。它们同样缺乏支持现实代理交互和评估的动态执行环境。

    **结论**：为了解决这些空白，DEVOPS-GYM 被构建出来，旨在实现对 AI 代理在端到端、真实世界 DevOps 任务中的全面评估，并提供适当的评估指标、工具调用支持和执行环境。


### **第 3 章：DevOps-Gym 的构建 (Construction of DevOps-Gym)**

本章是论文的核心，详细介绍了 DEVOPS-GYM 基准测试的构建过程，包括其设计原则、任务构成、技术挑战与解决方案。

#### **3.1 概述 (Overview)**

**设计原则**:
1.  **现实主义 (Realism)**: 任务源于真实的 GitHub 项目和 issue，或由专家模拟常见的生产问题，以确保任务的实用性。
2.  **代理式评估 (Agentic evaluation)**: 任务被设计为需要顺序决策的过程，涉及多种命令行工具的使用，考验代理的规划和工具交互能力。
3.  **完整的 DevOps 周期覆盖 (Complete DevOps cycle coverage)**: 精炼出四个关键阶段：
    *   **构建与配置**: 编译项目、修复构建失败。
    *   **监控**: 在项目运行时动态检测性能和资源问题。
    *   **问题解决**: 调试并修复项目中的 bug。
    *   **测试生成**: 创建回归测试以验证修复。

**基准测试概览**:
*   **语言与项目**: 选择 Java 和 Go，因为它们代表了具有标准化构建系统和强大监控工具的大型企业项目。
*   **任务来源**: 为了确保场景多样性，团队手动分析了 1000 多个 GitHub issue，特别是在现有基准中覆盖不足的监控和构建领域。
*   **任务构成**:
    *   **构建与配置**: 54 个任务（包括修复构建失败和实现新功能）。
    *   **监控**: 34 个任务（覆盖性能、资源等多类异常）。
    *   **问题解决**: 310 个任务。
    *   **测试生成**: 310 个任务。
    *   **端到端流水线**: 18 个任务，将上述四个阶段串联起来，测试代理的上下文维持能力。
*   **任务流程**: 代理首先需要成功构建项目，然后监控系统状态以发现异常，接着定位并解决问题，最后生成测试来验证修复。

**技术挑战与解决方案**:
1.  **数据污染防治 (Data contamination prevention)**:
    *   **挑战**: LLM 的大规模预训练可能已包含目标仓库的代码，导致评估结果虚高。
    *   **方案**: 采用系统性的前缀补全分析来识别和过滤掉可能存在于训练数据中的仓库。同时，移除仓库的 git 元数据，防止代理“作弊”查看历史提交记录。
2.  **任务复现 (Task reproduction)**:
    *   **挑战**: 复现真实世界的故障（特别是监控和构建类）极其困难，因为 issue 描述通常不完整。
    *   **方案**: 投入大量专家精力进行反复试验，重建正确的环境、依赖和配置。一个复杂的监控或构建问题的复现，往往需要专家花费超过 10 小时。
3.  **实现严谨的评估 (Enable rigorous evaluation)**:
    *   **挑战**: 设计和实现可扩展且严谨的评估流程本身就充满挑战，例如构建任务需要动态执行和静态配置相结合的验证。
    *   **方案**: 为不同阶段和场景设计了专门的评估指标（详见后续小节），并提供标准化的工具集接口，将基准测试格式转换为通用的 `terminal-bench` 格式，便于代理执行。

#### **3.2 构建与配置任务的构建 (Task Construction for Build and Configuration)**

本节详细阐述了构建与配置阶段的任务设计。这是 DevOps 周期中的关键一步，涉及代码编译、依赖管理、测试和产物创建，也是错误频发环节。

**任务设计**:
设计了两类任务以反映日常 DevOps 场景：
1.  **修复任务 (Repair tasks)**: 旨在解决五种常见的项目构建错误：
    *   依赖版本冲突
    *   构建配置错误
    *   编译错误
    *   工具链不匹配
    *   依赖资源不可用
    代理需要通过分析错误日志来诊断失败原因，应用针对性的修复，并成功重建项目。
2.  **实现任务 (Implementation tasks)**: 要求代理从头开始或修改现有配置以引入新功能，例如：
    *   在不同构建系统间迁移（如 Maven 迁移到 Gradle）。
    *   为特定用例配置发布目标。
    *   集成新插件。
    *   升级依赖版本。
    这类任务要求代理理解构建系统的语义和配置最佳实践。

**技术挑战与解决方案**:
*   **问题复现**: 真实 GitHub issue 的环境信息不完整，需要专家投入大量精力（约 40% 的初步选定问题需多次迭代）来重建工具链、编译配置等，以确保问题可稳定复现。
*   **合成任务设计**: 创建全面的合成构建任务是一项重大挑战，需要对仓库配置和编译机制有深入理解，并进行大量 DevOps 实验，才能构建出有意义且具挑战性的真实场景。
*   **精确评估**: 构建任务的评估非常复杂，需要多维度的评估指标。例如，对于迁移任务，需要设计统一的指标来衡量跨多种配置工具的等价性。为每个任务场景构建有效的评估流程都需要大量的工程和手动努力。

**任务详情与评估**:
*   **输入**: 包含构建失败配置的仓库（修复任务）或新构建需求的规范（实现任务），以及带有构建工具的终端访问权限。
*   **预期输出**: 修复构建失败的补丁（修复任务），或满足需求的完整配置文件（实现任务）。
*   **评估指标**:
    *   一个成功的构建过程包含两个子指标：1）构建过程无任何错误地执行；2）构建出的产物正确地实现了其所需功能。
    *   **修复任务**: 由于构建命令是标准化的，只要代理能无误地执行项目原有的构建命令，即可视为成功。
    *   **实现任务**: 不同的场景有不同的功能性衡量标准。例如，迁移任务要确保迁移后的实现与原始框架功能等价；插件集成要确保插件功能正常。这些指标最终都具体化为构建产物是否能通过其专属的测试用例。

#### **3.3 监控任务的构建 (Task Construction for Monitoring)**

本节专注于监控阶段的任务设计，这是 DevOps 中用于发现和诊断生产问题的关键环节。

**任务设计**:
*   **目标**: 监控任务要求代理在一个受控环境（如容器）中运行一个项目应用，通过外部命令行工具来（1）捕捉运行时执行状态和底层系统状态，以及（2）检测潜在的性能和资源使用异常。
*   **异常类型**: 任务侧重于**性能和资源异常**，而非直接导致系统崩溃的故障。因为即时崩溃可以通过简单的控制台输出来识别，评估价值有限；而性能和资源异常表现为系统逐渐退化，需要仔细分析才能发现，更能考验代理的诊断能力。
*   **具体异常场景**:
    1.  **资源使用问题 (Resource usage problems)**:
        *   内存泄漏 (Memory leaks)
        *   磁盘泄漏 (Disk leaks)
        *   系统句柄（文件、套接字）耗尽 (Handle exhaustion)
        *   CPU 峰值 (CPU spikes)
        这些问题代表了生产中最常见的资源相关故障，并且具有渐进性，例如内存泄漏可能需要数小时才能耗尽资源。
    2.  **性能下降 (Performance degradations)**:
        *   I/O 瓶颈 (I/O bottlenecks)
        *   低效的 SQL 查询处理 (Inefficient SQL query handling)
        这类问题不会导致直接失败，但会降低用户体验，表现为延迟增加或吞吐量下降。
    *   任务中也包含了**无异常的健康系统**，要求代理能正确识别出系统正常，避免误报。

**技术挑战与解决方案**:
*   **问题复现与合成任务设计**: 创建现实的、由专家注入的异常需要对代码库有深入的理解，以便在不破坏正常应用行为的情况下植入问题。设计的异常应在 5-15 分钟内通过标准监控工具显现，同时需要复杂的分析推理而非简单的检测。
*   **可观测性验证 (Observability validation)**: 专家需手动验证每个任务中的异常，确保它们可以通过提供的监控工具集（如 `top`, `free`, `ps`）被检测到，同时代理无法访问源代码或触发脚本。

**任务详情与评估**:
*   **输入**: 运行着带 bug 应用的容器化环境，以及一组标准系统监控工具的终端访问权限。
*   **预期输出**: 一份结构化的诊断报告，明确指出具体问题类型（如 `memory-leak`）并提供量化证据（如内存增长率、受影响的进程 ID）。
*   **评估指标**:
    *   主要指标是**二进制准确率 (binary accuracy)**，即代理是否正确识别出异常的具体类型。
    *   为了标准化评估，prompt 中明确定义了异常类型，并要求模型将诊断结果写入指定文件，不加任何解释。
    *   评估通过自动化的 `pytest` 脚本进行，检查诊断文件是否存在以及诊断类型是否与基准答案匹配。

#### **3.4 问题解决和测试生成任务的构建 (Task Construction for Issue Resolving and Test Generation)**

本节介绍了 DevOps-Gym 中问题解决与测试生成这两个阶段的设计。这两个阶段在现有的基准测试（如 SWE-bench 和 SWT-bench）中已有成熟的评估范式。

**任务设计**:
1.  **问题解决 (Issue Resolving)**:
    *   **目标**: 要求代理将 bug 的描述（可能来自监控告警或用户报告）转化为实际的代码修复。
    *   **流程**: 代理接收一个有 bug 的仓库和一份自然语言描述的 bug 说明，然后必须生成一个能够通过“失败到通过”（fail-to-pass）测试验证的补丁。
2.  **测试生成 (Test Generation)**:
    *   **目标**: 要求代理创建回归测试，以防止问题复现并确保功能的正确性。
    *   **流程**: 代理仅根据 bug 描述来创建测试用例，以确保全面的验证覆盖。这些生成的测试必须在有 bug 的代码上失败，但在打上补丁的代码上通过。

**与现有基准测试的主要区别**:
1.  **全面的去污染措施**: DevOps-Gym 实施了更严格的数据去污染程序，以减轻 LLM 训练数据可能带来的影响，确保评估结果的可靠性。
2.  **跨语言性能差距的关注**: 论文指出，LLM 代理在处理非 Python 语言（如 Java 和 Go）时性能会显著下降。DevOps-Gym 专注于这些语言，旨在揭示和量化这一差距，这可能源于训练数据中 Python 的主导地位。

**数据收集与评估**:
*   **数据收集**:
    *   **问题解决**: 采用了 Multi-SWE-bench 的收集流程，从未经良好维护的 Java 和 Go项目中筛选出那些通过 PR 解决了 issue 并有测试验证修复的案例，确保了问题的可复现性。
    *   **测试生成**: 任务来源于与问题解决相同的 issue 集合，为每个待修复的 bug 创建一个对应的测试生成场景。
*   **评估指标**:
    *   **问题解决**: 成功的标准是代理生成的补丁能够使所有给定的测试用例通过。
    *   **测试生成**: 成功的标准是代理生成的测试能够在原始的、有 bug 的代码上失败，但在应用了修复补丁的代码上成功通过。

#### **3.5 端到端任务 (End-to-End Tasks)**

除了评估单个 DevOps 阶段的能力，DevOps-Gym 还构建了端到端的流水线任务，以测试代理在多个阶段中维持上下文和解决连锁问题的能力。

**任务设计**:
*   **目标**: 模拟现实世界中问题从发现到解决的完整 DevOps 工作流。
*   **四阶段序列**: 每个端到端任务都遵循一个严格的四阶段序列：
    1.  **阶段 1: 构建 (Build)**
        *   **初始状态**: 提供一个已准备好用于监控的仓库，但作者故意注入了阻止编译的构建配置错误（如缺少依赖、路径不正确）。
        *   **任务**: 代理必须首先诊断并修复这些构建问题，建立一个可运行的工作基线。
    2.  **阶段 2: 监控与检测 (Monitoring and Detection)**
        *   **状态**: 构建成功后，代理部署并监控正在运行的系统。该系统中潜藏着一个来自监控任务集的性能或资源问题。
        *   **任务**: 代理必须使用系统工具来识别异常，此时它对问题的类型一无所知。
    3.  **阶段 3: 问题解决 (Issue Resolving)**
        *   **状态**: 基于在监控阶段的发现。
        *   **任务**: 代理需要实施代码级别的修复来解决问题的根源。
    4.  **阶段 4: 验证与测试 (Validation and Testing)**
        *   **状态**: 修复完成后。
        *   **任务**: 代理用其修复重新构建系统，并创建回归测试来验证问题已得到解决。测试必须专门针对在第 2 阶段识别出的性能特征，以确保问题不会再次发生。

**评估要点**:
*   **评估 DevOps 关键能力**: 这种流水线设计评估了在孤立任务中无法体现的关键 DevOps 能力，例如：
    *   在不同工具和阶段之间维持问题上下文。
    *   在阶段之间传递诊断信息。
    *   验证端到端的解决方案正确性。
*   **严格的成功标准**: 任务成功的条件是**必须顺序完成所有四个阶段**。在任何一个点失败都将终止整个流水线，这反映了现实世界中部署的约束——部分解决方案是无效的。

### **第 4 章：评估 (Evaluation)**

本章详细介绍了在 DEVOPS-GYM 基准测试上进行的综合评估，旨在揭示当前最先进的代理框架和语言模型在自动化端到端 DevOps 生命周期中的优势和不足。

#### **4.1 实验设置 (Experimental Setup)**

*   **代理与模型 (Agents and Models)**:
    *   评估了三种表现最佳的代理框架：**OpenHands**, **mini-SWE-agent**, 和 **Claude Code**。
    *   为了全面评估底层大语言模型（LLM）的影响，将这些框架与多个领先的 LLM 配对。具体来说，OpenHands 框架与五个模型进行了评估：Claude-4-sonnet, o4-mini, Gemini-2.5-Pro, Deepseek-v3.1, 和 Qwen3-Coder-30B。
    *   为了在相同模型下比较不同代理框架，mini-SWE-agent 和 Claude Code 都使用 Claude-4-Sonnet 作为其核心 LLM。
    *   初始评估集中在最先进的代理和模型上，以确定 DEVOPS-GYM 性能的上限。

*   **生成与环境 (Generation and Environment)**:
    *   模型通过其官方 API 访问，并设置了统一的超参数（温度 0.7，top-p 0.95）以确保结果的可比性。
    *   所有实验都在由 DEVOPS-GYM 提供的隔离且可复现的 Docker 容器环境中执行，以保证评估的公平性和一致性。

#### **4.2 结果 (Results)**

评估结果揭示了当前 AI 代理在处理复杂 DevOps 任务时的显著不足。

*   **构建与配置任务是新的挑战**:
    *   模型在该阶段表现差异巨大。Claude-4-Sonnet 表现远超其他模型，显示出其在处理大型项目配置管理和调试方面的卓越能力。
    *   即使使用相同的强大模型（Claude-4-Sonnet），代理框架的设计也至关重要。Claude-Code 的表现显著优于 mini-SWE-Agent，表明复杂的构建任务需要具有复杂工具调用能力和精心设计的架构的代理。
    *   所有模型在构建的“实现”任务（特别是迁移和目标发布任务）中表现都很差。这表明代理难以理解 Maven、GoReleaser 等构建工具的内部机制和实际使用模式，而不仅仅是解析错误日志。

*   **系统监控的动态性揭示了代理的关键失败点**:
    *   代理在监控任务上表现极差，部分最先进模型的成功率甚至为 0%。
    *   失败的根本原因在于监控任务的动态性，这对当前的 LLM 代理构成了三大挑战：
        1.  **处理连续时间输入**: 异常（如内存泄漏）是逐渐显现的，代理需要处理持续的、演变的系统状态信息流，这很快会耗尽其上下文长度。
        2.  **维持持续的注意力**: 代理即使被提示要持续监控，也常常会“分心”，过度分析早期的观察结果而忽略了实时的系统状态变化。
        3.  **基线辨别能力差**: 代理难以区分正常的系统操作波动和真正的异常，导致大量的误报。

*   **问题解决与测试生成任务的跨语言差距**:
    *   与在基于 Python 的 SWE-Bench 上的高成功率相比，代理在 DEVOPS-GYM 的 Java 和 Go 任务上解决问题的成功率急剧下降。这表明现有 LLM 在跨语言能力上存在巨大差距，可能由于训练数据中 Python 代码占主导地位。
    *   生成高质量的测试比解决问题本身更具挑战性。这要求代理不仅要静态理解代码，还需要动态分析能力来推断 bug 如何被触发以及如何被修复。

*   **端到端流水线任务的彻底失败**:
    *   在要求连续完成四个阶段的端到端任务中，所有代理的**成功率均为 0%**。
    *   尽管代理有时能成功完成第一阶段（构建修复），但它们几乎总是在第二阶段（监控）失败。
    *   这个结果凸显了一个关键差距：当前代理缺乏将单个 DevOps 任务串联成反映真实世界部署流程的、连贯工作流所需的上下文传播和多阶段规划能力。

#### **4.3 代理性能的稳定性 (Stability of Agent Performance)**

为了评估代理性能的稳定性，研究者从 DEVOPS-GYM 中随机抽样了 50 个任务，并让两个代表性的代理-模型组合（Claude Code + Claude-4-Sonnet 和 OpenHands + o4-mini）独立运行五次。

*   **结果**:
    *   两个代理在单次运行中的表现都相当一致，波动范围很小。
    *   多次运行的平均成功率与标准差表明，运行间的方差很小。
*   **结论**:
    *   这些发现证实了 DEVOPS-GYM 能够产生稳定的结果，并且模型性能对执行中的随机性不敏感。这增强了基准测试结果的可靠性和可信度。

### **第 5 章：结论与未来工作 (Conclusion and Future Works)**

本章对整篇论文进行了总结，并对未来的研究方向提出了展望。

**结论**:
作者成功提出了 **DEVOPS-GYM**，这是一个全面的基准测试，首次系统性地评估了 AI 代理系统在软件 DevOps 全生命周期中的能力。该基准覆盖了四个相互关联的关键阶段：构建与配置、监控、问题解决和测试生成。

通过对当前最先进的 AI 代理进行评估，研究揭示了它们的**显著局限性**。特别是在**监控**以及**构建与配置**这两个操作密集型（Ops）任务上，代理的表现尤其糟糕。这一发现凸显了当前 AI 技术的能力与实现真实世界 DevOps 自动化需求之间的**巨大鸿沟**。

作者强调，DEVOPS-GYM 精心选择了四个核心工作流。这些阶段既能代表 DevOps 自动化所需的核心推理和工具使用能力，又能在实践中保持严格和可复现的基准测试特性，避免了像 CI/CD 自动化或基础设施管理这类依赖于易变外部系统（如云 API）而难以评估的问题。

**未来工作**:
基于当前的研究，作者为未来的发展指出了几个充满希望的方向：
1.  **扩展任务集**: 遵循论文中提出的方法论，继续在每个阶段构建更多的任务，以覆盖更广泛的错误类型和更多样化的项目。这将进一步增强基准测试的全面性和挑战性。
2.  **扩展语言和阶段**: 将 DEVOPS-GYM 的覆盖范围扩展到其他流行的编程语言（如 Rust, C++ 等）和更多的 DevOps 阶段（如 CI/CD 自动化、基础设施管理等，尽管存在挑战）。
3.  **呼吁社区协作**: 论文最后呼吁整个研究社区共同努力，以推动 DevOps 自动化的发展。具体包括：
    *   通过贡献更多的场景和评估指标来**丰富 DEVOPS-GYM**。
    *   开发**能力更强、架构更优**的 AI 代理。
    *   训练专门为复杂代理工作流**优化的专用模型**。

总之，DEVOPS-GYM 不仅是一个评估工具，更是一个研究路线图，旨在引导和激励学术界与工业界共同攻克实现完全自主的 DevOps 这一宏大目标。
