## 指定章节详解

### 第一章：引言 (Introduction)

本章首先开宗明义，指出现代软件工程中“软件构建产物完整性”面临的
严峻挑战。SolarWinds 供应链攻击和 xz 压缩工具中的后门事件等近期
安全事件，凸显了信任不透明或未经验证的构建产物所固有的风险。
为了应对这些挑战，可复现构建（Reproducible Builds, R-B）被视为
一种至关重要的机制。R-B 的核心理念在于，通过允许独立方从源代码
重新生成二进制文件并验证其位对位等价性，从而将信任从分发者
转移到可审计和透明的构建过程。这种模式不仅通过验证所构建的
内容（而不仅仅是谁构建了它）来补充加密签名，还能确保软件的
可重构性，这对于软件的长期保存至关重要。

然而，尽管 R-B 具有巨大潜力，但其大规模实际应用面临两大核心挑战：
1.  **实现高可复现性率**: 在庞大且多样化的软件包集合中达到高可复现性
    仍然存在疑问。不过，最近对 Nix 包集合的研究表明，在超过 80,000 个
    软件包中实现了 90% 以上的可复现性，证明了大规模可复现性在技术上是
    可行的。
2.  **大规模监控可复现性**: 如何在大型软件生态系统中有效监控可复现性
    是一个较少被探讨但同样重要的问题。现有的监控基础设施通常是中心化的，
    计算成本高昂，且往往局限于部分软件包或近期版本。

本文提出的 Lila 系统，正是为了解决第二个挑战。Lila 是一个去中心化系统，
专门用于评估和聚合函数式包管理生态系统中的可复现性结果。它通过分布式
报告构建结果并聚合到一个可复现性数据库中，为开发者、维护者和未来的
可复现性研究提供支持。

### 第二章：背景 (Background)

本章为理解 Lila 的设计和影响奠定了基础，详细介绍了两个关键概念：
可复现构建和函数式包管理。

#### 2.1 可复现构建 (Reproducible builds)

可复现构建是一种软件工程实践，旨在确保在给定源代码和明确定义的
软件环境中，无论在哪台机器上、何时构建，都能可靠地生成位对位相同
的二进制产物。其主要目标是建立源代码与其编译输出之间可验证的链接，
允许第三方独立验证分布式软件的完整性。这直接解决了软件供应链中
一个根本性的信任假设：即发布的二进制文件忠实地对应其声明的源代码。

然而，传统的软件构建过程往往由于各种“非确定性”来源而产生非相同
的输出。这些非确定性因素包括：
*   **嵌入式构建时间戳 (Embedded build timestamps)**：构建过程中自动
    嵌入的时间信息。
*   **非确定性编译器行为 (Non-deterministic compiler behavior)**：
    编译器在不同运行中可能产生细微不同的输出。
*   **文件系统实现差异 (Differences in filesystem implementations)**：
    不同文件系统或其配置可能导致构建环境的差异。
*   **其他环境特定变量 (Other environment-specific variations)**：
    包括环境变量、本地化设置等。

可复现构建项目（Reproducible Builds Project）自 2015 年启动以来，
一直在定义实现构建可复现性所需的目标、方法和最佳实践方面发挥
核心作用。该项目提供了工具、指南和大量文档，帮助开发者识别和
消除构建非确定性来源。

#### 2.2 函数式包管理与 Nix (Functional package management and Nix)

Nix 和 Guix 是函数式包管理范式的实现，其核心思想是包被视为其声明
的构建时和运行时依赖关系的“纯函数”。这意味着：
*   **纯函数特性**: 包的构建输出仅取决于其输入（源代码和依赖），
    与构建环境中的其他外部因素无关。
*   **派生 (Derivations)**: Nix 表达式可重现地评估为精确定义的构建食谱，
    称为“派生”。派生指定了构建所需的准确软件环境。
*   **唯一标识**: 每个派生都通过加密的派生哈希值唯一标识。
*   **输出哈希**: 当派生被构建时，会产生包含最终构建产物的目录。
    这些输出也通过其“输出哈希”唯一标识，该哈希值来源于派生元数据，
    而非输出内容本身。这与传统包管理模型有根本不同。

Nixpkgs 是 Nix 社区维护的一个庞大的 Nix 表达式集合，它能够构建
各种软件。Nixpkgs 不仅是 Nix 生态系统和 NixOS Linux 发行版的包仓库，
目前包含超过 80,000 个软件包，是最大的 Linux 发行版包集合之一。

**存储替换机制 (Store Substitution Mechanism)**: 尽管 Nixpkgs 主要
是一个基于源代码的发行版，但 Nix 也支持二进制替换机制。这允许
用户用二进制缓存中的预构建二进制文件替换本地构建的输出。这
使得用户可以避免繁重的编译工作，但代价是必须信任第三方二进制
分发者。例如，`cache.nixos.org` 是一个中央二进制缓存，包含了
项目持续集成平台构建的所有包。

### 第三章：Lila (Lila)

本章深入探讨 Lila 的设计和其在实际应用中的潜力。

#### 3.1 通用设计 (General design)

Lila 的核心设计目标是实现可复现性证明的去中心化收集。图 2 提供了
Lila 整体设计的概览。

**客户端组件**:
Lila 客户端直接从 Nix 客户端的构建过程中捕获并报告构建结果。
其目标是为每个完成的构建记录一个可验证的声明，将构建食谱
（派生，derivation）与其产生的输出绑定起来。
为了实现这一点，Lila 利用 Nix 构建系统的 `post-build-hook` 机制。
这是一个 Nix 的特性，允许在本地构建完成后执行任意代码。当被
调用时，Lila 检查构建生成的所有输出路径，并创建一个证明。
这个证明包含：
*   定义构建的派生的加密哈希值。
*   所有生成输出内容的加密哈希值。
这个证明通过构建者特定的私钥进行签名，并通过认证的 API 请求
提交给聚合服务器。

**服务器组件**:
Lila 的服务器组件充当报告构建哈希的中心聚合器。它公开了一个
REST API（详见表 2），接受来自经过身份验证客户端的签名证明。
客户端使用系统管理员分配的私有令牌进行身份验证。服务器验证
报告的有效性，并将其存储以供后续查询和比较。此外，Lila 还可以
从 NixOS 持续集成平台中导入构建哈希，从而利用该基础设施执行的
全面构建集。表 1 详细描述了 Lila 数据库中证明条目的模式。

**Web 界面**:
服务器还包含一个用于生成和显示可复现性仪表板和报告的 Web 界面。
此界面使用户能够监控感兴趣的软件包集合，并及时发现可复现性退化。

#### 3.2 潜在用途 (Potential usage)

Lila 被设计为一个基础组件，既服务于寻求提升软件供应链安全性和
研究构建可复现性的从业者，也服务于研究人员。

**对于研究人员**:
Lila 为研究人员提供了宝贵的经验数据来源，有助于更好地理解大规模
构建可复现性所面临的挑战。其去中心化架构促进了从运行在各种
硬件和环境中的多样化用户群体收集可复现性报告，特别适合分析
特定软件生态系统中的构建可复现性。此外，Lila 可扩展的设计支持
集成额外的元数据收集机制，这有助于收集专门用于研究特定现象的
数据。例如，证明中包含的构建环境元数据可以帮助回答学术界此前
难以解决的研究问题，如位级可复现性对最终产物的影响。

**对于从业者和维护者**:
Lila 旨在成为可复现构建领域专家和维护者的实用工具。通过提供
可复现性仪表板，Lila 促进了对可复现性回归的及时识别，从而
实现更快的响应和修复。其分布式验证工作负载模型扩展了 NixOS
社区内可复现性监控的范围和时间跨度。这不仅允许监控更广泛的
软件包集合，还支持对旧版本软件包进行历史跟踪，从而实现对
软件包可复现性状态的长期分析。

最终，Lila 有望成为开发面向用户的工具的基础模块，这些工具将
利用位级可复现性来最小化或消除 Nix 二进制分发供应链中的信任关系。

### 第四章：相关工作 (Related work)

本章将 Lila 的工作与现有的位级构建可复现性（bitwise build reproducibility, R-B）
研究进行比较，突出了其独特的贡献。

**R-B 与软件供应链完整性**:
Lamb 等人[6]阐述了基于 R-B 的二进制分发系统的愿景，强调了其在消除
对分发中介盲目信任方面的作用。然而，实现这一愿景仍存在挑战，且目前
尚未完全实现。Lila 通过提供去中心化的监控机制，进一步推动了这一愿景。

**可复现性监控系统**:
当前的可复现性监控解决方案大多是中心化且项目特定的。例如：
*   `rebuilderd` 在 Arch Linux 生态系统中自动化重建和产物验证[11]。
*   Debian 和其他发行版维护着内部的可复现性检查基础设施。

这些系统在其适用范围内是有效的，但它们依赖于中心化的构建者，
这限制了其可扩展性。相比之下，Lila 旨在去中心化证明和监控过程，
从而分配计算工作负载和信任。这种设计选择受到了中心化监控系统
局限性的启发。

**R-B 的实证研究**:
实证研究为实现大规模软件包集合的可复现性提供了宝贵的见解。
*   Malka 等人[10]的研究表明，超过 90% 的 Nix 软件包集合是可复现的，
    证明了大规模 R-B 的可行性。
*   Sharma 等人[12]研究了 Maven 等特定生态系统中的可复现性。
*   Benedetti 等人[2]研究了六个流行软件生态系统中的可复现性变异性。

这些研究虽然富有洞察力，但由于其中心化性质，在范围上（软件包集
或时间范围）受到限制。Lila 通过提供一个可以从去中心化来源收集
可复现性数据的基础设施，补充了这些实证工作，既可用于操作监控，
也可用于未来的学术研究。

### 第五章：讨论与未来工作 (Discussion and future work)

本章讨论了 Lila 在 Nix 社区中的部署现状、其在研究中的潜力以及当前设计的
局限性，并展望了未来的发展方向。

**部署现状**:
Lila 正在 Nix 社区中逐步部署，目标是最终取代现有的、有限的可复现性
仪表板。其实验性实例已证明其有效性，已收集超过 150,000 份可复现性报告，
并生成了多个可复现性仪表板，这些仪表板帮助专家识别可复现性问题。
Lila 覆盖整个 Nix 生态系统这一宏大目标的成功，取决于可信第三方贡献
计算资源进行可复现性监控的意愿。目前已有数个参与方表达了兴趣，
这增强了我们对这种方法可行性的信心。从技术角度看，我们对少数用户
的初步部署并未发现设计固有的可扩展性问题。

**未来工作**:
*   **大规模研究**: 我们计划利用 Lila 数据库对函数式包管理生态系统中
    的位级可复现性及其随时间演变进行大规模研究。
*   **生态系统扩展**: 尽管 Lila 最初是为 Nix 生态系统开发的，但其设计
    与 Guix 完全兼容，甚至可以作为这两个生态系统之间共享的基础设施。
*   **去中心化程度的局限性**: Lila 的一个局限是，它仍然依赖中心化聚合
    服务器来收集和提供可复现性证明，仅在独立参与者之间实现了构建执行
    和证明的去中心化。实现完全点对点（peer-to-peer）的聚合将需要大量
    的工程投入，并带来协调和共识方面的挑战。

我们将目前的设计视为一个务实的初步步骤，它可以在当下用于改进软件
供应链安全，同时为未来更具雄心的完全去中心化可复现性跟踪架构留有余地。

### 第六章：结论 (Conclusion)

本文提出了 Lila，一个用于监控函数式包管理生态系统中构建可复现性
的去中心化基础设施。

通过利用 Nix 等函数式包管理器的特性，Lila 实现了跨不同环境和参与者
的可复现性证明的可扩展和去中心化收集。

我们的系统通过分布式验证工作负载，补充了现有的中心化可复现性监控
解决方案，提供了一个持久且可查询的构建报告数据库，并允许更安全的
包管理器实现基于该数据库进行构建。我们设想 Lila 既可以作为一个工具，
也可以作为一个数据库，支持未来的构建可复现性实证研究，从而能够
复现现有方法论，达到前所未有的大规模，并回答新的研究问题。
