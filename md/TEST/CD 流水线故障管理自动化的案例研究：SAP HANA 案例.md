# Using Large Language Models to Support Automation of Failure Management in CI/CD Pipelines: A Case Study in SAP HANA
# 使用大型语言模型支持 CI/CD 流水线故障管理自动化的案例研究：SAP HANA 案例

## 摘要
CI/CD 流水线故障管理若由人工执行，耗时且费力。由于有效故障管理所需的信息是非结构化的，无法被传统程序自动处理，因此自动化这一过程并非易事。鉴于大型语言模型（LLM）处理非结构化数据的能力，先前的工作已显示出其在自动化故障管理方面的良好前景。继这些研究之后，本文评估了一个基于 LLM 的系统是否能在大型工业软件项目（即 SAP HANA）的背景下自动化 CI/CD 流水线中的故障管理。我们评估了基于 LLM 的系统识别错误位置并提出不包含不必要操作的准确解决方案的能力。为了支持 LLM 生成准确的解决方案，我们为其提供了不同类型的领域知识，包括流水线信息、故障管理指令和历史故障数据。我们进行了一项消融研究，以确定哪种类型的领域知识对解决方案的准确性贡献最大。结果表明，来自历史故障的数据对系统的准确性贡献最大，使其能够在 92.1% 的数据集中生成准确的解决方案。当提供领域知识时，该系统以 97.4% 的准确率正确识别了错误位置，而在没有领域知识的情况下，准确率为 84.2%。总之，我们的研究结果表明，当提供历史故障数据时，LLM 是自动化 CI/CD 流水线故障管理的一种有前途的方法。

## 各章节主旨内容
*   **I. 引言 (Introduction)**：介绍了 CI/CD 流水线故障管理的挑战，人工管理的弊端（耗时、易错），以及 LLM 在处理非结构化日志方面的潜力。提出了本文的研究问题（RQ1-RQ3）。
*   **II. 背景 (Background)**：介绍了 SAP HANA 的交付流程、Jenkins 流水线及其在 SAP HANA 中的应用，以及相关工作。特别描述了复杂的流水线层级结构（主流水线、子流水线、远程流水线）。
*   **III. 方法论 (Methodology)**：详细阐述了基于 LLM 的故障管理系统的设计，包括如何定位最下游的失败任务（Most Downstream Failed Job）、日志预处理步骤、提示工程（Prompt Engineering）以及使用的领域知识类型（流水线信息 PI、故障管理指令 FMI、历史记录 HR）。
*   **IV. 实验结果 (Experiment Results)**：回答了三个研究问题：识别了 SAP HANA CI/CD 流水线中最常见的故障原因（RQ1）；评估了 LLM 定位错误位置的准确性（RQ2）；通过消融研究分析了不同领域知识对解决方案准确性的贡献（RQ3）。
*   **V. 经验教训 (Lessons Learned)**：总结了 LLM 支持自动化故障识别和修复的可行性，强调了函数调用（Function Calling）和领域知识的重要性。
*   **VI. 有效性威胁 (Threats to Validity)**：讨论了构造效度、内部效度和外部效度，指出了数据集限制和模型特定性等潜在问题。
*   **VII. 结论 (Conclusion)**：总结了研究发现，指出远程服务错误是主要故障原因，LLM 在定位故障和提供解决方案方面表现出色，尤其是结合历史故障记录时。建议采用混合方法（非 LLM + LLM）以提高成本效益。

## 结论
本研究验证了大型语言模型（LLM）在 SAP HANA 交付的 CI/CD 流水线中自动化故障管理的潜力。实验结果表明，最常见的故障原因是由远程服务触发的错误，因此无法完全防止复发。其次，具有函数调用功能的 LLM 可以正确识别最下游的失败任务，在提供领域知识的情况下表现更好（准确率 98%）。第三，相关失败构建的历史记录对 LLM 提出解决方案的准确性贡献最大。包含历史记录的消融条件在不同根本原因下产生了更高的准确性和更低的方差。鉴于某些故障频繁复发，作者建议未来采用混合方法，优先尝试非 LLM 解决方案，仅在失败时调用 LLM，以实现成本效益最大化。

# 章节详解

## I. 引言 (Introduction)

本章首先确立了 CI/CD 流水线在现代软件开发中的关键地位，同时也指出了其维护的成本高昂。作者强调，尽管有重试和断路器等弹性模式，但它们无法应对所有类型的故障。人工介入（On-call 工程师）不仅耗时，而且需要深厚的专业知识来从海量且充满干扰信息的日志中识别根本原因。

### 核心痛点
1.  **非结构化数据**：故障信息（如日志）通常是非结构化的，传统自动化工具难以处理。
2.  **环境动态性**：CI/CD 流水线频繁更新，导致日志格式变化，使得基于特定规则或需要大量训练数据的传统机器学习模型难以适应（“Unseen log formats”）。
3.  **数据稀缺性**：日志格式更新后，新样本数量少，重新训练模型成本高且困难。

### 解决方案方向
LLM 凭借其处理非结构化数据且无需特定任务训练的能力，成为解决上述问题的理想选择。本文以 SAP HANA 这一大型工业项目为例，探讨 LLM 在实际场景中的应用效果。

## II. 背景 (Background)

### SAP HANA 与 Jenkins 流水线
SAP HANA 是一个内存数据库管理系统，分为 On-Premise 和 Cloud 两个版本，分别有不同的发布周期（月度和周度）。其交付依赖于 Jenkins CI/CD 流水线。
文章详细描述了 Jenkins 流水线的结构：
*   **层级化设计**：流水线可以调用其他流水线（下游流水线或子流水线），形成层级结构。
*   **Freestyle Job**：不调用子流水线的具体任务步骤被称为 Jenkins freestyle job，通常执行具体的脚本或命令。
*   **远程流水线**：除了本地子流水线，还会触发运行在不同 Jenkins 服务器上的远程流水线，这些通常涉及版本控制（Git）、代码审查（Gerrit）等外部服务，且由其他团队管理。

### 故障诊断的复杂性
由于流水线的层级结构（Main Pipeline -> Sub-pipeline -> Job），错误往往发生在最末端的某个节点。
*   **嵌套日志**：人工排查需要逐层点击控制台输出链接，直到找到包含错误信息的“最下游失败任务”（Most Downstream Failed Job）。
*   **工具局限性**：现有的 Jenkins 插件（如 Blue Ocean）无法完整展示包含远程流水线和子流水线的整个层级结构，导致无法直观定位故障点。

## III. 方法论 (Methodology)

本章是论文的核心，详细介绍了系统的设计与实现。系统利用 OpenAI 的 GPT-4o 模型，通过 Jenkins Pipeline 实现自动化触发。

### 1. 定位最下游失败任务 (Most Downstream Failed Job Finding)
这是故障诊断的第一步。由于控制台日志只显示下游任务失败的事实，而不显示根本原因，因此必须找到最底层的任务日志。
*   **Coordinator (协调器)**：具有函数调用能力的 LLM Agent，负责迭代式地寻找故障点。
*   **Downstream Job Finder**：一个子系统（或工具），被协调器调用。它读取当前层级的日志，利用 LLM 分析并返回失败的下游任务名称（如 DS-1）。
*   **迭代流程**：Coordinator 调用 Finder -> Finder 返回子任务 DS-1 -> Coordinator 再次调用 Finder 检查 DS-1 -> 直到 Finder 返回“当前已是最下游任务”。
*   **备用方案**：虽然主要依赖正则匹配日志中的任务链接，但为了应对日志格式变化，系统设计了基于 LLM 的查找机制作为补充。

### 2. 日志预处理 (Log Preprocessing)
为了适应 LLM 的上下文窗口并降低成本，原始日志必须经过精简：
*   **过滤无关信息**：移除 Jenkins 的状态更新关键字（如时间戳、echo 命令）。
*   **差分过滤**：对比失败构建与上一次成功构建的日志，移除相同的行（假设相同行代表正常流程）。虽然这种方法不如基于模板的提取准确，但在频繁变更的 CI/CD 环境中更具成本效益。
*   **关键字提取**：保留包含 "failure", "error", "exception" 的行及其前后 5 行上下文。
*   **脱敏处理**：使用 StarPII 模型移除敏感信息（如 IP、邮箱）。

### 3. 基于 LLM 的原因分析与解决方案生成
使用角色扮演（Role-playing）提示技术，让 LLM 扮演软件工程师。
*   **输入**：流水线结构信息、预处理后的最下游任务日志、领域知识。
*   **输出**：故障原因（Causes of Failures）和推荐解决方案（Recommended Solutions）。

### 4. 领域知识 (Domain Knowledge)
为了提高准确性，系统集成了三类知识：
*   **流水线信息 (PI)**：流水线结构、命名规范、远程 URL 等静态信息。
*   **故障管理指令 (FMI)**：资深工程师的处理步骤和常见故障解决方案。
*   **历史故障记录 (HR)**：**关键设计**。使用 RAG（检索增强生成）技术，检索与当前日志最相似的前 3 个历史故障案例（包含根因、日志行、解决方案），作为 Few-shot 示例提供给 LLM。

## IV. 实验结果 (Experiment Results)

### RQ1: 常见故障原因
*   分析了 6 个月内的 76 次故障。
*   **Top 1 原因 (16次)**：Gerrit 代码审查过程中的容器镜像版本设置错误。这是跨团队协作的典型问题。
*   **Top 2 原因 (13次)**：轮询构建服务器状态时失败（通常因编译错误导致）。
*   **结论**：大部分故障源于远程服务或外部依赖，而非流水线脚本本身的逻辑错误。

### RQ2: 错误位置定位准确性
*   **无领域知识**：准确率 84.2%（64/76）。主要错误在于“幻觉”，即 LLM 臆造了不存在的下游任务。
*   **有领域知识**：准确率 **97.4%**（74/76）。领域知识（特别是日志格式和命名规范）显著帮助 LLM 避免了错误的路径探索。
*   **稳定性**：有领域知识的情况下，系统性能的方差更小，表现更一致。

### RQ3: 领域知识的贡献 (消融研究)
作者将解决方案质量分为三级：红色（错误）、黄色（间接解决/有冗余）、绿色（精确解决）。
*   **HR (历史记录) 的决定性作用**：包含 HR 的组合（HR, PI+HR, FMI+HR, PI+FMI+HR）表现最佳，准确率在 90% 以上。
*   **单项对比**：
    *   仅 HR：70 个精确解。
    *   仅 FMI：40 个精确解。
    *   仅 PI：15 个精确解。
    *   无知识：仅 13 个精确解，50 个错误解。
*   **结论**：历史故障记录是提升 LLM 解决特定领域复杂问题能力的最关键因素。相比之下，单纯的静态流水线信息帮助有限。

## V. 经验教训 (Lessons Learned)

1.  **端到端自动化的可行性**：通过 Function Calling，LLM 不仅能分析，理论上也能执行修复操作（如重启任务、回滚版本）。
2.  **知识库的动态维护**：历史记录库需要持续更新。对于新出现的故障类型，系统初次可能失败，但一旦人工解决并录入库中，系统即可学会处理，形成正向循环。

## VI. 有效性威胁 (Threats to Validity)

*   **数据集规模**：仅使用了 76 个故障案例，样本量相对较小。但这是实际生产环境半年的数据，具有代表性。
*   **模型特定性**：仅测试了 GPT-4o (temperature 0.0)，未对比其他模型。
*   **外部效度**：研究高度定制于 SAP HANA 的 Jenkins 环境，推广到其他 CI/CD 工具或架构可能需要调整。

## VII. 结论与未来展望 (Conclusion)

文章最后提出了一个务实的**混合系统（Hybrid System）**构想：
鉴于许多故障是重复的（如网络抖动、特定编译错误），直接调用昂贵的 LLM 并不划算。
**建议流程**：
1.  先尝试传统的非 LLM 方法（如正则匹配历史指纹）。
2.  如果非 LLM 方法失败（无法定位或无匹配历史），再启动 LLM Agent 进行深度分析。
这种分层处理策略能在保证高解决率的同时，大幅降低运营成本。
