## 论文整体骨架

### 中文标题
弥合差距：揭示异构 AI 加速器间的行为不一致性

### 中文摘要
尽管 NVIDIA 仍然是云数据中心 AI 加速器的主导供应商，但 AMD、Intel、Mac 和华为等新兴供应商正以具有竞争力的价格提供声称兼容且高性能的替代方案。本文首次对异构 AI 加速器上机器学习模型的行为差异（divergence）进行了实证研究。我们利用一个自动化流水线，从 4000 个真实世界模型中合成了超过 10 万个变体模型，并在五个不同的企业级加速器上执行它们。研究结果表明，来自 Mac 和华为的新 AI 平台所支持的算子（operators）数量比 NVIDIA 至少少 17%。这些平台还表现出更高的输出差异率（超过 5%），这源于算子实现的差异、对异常数值的处理方式不同以及指令调度上的变化。此外，它们在基于模型编译的加速过程中更容易出现故障；在某些情况下，编译后的模型产生的输出与标准执行模式下的输出有显著不同。我们还发现了 PyTorch 框架中的 7 个实现缺陷以及跨厂商的 40 个平台特定问题。这些结果凸显了在日益多样化的硬件生态系统中实现一致的机器学习行为所面临的挑战。

### 各章节主旨内容
*   **第 1 章：引言 (Introduction)**
    *   本章介绍了研究背景：随着机器学习的普及，除了主流的 NVIDIA GPU，市场上出现了越来越多来自 AMD、Intel、Mac、华为等厂商的 AI 加速器。这些新兴平台声称能以更低的成本提供兼容 PyTorch 的高性能计算。然而，硬件的多样性也引发了对其行为一致性和可靠性的担忧。本研究旨在通过实证研究，回答一个核心问题：机器学习模型在不同硬件平台上的执行结果一致性如何？潜在的不一致性来源是什么？

*   **第 2 章：背景与相关工作 (Background and Related Work)**
    *   本章首先简要介绍了 PyTorch 框架，解释了从高级 Python 代码到低级 ATen 算子构成的计算图的转换过程，这是后续分析的基础。接着，本章回顾了现有的机器学习框架缺陷检测技术，如 CRADLE、AUDEE、LEMON 等，这些工作大多通过模型诱变或差分测试来发现软件层面的 bug。本研究在此基础上进行了扩展，将差分测试的维度从软件框架之间转移到了不同硬件加速器之间，旨在揭示由底层硬件或驱动差异引起的不一致性。

*   **第 3 章：流水线实现 (Pipeline Implementation)**
    *   本章详细描述了用于实验的自动化测试流水线。该流水线包含以下几个关键步骤：
        1.  **数据集构建 (Dataset Curation)**：从 Hugging Face、TorchHub 和 Papers with Code 等来源收集了近 4000 个具有独特架构的真实世界 PyTorch 模型，构建了一个庞大的模型语料库。
        2.  **计算图提取 (Computation Graph Extraction)**：通过插桩（instrumenting）PyTorch 的 ATen 分发器，精确地捕获模型的计算图，包括算子调用、参数和张量依赖关系。
        3.  **变体模型生成 (Variant Model Generation)**：采用图融合和算子诱变技术，从语料库中随机组合和修改计算图，自动生成超过 10 万个用于测试的变体模型。
        4.  **模型执行 (Model Execution)**：在默认（Eager）、JIT 编译和 TorchDynamo 编译三种模式下，在五个不同的硬件平台（NVIDIA H200, AMD MI300X, Intel MAX 1100, Huawei Ascend 910B, Mac M4 Pro）上执行生成的模型。
        5.  **输出比较与差异分析 (Output Comparison and Analysis Discrepancy)**：使用 TensorFlow 和 PyTorch 的标准公差比较不同平台间的输出结果。一旦发现差异，通过逐层对比数据流图，精确定位到第一个产生分歧的算子。

*   **第 4 章：评估 (Evaluation)**
    *   本章展示并分析了实验结果。首先，分析了模型在不同平台上的执行失败情况，发现 Mac 和华为平台因“算子不支持”导致的失败率显著更高（超过 10%）。深入分析揭示，这两个平台在量化、稀疏计算、注意力机制、NLP 嵌入和高级线性代数等方面的算子支持严重不足。此外，还发现了 AMD 平台上边界检查机制的缺失问题。接着，通过对比成功执行的模型的输出，发现 Mac 和华为平台的输出不一致性比率最高（分别达到 14.1% 和 5.1%）。通过案例分析，将不一致的原因归结为四类：错误的算子实现、异常数值处理的差异、指令调度的不确定性以及未定义行为的处理。最后，本章还评估了 JIT 和 TorchDynamo 编译模式下的可靠性，发现编译会引入更多的不一致性，尤其是在 Mac 和华为平台上。

*   **第 5 章：结论 (Conclusion)**
    *   本章总结了研究的主要发现：新兴的 AI 加速器平台（尤其是 Mac 和华为）在算子支持、行为一致性和编译可靠性方面与成熟的 NVIDIA 平台存在显著差距。这些不一致性主要源于算子实现、异常值处理和指令调度的差异。研究结果凸显了在日益异构的硬件环境中确保机器学习行为一致性的巨大挑战，并为开发者在选择硬件平台和进行代码迁移时提供了重要的实践参考。

### 中文结论
在本文中，我们探讨了机器学习模型在五大主流云数据中心 AI 加速器上的输出一致性。我们的研究结果表明，与 NVIDIA 相比，来自 Mac 和华为的新兴 AI 平台所支持的算子至少要少 17%，尤其是在对训练至关重要的算子方面。这些平台也表现出更高的输出不一致率（超过 5%），这可以归因于算子实现的差异、对异常数值的处理方式不同以及指令调度的变化。此外，在启用基于模型编译的加速时，它们更容易出现故障。这些结果凸显了在日益多样化的硬件平台范围内确保机器学习行为一致性的困难。

## 指定章节详解

### 第 1 章：引言 (Introduction)

本章为整篇论文的研究奠定了背景和动机。随着机器学习（ML）技术的飞速发展，对计算能力的需求也急剧增长。为了满足这一需求，云服务提供商除了广泛采用的 NVIDIA GPU 之外，还开始提供越来越多来自不同供应商（如 AMD, Intel, Mac, Huawei）的 AI 加速器作为替代方案。这些新兴平台通常以更具竞争力的价格，声称提供与主流 ML 框架（特别是 PyTorch）兼容的高性能计算能力。

然而，硬件平台日益增长的多样性给开发者带来了新的挑战，核心是**一致性 (consistency)** 和**可靠性 (reliability)** 的问题。当开发者希望摆脱对单一供应商（NVIDIA）的依赖（即 vendor lock-in），转而采用更具成本效益的异构硬件方案时，他们必须确信自己的模型在不同硬件上的行为是可预测且一致的。

基于此背景，本文提出了该领域的第一个大规模实证研究，旨在回答以下核心研究问题：
1.  **机器学习模型在不同硬件平台上的输出有多一致？**
2.  **导致不一致的潜在根源是什么？**

为了回答这些问题，研究者采用了一种**差分测试 (differential testing)** 的方法。他们设计并实现了一个自动化的流水线（如图 1 所示）：
*   首先，从网络上搜集了约 4000 个真实世界的 ML 模型。
*   然后，利用这些模型作为“基因库”，通过图合并和节点诱变技术，自动“合成”出超过 10 万个具有多样性的新模型。
*   接着，用相同的随机输入，在五个不同的企业级 AI 加速器上运行这些合成模型。
*   最后，跨平台比较模型的输出结果，对检测到的任何显著差异进行深入分析，以追溯其根本原因。

本章最后预告了研究的主要发现，包括：1) 新平台（Mac, Huawei）在算子支持上存在明显短板；2) 这些新平台在输出结果上表现出更高的不一致性，其根源在于算子实现缺陷、异常值处理差异等；3) 启用编译加速功能时，这些平台的可靠性会进一步下降。这些发现共同揭示了在异构硬件生态中实现一致性 ML 行为的巨大挑战。

### 第 2 章：背景与相关工作 (Background and Related Work)

本章首先为非专业的读者提供了理解后续内容所需的基础知识，然后回顾了学术界在相关领域已有的研究工作，从而凸显本研究的创新性和贡献。

#### ML 模型与计算图 (ML Model and Computation Graph)

本节以 PyTorch 为例，解释了现代深度学习框架是如何工作的。
*   **高层定义**：在 PyTorch 中，一个模型通常被定义为一个继承自 `nn.Module` 的 Python 类。模型的网络层（如线性层、卷积层）在 `__init__` 方法中声明，而模型的计算逻辑（数据如何流经这些层）则在 `forward` 方法中定义。
*   **底层表示**：当高层的 Python 代码执行时，它会被 PyTorch 的后端转换成一个**计算图 (Computation Graph)**。这是一个有向无环图（DAG），其中**节点 (node)** 代表一个具体的操作（如加法、矩阵乘法），这些操作在 PyTorch 的 C++ 后端被称为 **ATen 算子**（例如 `aten::add`, `aten::addmm`）。图中的**边 (edge)** 代表数据流，即一个算子的输出作为另一个算子的输入。输入和输出张量可以被看作是图的特殊源节点和汇节点。（如图 2 所示）
*   **硬件调度**：最终，这些 ATen 算子会被进一步分发到与具体硬件匹配的底层实现上，例如在 NVIDIA GPU 上会调用对应的 CUDA 核函数（如 `relu_cuda`）。这个从高级 Python API 到硬件特定指令的转换过程是所有优化的基础。

#### 框架缺陷检测 (Framework Bug Detection)

本节回顾了学术界在自动检测深度学习框架（如 TensorFlow, PyTorch）自身 bug 方面的研究进展。这些工作为本论文的研究方法提供了灵感。
*   **跨框架差分测试**：早期的工作如 CRADLE，通过在不同框架（例如 PyTorch vs. TensorFlow）上运行相同的模型，比较它们的输出差异来发现 bug。
*   **模型生成与诱变**：后续工作如 AUDEE, LEMON, Muffin, EAGLE 等，不再局限于使用真实世界的模型，而是通过遗传算法或图论技术自动生成或“诱变”（mutate）出大量新模型作为测试用例，以探索更广泛的代码路径，从而发现更多隐藏的 bug。NNSmith 在此基础上引入了更高级的抽象算子建模，进一步增强了生成模型的多样性。
*   **算子级别差分测试**：近期的研究如 FreeFuzz, SkipFuzz, IvySyn 则将测试的粒度从模型级别下沉到算子级别，通过对单个算子的参数进行随机变异来发现问题。

**本研究的贡献**：之前的工作主要集中在**软件层面**，即在同一硬件上比较不同框架或框架版本的行为。本研究则开辟了一个新的维度：**硬件层面**的差分测试。它使用统一的软件栈（PyTorch），但在不同的硬件加速器上运行模型，从而专门揭示由于硬件、驱动或底层库的差异所导致的**行为不一致性**。这是一个在文献中尚未被充分探讨、但对实际应用具有重要意义的补充性研究方向。

### 第 3 章：流水线实现 (Pipeline Implementation)

本章详细介绍了用于自动化测试的流水线（Pipeline）的具体实现方法，这是整个研究的技术核心。

#### A. 数据集构建 (Dataset Curation)

为了生成多样化的测试模型，首先需要一个丰富且真实的模型“语料库”。研究者通过以下方式构建了这个语料库：
1.  **网络爬虫**：开发了一个爬虫程序，从两个主流的模型分发平台 Hugging Face 和 TorchHub 上收集了约 17 万个预训练的 PyTorch 模型。
2.  **去重**：由于许多模型仅仅是权重不同但架构相同（例如，在不同数据集上微调的同一个模型），研究者通过分析模型的配置文件（`config.json`）和基类（如 `BertModel`）来识别并剔除架构重复的模型，最终保留了 3,711 个具有独特架构的模型。
3.  **补充来源**：为了增加代码结构和优化技术的多样性，研究者还从学术平台 Papers with Code 中手动挑选了 300 个与论文相关的代码仓库。

#### B. 计算图提取 (Computation Graph Extraction)

为了能够对模型进行分析和操作，需要将其从 Python 代码形式转换为结构化的计算图。研究者通过**插桩 (instrumenting)** PyTorch 底层的 ATen 分发函数（如 `kernel.call`）来实现这一点。每当模型执行一个算子时，插桩代码就会记录下这次调用的详细信息，包括：
*   算子名称。
*   输入参数（如果是普通值则记录值，如果是张量则记录其唯一标识符、形状、数据类型等元数据）。
*   返回值（同样记录张量的元数据）。

通过追踪张量在不同算子调用之间的产生和消耗关系，就可以精确地、无侵入地重建出模型的完整计算图，且该方法同时支持前向和后向传播图的提取。

#### C. 变体模型生成 (Variant Model Generation)

这是流水线创造力的核心。基于 GraphFuzz 的思想，通过以下两种方式生成大量用于测试的新模型：
1.  **图融合 (Graph Merging)**：从一个空图开始，迭代地从语料库中随机抽取一个模型的计算图的某个子图，然后将其“嫁接”到当前正在生成的图中。如果接口处的张量维度不匹配，会自动插入“胶水节点”（如 `flatten`, `slice`, `pad`, `reshape` 等算子）来解决冲突。这个过程持续进行，直到图的大小达到预设的阈值（本研究中设为 1000 个节点）。
2.  **算子诱变 (Operator Mutation)**：为了进一步增加多样性，以一定的概率（如 25%）对图中的算子进行替换。例如，将一个“非原地”（out-of-place）操作 `aten::add` 替换为其“原地”（in-place）版本 `aten::add_`，或者将一个激活函数 `aten::relu` 替换为 `aten::hardtanh`。

#### D. 模型执行 (Model Execution)

生成的模型会在三种不同的 PyTorch 执行模式下运行：
1.  **默认（Eager）模式**：标准的解释执行模式，操作逐条执行，不进行优化。
2.  **即时编译（JIT）模式**：通过 `torch.jit`，PyTorch 会追踪模型的执行，进行一些图优化（如常量折叠），然后编译成更高效的形式。
3.  **TorchDynamo 编译模式**：这是 PyTorch 最新的编译模式，它能将计算图交给特定于硬件厂商的后端（如 Intel 的 IPEX）进行深度优化和编译。

#### E. 输出比较与差异分析 (Output Comparison and Analysis Discrepancy)

这是发现问题的最后一步。
1.  **输出比较**：采用 TensorFlow 和 PyTorch 测试套件中标准的容差比较公式 `|Op1 - Op2| ≤ atol + rtol· |Op2|` 来判断两个平台的输出是否一致。
2.  **差异定位**：一旦检测到最终输出不一致，流水线会回溯整个模型的**数据流图**。从模型的第一个算子开始，逐层比较每个算子在不同平台上的输出。当第一次发现某个算子的输出开始出现差异时，就将这个算子标记为导致不一致的根源。通过计算输出差异的平均绝对距离（MAD）的变化率，可以精确地定位到问题的源头。

这个设计精巧的自动化流水线，使得研究者能够系统性、大规模地对异构 AI 加速器的行为一致性进行深入的探索。

### 第 4 章：评估 (Evaluation)

本章是论文的核心，展示了通过前述流水线发现的关于五个 AI 加速器平台（NVIDIA H200, AMD MI300X, Intel MAX 1100, Huawei Ascend 910B, Mac M4 Pro）行为差异的实证结果。

#### A. 模型执行的差异 (Discrepancy in Model Execution)

本节分析了在默认（Eager）执行模式下，各平台运行10万个合成模型时的表现。

*   **执行失败率 (Failure Rate)**：
    *   **NVIDIA (基准)**：失败率最低，约为 2.5%，失败原因主要是随机输入导致的合法错误，如索引越界或形状不匹配。
    *   **Mac 和 Huawei**：失败率显著更高，均超过 10%（图 3）。失败的主要原因是**算子不支持 (Unsupported Ops)**。这意味着这两个新兴平台尚未实现 PyTorch 的全部算子集。
    *   **算子支持情况**：深入分析发现，与 NVIDIA 支持 488 个可加速算子相比，华为只支持 407 个（少 17.5%），Mac 只支持 332 个（少 34.0%）。缺失的算子主要集中在以下几个领域：
        1.  **量化推理**：用于模型压缩和加速。
        2.  **稀疏运算**：处理含大量零值的张量。
        3.  **注意力机制**：如 FlashAttention 等高性能实现。
        4.  **NLP 嵌入**：特别是一些后向传播（训练）相关的算子。
        5.  **融合优化器**：如 Fused Adam。
        6.  **高级线性代数**。
        此外，这两个平台对 64 位浮点数和 64 位整数索引的支持也非常有限。

*   **特殊的平台行为**：
    *   **AMD 边界检查缺失**：AMD 平台报告的“索引越界”错误比 NVIDIA 少得多。这意味着在某些本应报错的情况下，AMD 平台继续执行，这可能导致难以察觉的内存访问错误，是一个潜在的安全漏洞 (**发现 3**)。

#### B. 模型输出的不一致性与案例研究 (Inconsistency in Model Outputs and Case Studies)

本节分析了在所有平台上都成功执行的 87,840 个模型，并比较了它们的输出。

*   **不一致率**（图 4）：
    *   **AMD 和 Intel**：表现最好，与 NVIDIA 的输出差异率分别仅为 0.2% 和 0.4% (**发现 4**)。
    *   **Huawei 和 Mac**：表现较差，不一致率分别高达 5.1% 和 14.1%。

*   **不一致的根源分析** (**发现 5**)：通过手动分析案例，研究者将不一致的原因归为四类：
    1.  **算子实现缺陷**：某些算子在特定平台上的实现是错误的。例如，华为的 `max_pool1d_with_indices` 在输入为 64 位整数时返回了错误的类型（8位无符号整型）；Mac 的 `aten::pad` 在填充尺寸过大时会产生数据损坏。
    2.  **异常数值处理**：不同平台对 `Inf` (无穷大) 和 `NaN` (非数值) 的处理和传播方式不同。例如，NVIDIA 上 `Inf / Inf` 得到 `NaN`，而在 AMD 和 Intel 上得到 `Inf`。更糟糕的是，AMD 的 `batch_norm` 会用插值替换 `NaN`，而 Mac 的 `reshape` 会将 `NaN` 静默地转换为 0，这都掩盖了潜在的数值问题。
    3.  **指令调度**：由于并行计算的内在不确定性，某些操作（如 `max_unpool2d`）在原子操作或内存访问顺序上的微小差异，会导致在不同平台或不同运行中产生略有不同的结果。NVIDIA 在这方面表现出最好的确定性，而 AMD 则会产生多种不同的合法输出。
    4.  **未定义行为**：例如，正整数除以零，在 NVIDIA 上返回 32 位无符号整数的最大值，而在 macOS 上返回 0，行为各异。

#### C. 模型编译问题分析 (Analysis of Model Compilation Issues)

### 第 5 章：结论 (Conclusion)

本章是对整个研究工作的总结陈词。论文通过大规模的实证研究，系统地探讨了在五大主流 AI 加速器平台上机器学习模型的行为一致性问题。

研究的核心发现可以概括如下：
1.  **新兴平台存在显著功能差距**：与市场领导者 NVIDIA 相比，新兴的 AI 加速器平台，特别是 Mac 和华为，在算子支持方面存在巨大鸿沟（至少少 17%）。缺失的算子主要集中在对模型训练和高级推理（如量化）至关重要的领域。这意味着开发者如果想将模型从 NVIDIA 迁移到这些平台，很可能需要进行大量代码重构，甚至牺牲性能和精度。
2.  **输出不一致性问题普遍存在**：即使模型能够在所有平台上成功运行，其输出结果也未必一致。Mac 和华为平台表现出超过 5% 的高不一致率。这些差异的根源多种多样，包括：
    *   算子本身的实现缺陷。
    *   对 `NaN` 和 `Inf` 等异常数值的不同处理逻辑。
    *   由并行指令调度引入的细微不确定性。
3.  **编译加速功能尚不成熟**：当启用 PyTorch 的 JIT 或 TorchDynamo 编译功能以追求更高性能时，这些平台，尤其是 Mac 和华为，暴露出更多的可靠性问题，包括更高的输出不一致率和大量的编译失败。

总而言之，这项研究清晰地描绘了当前异构 AI 硬件生态的现状：虽然多样化的硬件选择为市场带来了活力和成本效益，但其背后的一致性和可靠性问题不容忽视。对于追求多平台部署和避免供应商锁定的开发者而言，这些发现提供了宝贵的警示和实践指导。它强调了在异构计算时代，建立标准化测试、确保跨平台行为一致性将是整个行业需要共同面对的重要挑战。
