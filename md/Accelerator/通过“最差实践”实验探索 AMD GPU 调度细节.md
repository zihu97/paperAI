**中文标题**: 通过“最差实践”实验探索 AMD GPU 调度细节
**中文摘要**: 图形处理单元（GPU）已成为近期大量实时研究的目标，但这类研究常因 GPU 软硬件的“黑盒”性质而受阻。尽管 AMD 公司已采用开源软件栈，人们可能期望针对 AMD GPU 的实时研究会增多，然而现实更为复杂。在不了解内部细节差异的情况下，研究人员无法假定基于 NVIDIA GPU 的观察结论同样适用于 AMD GPU。此外，AMD 软件的开放性并不意味着其调度行为是显而易见的，特别是其文档稀疏且分散。本文中，我们将分散的文档整合为单一的连贯来源，提供了一个关于 AMD GPU 如何调度计算工作的端到端描述。我们首先通过一个具体示例，展示了不正确的管理方式如何在共享的 AMD GPU 中引发极端的“最差实践”行为。随后，我们解释了导致这些“最差实践”的 AMD GPU 内部调度规则，并阐述了如何正确管理 AMD GPU 共享中一些最关键的性能影响因素。
**各章节主旨内容**:
*   **第1章 引言**: 介绍了在实时系统中理解 GPU 调度行为的重要性，并指出尽管 AMD 提供了开源软件栈，但其内部调度机制仍缺乏清晰的、集中的文档，导致研究人员难以构建精确的性能模型。
*   **第2章 背景**: 概述了通用 GPU 编程的基本模式（使用 HIP API）、关键抽象（线程、块、流），并回顾了以往针对 GPU（主要是 NVIDIA）调度行为的研究工作，为本文的研究提供了上下文。
*   **第3章 激励性实验**: 通过一组对比实验，展示了在共享 AMD GPU 时，不当的资源划分（所谓的“最差实践”）会导致灾难性的性能下降，从而引出深入探究其背后调度机制的必要性。
*   **第4章 AMD GPU 上的计算内核调度**: 详细揭示了从用户态 API 调用到硬件执行的完整调度流程，解释了硬件队列、异步计算引擎（ACE）、着色器引擎（SE）以及工作负载管理器（WLM）之间的交互机制，阐明了第3章中观察到的性能异常现象的根本原因。
*   **第5章 实际应用**: 基于前文章节的分析，提供了使用计算单元（CU）掩码 API 的实用指导。对比了两种不同的 CU 划分策略（SE-packed vs. SE-distributed），并展示了在存在竞争任务时，合理的 CU 分布对于性能至关重要。
*   **第6章 结论**: 总结了本文的工作，强调了理解 GPU 内部调度行为对于构建可靠实时系统的重要性，并提供了首个关于如何正确应用 AMD GPU CU 掩码功能的公开指南。
**中文结论**: 本文通过清晰的演示，揭示了了解或不了解 GPU 内部行为对实践产生的巨大影响。我们结合公开（但分散）的信息、第一方源代码和实验证据，提供了关于 AMD GPU 的详细信息。此外，我们还首次发布了关于如何正确应用 AMD GPU 的 CU 掩码功能的指南。不了解这些信息，开发可靠的实时系统几乎是不可能的，因为存在灾难性性能陷阱的风险。从另一个角度看，掌握了这些知识，开发者就能够以更高的置信度预测其系统的性能，这是使用闭源平台所无法获得的。

## 各章节详解

### 第1章：引言 (Introduction)

本章首先强调了在日益复杂的计算硬件上实现可预测定时是实时系统研究的核心。与 CPU 相比，GPU 的内部行为更像一个“黑盒”，这给性能建模带来了巨大挑战。尽管 AMD 的 GPU 计算软件栈（ROCm）是开源的，但这并未完全解决信息获取的难题。其庞大的代码库和稀缺的文档使得理解其调度行为依然困难。研究者不能简单地将从 NVIDIA GPU 上得到的经验推论到 AMD GPU 上。因此，本文旨在填补这一信息鸿沟，通过整合多方资料（公开演示、白皮书、源代码），提供一个关于 AMD GPU 从上到下如何调度计算任务的完整视图，并解释不当的资源管理为何会导致严重的性能问题。

### 第2章：背景 (Background)

本章为后续内容铺垫了基础知识。
1.  **GPU 编程模型**: 讲解了典型的 GPU 计算流程：数据从 CPU 拷贝到 GPU -> 调用 GPU 内核（Kernel）-> 等待内核执行完成 -> 将结果从 GPU 拷回 CPU。文中以 AMD 的 HIP API（类似于 NVIDIA 的 CUDA）为例，介绍了几个核心概念：
    *   **线程 (Thread)**: GPU 上并行计算的最小逻辑单位。
    *   **块 (Block)**: 一组线程的集合，是 GPU 上的基本调度单元。
    *   **流 (Stream)**: 一个 GPU 操作队列，同一流内的操作按 FIFO（先进先出）顺序执行，不同流的操作可能并发执行。
2.  **前期工作 (Prior Work)**: 回顾了学术界在揭示 NVIDIA GPU 调度行为方面所做的努力，这些工作大多依赖于黑盒实验和逆向工程。同时，也提到了与 NVIDIA 合作的研究，这些研究获得了更精确的内部信息。这些工作凸显了为 AMD GPU 提供类似深入分析的必要性和价值。

### 第3章：激励性实验 (Motivating Experiments)

本章通过实验设计，有意地触发了 AMD GPU 在共享时的高度破坏性干扰，以展示理解其内部调度的重要性。
*   **实验平台**: 使用了 AMD Radeon VII GPU，它拥有 60 个计算单元（Compute Units, CU）。软件栈为 ROCm 4.2。
*   **工作负载**: 实验任务为矩阵乘法。定义了两种任务：`MM1024`（使用 1024 个线程的大块）和 `MM256`（使用 256 个线程的小块）。
*   **实验场景**:
    1.  **完全共享 (Full GPU Sharing)**: 两个任务共享所有 60 个 CU。
    2.  **均匀分区 (Even Partitioning)**: 将 60 个 CU 均分为两个不重叠的分区，每个任务独占 30 个 CU。
    3.  **非均匀半分区 (Uneven Semi-Partitioning)**: 在均匀分区的基础上，允许被测任务额外访问对方分区中的一个 CU（即一个 CU 被共享）。
*   **“反常”结果**: 实验数据显示（Table 1），在“非均匀半分区”场景下，`MM1024` 任务的响应时间急剧恶化，比完全共享时慢了约 10 倍，性能表现极差。这种现象被称为“最差实践” (worst practice)。这些令人惊讶的结果表明，对 CU 的简单、看似无害的划分可能会导致灾难性的后果，从而激励作者必须深入挖掘其背后的调度原理。

### 第4章：AMD GPU 上的计算内核调度 (Scheduling Compute Kernels on AMD GPUs)

本章是论文的核心，详细拆解了 AMD GPU 的调度机制，以解释第3章中的反常现象。
1.  **用户态队列处理 (Queue Handling in Userspace)**: 一个内核启动请求 (`hipLaunchKernelGGL`) 首先进入 HIP 运行时管理的软件队列（与 HIP Stream 对应）。之后，ROCm 运行时（ROCclr）将该请求转换为一个 AQL 数据包，并放入一个 HSA 队列。HSA 队列是用户态内存中的一个环形缓冲区，可被 GPU 直接访问。为了减少 HSA 队列的数量，默认情况下，多个 HIP Stream 会共享一个 HSA 队列池（默认4个）。但如果一个 Stream 创建时指定了 CU 掩码，它会获得一个专属的 HSA 队列。
2.  **队列分配到硬件 (Assigning Queues to GPU Hardware)**: GPU 硬件（通过驱动和固件）将用户态的 HSA 队列（由内存队列描述符 MQD 表示）映射到硬件队列描述符（HQD）上。Radeon VII GPU 有4个异步计算引擎（Asynchronous Compute Engines, ACE），每个 ACE 能处理8个队列，总共支持32个并发队列。如果活动队列超过32个，就会发生“过订阅”（Oversubscription），GPU 会在这些队列间进行时间分片切换，导致显著的性能下降。
3.  **线程块调度 (Scheduling Thread Blocks)**: 这是解开性能之谜的关键。
    *   **ACE 到 SE 的分发**: 每个 ACE 以轮询（round-robin）方式，将内核的线程块（Blocks）依次分发给 GPU 的4个着色器引擎（Shader Engines, SE），就像发牌一样。**一个关键规则是：ACE 不能跳过任何一个 SE，除非该 SE 上的所有 CU 都被掩码禁用了。**
    *   **SE 内部的调度**: SE 内部的工作负载管理器（Workload Manager, WLM）负责将线程块分配给具体的 CU。如果一个 CU 没有足够的资源（如寄存器）来运行一个大的线程块（如 `MM1024`），它不会空闲等待。此时，如果队列中有资源需求更小的线程块（如 `MM256`），WLM 会让小块“插队” (cut ahead) 执行。
    *   **“最差实践”的根源**: 在“非均匀半分区”实验中，研究者创建了一个分区，其中一个 SE 上只有一个 CU 是活动的，并且这个 CU 还被共享，处理能力非常低下。当 ACE “发牌”轮到这个“慢速”的 SE 时，它必须等待那个唯一的、繁忙的 CU 完成任务才能继续向下一个 SE 发牌。由于 ACE 不能跳过这个慢速 SE，整个分发流水线被严重阻塞，导致 GPU 的其他三个功能完备的 SE 处于饥饿状态，从而造成了近乎14倍的性能下降。而 `MM256` 对 `MM1024` 的破坏性干扰也是因为 `MM256` 的小块不断“插队”，使得 `MM1024` 的大块 starved（饥饿）。

### 第5章：实际应用 (Practical Applications)

本章将理论应用于实践，探讨如何正确使用 CU 掩码（CU-Masking）进行性能隔离。
*   **CU 掩码 API**: HIP API 提供了 `hipExtStreamCreateWithCUMask` 函数，允许用户为 Stream 指定一个位掩码来限制其可用的 CU。这个位掩码是“扁平”的，而驱动程序会负责将其映射到具体的 SE 和 CU 上。映射规则是：bit 0, 4, 8, ... 对应 SE0 上的 CU；bit 1, 5, 9, ... 对应 SE1 上的 CU，以此类推。
*   **分区策略对比**:
    *   **SE-packed (SE打包)**: 尽可能将一个分区的所有 CU 集中在少数几个 SE 上。例如，一个大小为 16 CU 的分区，可能会占据 SE0 的全部15个 CU 和 SE1 的1个 CU。
    *   **SE-distributed (SE分布)**: 尽可能将 CU 均匀地分布在所有4个 SE 上。
*   **实验结论**: 单独运行时，两种策略差异不大。但在存在竞争任务时（如 `MM1024` vs. `MM256`），`SE-distributed` 策略表现得**远比 `SE-packed` 优越**。原因是 `SE-packed` 很容易无意中创造出第4章描述的那种“慢速 SE”陷阱（例如，一个 SE 上只有一个或很少几个 CU），从而导致性能瓶颈。而 `SE-distributed` 保证了每个 SE 都有相对均衡的处理能力，避免了 ACE 分发流水线被阻塞。

### 第6章：结论 (Conclusion)

本章总结了论文的贡献：
1.  **揭示了关键的性能陷阱**: 首次公开并详细解释了 AMD GPU 由于其硬件调度设计（ACE-SE 轮询分发、WLM 插队机制）而存在的严重性能问题。
2.  **提供了端到端的调度视图**: 整合了各种分散的信息，为研究社区提供了关于 AMD GPU 计算任务调度的连贯、详细的描述。
3.  **给出了实用的指导**: 提出了第一个关于如何正确使用 CU 掩码进行分区和性能隔离的公开指南，强调了采用 `SE-distributed` 策略的重要性。

最终，论文指出，只有深入理解这些底层细节，开发者才能在 AMD GPU 平台上构建真正可靠、可预测的实时系统，避免陷入意想不到的性能灾难。
