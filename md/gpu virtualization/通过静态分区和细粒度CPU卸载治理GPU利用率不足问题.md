# 论文分析：通过静态分区和细粒度 CPU 卸载治理 GPU 利用率不足

---

## 第一部分：论文整体骨架

### 1. 论文基本信息
- **文件名**：2604.08451v1.pdf
- **英文标题**：Taming GPU Underutilization via Static Partitioning and Fine-grained CPU Offloading
- **中文标题**：通过静态分区和细粒度 CPU 卸载治理 GPU 利用率不足

### 2. 中文摘要
随着 GPU 计算吞吐量和内存容量的飞速增长，如何高效利用这些资源成为一大挑战。英伟达的多实例 GPU（MIG）技术通过将资源划分为固定大小的隔离实例来提高利用率，但其在高性能计算（HPC）负载中的有效性和局限性尚不明确。本文对多种 GPU 共享机制（MIG、MPS、时间分片）进行了系统级表征分析，涵盖了科学计算、人工智能和数据分析等真实负载（如 NekRS、LAMMPS、Llama3 和 Qiskit）。
研究发现，虽然 MIG 能显著降低资源空置并提升能效，但仍存在共享资源（如功率限制）导致的干扰。此外，MIG 粗粒度的资源配置往往与应用的实际需求不匹配。为此，本文提出了一种基于缓存一致性 Nvlink-C2C 互连的内存卸载方案，旨在弥补粗粒度资源切片间的差距，进一步降低资源浪费。

### 3. 各章节主旨内容
- **第一章：引言 (Introduction)**：阐述了 GPU 资源快速增长背景下的利用率不足问题，介绍了时间共享与空间共享机制，并明确了本文研究 MIG 在 HPC 领域影响的研究目标。
- **第二章：动机与背景 (Motivation and Background)**：详细介绍了单应用与多应用 GPU 共享技术，包括 CUDA 流、Green Contexts、MPS 以及 MIG 的硬件机制，并介绍了 Grace Hopper 架构中的 Nvlink-C2C。
- **第三章：方法论 (Methodology)**：描述了实验环境（H100 GPU + Grace CPU）、利用率度量指标（基于 NVML GPM）以及选取的八种典型应用负载和 Benchmarks。
- **第四章：资源利用不平衡分析 (Imbalanced Resource Underutilization)**：定量分析了计算资源（SM 占用率）和内存资源（容量与带宽）的利用情况，揭示了性能随资源扩展的非线性特征。
- **第五章：GPU 共享的机遇与挑战 (Opportunities and Challenges)**：评估了并行运行多个任务时的系统吞吐量和能效增益，并识别出“功率降频（Power Throttling）”是主要的干扰来源。
- **第六章：基于 Nvlink-C2C 的内存卸载共享 (GPU Sharing with Nvlink-C2C Enabled Offloading)**：提出了将部分数据卸载至 CPU 内存的方案，并设计了一个奖励模型（Reward Model）来权衡性能与资源浪费。
- **第七章：相关工作 (Related Works)**：回顾了 GPU 共享、MIG 调度优化以及内存卸载技术的研究现状。
- **第八章：结论 (Conclusion)**：总结了 MIG 在解决资源失配方面的潜力，以及 Nvlink-C2C 卸载方案在提升资源 Continuity 方面的作用。

### 4. 中文结论
本文通过详尽的实验证明，MIG 能有效提升系统吞吐量和能效，但其固定的资源划分比例（如 1g.12gb, 2g.24gb 等）容易导致计算与内存需求的失配。通过引入基于 Nvlink-C2C 的细粒度内存卸载，可以使那些内存略微超过 MIG 实例限制的应用无需升级到更高一级的粗粒度切片，从而大幅提升系统的资源利用效率和灵活性。

## 第二部分：章节详解

### 1. 深度动机与背景：为什么 MIG 仍然不够？
作者首先指出，虽然现代 GPU（如 H100、B200）的算力和内存容量呈指数级增长，但不同领域的应用对资源的需求极不平衡。
- **设计痛点**：传统的 MPS（Multi-Process Service）虽然能共享计算资源，但缺乏内存隔离，且一个进程崩溃会导致所有任务失败，不适合多租户环境。MIG 提供了硬件级的计算和内存双重隔离，是解决之道。
- **局限性**：MIG 的划分是“粗粒度”且“刚性”的。在 H100 上，MIG 实例的内存容量通常以 12GB 为基准翻倍（12GB, 24GB, 48GB, 96GB）。如果一个应用的内存需求是 16GB，按照 MIG 的规则，它必须使用 2g.24gb 实例，这将导致 8GB（33%）的内存浪费。
- **Nvlink-C2C 的机遇**：Grace Hopper 架构引入了 450GB/s 的单向带宽互连，支持 CPU 和 GPU 之间的缓存一致性。这为“跨越”MIG 刚性边界提供了物理基础。

### 2. 利用率度量：如何客观评价“浪费”？
本文使用 Nvidia Management Library (NVML) 的 GPM 功能进行高频采样（0.2s 间隔），克服了以往 `nvidia-smi` 只能看内核级信息的局限。
- **关键技术指标**：
    - **SM Occupancy**：活动 Warp 占硬件上限的百分比。
    - **Pipeline Utilization**：区分了 FP32、Tensor Cores 等不同计算单元的活跃度。
    - **Wasted Resources**：定义了由于 MIG 划分导致的无法被任何实例分配的“静态浪费”。例如，在 H100 上配置 7 个 1g.12gb 实例，会产生 15% 的 SM 浪费和 17.5GiB 的内存闲置。

### 3. 应用表征：负载的真实面貌
作者选取了极具代表性的负载进行测试：
- **不平衡案例**：Qiskit（量子模拟）在全 GPU 上有 60% 的 SM 占用，但内存带宽需求高达 90%；而 NekRS（流体力学）在全 GPU 上的 SM 占用仅为 12%，大部分时间在等待 CPU 执行。
- **扩展性分析**：图 4 显示，并非所有应用都能随资源加倍而线性提速。NekRS 和 FAISS 对内存资源敏感但对计算核心不敏感，导致它们在更大的 MIG 切片中利用率急剧下降。这种“性能-资源失配”是本文提出卸载方案的核心依据。

### 4. 共享性能与干扰：功率墙的侧信道
在并行执行 7 个任务时，作者发现了有趣的现象：
- **吞吐量与节能**：平均系统吞吐量提升了 1.4 倍，能耗平均降低了 26%。
- **挑战：Power Throttling**：MIG 虽然在逻辑和硬件资源上实现了隔离，但无法隔离“功耗预算”。对于计算密集型负载（如 LLM Training），7 个实例同时运行时会触碰 700W 的总功率墙，导致频率从 1980MHz 降至 1815MHz。这意味着即便有 MIG 隔离，不同租户的任务仍会通过频率波动产生相互干扰。

### 5. 核心创新：基于 Nvlink-C2C 的内存卸载与奖励模型
为了解决 MIG 实例内存“跳变”导致的浪费，作者提出了将超出切片限制的内存需求通过 Nvlink-C2C 卸载到 CPU。
- **实现细节**：
    - 在 FAISS 和 Llama3 中使用 `cudaMallocManaged` 或更改 GGML 分配器，让大对象溢出到主机内存。
    - Qiskit 则利用其原生的状态向量交换策略。
- **奖励模型 (Reward Model)**：
    作者设计了一个公式：$R = \frac{P/P_{GPU}}{\alpha + W_{MEM} + W_{SM}}$
    - $P/P_{GPU}$：应用在当前配置下的相对性能。
    - $W_{MEM}, W_{SM}$：内存和计算资源的浪费系数。
    - $\alpha$：政策权重参数。当 $\alpha=0$ 时，模型完全优先考虑减少浪费；当 $\alpha$ 增大时，模型更倾向于性能。
- **权衡分析**：通过实验证明，对于 FAISS 等负载，使用“小切片 + C2C 卸载”的奖励得分高于直接使用“大切片”，因为它在极小的性能损失下极大提高了资源利用率。

### 6. 总结与权衡
本文展示了在 HPC 环境中，单纯的硬件分区是不够的。
- **结论与对比**：时间分片虽然看似减少了内存浪费，但巨大的上下文切换开销使其带宽利用率极低。MPS 缺乏隔离性。MIG 结合 C2C 卸载是目前在 Grace Hopper 平台上实现高效、灵活共享的最佳实践。
- **未来方向**：作者指出目前的卸载仍需应用层微调，未来需要更自动化的透明内存迁移机制来进一步降低 HPC 用户的门槛。
