# Nanvix: A Multikernel OS Design for High-Density Serverless Deployments
2604.11669v1.pdf
## Nanvix：面向高密度无服务器部署的多内核操作系统设计

### 中文摘要
无服务器（Serverless）计算提供商一直致力于通过提高“部署密度”（即单台服务器上托管的应用数量）来优化
资源利用率。然而，在不牺牲应用性能或强隔离性的前提下实现高密度部署仍是一个公开挑战。现有的共享组件
方案虽能提升密度，但容易引入租户间侧信道攻击风险，且处理不当时会引起严重的资源竞争。

本文提出了 Nanvix，这是一种专为高密度无服务器部署设计的新型多内核操作系统。Nanvix 的核心思想是
“执行状态解耦”：将每个应用调用独有的瞬态执行状态与同一租户内共享的长期持久状态分离。具体而言，应用
程序运行在轻量级的“用户 VM”中，该 VM 仅搭载一个实现基本线程和内存管理的微内核；所有的 I/O 请求则被
转发至一个“系统 VM”，该 VM 运行宏内核（包含丰富的设备驱动程序），并由同一租户的所有调用共享。这种
分层设计在保证硬件级隔离的同时，大幅减少了资源冗余。实验表明，Nanvix 实现了数量级级的冷启动时间
缩减，且在生产负载重放测试中，其对主机服务器的需求量比现有主流方案（如 Firecracker）减少了 20 到
100 倍。

### 各章节主旨内容
1. **引言 (Introduction)**：阐述了无服务器计算中冷启动延迟、租户隔离与部署密度之间的“不可能三角”，
   引出 Nanvix 的设计动机。
2. **迈向高密度无服务器部署 (Towards Dense Serverless Deployments)**：分析了为什么部署密度是供应商
   的核心效率指标，并探讨了现有技术（如容器化 VM、快照技术）在平衡兼容性与密度时的痛点。
3. **Nanvix 概述 (Nanvix Overview)**：介绍了多内核架构的核心理念，即通过解耦微内核与宏内核来兼顾
   低延迟启动、高密度部署和强租户隔离。
4. **用户 VM (User VM)**：详细描述了用户 VM 的架构，包括轻量化微内核的系统调用、基于信用的流量控制
   机制以及专门化的 VMM 设计。
5. **系统 VM (System VM)**：说明了系统 VM 如何多路复用来自不同用户 VM 的请求，并利用线程隔离和系统
   调用过滤来保证租户内的安全性。
6. **实现 (Implementation)**：介绍了使用 Rust 语言从零实现 Nanvix 的技术细节，以及如何集成 KVM 和
   Cloud Hypervisor。
7. **评估 (Evaluation)**：通过微基准测试和生产轨迹重放，验证了 Nanvix 在冷启动速度、内存占用和部署
   密度方面的显著优势。
8. **相关工作 (Related Work)**：对比了无服务器执行环境（如 Faasm、gVisor）、租户内共享机制及 OS 
   解耦的相关研究。
9. **讨论 (Discussion)**：探讨了单租户部署场景、不可信代码沙箱化以及硬件卸载的未来可能性。
10. **结论 (Conclusions)**：总结了 Nanvix 作为一种原理性的 VM 抽象拆分方案，为高效无服务器基建提供
    了新的可能。

### 中文结论
Nanvix 重新审视了长久以来的虚拟机抽象，并证明了通过有原则地拆分“瞬态执行状态”与“持久系统状态”，可以
从根本上构建一个更高效的无服务器基座。通过引入微内核用户 VM 和共享宏内核系统 VM，Nanvix 避免了传统
容器或全功能 VM 常见的资源搁置、调整大小开销以及复杂的快照管理问题。评估结果显示，Nanvix 不仅将应用
启动时间缩短至几毫秒，还显著提高了单位内存（每 GiB）能承载的沙箱数量，为生产规模的负载提供了极高的
资源利用率。

## 章节详解

### 1. 设计思想：状态与功能的彻底解耦

Nanvix 的设计初衷是为了打破无服务器计算中“兼容性、性能、密度”三者不可兼得的僵局。

*   **痛点分析**：传统方案（如 AWS Firecracker）在每个虚拟机内都运行完整的 Linux 内核。对于生命周期
    极短（通常仅几百毫秒）的 Serverless 函数来说，这种“厚重”的内核带来了巨大的内存浪费和启动开销。
    虽然快照技术（Snapshots）能加速启动，但快照的管理极其复杂，且容易造成内存重复。
*   **Nanvix 的解法**：作者观察到，应用真正需要的是计算环境（线程、内存），而外设、网络栈和文件系统
    是持久且复杂的。因此，Nanvix 采用多内核（Multikernel）架构，将 OS 功能一分为二：
    *   **User VM (轻量级)**：只负责执行应用逻辑，运行一个极简微内核，支持线程管理、内存分配和 POSIX 
        垫片层。它不包含任何驱动、网络栈或文件系统，其体积极小，甚至不需要通过传统的 PCI 枚举过程。
    *   **System VM (重量级)**：负责处理所有 I/O。它运行一个裁剪过的 Linux 宏内核，拥有完整的网络和
        存储能力。关键点在于：**同一个租户的所有 User VM 共享同一个 System VM**。

这种“一主多从”的模式，既保证了不同租户之间通过 Hypervisor 强隔离，又实现了租户内部资源的最大化复用。

### 2. 用户 VM 架构：极致的精简与高效通信

用户 VM 的核心是针对无服务器场景专门优化的微内核。

*   **关键技术：基于信用的流量控制 (Credit-based Flow Control)**：
    为了减少 Guest（微内核）与 Host（VMM）之间的 VM Exit 开销（这是虚拟化中的主要性能损耗点），
    Nanvix 引入了类似高速网络协议中的信用机制。
    *   微内核通过共享内存中的控制页来轮询响应，而不是每次 I/O 都触发昂贵的陷阱。
    *   VMM 维护一个信用计数器，只有当信用额度可用时才处理请求，从而实现了非阻塞的批量通信。
*   **微内核调用 (Kernel Calls)**：
    微内核仅实现了 20 余个关键调用，分为本地执行（如 `getpid`, `mmap`, `mutex_lock`）和远程执行
    （如 `send`, `recv`, `push`, `pull`）。远程调用会通过特定的 PMIO（Port-mapped I/O）接口转发给
    System VM。
*   **零拷贝大块数据传输 (Near zero-copy Bulk Data Transfer)**：
    对于大文件或网络包，Nanvix 利用 MMIO（Memory-mapped I/O）和 rendezvous（会晤）机制。控制消息
    包含物理页地址（GPA），VMM 直接在用户物理内存与系统传输缓冲区之间拷贝数据，极大地降低了 CPU 周期
    消耗。

### 3. 系统 VM 与多路复用：租户内的集中式管理

System VM 是 Nanvix 架构中的“功能中心”，它不运行任何租户提供的代码，仅执行受信任的系统逻辑。

*   **线程隔离与 1:1 映射**：
    System VM 为每个 User VM 维护一组对应的 Worker 线程。每个来自 User VM 的系统调用请求，都会由
    对应的 Worker 线程在宏内核中代为执行。通过 Linux 的 cgroups-v2 等机制，可以对这些线程进行精确
    的资源配额管理，防止同一个租户内的不同函数实例互相干扰。
*   **系统调用过滤 (Syscall Filtering)**：
    所有的 I/O 请求在进入宏内核之前，都要经过 Gateway 层的过滤。由于 System VM 了解 User VM 的身份，
    它可以轻松实现细粒度的安全策略，例如限制某个函数只能访问特定的网络端口或文件路径。
*   **快照与预热**：
    由于 System VM 的镜像对所有租户是通用的，它可以被预先加载并以只读方式共享物理内存页。这意味着
    当一个新租户的首个请求到来时，系统可以瞬间克隆出一个已经“热身”完毕的 I/O 环境。

### 4. 评估与权衡：密度与延迟的跨越式提升

Nanvix 的实验结果非常具有说服力，尤其是在资源受限的边缘或高并发场景。

*   **冷启动性能**：
    *   传统 Linux 引导需要 1 秒以上，Firecracker 约 100ms。
    *   Nanvix 的 User VM 启动仅需 **7ms (p50)**。如果排除掉控制平面的开销，微内核引导仅需 1-2ms。
*   **部署密度 (Deployment Density)**：
    *   在 1 GiB 内存中，Firecracker 仅能部署约 20 个实例，gVisor 约 43 个。
    *   **Nanvix 可以部署 134 个实例**。如果多个实例共享 System VM，密度优势甚至会扩大到百倍以上。
*   **权衡 (Trade-offs)**：
    *   **性能损失**：由于引入了跨 VM 的 I/O 转发，Nanvix 的网络往返时延（RTT）比原生 Linux 进程
        增加了约 50%。
    *   **兼容性挑战**：Nanvix 虽然支持 POSIX，但需要重新编译应用。对于需要深度定制内核特性的应用
        （如特定的 `setsockopt` 选项），可能需要扩展 System VM 的 Gateway 层。

### 5. 总结与启示

Nanvix 的成功展示了“解耦”在现代系统架构中的威力。它告诉我们，虚拟机不一定非要模拟一个完整的
计算机系统（包含显示器、磁盘控制器等），在云原生时代，虚拟机可以仅仅是一个受保护的执行上下文。

该研究的意义不仅在于提供了一个更快的 Serverless 运行时，更在于它为未来“软硬件协同设计”指明了
方向：如果未来的智能网卡（SmartNIC）或加速器能够直接处理 System VM 的 I/O 任务，那么 User VM 
将变得更加纯粹，甚至可以实现接近裸机性能的硬件级隔离。
