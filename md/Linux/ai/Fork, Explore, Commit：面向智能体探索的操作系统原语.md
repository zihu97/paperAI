# Fork, Explore, Commit：面向智能体探索的操作系统原语 (Fork, Explore, Commit: OS Primitives for Agentic Exploration)

## 第一部分：论文整体骨架

### 论文信息
- **文件名**：2602.08199v1.pdf
- **英文标题**：Fork, Explore, Commit: OS Primitives for Agentic Exploration
- **中文标题**：Fork, Explore, Commit：面向智能体探索的操作系统原语

### 中文摘要
AI 智能体（AI Agents）正越来越多地执行“智能体探索”任务：即并行追求多个解决方案路径，并仅提交成功的路径。由于每个探索
路径都可能修改文件并产生进程，智能体需要一个隔离的环境，能够对文件系统状态和进程状态提供原子的提交和回滚语义。本文介绍
了“分支上下文”（Branch Context），这是一种全新的操作系统抽象，提供：(1) 具有独立文件系统视图和进程组的写时复制（COW）
状态隔离；(2) 结构化的“分叉-探索-提交/终止”生命周期；(3) “首个提交获胜”的冲突解决机制，自动使兄弟分支失效；(4) 支持
分层探索的嵌套上下文。我们在 Linux 中通过两个互补组件实现了分支上下文：一是 BRANCHFS，一个基于 FUSE 的文件系统，为每个
分支提供 O(1) 创建、原子提交和无权限限制的 COW 工作区；二是 branch()，一个提议的 Linux 系统调用，用于将进程派生到具有
可靠终止和内核强制兄弟隔离的分支上下文中。初步评估显示，BRANCHFS 的分支创建时间低于 350 μs，小规模更改的提交开销低于 1 ms。

### 各章节主旨内容
1. **引言 (Introduction)**：阐述智能体探索的概念，指出当前操作系统在支持并行路径搜索和原子状态管理方面的缺失。
2. **动机 (Motivation)**：分析 AI 智能体执行模式（如 Tree-of-Thought），推导出智能体探索对环境的六大核心需求（R1-R6），
   并指出现有机制（如 OverlayFS、Btrfs、容器）的局限性。
3. **分支上下文 (Branch Contexts)**：定义新的 OS 抽象，包括其生命周期（Fork, Explore, Commit）和核心语义（冻结源、
   隔离执行、首个提交获胜、嵌套支持）。
4. **BRANCHFS 设计与实现 (BRANCHFS Design and Implementation)**：详细介绍基于 FUSE 的文件系统实现，采用文件级写时复制
   和墓碑标记（Tombstone）机制来处理删除。
5. **branch() 系统调用 (The branch() Syscall)**：提出一种新的 Linux 系统调用接口，用于协调跨分支的进程，解决用户态难以
   实现的原子组合、进程终止和兄弟隔离问题。
6. **初步评估 (Preliminary Evaluation)**：测试 BRANCHFS 的性能，证明其在分支创建和提交方面的低延迟，以及在智能体工作负载
   下的 I/O 吞吐量表现。
7. **相关工作 (Related Work)**：对比容器、gVisor、Speculator 和 TxOS 等现有的隔离和事务性 OS 机制。
8. **讨论与未来工作 (Discussion and Future Work)**：探讨网络/IPC 的副作用门控、语义合并策略以及 branch() 的内核实现路线图。
9. **结论 (Conclusion)**：总结分支上下文作为一种轻量级、原子化 OS 原语对加速智能体探索的重要性。

### 中文结论
本文提出并初步实现了“分支上下文”这一新的 OS 抽象，旨在解决 AI 智能体在复杂环境下进行并行探索时面临的状态隔离与原子提交
难题。通过结合用户态文件系统 BRANCHFS 和内核态系统调用 branch()，该方案不仅避免了传统容器或虚拟机方案的高昂开销，还通
过“首个提交获胜”的语义自然适配了智能体的决策模式。评估证明 BRANCHFS 具有极佳的响应速度，能够支持高频的分支创建与回滚，
为构建高效、可靠的自主智能体系统提供了坚实的底层操作系统支持。

## 第二部分：章节详解

### 1. 引言与背景：智能体探索的兴起
在软件工程（SWE-agent）、计算机操作（Computer Use）和操作系统调优等领域，AI 智能体不再仅仅执行线性指令，而是采取
“推理-行动”循环。为了提高成功率，现代智能体开始尝试并行探索多种策略，例如同时尝试三个不同的代码修复方案，并运行测试
来筛选唯一的正确答案。
**痛点分析**：
- **不可逆副作用**：智能体执行的 shell 命令、软件包安装或文件编辑具有副作用。一旦探索失败，难以干净地回退。
- **并发冲突**：多个探索路径如果共享同一个工作区，会产生严重的写冲突。
- **现有方案碎片化**：依赖 git stashing 或临时目录不仅速度慢，且无法捕获所有系统级的修改（如 `npm install` 导致的
  全局状态变化）。

### 2. 动机与需求分析：为什么需要 OS 原语？
作者归纳了智能体探索环境的六个核心需求（Requirements）：
- **R1: 隔离的并行执行**：各路径互不干扰。
- **R2: 带单赢解决机制的原子提交**：一个成功，其他兄弟分支必须自动失效，防止基于过期状态的错误提交。
- **R3: 分层嵌套**：支持“树状思考”（Tree-of-Thoughts）中的子策略探索。
- **R4: 完整文件系统覆盖**：必须捕获所有修改，不限于 git 跟踪的文件。
- **R5: 轻量、无特权且便携**：创建分支需达到亚毫秒级，且无需 root 权限。
- **R6: 进程协调**：当分支结束时，必须能够可靠地终止其派生的所有进程（防止僵尸进程或跨路径进程干扰）。

**现有技术局限**：
- **OverlayFS**：不支持原子的提交到父目录，且挂载需要 root 权限。
- **Btrfs/ZFS**：与特定文件系统绑定，缺乏原生的提交/失效语义。
- **容器/VM**：开销过大，启动延迟高，难以支持高频（如每步推理一个分支）的操作。

### 3. 分支上下文：定义与生命周期
分支上下文 $b_i$ 封装了两个核心要素：
1. **文件系统视图**：由原始文件（Origin）和写时复制的 Delta 层 $\Delta_i$ 叠加而成。
2. **进程组**：其产生的副作用仅限于当前分支。

**生命周期详解**：
- **Fork**：从冻结的父状态原子化地创建 $N$ 个兄弟分支。父状态变为只读，确保所有分支起点一致，从根源消除合并冲突。
- **Explore**：分支独立运行，所有修改被记录在各自的 Delta 层中。
- **Commit/Abort**：
    - **首个提交获胜 (First-commit-wins)**：一旦某个分支提交成功，父状态会被更新，所有其他处于运行中的兄弟分支会立即
      被标记为失效（Epoch Counter 增加），其后续的提交尝试将返回错误（如 -ESTALE）。
    - **原子性**：提交过程包含删除（应用墓碑文件）和修改（复制 Delta 文件）两个原子步骤。

### 4. BRANCHFS：基于 FUSE 的轻量级实现
BRANCHFS 是该方案的文件系统组件，使用 Rust 语言和 `fuser` 库实现。
- **文件级写时复制**：当分支内修改文件时，BRANCHFS 将整个文件从基目录拷贝到分支的 Delta 目录。这种粗粒度设计（Coarser
  granularity）虽然比块级 COW 稍微费空间，但在智能体典型的工作负载（中小规模文件编辑）下，实现更简单且开销可控。
- **查询逻辑**：Lookups 按照 `当前分支 Delta -> 祖先分支 Delta -> 基目录` 的顺序查找。
- **墓碑标记 (Tombstone)**：为了处理文件删除，系统会创建一个轻量的哨兵文件（Sentinel file）。如果查找过程中遇到墓碑，
  直接返回“文件不存在”，防止已删除文件从底层重新“浮现”。
- **隔离性**：每个智能体被放置在独立的挂载命名空间（Mount Namespace）中，通过绑定挂载（Bind-mount）实现视图隔离。

### 5. branch() 系统调用：内核态的进程协调
由于用户态无法完全保证进程的可靠终止和跨分支的原子组合操作，作者提议增加 `branch()` 系统调用。
- **API 设计**：模仿 `bpf(2)` 的多路复用模式，支持 `BR_CREATE`、`BR_COMMIT`、`BR_ABORT`。
- **核心逻辑**：
    - `BR_CREATE`：一次性派生出 $N$ 个子进程，每个进程拥有独立的挂载命名空间和 cgroup。
    - **内存分支 (BR_MEMORY)**：支持页表级的 COW。如果设置了该标志，父进程的内存页变为只读，子进程写入时触发 COW，这
      为带有重度内存状态（如 Python 解释器环境）的智能体提供了高效支持。
- **兄弟隔离 (BR_ISOLATE)**：内核通过 `branch_id` 检查强制隔离信号和 ptrace 调试，防止兄弟分支间互相杀掉进程。
- **文件系统中立**：通过通用的 ioctl（如 `FS_IOC_BRANCH_CREATE`）与底层文件系统（如 BRANCHFS 或 DAXFS）通信。

### 6. 实验评估与权衡
实验在 AMD Ryzen 5 5500U 环境下进行：
- **创建速度**：分支创建耗时约 310 μs，且与基目录下的文件数量无关（$O(1)$ 复杂度），显著优于容器启动。
- **提交开销**：1 KB 的修改提交仅需 317 μs，1 MB 修改需 2.1 ms。开销与修改量（Delta 大小）成比例。
- **I/O 性能**：
    - 使用 FUSE Passthrough 模式，读取吞吐量可达原生的 82%（7,236 MB/s）。
    - 写入性能有时略高于原生，原因是 BRANCHFS 将 `fsync` 视为无操作（因为分支是临时性的，仅在提交时保证持久化）。
- **局限性**：目前的 prototype 暂不支持网络和 IPC 的回滚（副作用门控），这是未来工作的重点。

### 7. 结论与启示
《Fork, Explore, Commit》提出了一种与传统 OS 哲学（如 `fork()` 的隐式资源继承）不同的新范式：**显式的、带生命周期的分支
管理**。
对于 AI 智能体而言，这种“分支”能力就如同人类大脑中的“虚拟推演”。通过将这种推演能力下沉到操作系统内核与文件系统层，
开发者可以构建更加大胆、高效的智能体，而无需担心破坏底层系统的完整性。这一研究为“Agent-Native OS”的未来奠定了基础。
