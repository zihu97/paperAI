
# 论文骨架

*   **论文文件名和英文标题**
    *   文件名：2601.20435v1.pdf
    *   英文标题：Rethinking Thread Scheduling under Oversubscription: A User-Space Framework for Coordinating Multi-runtime and Multi-process Workloads

*   **中文标题**
    *   超额订阅下线程调度的反思：一个用于协调多运行时和多进程工作负载的用户空间框架

*   **中文摘要**
    *   高性能计算（HPC）与人工智能（AI）的融合催生了日益复杂的并行应用和工作负载。这些工作负载通常在同一应用或并置的任务中结合多个并行运行时，对传统操作系统（OS）调度器提出了巨大的挑战。当发生超额订阅（就绪线程数多于核心数）时，OS调度器依赖周期性抢占来复用核心，但这常引入干扰并可能降低性能。本文提出了（1）用户空间调度框架（User-space Scheduling Framework, USF），这是一个完全在用户空间实现的无缝进程调度框架。USF允许用户在没有特殊权限的情况下实现自己的进程调度算法。我们使用其默认的协作策略（2）SCHED_COOP对USF进行了评估，该策略旨在通过仅在线程阻塞时进行切换来减少干扰。这种方法缓解了锁持有者抢占（LHP）、锁等待者抢占（LWP）和可扩展性崩溃等众所周知的问题。我们通过使用nOS-V运行时扩展GNU C库来实现USF和SCHED_COOP，从而在多个运行时（如OpenMP）之间实现无缝协调，无需对应用进行侵入式修改。评估显示，在超额订阅的多进程场景中（包括嵌套BLAS工作负载、使用LLaMA-3的多进程PyTorch推理以及分子动力学模拟），性能提升高达2.4倍。

*   **各章节主旨内容**
    *   **第一章：引言**
        *   介绍了在HPC与AI融合的背景下，多运行时和多进程工作负载导致的线程超额订阅问题。传统OS调度器基于抢占的机制会引起性能下降。为此，本文提出一个完全在用户空间实现的、无需特殊权限、透明的调度框架USF及其协作式策略SCHED_COOP。
    *   **第二章：背景**
        *   阐述了实现USF和SCHED_COOP所依赖的基础技术。主要包括：1）Linux内核调度器（默认为EEVDF），作为最终的仲裁者；2）GNU C库（glibc），作为拦截标准API调用的理想收敛点；3）nOS-V库，一个提供用户空间多进程任务调度核心能力的灵活线程与任务库。
    *   **第三章：SCHED_COOP**
        *   从用户视角描述了SCHED_COOP策略。它是一个虚拟的用户空间调度器，其核心特点是管理的线程除非自愿阻塞或放弃CPU，否则不会被其他SCHED_COOP线程抢占，从而实现了非侵入式的协作式调度，减少了上下文切换开销。
    *   **第四章：设计**
        *   详细阐述了USF和SCHED_COOP的实现架构。核心思想是通过扩展glibc来拦截pthreads相关的API调用（如线程创建、锁、条件变量等），并将这些线程无缝转换为由nOS-V管理的协作式任务。这使得USF能够在一个集中的用户空间调度器中管理来自不同进程的线程，而无需修改应用代码。
    *   **第五章：评估**
        *   通过一系列复杂的实验场景，全面评估了USF和SCHED_COOP的性能。实验覆盖了嵌套运行时（矩阵乘法）、多运行时组合（Cholesky分解）、多进程AI推理（PyTorch）和分子动力学模拟（LAMMPS）。结果证明，相较于Linux默认调度器，SCHED_COOP在超额订阅场景下能带来显著的性能提升（最高达2.4倍）。
    *   **第六章：相关工作**
        *   回顾了在用户空间调度、运行时协调和超额订阅管理领域的相关研究。  通过对比，凸显了USF的独特性，即它是一个无需内核修改、支持多进程、且完全兼容TLS（线程局部存储）的无缝解决方案。
    *   **第七章：结论和未来工作**
        *   总结了本文的贡献：提出了USF框架和SCHED_COOP策略，证明了在用户空间进行协作式调度是解决超额订阅问题的有效途径。未来的工作方向包括支持io_uring、futex等更多阻塞原语，并探索更多定制化的调度策略。

*   **中文结论**
    *   本文提出了USF，一个用于开发无缝用户空间调度策略的框架，并介绍了其默认的协作式策略SCHED_COOP。两者都完全在用户空间通过扩展glibc实现，无需Linux内核扩展，并支持TLS。我们的框架允许开发特定的调度策略，并提供了定义标准阻塞API行为的方法。它支持任何依赖标准glibc同步原语的应用和运行时，尽管自定义的忙等待（busy-wait）屏障需要适配。我们广泛测试了SCHED_COOP在多进程和多运行时场景下的表现，并证明了它在管理超额订阅方面是有效的，性能提升高达2.4倍。未来的工作包括：增加对Linux I/O（通过io_uring）的阻塞支持；扩展支持的阻塞调用集以包含futex；向USF核心暴露更多glibc API；探索SCHED_COOP的内核实现并与glibcv版本进行比较；评估USF中的基于信号的抢占；以及评估针对不同工作负载（如混合关键系统或交互式工作负载）的新的自定义调度策略。

# 指定章节详解

*   **第一章：引言 详解**
    *   **问题背景**：现代HPC和AI应用常常混合使用不同的并行库（如OpenMP、PyTorch内置的pthreadpool），这会导致创建的线程总数远超CPU核心数，即“超额订阅”。
    *   **传统OS调度器的问题**：Linux等操作系统的默认调度器（如EEVDF）采用公平共享和时间分片的策略。它会周期性地抢占正在运行的线程，以让其他线程获得执行机会。这种“一刀切”的抢占策略不了解应用内部的逻辑，可能在关键时刻（如线程持有锁时）中断关键线程，导致其他等待该锁的线程也无法前进，造成严重的性能抖动和下降（即LHP和LWP问题）。
    *   **本文方案**：为了解决此问题，论文提出了USF（用户空间调度框架）。它是一个“寄生”在glibc中的框架，能够在不修改内核、不需要root权限的情况下，接管应用的线程调度。其默认策略SCHED_COOP的核心思想是“协作”，即线程仅在遇到阻塞操作（如等待锁、IO）或主动放弃（yield）时才让出CPU，避免了破坏性的强制抢占。这种方式为应用提供了更稳定、可预测的执行环境。

*   **第二章：背景 详解**
    *   **Linux内核调度器 (EEVDF)**：它是系统中所有线程调度的最终决策者。USF运行在其之上，并且不会干扰内核管理的关键OS服务或其他非USF管理的线程。USF的目标是管理好“自己的”线程，避免它们之间互相干扰。
    *   **GNU C库 (glibc)**：几乎所有Linux下的C/C++程序都会默认链接glibc。它提供了pthreads、I/O、内存管理等核心API的实现。论文巧妙地选择扩展（或称为“劫持”）glibc，因为这样可以透明地拦截所有应用的线程创建和同步行为，从而将它们纳入USF的管理范围，而应用本身及其所依赖的库则完全无感知。
    *   **nOS-V库**：这是实现USF协作式调度的核心引擎。nOS-V是一个轻量级的用户空间任务库，它提供了一套API来创建“任务”，并将这些任务调度到其管理的“工作线程”上执行。它本身支持多进程协调（通过共享内存），并实现了工作窃取、任务暂停/恢复等高级功能。在USF中，每个pthread被巧妙地包装成一个nOS-V任务。

*   **第三章：SCHED_COOP 详解**
    *   **虚拟调度器**：SCHED_COOP并非一个真正的OS级调度策略，而是一个在USF框架下实现的用户空间调度算法。用户可以通过简单的命令行 `chrt -c <app>` 来启动它。
    *   **非抢占式行为**：在SCHED_COOP的管理下，一个线程一旦开始在某个CPU核心上运行，就会一直运行下去，直到它因等待资源而必须阻塞（例如 `pthread_mutex_lock` 一个已经被占用的锁），或者它自愿调用 `sched_yield`。此时，USF会捕获到这一事件，并从就绪队列中选择另一个线程在该CPU核心上运行。
    *   **与OS调度的关系**：SCHED_COOP线程虽然不会互相抢占，但仍可能被OS调度器抢占，以运行更高优先级的系统任务或其他普通进程。但关键在于，参与协作调度的应用线程形成了一个“内部和平”的孤岛，避免了最主要的性能干扰来源——即应用内部线程间的混乱抢占。

*   **第四章：设计 详解**
    *   **架构概览**：整体架构可以看作一个“三明治”模型：顶层是应用代码，底层是nOS-V调度引擎，中间是经过扩展的glibc（论文中称为glibcv）。
    *   **API拦截机制**：
        1.  **线程创建 (`pthread_create`)**：当应用调用 `pthread_create` 时，glibcv会拦截这个调用。它并不直接创建一个自由运行的OS线程，而是创建一个线程并立即将其注册为nOS-V的一个“工作线程”，并将用户指定的函数包装成一个nOS-V任务。该任务被提交到nOS-V的调度队列中，等待被调度。
        2.  **阻塞操作 (如 `pthread_mutex_lock`)**：当一个线程尝试获取一个已经被占用的锁时，glibcv会拦截此调用。它会将当前线程对应的nOS-V任务状态置为“暂停”，并通知nOS-V调度器当前CPU核心已空闲。nOS-V会立刻在 该核心上调度一个新的就绪任务（即另一个线程）来运行，从而实现了CPU资源的无缝利用。
        3.  **解锁操作 (如 `pthread_mutex_unlock`)**：当一个线程释放锁时，glibcv会检查是否有其他线程正在等待这个锁。如果有，它会将等待队列中的一个（或多个）线程对应的nOS-V任务重新放入就绪队列，等待下一次调度机会。
    *   **多进程支持**：nOS-V使用共享内存来维护一个全局的任务队列和调度器状态。当一个新进程（通过`fork`创建）启动时，如果设置了USF的环境变量，它的glibcv实例会自动连接到这个共享内存区域，从而将新进程的所有线程也纳入同一个全局调度器的管理之下。
    *   **线程缓存**：为了避免频繁创建和销毁线程的开销，USF实现了一个线程缓存。当一个线程函数执行完毕后，它不会立即退出，而是被置于一个“缓存池”中。下次应用请求创建新线程时，USF会直接从缓存池中复用一个，极大地提升了效率。

*   **第五章：评估 详解**
    *   **实验设计思想**：评估遵循“从简到繁”的原则。首先，在一个相对简单的嵌套运行时矩阵乘法场景中，验证两个核心假设：1）通过手动修改代码以适应协作式调度（Manual版），可以提升性能；2）使用USF无缝、透明地实现同样的效果（SCHED_COOP版），同样能提升性能。
    *   **关键实验场景分析**：
        1.  **矩阵乘法 (Matmul)**：这个实验展示了在嵌套并行（外层OmpSs-2任务 + 内层OpenMP并行BLAS）中，默认Linux调度器由于频繁抢占导致性能很差。而SCHED_COOP通过避免内部抢占，让内层OpenMP线程能够稳定执行，性能接近于需要大量手动修改代码的Manual版本，验证了USF的有效性和透明性。
        2.  **AI推理微服务 (Python Microservices)**：这个场景模拟了真实世界的多进程AI应用。多个Python进程（每个进程内部使用PyTorch进行并行计算）同时处理请求。由于Python的GIL限制，多进程是常用模式。实验表明，当请求频率增高导致严重超额订阅时，Linux调度器会因“公平”地在所有线程间切换而导致所有请求都变慢（同步崩溃）。而SCHED_COOP能够让一部分请求优先完成，然后处理下一批，形成良性的“流水线”，从而在高负载下依然保持高吞吐和低延迟，性能提升最高达到2.4倍。
        3.  **分子动力学 (LAMMPS)**：这个实验展示了在复杂的MPI+OpenMP混合并行应用中，通过同时运行两个模拟任务（Co-execution），可以更好地利用机器资源。SCHED_COOP在这种场景下，既允许多个进程的任务互相填充计算空隙，又避免了它们之间的调度干扰，最终获得了比传统的独占执行或简单并置更高的总吞吐量。
    *   **核心发现**：评估证明，超额订阅场景下的性能瓶颈主要来自OS调度器的“无知”抢占所导致的同步噪声。协作式调度通过赋予应用“知情权”，能够有效消除这种噪声，从而释放并行应用的全部潜力。

*   **第六章：相关工作 详解**
    *   **对比ghOSt, F4, Syrup**：这些也是先进的用户空间调度框架，但它们通常需要对Linux内核进行大量修改，或者依赖eBPF等较新的内核特性，这限制了它们的广泛部署和适用性。USF的优势在于其对内核“零要求”，仅通过扩展glibc即可工作。
    *   **对比Arachne, DLB**：这些工作通过资源分区来管理超额订阅，例如动态地为不同进程分配CPU核心。这种方法是有效的，但通常需要应用或运行时库集成它们的特定API，不具备USF的透明性。
    *   **对比用户级线程 (ULTs)**：像Argobots这样的ULT库也采用协作式调度，但它们通常不支持线程局部存储（TLS），这使得许多现有代码（尤其是大量使用TLS的代码）无法直接在ULT上运行。USF通过调度真正的OS线程，完美地保留了对TLS的兼容性。
    *   **独特定位**：USF是第一个在不修改内核、完全兼容TLS、且对应用透明的前提下，实现多进程协作式调度的框架。

*   **第七章：结论和未来工作 详解**
    *   **核心贡献总结**：本文为解决HPC+AI融合带来的调度难题提供了一个实用且创新的解决方案。它证明了在用户空间层面，通过智能的协作式调度，可以安全、高效地管理超额订阅，其效果甚至优于传统的独占或分区资源使用方式。
    *   **未来展望**：
        1.  **I/O支持**：当前USF主要拦截同步原语，但对阻塞式I/O操作（如`read`, `write`）的拦截尚不完善。未来计划集成`io_uring`，将I/O操作也变成可被调度的异步任务。
        2.  **更广泛的API支持**：计划支持`futex`，这是许多现代运行时（如Go）使用的底层同步原语，从而扩大USF的适用范围。
        3.  **更丰富的调度策略**：SCHED_COOP只是一个开始。USF框架本身是可扩展的，未来可以基于它为实时系统、交互式应用等开发更多定制化的、更智能的调度策略。
        4.  **安全性和内核集成**：探索与内核更紧密的集成方式，以克服当前基于共享内存方案的安全性限制，并评估内核级实现的性能潜力。
