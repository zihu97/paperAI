# 2511.11621v1.pdf - AIVAILABLE: A SOFTWARE-DEFINED ARCHITECTURE FOR LLM-AS-A-SERVICE ON HETEROGENEOUS AND LEGACY GPUS

## 中文标题：AIvailable：面向异构及老旧 GPU 的软件定义 LLM 即服务架构

### 中文摘要
大语言模型（LLM）的兴起增加了对可扩展、高性能推理系统的需求，但现有的多数框架往往假设硬件环境是同构且资源丰富的，这在
学术界或资源受限的场景中并不现实。本文引入了 AIvailable，这是一个低成本、高可用的 LLM 即服务（LLMaaS）平台。它采用
软件定义的方法，支持在异构及老旧 GPU 节点（包括 NVIDIA 和 AMD 设备）上运行 LLM，重点在于充分利用每个节点的显存
（VRAM）。AIvailable 实现了全 GPU 加速推理，无需 CPU 回退，并通过统一的客户端接口实现与所有已部署模型的无缝交互。
其架构包含四个核心组件：客户端接口、服务前端、SDAI 控制器（负责编排、部署和监控）以及执行工作负载的异构 GPU 后端。
通过抽象 GPU 特定细节并提供感知显存的动态模型分配与重分配，AIvailable 确保了资源的高效利用以及对故障或工作负载波动的
弹性。该系统旨在通过重新利用老旧 GPU，帮助学术实验室和中小企业实现生成式 AI 的民主化。

### 各章节主旨内容
- **Chapter 1: Introduction** - 阐述了低成本、本地部署 LLM 解决方案的必要性，特别是针对中小企业和学术机构。
- **Chapter 2: Related Work** - 回顾了分布式推理、调度策略及异构计算的现状，指出其通常依赖高性能数据中心级硬件。
- **Chapter 3: Proposed Architecture** - 详细介绍了 AIvailable 的四层软件定义架构及其高可用、高效利用和易用的目标。
- **Chapter 4: Prototype** - 描述了基于 Python、Flask、HAProxy、OpenWebUI 和 Ollama 构建的原型系统及硬件配置。
- **Chapter 5: SDAI Interface** - 展示了软件定义 AI 接口的功能，包括实时监控、配置向导和 GPU 实例管理。
- **Chapter 6: Discussion** - 探讨了软件定义方法在降低编排复杂性、处理故障以及保障隐私方面的优势与局限性。
- **Chapter 7: Conclusion** - 总结了主要贡献，强调了在混合架构硬件上实现可靠的多智能体 LLM 服务的可行性。

### 中文结论
AIvailable 成功证明了通过软件定义架构可以在异构且部分老旧的 GPU 集群上实现可行的多模型 LLM 服务。其核心贡献在于耦合了
轻量级控制平面（SDAI 控制器）与基于标准的标准数据平面（HAProxy），实现了跨多元节点的统一访问与高可用性。实验表明，
即使没有数据中心级 GPU，通过感知识别显存的模型放置策略和自动化配置生成，也能实现可靠的推理服务。未来工作将聚焦于
大规模验证、推理延迟基准测试以及更先进的动态资源分配策略。

# 章节详细解读

## Chapter 1 & 2: 背景与相关工作

### 1. 设计思想 (Design Philosophy)
- **痛点分析**：顶级 GPU（如 H100, A100）价格昂贵，学术机构和中小企业难以负担。现有推理框架（如 vLLM, TGI）虽然
  强大，但往往针对高性能、同构硬件优化。老旧 GPU（如 GTX 系列）或异构环境（NVIDIA 与 AMD 混用）被现有主流方案忽视，
  导致大量算力资源闲置。
- **核心动机**：提出“软件定义”的思想，将推理能力的提供与底层硬件的物理限制解耦。通过抽象层屏蔽不同显卡驱动（CUDA 
  与 ROCm）的差异，实现老旧硬件的“变废为宝”。
- **设计目标**：在不依赖昂贵设备的前提下，提供具备高可用性（HA）、显存感知调度和统一 API 的生产级 LLM 服务。

### 2. 关键技术 (Key Technologies)
- **异构抽象**：识别不同厂商（NVIDIA/AMD）及不同架构（Ampere/Pascal/RDNA）的 GPU 特性。
- **SDAI 概念**：引入软件定义 AI（Software-Defined AI, SDAI）理念，类似于 SDN 将控制面与数据面分离，SDAI 
  将模型编排逻辑与具体推理引擎分离。

### 3. 实现细节 (Implementation Details)
- **对比与权衡**：
    - **vs 数据中心方案**：放弃了对超大规模并行（如 Tensor Parallelism）的追求，转而侧重于在单个节点内最大化 
      VRAM 利用率，支持多实例并发。
    - **权衡**：不使用 CPU 回退（Fallback），以保证推理速度，但这限制了模型大小必须适配物理 VRAM。

## Chapter 3: 架构设计 (Proposed Architecture)

### 1. 设计思想 (Design Philosophy)
- **痛点分析**：管理多个异构节点通常涉及极其复杂的配置（端口、模型路径、负载均衡等）。单点故障（SPOF）会导致服务
  完全中断。
- **核心动机**：构建一个四层解耦架构，将“谁来访问”、“如何路由”、“谁来决策”、“谁来计算”四个功能彻底分离。
- **设计目标**：实现“透明推理”，即用户只需向一个逻辑终点发请求，无需关心该请求最终落在哪个显卡上。

### 2. 关键技术 (Key Technologies)
- **架构拆解**：
    - **Client Interface**：用户交互层，提供 WebUI 或 API 接入。
    - **Service Frontend**：安全网关与路由层。集成 HA 和 LB，防止节点过载。
    - **SDAI Controller**：编排核心。负责节点发现、模型部署决策和持续监控。
    - **Service Backend**：计算执行层。由异构 GPU 节点组成，支持多实例并发执行。

### 3. 实现细节 (Implementation Details)
- **显存感知分配策略**：SDAI 控制器会检测每个节点的可用显存，根据模型所需显存大小进行静态或动态放置。
- **双向同步**：控制器与前端紧密集成，确保路由决策基于节点的实时健康状态和负载情况。

## Chapter 4 & 5: 原型实现与接口功能 (Prototype & SDAI Interface)

### 1. 设计思想 (Design Philosophy)
- **痛点分析**：配置异构集群的 HAProxy 和推理引擎手动工作量巨大且易出错。非专业用户难以直观理解集群资源分布。
- **核心动机**：开发一个高度自动化的控制台（Wizard-based），将复杂的配置流程简化为“选择-配置-生成”三个步骤。
- **设计目标**：实现“零脚本”部署，让非技术人员也能管理 LLM 集群。

### 2. 关键技术 (Key Technologies)
- **技术栈组合**：
    - **推理引擎**：使用 Ollama 框架，因其对多种 GPU 架构的优秀兼容性。
    - **负载均衡**：HAProxy 负责流量分发和健康检查。
    - **后端框架**：Python + Flask。
- **配置向导逻辑**：根据模型权重（如 7B, 1.5B）计算所需的 VRAM，自动生成每个节点的定制化启动脚本和端口映射。

### 3. 实现细节 (Implementation Details)
- **硬件异构性实验配置**：
    - 节点涵盖了从 RTX 3070 到 GTX 1660 Super，以及 AMD RX 6600/6800，跨越了 2019 年至 2021 年的软硬件。
- **自动化流程**：SDAI 接口会生成经过优化的 HAProxy 配置，自动处理模型副本在不同节点间的冗余部署。

### 4. 权衡与对比 (Trade-offs & Comparison)
- **易用性提升**：通过 WebUI 管理，相比于 Kubernetes 管理复杂的 GPU 调度，显著降低了运维门槛。
- **性能损耗**：由于引入了多层路由（Frontend -> Backend Proxy），在高并发下会有微小的网络开销，但在低成本场景下
  这种折中是完全可以接受的。

## Chapter 6 & 7: 讨论与总结

### 1. 设计思想 (Design Philosophy)
- **痛点分析**：虽然系统提高了可用性，但无法完全避免“协调性故障”（如同机房断电）。
- **核心动机**：评估软件定义方法在真实科研/企业环境中的鲁棒性。
- **设计目标**：界定系统的边界，为后续的自动化运维奠定理论基础。

### 2. 关键技术 (Key Technologies)
- **故障屏蔽机制**：利用副本级负载均衡（Replica-level LB）掩盖单实例崩溃。
- **隐私保护**：通过本地托管模型，确保数据流向不出本地局域网，满足中小企业的合规性要求。

### 3. 权衡与对比 (Trade-offs & Comparison)
- **局限性**：目前的观察指标主要集中在“存活性”和“容量”，缺乏对推理生成内容质量（LLM Quality）的监控。
- **未来方向**：作者指出成本分析（能耗、折旧等）尚未纳入，未来需加入更精细的经济调度模型。
