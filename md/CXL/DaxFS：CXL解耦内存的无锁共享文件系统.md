# DAXFS: A Lock-Free Shared Filesystem for CXL Disaggregated Memory
2604.01620v1.pdf
## 中文标题：DaxFS：CXL 解耦内存的无锁共享文件系统

### 中文摘要
CXL（Compute Express Link）技术实现了多个主机共享具有硬件缓存一致性的字节寻址内存，但现有的文件系统未能
有效利用这一特性进行无锁的多主机协作。本文提出了 DaxFS，这是一款专为 CXL 共享内存设计的 Linux 文件系统。
DaxFS 将 CXL 3.0 跨主机一致的 `cmpxchg` 原子操作作为唯一的协调原语。通过基于 CAS 的哈希覆盖层（Hash 
Overlay），DaxFS 实现了多主机并发写入而无需中心协调器。此外，它引入了一种新型的多主机 CLOCK 驱逐算法 
(MH-clock)，在共享 DAX 内存中提供协作式共享页缓存，并通过 `cmpxchg` 实现完全去中心化的驱逐选择。在 
QEMU 模拟的 CXL 3.0 环境下，DaxFS 在多主机争用下保持了 >99% 的 CAS 准确性且无更新丢失。在单主机测试中，
DaxFS 在大多数负载下超过了 tmpfs 的吞吐量，随机写入性能提升达 2.68 倍。初步的 GPU 微基准测试显示，其
基于 `cmpxchg` 的设计可扩展至 GPU 线程，以 PCIe 5.0 带宽极限执行页缓存操作。

### 各章节主旨内容
- **第 1 章: 引言 (Introduction)** - 阐述 CXL 3.0 带来的硬件一致性共享内存机遇，指出传统文件系统在 CXL 环境
  下的局限性，并概述 DaxFS 的核心贡献：无锁多主机写入、去中心化驱逐算法 MH-clock 及 GPU 协同设计。
- **第 2 章: 背景与动机 (Motivation)** - 分析 CXL 3.0 的 GFAM 模型及 `cmpxchg` 原子的重要性。识别了现有系统
  的三个核心痛点：每主机状态冗余、缺乏多主机协作机制以及网络协议栈/GPU 数据路径的巨大开销。
- **第 3 章: 设计 (Design)** - 详细描述 DaxFS 的三大设计原则：内存即存储、读操作免费及仅使用硬件原子操作协作。
  介绍其扁平的磁盘格式、基于 CAS 的哈希覆盖层实现、MH-clock 协作页缓存以及跨主机一致性保障机制。
- **第 4 章: 实现 (Implementation)** - 介绍 DaxFS 作为 Linux 内核模块的实现细节（约 2500 行 C 代码），
  包括内存映射模式、VFS 集成方式、挂载时校验以及与 GPU P2PDMA 的集成。
- **第 5 章: 评估 (Evaluation)** - 通过单主机 DRAM-backed DAX 和 QEMU 模拟的 CXL 3.0 多主机环境，对比测试 
  DaxFS 与 tmpfs、ext4-dax 的性能，并验证了其在 GPU 端的原子操作延迟与吞吐量。
- **第 6 章: 相关工作 (Related Work)** - 对比持久性内存文件系统（NOVA、PMFS）、CXL 共享内存系统（FamFS）
  及传统分布式文件系统，强调 DaxFS 在无锁协作方面的独特性。
- **第 7 章: 讨论与局限性 (Discussion and Limitations)** - 分析扁平目录结构的扩展性、固定大小哈希表的限制、
  持久化保证、内存池碎化以及 POSIX 合规性缺失等问题。
- **第 8 章: 结论 (Conclusion)** - 总结 DaxFS 通过利用 CXL 硬件原子操作实现的性能突破，证明了在解耦内存
  池中构建统一文件系统命名空间和缓存的可行性。

### 中文结论
DaxFS 证明了利用 CXL 3.0 提供的硬件一致性原子操作可以构建出完全无锁、去中心化的共享文件系统。它成功解决了
CXL 环境下多主机共享内存的“孤岛”问题，通过 MH-clock 算法实现了高效的协作缓存，并为 GPU 直接参与文件系统
协作开辟了道路。实验表明，这种去中心化的设计不仅在多主机争用下表现稳健，而且在单机性能上也由于消除了传统
文件系统的锁开销而优于 tmpfs，是未来 disaggregated memory 架构下系统软件演进的重要方向。

# 章节详细解读

## 第一章与第二章：背景、动机与设计需求

### 1. 设计思想 (Design Philosophy)
- **痛点分析**：
    - **每主机冗余 (Per-Host Duplication)**：tmpfs 或 ext4-dax 在每个主机上维护独立的元数据和页缓存。即使
      物理内存是共享的，不同主机仍会缓存同一数据的多个副本，浪费昂贵的 CXL 内存。
    - **缺乏多主机协作**：现有 DAX 文件系统假设单主机独占。FamFS 虽然支持多主机挂载，但采用单主读写模型，
      客户端无法并发创建或写入文件。
    - **协议栈开销**：NFS/Ceph 等分布式文件系统引入了微秒级的网络延迟，将纳秒级的内存访问降级，抵消了 CXL 的优势。
- **核心动机**：利用 CXL 3.0 引入的 GFAM（全局架构连接内存）和 `cmpxchg` 原子操作，将文件系统的协调逻辑直接
  下沉到硬件一致性协议中，实现真正的“多核多主机”无缝协作。
- **设计目标**：实现无中心协调器的多主机并发写入、共享页缓存（避免冗余）以及零拷贝的用户态访问（mmap 直接映射）。

### 2. 关键技术 (Key Technologies)
- **CXL 3.0 原子性**：DaxFS 依赖 CXL 3.0 的 DCOH（设备一致性引擎）确保跨主机的 64 位 `cmpxchg` 操作是原子的。
  这使得不同服务器的 CPU 可以像访问本地内存中的锁一样进行同步。
- **GPU 路径协同**：由于 PCIe 3.0+ 已经支持原子操作 TLP（事务层包），DaxFS 的无锁设计可以天然扩展到 GPU，
  允许 GPU 线程直接在 CXL 内存上执行 `atomicCAS` 来查找或更新页缓存状态。

### 3. 实现细节 (Implementation Details)
- **DCOH 与 snoop filter**：CXL 设备维护监听过滤器，负责处理跨主机的硬件缓存回刷（back-invalidation），
  从而保证元数据修改在所有主机间立即可见。

## 第三章：DaxFS 核心设计

### 1. 设计思想 (Design Philosophy)
- **痛点分析**：传统的树状元数据结构（如 B-tree）在多主机无锁环境下极难维护，且容易产生循环引用或悬空指针。
- **核心动机**：采用“分层存储”与“哈希覆盖层”结合的设计。基础映像（Base Image）是只读且扁平的，所有修改均通过
  哈希覆盖层进行，利用原子操作实现无冲突的条目插入。

### 2. 关键技术 (Key Technologies)
- **CAS-Based Hash Overlay**：
    - 使用开放寻址、线性探测的哈希表存储所有元数据变更（inode、dirent、data page 指针）。
    - 每个桶（Bucket）16 字节，包含 `state_key`（状态位 + 63 位查找键）和 `pool_off`（内存池偏移）。
    - 插入流程：计算哈希 -> 读取桶 -> 使用 `cmpxchg` 将 FREE 状态原子切换为 PENDING/USED。
- **MH-clock 驱逐算法**：
    - **痛点**：标准 CLOCK 算法依赖单机内核的中心化扫描，无法跨主机工作。
    - **创新**：去中心化。每个主机在发生缓存缺失时，独立扫描其局部的 8 槽位探测窗口。通过三个阶段：
      1) 寻找冷条目（ref_bit=0）；2) 清理并让出（Clear and yield）给其他主机接触机会；3) 强制驱逐。
    - **全局进度**：一个共享的原子 `evict_hand` 计数器周期性推进，确保全局范围内的扫描覆盖。
- **页缓存状态机**：
    - 状态包括 FREE (0) -> PENDING (1) -> VALID (2)。
    - PENDING 状态用于处理耗时的后台填充（如从 SSD 加载到 CXL 内存），其他主机看到 PENDING 会忙等而非发起重复读取。

### 3. 实现细节 (Implementation Details)
- **Flat On-Disk Format**：文件系统分为四个顺序排列的区域：Superblock、Base Image、Hash Overlay、Page Cache。
  这种布局便于在挂载时进行单次顺序扫描验证，确保安全性。
- **Key Encoding**：将 (inode << 20 | pgoff) 编码为 63 位键，区分数据页、inode 元数据和目录项。
- **Pool Allocator**：使用原子 `cmpxchg` 操作全局 `pool_alloc` 指针进行 bump 指针分配，实现无锁的内存申请。

## 第四章与第五章：实现与评估

### 1. 关键技术 (Key Technologies)
- **P2PDMA 集成**：通过 `dma-buf` 将 CXL 内存区域导出给 GPU 驱动，利用 NVIDIA 开源内核模块的修改版本实现
  GPU 显存与 CXL 内存间的直接拷贝，避开 CPU 中转。
- **VFS 适配**：实现了标准的 `read_iter`。对于覆盖层数据，它会自动检测物理连续的页面并将其合并为单次 
  `copy_to_iter` 调用，提升大文件读取效率。

### 2. 评估结果 (Evaluation Results)
- **性能优势**：
    - **随机写入**：在 4 线程 4KB 写入下，DaxFS 达到 4830 MiB/s，而 tmpfs 仅为 1803 MiB/s（2.68倍）。
      这是因为 tmpfs 受限于页面锁和全局 `i_pages` 锁。
    - **元数据操作**：`stat` 延迟比 ext4-dax 低 4.6%，归功于其扁平的哈希查找逻辑。
- **GPU 测试**：GPU 端的 CAS 操作延迟约 1811 ns，远高于 DRAM 访问，但其高并发性可以在 1024 线程下提供
  500+ Mops/s 的页缓存查找吞吐量。
- **多主机验证**：在 QEMU TCP 转发原子的环境下，DaxFS 证明了在高并发争用下依然能保证元数据的完整性。

### 3. 权衡与对比 (Trade-offs & Comparison)
- **开销**：DaxFS 引入了哈希表探测的开销。对于 16MB 的大顺序读取，其吞吐量低于 tmpfs，因为每页都需要进行
  覆盖层查找。但在 1MB 小文件场景下，该开销被摊销，性能与 tmpfs 持平。
- **持久化考量**：目前依赖 ADR/eADR 技术保证断电时的内存刷回，尚未在软件层主动调用 `clflush`，以追求极致性能。

## 第七章：讨论与局限性

### 1. 优劣势分析
- **优势**：极高的写入并发度；天然支持 GPU 协同；消除了分布式锁管理的复杂度和延迟。
- **劣势**：
    - **目录扩展性**：扁平目录结构在超过 10K 条目时线性扫描变慢（O(n)）。
    - **哈希表固定大小**：创建后无法在线动态调整大小，需要预先规划。
    - **内存碎片**：目前的内存池回收逻辑简单，长期运行可能产生碎片，需要未来引入在线压缩机制。

### 2. 设计取舍 (Trade-offs)
- **POSIX 兼容性**：为了性能舍弃了 `mknod`（设备节点）、FIFO、Socket 等特性，专注于高性能计算和容器镜像加载场景。
- **文件名长度**：限制为 255 字符，以保持扁平条目的固定大小，简化原子操作逻辑。
