### 中文标题
Cohet：一个带有硬件校准全系统仿真的 CXL 驱动的相干异构计算框架

### 中文摘要
传统的基于 PCIe 互连的异构计算系统在细粒度的主机-设备交互和复杂的编程模型方面效率低下。近年来，出现了许多专有和开放的缓存一致性互连标准，其中 Compute Express Link (CXL) 在收购了几个竞争方案后，在开放标准领域占据主导地位。尽管基于 CXL 的相干异构计算有潜力从根本上改变 CPU 和 XPU 的协同计算模式，但这方面的研究因缺乏可用的 CXL 支持平台、不成熟的软硬件生态系统以及不明确的应用前景而受到阻碍。本文提出了 Cohet，这是第一个由 CXL 驱动的相干异构计算框架。Cohet 将计算和内存资源解耦，形成无偏见的 CPU 和 XPU 池，它们共享一个统一且相干的内存池。它为 CPU 和 XPU 计算线程提供标准的 malloc/mmap 接口，这些线程共享单个每进程页表，由操作系统处理智能内存分配、页面自动迁移和异构资源管理。这种设计将异构并行编程显著简化到与同构编程相当的水平。为了促进 Cohet 的研究，我们还提出了一个名为 SimCXL 的全系统周期级模拟器，它能够建模所有 CXL 子协议和设备类型。SimCXL 已通过真实的 CXL 测试平台进行了严格校准，该平台包含各种 CXL 内存和加速器，平均模拟误差为 3%。我们的评估显示，在缓存行粒度下，与 DMA 传输相比，CXL.cache 将延迟降低了 68%，带宽增加了 14.4 倍。基于这些见解，我们通过两个“杀手级应用”——远程原子操作 (RAO) 和远程过程调用 (RPC)——展示了 Cohet 的优势。与 PCIe-NIC 设计相比，CXL-NIC 在 RAO 卸载方面实现了 5.5 到 40.2 倍的加速，在 RPC (反)序列化卸载方面实现了平均 1.86 倍的加速。

### 各章节详解

*   **第一章：引言 (Introduction)**
    *   **设计思想**：论文开篇指出现有基于 PCIe 的异构系统（CPU + XPU）面临两大瓶颈：1) 高延迟和低带宽的交互方式，不适合细粒度任务；2) 复杂的编程模型，需要程序员显式管理数据移动，开发效率低。CXL 作为新兴的开放标准，通过其缓存一致性协议（CXL.cache）和内存协议（CXL.mem）为解决这些问题提供了可能。作者认为，CXL 有潜力实现真正的相干异构计算，从根本上改变 CPU 和 XPU 的协作方式。
    *   **关键技术**：为了探索这一潜力，本文提出了 Cohet 框架，其核心思想是构建一个统一的、相干的内存池，让 CPU 和各种 XPU（如 GPU, DPU, FPGA）可以像访问本地内存一样互相访问数据，无需复杂的数据拷贝。同时，为了在缺乏真实硬件的情况下进行研究，作者开发了一个名为 SimCXL 的高保真全系统模拟器。
    *   **实现细节**：论文的主要贡献包括：1) 提出 Cohet 计算框架，实现优化的协作模式和简化的编程模型；2) 开发支持所有 CXL 协议和设备类型的全系统模拟器 SimCXL；3) 全面表征 CXL.cache 的延迟和带宽特性，并用此数据校准 SimCXL；4) 设计了一个 CXL-NIC 并展示了其在 RAO 和 RPC 两个应用中相比 PCIe-NIC 的巨大性能优势。

*   **第二章：背景 (Background)**
    *   **设计思想**：本章旨在为后续内容铺垫基础。首先解释了当前主流的 PCIe 异构计算模式，即“CPU主导、XPU辅助”的模式。CPU 通过 MMIO（内存映射 I/O）向 XPU下发任务描述符，XPU 通过 DMA（直接内存访问）从主机内存读写数据。这种模式适用于大块数据的批量传输，但对于需要频繁、小数据量交互的任务则效率低下。
    *   **关键技术**：详细介绍了 CXL 协议。CXL 构建于 PCIe 物理层之上，定义了三个关键的子协议：
        1.  **CXL.io**：提供与 PCIe 相当的设备发现、配置和 I/O 功能。
        2.  **CXL.cache**：允许设备（XPU）缓存主机（CPU）内存中的数据，并与 CPU 缓存保持硬件级别的一致性。这是实现细粒度、低延迟交互的关键。
        3.  **CXL.mem**：允许主机 CPU 直接通过 Load/Store 指令访问设备上的内存，将设备内存扩展为系统内存的一部分。
    *   **实现细节**：根据支持的协议组合，CXL 设备被分为三种类型：Type-1（CXL.io + CXL.cache，如智能网卡），Type-2（支持全部三种协议，如带内存的加速器），Type-3（CXL.io + CXL.mem，如内存扩展器）。本文重点关注可作为加速器使用的 Type-1 和 Type-2 设备。

*   **第三章：相干异构计算框架 (Coherent Heterogeneous Computing Framework)**
    *   **设计思想**：首先，论文系统性地指出了传统异构计算的四个核心局限性（Limitations）：
        1.  **L1: 计算与内存耦合**：计算单元和内存紧密绑定，导致资源无法灵活扩展和共享，容易造成资源浪费。
        2.  **L2: 低效的协作模式**：基于描述符和 DMA 的交互模型延迟高，阻碍了 CPU 和 XPU 之间的紧密协作。
        3.  **L3: CPU 承受“加速器税”**：所有任务分发和同步都需 CPU 干预，当任务粒度变小时，CPU 开销可能抵消加速带来的好处。
        4.  **L4: 复杂的编程模型**：缺乏硬件一致性导致需要复杂的软件栈和程序员手动管理，增加了开发难度。
    *   **关键技术**：针对以上问题，Cohet 框架提出了四个对应的设计哲学（Design Philosophy）：
        1.  **S1: 资源解耦与池化**：利用 CXL 将计算资源和内存资源解耦，形成可独立扩展的池，通过统一的硬件一致性内存视图，避免数据拷贝。
        2.  **S2: 优化协作模式**：利用 CXL.cache/mem 提供的低延迟内存语义访问，实现 CPU 与 XPU 之间的高效、细粒度交互。
        3.  **S3: 去中心化的控制与同步**：将 XPU 提升为与 CPU 对等的角色，它们可以通过共享内存和硬件原子操作直接协调，无需 CPU 作为中介。
        4.  **S4: 简化的编程模型**：硬件维护的内存一致性使得操作系统可以将异构计算单元视为标准的 NUMA 节点，为用户提供统一的内存分配接口（如 `malloc`），实现向后兼容和无缝加速。
    *   **实现细节**：Cohet 的垂直堆栈分为三层：1) **硬件层**：通过 CXL Fabric 连接 CPU 和 XPU，形成统一的物理内存池，硬件负责维护缓存一致性。2) **操作系统层**：Linux 内核将 XPU 识别为 NUMA 节点，并通过异构内存管理（HMM）模块提供统一的页表和内存分配接口。3) **用户层**：拥抱 OpenCL 3.0 等开放标准，允许应用程序像在普通多核 CPU 系统上一样使用 `malloc`/`mmap` 等标准接口，无需关心底层数据迁移，大大简化了编程。

*   **第四章：SimCXL 设计与实现 (SimCXL Design and Implementation)**
    *   **设计思想**：为了在缺少成熟硬件的条件下验证 Cohet 框架，作者基于业界标准的 gem5 模拟器开发了 SimCXL。SimCXL 的设计目标是提供一个全系统、周期精确的模拟环境，能够真实地模拟包含 CXL 互连的异构系统中的软硬件交互。
    *   **关键技术**：SimCXL 的架构构建在 gem5 的全系统（Full-System）模式之上。其核心扩展包括：
        1.  **XPU 设备模型**：创建了一个代表 CXL 加速器的 PCIe 设备模型，包含计算单元、设备内存和本地缓存（HMC）。
        2.  **完整的 CXL 协议支持**：
            *   **CXL.io**：扩展 gem5 原生的 PCIe 协议，支持设备枚举、配置、MMIO 和 DMA。
            *   **CXL.cache & CXL.mem**：在 Ruby 缓存一致性模型中实现了一个新的异构 MESI 协议，使得 XPU 的 HMC 和 CPU 的缓存能够作为对等缓存在 LLC（Last-Level Cache）层面进行一致性同步。
        3.  **OS 内核驱动**：在模拟的客户机操作系统（Linux）中开发了专门的驱动程序，与 HMM 模块协同工作，向用户空间暴露统一的内存接口。
    *   **实现细节**：模拟器通过 gem5 的 SLICC 语言定义了缓存状态转换。当 XPU 访问主机内存时，请求通过 CXL.cache 协议发送到主机 LLC。如果 XPU 要修改数据，它会请求数据的所有权；修改后的数据会暂时保留在 XPU 的 HMC 中，并在需要时通过一致性协议写回主机内存。CXL.mem 则通过配置地址映射，让主机 LLC 可以直接将请求路由到设备内存。这套机制完整地模拟了 CXL 的核心功能。

*   **第五章：Cohet 杀手级应用 (Cohet Killer Apps)**
    *   **设计思想**：为了具体展示 Cohet/CXL 框架的优势，本章设计了一个 CXL-NIC（支持 CXL 的智能网卡），并用它来加速两个对分布式系统至关重要的应用：远程原子操作（RAO）和远程过程调用（RPC）。
    *   **关键技术**：
        1.  **远程原子操作 (RAO)**：在分布式计算中，RAO 用于保证对远程节点共享数据的原子性更新。
            *   **PCIe-NIC 实现**：需要两次独立的 DMA 操作（一次读，一次写），并且需要复杂的软件逻辑来保证原子性，开销巨大。
            *   **CXL-NIC 实现**：利用 CXL.cache，CXL-NIC 可以将目标数据缓存到其片上 HMC。后续的原子操作可以直接在 HMC 中以极低的延迟完成，并通过硬件一致性协议自动将结果同步回主机。这避免了昂贵的 DMA 和软件同步开销。
        2.  **远程过程调用 (RPC)**：RPC 是微服务架构的基石，但其参数的序列化/反序列化开销巨大。
            *   **PCIe-NIC 实现**：数据需要在主机内存和网卡内存之间通过 DMA 多次拷贝，CPU 开销显著。
            *   **CXL-NIC 实现**：
                *   **反序列化**：CXL-NIC 直接将解码后的字段通过低延迟的 CXL.cache `NC-P`（Non-cacheable Push）指令推送到主机 CPU 的 LLC 中，供 CPU 直接使用。
                *   **序列化**：CPU 可以通过 CXL.mem 直接在 CXL-NIC 的设备内存上构建应答消息，或者通过 CXL.cache 让 NIC 直接读取主机内存中的消息结构，大大减少了数据移动和 CPU 的参与。
    *   **实现细节**：作者在 SimCXL 中详细实现了这两种 NIC 的逻辑，包括 RAO 处理单元（PEs）和 RPC 的（反）序列化流水线，并对比了它们与传统 PCIe 设计的交互流程，突出了 CXL 设计在减少数据移动、降低延迟和简化逻辑方面的优势。

*   **第六章：实验与评估 (Experiments and Evaluation)**
    *   **设计思想**：本章的目标是：1) 验证 SimCXL 模拟器的准确性；2) 量化 CXL.cache 相对于传统 DMA 的性能优势；3) 评估 Cohet 驱动的 RAO 和 RPC 应用的加速效果。
    *   **关键技术**：
        1.  **硬件校准**：作者搭建了一个包含 Intel 第四代至强处理器和 Intel Agilex FPGA（可配置为 CXL 或 PCIe 设备）的真实硬件平台。通过在这个平台上运行微基准测试（microbenchmarks），测量了不同内存访问场景（HMC命中、LLC命中、主存命中）下的延迟和带宽。然后，通过调整 SimCXL 中的协议参数，使模拟器的性能数据与真实硬件的测量结果相匹配。校准结果显示，平均误差仅为 3%，证明了 SimCXL 的高保真度。
        2.  **性能评估**：
            *   **CXL.cache vs. DMA**：结果显示，对于小数据块（如 64B），CXL.cache 的内存访问延迟比 DMA 低 68%，带宽是 DMA 的 14.4 倍。这证实了 CXL 在细粒度访问上的巨大优势。
            *   **RAO 评估**：在多种访问模式下，CXL-NIC 的 RAO 吞吐量比 PCIe-NIC 高出 5.5 倍至 40.2 倍，尤其是在存在热点数据（hotspot）的场景下，CXL 的片上缓存优势尽显。
            *   **RPC 评估**：CXL-NIC 的（反）序列化操作平均比 PCIe-NIC 快 1.86 倍，因为它通过统一内存和低延迟访问，消除了大量的数据拷贝和 CPU 开销。
    *   **实现细节**：实验详细配置了硬件平台的 SNC（Sub-NUMA Clustering）模式以模拟不同的 NUMA 延迟。在 SimCXL 中，对 CXL 控制器和 Ruby 子系统中的参数进行了微调，以匹配硬件测量值。评估基准使用了 CircusTent（用于 RAO）和 HyperProtoBench（用于 RPC）。

*   **第七章：相关工作 (Related Work)**
    *   **设计思想**：本章将本文的工作置于现有研究的背景下，通过对比彰显其创新性和独特性。
    *   **关键技术**：主要从两个方面进行比较：
        1.  **CXL 模拟/仿真方法**：现有的工作要么使用“远程 NUMA 节点”来模拟 CXL 设备，这种方法不准确，无法反映 CXL 设备的真实特性；要么只关注 CXL Type-3（内存扩展）设备，忽略了 Type-1/2 的异构计算场景。SimCXL 是第一个能够对 CXL XPU（Type-1/2 设备）和所有三种 CXL 子协议进行建模的全系统模拟器，且经过了硬件校准。
        2.  **CXL 异构计算研究**：之前的研究大多局限于利用 CXL.cache 的某些特性进行特定的硬件定制或应用优化，缺乏一个全局的、贯穿软硬件栈的计算框架视角。而 Cohet 是第一个从框架层面提出完整解决方案的工作，极大地简化了编程模型和内存管理。

*   **第八章：讨论与未来工作 (Discussion and Future Work)**
    *   **设计思想**：坦诚地分析当前 CXL 技术的局限性，并为社区指明未来的研究方向。
    *   **实现细节**：
        *   **挑战**：文章指出，当前 CXL 仍面临一些挑战，例如地址翻译服务（ATS）可能引入的 ATC（Address Translation Cache）未命中开销，以及操作系统和编程框架对 CXL 的支持尚不成熟。
        *   **未来工作**：作者计划扩展 SimCXL 以支持 CXL 3.x 标准及其 Fabric 互连特性，从而能够模拟更大规模的多节点系统。他们还计划探索更广泛的应用场景，如图计算、内存键值存储、大语言模型推理等，这些应用都能从 CXL 提供的低延迟、细粒度内存访问中获益。
