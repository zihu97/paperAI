- **英文标题**: Exploring Performance and Cost Optimization with ASIC-Based CXL Memory
- **中文标题**: 基于ASIC的CXL内存的性能与成本优化探索

## 中文摘要
随着内存密集型应用的持续增长，Compute Express Link (CXL) 作为一种有前途的互连技术，实现了主机处理器与外围设备之间的高速、低延迟通信。
本研究深入探讨了基于ASIC的CXL内存在各种数据中心场景下的应用性能。我们进一步通过精心设计的策略，探索了使用CXL内存的多种潜在影响（如吞吐量、
延迟和成本降低）。我们的实证结果显示了CXL内存的巨大潜力，揭示了关于CXL内存的多个有趣观察，并为CXL内存在真实世界部署环境中的广泛应用做出了贡献。
基于我们的基准测试，我们还开发了一个抽象成本模型，可以估算使用CXL内存带来的成本效益。

## 各章节主旨内容
*   **第1章：引言 (Introduction)**
    *   指出现代应用（如机器学习、HPC）对内存容量和带宽的巨大需求，这超出了单机DDR内存的物理和成本限制。CXL技术作为一种突破性的互连标准被引入，
        用以扩展内存。论文指出现有研究多基于仿真或FPGA，缺乏对商用ASIC CXL硬件的实证评估，并明确了本文的三大贡献：对ASIC CXL硬件的实证评估、
        成本效益分析、以及开源所有测试数据和配置。

*   **第2章：背景与方法论 (Background and Methodology)**
    *   介绍了CXL技术的基础，包括其基于PCIe 5.0物理层，以及CXL.io, CXL.cache, CXL.mem三种协议。重点关注CXL 1.1 Type-3设备，
        即内存扩展设备。接着介绍了实验所用的硬件（Intel Sapphire Rapids CPU, AsteraLabs A1000 CXL扩展卡）和软件支持（Linux内核对分层内存的最新支持，
        如N:M交错策略和热页提升机制）。

*   **第3章：CXL 1.1 性能特征 (CXL 1.1 Performance Characteristics)**
    *   通过微基准测试（Intel MLC）详细测量了CXL内存的延迟和带宽。研究对比了本地DDR内存（MMEM）、远程DDR内存（MMEM-r）、
        本地CXL内存（CXL）和远程CXL内存（CXL-r）的性能。核心发现是：本地CXL访问延迟高于本地DDR，但与远程DDR访问延迟相当；
        远程访问CXL内存会导致性能急剧下降，应尽量避免。

*   **第4章：内存容量受限型应用 (Memory Capacity-bound Applications)**
    *   探讨CXL如何为容量受限的应用带来好处。以两个真实世界应用为例：
        1.  **KeyDB (键值存储)**：证明了使用CXL作为内存扩展层，可以避免将数据溢出到慢速的SSD上，从而显著提升性能。
        2.  **Spark SQL (大数据分析)**：展示了通过使用CXL内存，可以将原本需要3台服务器的工作负载整合到2台服务器上运行，虽然有一定性能损失，
            但远优于溢出到SSD，从而实现成本效益。
        3.  **虚拟机 spare cores**: 论证了CXL可以帮助云厂商以更低成本满足虚拟机的内存需求，从而“解锁”因内存不足而闲置的CPU核心，增加收入。

*   **第5章：内存带宽受限型应用 (Memory Bandwidth-Bound applications)**
    *   探讨CXL如何为带宽受限的应用带来好处。以**LLM（大语言模型）推理**为例，当推理请求增加时，内存带宽成为瓶颈。
        实验证明，通过将内存访问在本地DDR和CXL之间进行交错（interleaving），可以利用CXL提供的额外带宽，提升系统的总吞吐量（每秒处理的token数），
        性能甚至超过了仅使用DDR内存的配置。

*   **第6章：成本影响 (Cost Implications)**
    *   提出了一个**抽象成本模型（Abstract Cost Model）**，用于估算采用CXL带来的总拥有成本（TCO）节省。该模型仅使用少数几个可以从微基准测试中获得的参数
        （如纯DDR、纯CXL、纯SSD运行时的相对性能），即可计算出为达到相同性能目标，使用CXL方案可以减少多少服务器数量，并最终估算出TCO节省的百分比。

*   **第7章：讨论 (Discussion)**
    *   将论文的发现扩展到未来的CXL 2.0/3.0及更高版本。探讨了未来数据中心向内存池化和资源解耦架构演进的趋势，CXL在其中扮演关键角色。
        同时分析了实现这一愿景所需面临的硬件、软件和互连技术的挑战。

*   **第8章：相关工作 (Related Work)**
    *   简要回顾了内存解耦、CXL技术研究的现有工作，包括基于NUMA仿真、软件模拟器和FPGA原型的研究。

*   **第9章：结论 (Conclusion)**
    *   总结了本文工作：首次对商用ASIC CXL内存在真实世界数据中心应用中进行了全面的实证评估，填补了理论研究与实践之间的空白。
        论文的发现揭示了CXL的潜力和局限性，为如何有效使用CXL提供了可操作的建议。最后，开发的抽象成本模型为评估CXL的经济可行性提供了重要指导。

## 中文结论
我们对Compute Express Link (CXL)在真实世界数据中心应用中进行了全面的实证评估，填补了先前理论研究留下的关键知识空白。我们的研究结果揭示了CXL的潜力与局限性，
为其持续发展提供了可行的建议，以更好地服务于以数据为中心的计算环境。基于我们的基准测试，我们还开发了一个抽象成本模型，该模型可以在不依赖内部或敏感数据的情况下估算总拥有成本（TCO）的节省，
为我们下一代基础设施的设计提供了重要指导。

# 各章节详解

## 第1章：引言 (Introduction)
随着大语言模型（LLM）和高性能计算（HPC）等应用的兴起，现代服务器对内存容量和带宽的需求急剧增加。例如，一个175B参数的LLM模型仅参数就需要700GB内存。
传统的做法是通过在主板上插满高密度DDR内存条，但这面临物理插槽数量有限、散热问题以及高昂的成本等多重制约。

Compute Express Link (CXL) 应运而生，它是一种基于PCIe物理层的高速互连标准，允许CPU访问挂载在PCIe插槽上的外部内存设备，从而突破主板的内存容量限制。
尽管学术界和工业界对CXL寄予厚望，但之前的研究大多依赖于软件仿真或FPGA原型，这些方法无法完全模拟真实ASIC（专用集成电路）硬件的性能特征。

本文旨在填补这一空白，通过在商用ASIC CXL硬件上进行广泛的实验，探索其在真实数据中心负载下的性能表现和成本效益。论文的核心贡献在于：
1.  **对ASIC CXL硬件的实证评估**：首次全面地、系统地测试了商用CXL 1.1 Type-3内存扩展设备在各种应用场景下的性能表现。
2.  **成本效益分析**：不仅评估性能，还建立了一个抽象成本模型，量化分析采用CXL技术能在多大程度上降低数据中心的总拥有成本（TCO）。
3.  **开放研究数据**：开源了所有的测试数据、脚本和服务器配置，为社区进一步研究CXL技术提供宝贵的实践基础。

一个有趣的初步发现是，与普遍认为“CXL内存是慢速内存，应在本地DDR用尽后才使用”的观点不同，本文发现**即使本地DDR内存未被充分利用，将部分负载转移到CXL内存有时也能提升整体性能**，
因为它有助于缓解DDR内存通道的带宽争用。

## 第2章：背景与方法论 (Background and Methodology)
本章介绍了CXL技术的基本概念以及实验所采用的软硬件环境。

*   **CXL技术概述**:
    *   **物理层**: CXL建立在PCIe 5.0物理层之上，提供高带宽连接。
    *   **三大协议**: CXL定义了三种协议来满足不同需求：
        *   `CXL.io`: 用于设备发现、配置和I/O，功能上类似标准的PCIe。
        *   `CXL.cache`: 允许设备（如加速器）与CPU共享缓存，保证缓存一致性。
        *   `CXL.mem`: 允许CPU通过加载/存储指令直接访问挂载在CXL设备上的内存。
    *   **三种设备类型**:
        *   `Type-1`: 加速器（如智能网卡），使用CXL.io和CXL.cache。
        *   `Type-2`: GPU、FPGA等，使用全部三种协议，能与CPU共享内存。
        *   `Type-3`: 内存扩展设备（本文焦点），使用CXL.io和CXL.mem，作为内存缓冲和扩展。

*   **硬件实验平台**:
    *   **服务器配置**: 实验平台包含两类服务器：CXL服务器和基线服务器。
    *   **CXL服务器**: 配备双路第四代Intel Xeon CPU（Sapphire Rapids），1TB DDR5内存，以及两张来自AsteraLabs的A1000 CXL Gen5内存扩展卡，每张卡上带256GB DDR5内存，
        总计为服务器增加了512GB的CXL内存。
    *   **基线服务器**: 配置与CXL服务器相同，但不安装CXL扩展卡，用于性能对比。

*   **软件支持**:
    *   由于CXL作为新兴技术，操作系统（OS）的支持仍在快速演进中。本文利用了Linux内核（v6.1及以上）中最新的内存管理特性，这些特性对管理CXL这种新型分层内存至关重要。
    *   **N:M Interleave Policy**: 传统的内存交错策略在所有内存通道上均匀分配数据。新的N:M策略允许将N页数据分配到高速内存（如DDR），M页数据分配到低速内存（如CXL），
        从而可以根据应用的特性灵活调整数据在不同内存层级间的分布比例。
    *   **Hot Page Selection (热页选择与提升)**: 内核能够识别被频繁访问的“热页”，并自动将它们从CXL这样的慢速内存迁移到DDR这样的快速内存中，以优化性能。
        这一机制对于延迟敏感型应用至关重要。

## 第3章：CXL 1.1 性能特征 (CXL 1.1 Performance Characteristics)
本章通过Intel Memory Latency Checker (MLC) 工具对CXL内存的基础性能进行了精细测量，旨在回答以下问题：CXL内存与DDR内存相比性能如何？
不同访问模式（读写比例、随机/顺序）有何影响？

实验对比了四种内存访问场景：
1.  **MMEM (Main Memory)**: 访问与CPU在同一个Socket上的本地DDR5内存。
2.  **MMEM-r (Remote Main Memory)**: 跨Socket访问另一个CPU上的远程DDR5内存。
3.  **CXL**: 访问与CPU在同一个Socket上的CXL内存。
4.  **CXL-r**: 跨Socket访问另一个CPU上的CXL内存。

**核心性能发现** (参考图3和图4):
*   **延迟 (Latency)**:
    *   **本地访问**: 本地CXL的空载延迟约为250ns，显著高于本地DDR的97ns。这是因为CXL访问需要经过PCIe控制器和CXL控制器，路径更长。
        但有趣的是，本地CXL的延迟（~250ns）与远程DDR的延迟（~130ns-200ns）在同一数量级，这意味着**对于需要跨Socket访问内存的应用，使用本地CXL可能是个不错的折衷**。
    *   **远程访问**: 远程CXL（CXL-r）的延迟极高，达到485ns。这是性能上的巨大瓶颈。

*   **带宽 (Bandwidth)**:
    *   **本地访问**: 本地DDR的读带宽最高可达67 GB/s。本地CXL的带宽约为54.6 GB/s，达到了DDR带宽的80%左右，性能相当可观。
    *   **远程访问**: 远程DDR的带宽受限于UPI互连总线，但仍能达到较高水平。然而，**远程CXL（CXL-r）的带宽出现断崖式下跌，仅为20.4 GB/s**。
        论文作者与Intel沟通后确认，这可能是当前CPU平台上的远程Snoop Filter（RSF）实现限制所致，预计在未来CPU代中会得到解决。

**关键启示 (Key Insights)**:
*   **避免跨Socket访问CXL内存**: 当前硬件下，任何需要跨Socket访问CXL内存的设计都应极力避免，因为它会同时带来极高的延迟和极低的带宽。
*   **CXL可作为负载均衡资源**: 即使本地DDR带宽未用满，将部分对延迟不敏感但消耗带宽的负载分流到CXL上，可以为DDR内存释放出带宽，
    从而让真正需要低延迟的应用受益。这挑战了“CXL只是DDR用尽后的次级选项”的传统观念。

## 第4章：内存容量受限型应用 (Memory Capacity-bound Applications)
本章探讨了CXL扩展内存容量带来的实际价值，核心思想是**用CXL内存替代更慢的存储层（如SSD）或避免因内存不足导致的资源浪费**。

*   **案例一：KeyDB (高性能键值数据库)**
    *   **场景**: 许多内存数据库（如Redis/KeyDB）在内存不足时，会将冷数据“溢出”（spill）到SSD上。虽然这扩展了容量，但SSD的访问延迟是内存的上千倍，
        导致性能急剧下降。
    *   **实验**: 论文比较了三种配置：1) 数据完全在DDR内存中；2) 部分数据在DDR，部分在CXL；3) 部分数据在DDR，部分溢出到SSD。
    *   **结果 (图5)**:
        *   纯DDR内存性能最佳。
        *   **DDR+CXL的性能远超DDR+SSD**。具体来说，DDR+SSD的配置性能下降了约1.8倍，而DDR+CXL（通过N:M交错或热页提升）的性能损失要小得多，
            几乎接近纯DDR内存的性能。
        *   **结论**: CXL为容量受限的内存数据库提供了一个远优于SSD的扩展层，可以用可接受的性能损失换取巨大的容量提升。

*   **案例二：Spark SQL (大数据分析)**
    *   **场景**: Spark在处理大规模数据Shuffle时，如果内存不足，同样会将中间数据溢出到磁盘，严重影响查询性能。数据中心通常通过增加服务器数量来解决内存容量问题，
        但这会增加成本。
    *   **实验**: 论文模拟了一个场景，比较了“3台无CXL的服务器”和“2台有CXL的服务器”运行相同的Spark TPC-H查询。总内存容量近似，但后者服务器数量更少。
    *   **结果 (图7)**:
        *   在2台CXL服务器上运行，虽然性能比在3台纯DDR服务器上慢（慢1.4倍到9.8倍，取决于内存交错比例），但**仍然远远快于在内存不足的机器上溢出到SSD的场景**。
        *   **结论**: CXL使得用更少的服务器来处理同样规模的数据集成为可能。虽然单次任务执行时间可能变长，但从**吞吐量/成本**的角度看，
            这是一种非常划算的部署方式，能显著提高数据中心的服务器利用率和成本效益。

*   **案例三：为虚拟机提供“备用核心” (Spare Cores for Virtual Machine)**
    *   **场景**: 云服务商（CSP）售卖虚拟机时，通常遵循一个固定的CPU与内存比，如1:4（1个vCPU配4GB内存）。
        随着现代CPU核心数飞速增长（如Intel Sierra Forest有上千核心），主板能承载的DDR内存容量增长速度跟不上，导致服务器上出现大量“有核无存”的尴尬局面，
        这些核心因无法配足内存而不能售卖，造成收入损失。
    *   **实验**: 论文模拟了云厂商使用CXL内存为虚拟机提供服务的场景。
    *   **结果 (图8)**:
        *   在CXL上运行的应用（KeyDB）性能仅比在DDR上慢12.5%。
        *   **成本分析**: 即使云厂商为运行在CXL内存上的虚拟机提供20%的折扣，他们仍然可以通过销售原本因内存不足而闲置的CPU核心，将总收入提升27%。
        *   **结论**: CXL为云服务商提供了一种低成本、高效率的方式来平衡CPU与内存资源，将闲置的计算资源转化为实实在在的收入。

## 第5章：内存带宽受限型应用 (Memory Bandwidth-Bound applications)
本章的核心论点是：**CXL不仅能扩展容量，还能提供额外的带宽**。当应用性能的瓶颈从计算转向内存带宽时，CXL可以作为一个有效的“加速器”。

*   **案例：大语言模型（LLM）推理**
    *   **场景**: LLM推理过程中的KV Cache（键值缓存）访问是一个典型的内存带宽密集型操作。当并发推理请求增多时，CPU到DDR内存的带宽很快就会饱和，
        成为系统吞吐量（每秒能生成多少token）的瓶颈。
    *   **实验**:
        *   实验环境被配置为SNC-4模式，即将一个CPU Socket划分为4个独立的NUMA域，每个域只有2个DDR内存通道，以此**人为地快速制造出内存带宽瓶颈**。
        *   在这样一个带宽受限的NUMA域里，除了DDR内存，还挂载了一块CXL内存扩展卡。
        *   实验比较了不同内存分配策略下的LLM推理吞吐量：
            1.  **MMEM only**: 所有推理负载只使用DDR内存。
            2.  **Interleave (3:1, 1:1, 1:3)**: 将内存访问按照不同比例交错分配在DDR和CXL内存上。

    *   **结果 (图10)**:
        *   在并发请求较少时，`MMEM only` 性能最好，因为DDR延迟最低。
        *   随着并发请求数增加（超过48个线程），DDR内存带宽逐渐饱和，`MMEM only` 配置的吞吐量开始停滞不前。
        *   此时，**Interleave配置的性能反超了`MMEM only`**。例如，`3:1`（75% DDR, 25% CXL）的交错配置，其峰值吞吐量比`MMEM only`高出95%。
        *   **原因**: Interleave配置的总可用带宽是“DDR带宽 + CXL带宽”。当DDR带宽饱和时，CXL提供的额外带宽使得系统总吞吐量得以继续提升。
            尽管CXL的延迟更高，但在高并发、吞吐量为王的场景下，总带宽比低延迟更重要。

**关键启示**:
*   **重新思考分层内存**: 传统的内存分层管理策略（如热页提升）总是倾向于将数据往最快的内存层（DDR）上放。但本章的实验证明，这种策略在带宽受限时可能不是最优的。
    **一个更优的策略应该同时考虑延迟和带宽**。当高层内存的带宽利用率过高时，即使它还有容量，也应该主动将一些负载分配到下一层内存（CXL），以避免高层内存因带宽争用而导致所有访问的延迟急剧上升。

## 第6章：成本影响 (Cost Implications)
本章旨在回答管理者最关心的问题：**使用CXL到底能省多少钱？** 为此，论文提出了一个**抽象成本模型（Abstract Cost Model）**，其优点是不需要复杂的模拟或访问公司内部敏感的财务数据，
只需几个通过简单基准测试就能得到的性能参数即可。

**模型的核心思想**:
假设一个工作负载的总大小为`W`，我们需要达到的性能目标是`T`。
*   **基线场景 (不用CXL)**: 我们有一堆服务器，每台服务器的DDR内存容量为`D`。当内存不足时，数据溢出到SSD。
*   **CXL场景**: 我们有一堆CXL服务器，每台服务器有容量为`D`的DDR和容量为`D/C`的CXL内存。

**模型参数 (参考表3)**:
*   `Ps`: 性能基准，当数据完全溢出到SSD时的吞吐量，归一化为1。
*   `Rd`: 相对性能，数据完全在DDR内存中时的吞吐量，相对于`Ps`的倍数 (例如, `Rd=10` 表示DDR比SSD快10倍)。
*   `Rc`: 相对性能，数据完全在CXL内存中时的吞吐量，相对于`Ps`的倍数 (例如, `Rc=8` 表示CXL比SSD快8倍)。
*   `C`: DDR内存与CXL内存的容量比。
*   `Rt`: 单台CXL服务器相对于基线服务器的成本比率 (例如, `Rt=1.1` 表示CXL服务器贵10%)。

**推导与应用**:
通过一系列公式推导，模型可以计算出要达到相同的总性能目标，所需的CXL服务器数量（`Ncxl`）与基线服务器数量（`Nbaseline`）之间的比率 `Ncxl / Nbaseline`。
这个比率完全由 `Rd`, `Rc`, `C` 这几个性能参数决定。

最终，**总拥有成本节省（TCO saving）** 可以表示为：
`TCO saving = 1 - (Ncxl * Rt) / Nbaseline`

**示例**:
假设 `Rd=10`, `Rc=8`, `C=2` (DDR容量是CXL的两倍)，`Rt=1.1` (CXL服务器贵10%)。
1.  模型计算出 `Ncxl / Nbaseline = 0.6729`。这意味着，使用CXL后，只需要原先**67.3%**数量的服务器就能完成同样的工作。
2.  代入成本比率`Rt`，最终的TCO节省为 `1 - (0.6729 * 1.1) = 0.2598`，即**节省了约26%的总拥有成本**。

这个模型为决策者提供了一个简单而强大的工具，可以快速评估在特定工作负载和硬件配置下，引入CXL技术的经济可行性。

## 第7章：讨论 (Discussion)
本章将视野从CXL 1.1扩展到CXL 2.0/3.0，并探讨了数据中心内存架构的未来演进。

*   **CXL 2.0/3.0及未来**:
    *   **内存池化 (Memory Pooling)**: CXL 2.0/3.0最核心的特性是支持**交换（Switching）**和**内存池化**。
        这意味着多台服务器可以通过一个CXL交换机共享一个巨大的内存池。内存不再是单个服务器的私有财产，而是整个机架乃至数据中心的共享资源。
    *   **资源解耦 (Disaggregation)**: 这是内存池化的最终目标。计算（CPU）、内存（DRAM, CXL-PMEM）和存储（SSD）可以独立地进行扩展和升级。
        数据中心可以按需采购和分配资源，极大地提升了灵活性和成本效益。例如，可以构建一个主要由内存组成的机架和一个主要由计算组成的机架。
    *   **Nvidia GH200的类比**: 论文将Nvidia的Grace Hopper超级芯片架构与CXL的愿景进行了类比。GH200内部通过NVLink-C2C高速互连，实现了GPU HBM、CPU DDR和远程GPU/CPU内存的统一访问，
        形成了一个分层的内存架构，这与CXL所倡导的异构、分层内存思想不谋而合。

*   **实现挑战**:
    要实现完全的内存解耦，仍面临诸多挑战：
    1.  **硬件挑战**: 需要更高效的CXL控制器、交换机，以及与处理器更紧密的集成。同时，需要新的内存技术（如MRAM, ReRAM）融入CXL生态。
    2.  **软件挑战**: 操作系统和虚拟机监视器（Hypervisor）需要进行重大改造，以支持对远程池化内存的高效分配、迁移、监控和故障处理。需要一个统一的内存管理框架来屏蔽底层硬件的复杂性。
    3.  **互连技术挑战**: 当前基于PCIe电缆的CXL连接方案在扩展到机架外时面临限制。需要新的互连技术，如**光互连（Optical Interconnects）**和**Ultra Ethernet (UEC)**，
        来实现数据中心规模的内存互联。

## 第8章与第9章：相关工作与结论
*   **相关工作 (Related Work)**:
    本章简要梳理了在内存解耦领域的前人工作。许多早期研究，如Meta的`Pond`和微软的研究，都使用了NUMA服务器来模拟CXL的内存池化，因为在真实CXL硬件稀缺时，
    跨Socket访问远程内存是模拟高延迟内存访问的权宜之计。本文的工作则是在真实的ASIC硬件上进行了验证，使研究从模拟走向现实。

*   **结论 (Conclusion)**:
    论文最后对全文进行了总结。本文的核心贡献在于，它没有停留在理论或仿真层面，而是**首次提供了关于商用ASIC CXL内存在真实数据中心应用中的全面、深入的实证评估报告**。
    研究不仅揭示了当前CXL 1.1硬件的性能边界和使用上的“最佳实践”（比如避免跨socket访问），还通过对容量和带宽受限应用的分析，展示了CXL在提高资源利用率和降低成本方面的巨大潜力。
    最后，提出的抽象成本模型为业界提供了一个实用的决策工具。总而言之，这项工作为CXL技术从概念走向大规模商业部署铺平了道路。
